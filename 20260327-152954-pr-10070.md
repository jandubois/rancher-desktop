## Executive Summary

This PR fixes the broken no-proxy list UI (missing `Array.isArray` guard on the StringList `@change` handler), adds input validation for noproxy entries, and routes domain-based entries through moproxy's `--policy` flag while keeping IP/CIDR entries on the iptables path. The implementation is well-structured, and the validator has thorough test coverage. The main concerns are that the validator explicitly accepts IPv6 addresses but the init script feeds them to IPv4-only `iptables`, and that the policy.rules fallback content uses `;` as a comment character, which may not be valid in moproxy's policy parser.

2. **Policy.rules fallback uses `;` comment character, which may crash moproxy** — `wsl.ts:792-793` [Claude Opus 4.6, Gemini 3.1 Pro] (important, regression)

```typescript
await this.writeFile('/etc/moproxy/policy.rules',
  domainRules.length > 0 ? `${ domainRules.join('\n') }\n` : '; no domain noproxy rules\n');
```

When no domain rules exist (the default state), this writes `; no domain noproxy rules` to the policy file. The `;` character is a comment delimiter in INI files (like `proxy.ini`), but moproxy's policy parser uses `#` for comments. If the parser rejects `;`, moproxy fails to start for every user who does not configure domain-based noproxy entries — that is, the common case.

Fix: Replace the comment prefix:

```diff
-  domainRules.length > 0 ? `${ domainRules.join('\n') }\n` : '; no domain noproxy rules\n');
+  domainRules.length > 0 ? `${ domainRules.join('\n') }\n` : '# no domain noproxy rules\n');
```

1. **Support `.example.com` wildcard convention** — `settingsValidator.ts:695` [Gemini 3.1 Pro] (suggestion, enhancement)


The validation accepts `*.example.com` but rejects `.example.com`, because splitting on `.` produces an empty first label that fails the regex.
```diff
-    const domain = entry.startsWith('*.') ? entry.substring(2) : entry;
```
