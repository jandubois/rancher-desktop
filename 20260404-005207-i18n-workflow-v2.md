## Executive Summary

This branch adds comprehensive i18n support to Rancher Desktop: a Go maintenance CLI (`i18n-report`), main-process and renderer translation infrastructure, a locale setting with language picker, and translated strings across ~40 Vue components. The architecture is well-designed, the commit history is clean, and the feature gate is pragmatic. The review found one navigation regression (broken tab-level `data-navigate`), several data-integrity bugs in the Go tooling (multiline handling, metadata parsing, missing manifest entry), a locale-revert gap in the main process, and various moderate issues around initialization ordering and translation-string safety.

## Critical Issues

1. **`data-navigate` click handler passes comma-separated string as single arg** — `BodyKubernetes.vue:78-83`, `ContainerEngineGeneral.vue:41-46` [Claude Opus 4.6] (critical, regression)

```js
onWarningClick(event: MouseEvent) {
  const target = event.target as HTMLElement;
  if (target?.getAttribute('data-navigate')) {
    this.$root.navigate(target.getAttribute('data-navigate'));
  }
}
```

The old code called `$root.navigate('Container Engine', 'general')` with two separate arguments. The new code reads `data-navigate="Container Engine,general"` from the translation string (`en-us.yaml:795`) and passes the entire string as one argument. The `navigate` function at `ModalBody.vue:49` expects `(navItem: string, tab: string)`, so `navItem` becomes `"Container Engine,general"` and `tab` becomes `undefined`. Tab-level navigation from the Spinkube warning banner is broken.

The `ContainerEngineGeneral.vue` case (`data-navigate="Kubernetes"`) navigates to a top-level page with no tab, so it works by accident.

Fix:
```diff
 onWarningClick(event: MouseEvent) {
   const target = event.target as HTMLElement;
-  if (target?.getAttribute('data-navigate')) {
-    this.$root.navigate(target.getAttribute('data-navigate'));
+  const nav = target?.getAttribute('data-navigate');
+  if (nav) {
+    const [navItem, tab] = nav.split(',');
+    this.$root.navigate(navItem, tab);
   }
 }
```

## Important Issues

1. **Locked-field regex breaks under non-English locales** — `settingsValidator.ts:287-290`, `commandLineOptions.ts:142`, `backendHelper.ts:171` [Claude Opus 4.6, Codex GPT 5.4] (important, gap)

```ts
errors.push(t('validation.fieldLocked', { field: `${ prefix }.${ k }` }));
```

`commandLineOptions.ts:142` detects locked-field errors via `/field ".+?" is locked/`, and `backendHelper.ts:171` via `/field ".*" is locked/`. These regexes match only the English text. When a non-English locale is active, the translated string will not match, so `LockedFieldError` will not be thrown. Deployment profile enforcement silently degrades. The comment at lines 287-289 acknowledges this and defers the fix, but the feature gate does not prevent activation via `rdctl` or `settings.json`.

Fix: Keep the `fieldLocked` error in English regardless of locale, or add a structured error type now.

2. **Main process fails to revert to default locale when set to 'none'** — `i18n.ts:125-130` [Gemini 3.1 Pro] (pass 2) (important, regression)

```typescript
export async function onSettingsUpdate(settings: RecursivePartial<Settings>) {
  const locale = settings?.application?.locale;
  if (locale && locale !== 'none') {
    await switchTo(locale);
  }
}
```

When a user switches the locale back to "none" (system default), the renderer correctly reverts to `en-us`, but the main process ignores `'none'` entirely. The main process stays in the previously selected language until the application restarts — a split-brain state where menus and tray show German while the renderer shows English.

Fix:
```diff
-  const locale = settings?.application?.locale;
-  if (locale && locale !== 'none') {
+  const raw = settings?.application?.locale;
+  const locale = (!raw || raw === 'none') ? 'en-us' : raw;
+  if (locale !== currentLocale) {
     await switchTo(locale);
   }
```

3. **Shipped locale files are incomplete and fail the new checker** — `de.yaml`, `zh-hans.yaml` [Codex GPT 5.4] (important, regression)

`de.yaml` lacks `containerInfo.shell` and the new `deploymentProfile.noVersion*` keys. `zh-hans.yaml` lacks those keys and still uses stale `spinkubeWarningPrefix|Link|Suffix` keys instead of the new single `spinkubeWarning` key. Running `i18n-report check` on both locales reports failures.

Fix: Add the missing keys and replace stale keys. Make `i18n-report check` part of CI for every shipping locale.

4. **`meta/locales.yaml` registers `fa` but `fa.yaml` does not exist** — `meta/locales.yaml:7-12`, `manifest.go:89-97` [Codex GPT 5.4] (pass 2) (important, regression)

```yaml
locales:
  en-us:
    status: source
  de:
    status: experimental
  fa:
    status: experimental
```

`Manifest.validate()` rejects any non-source locale without a matching translation file. Since `fa.yaml` does not exist, `loadManifest()` fails globally, breaking `manifest`, `validate`, `check --policy`, and `meta --locale=all` for all locales.

Fix: Remove `fa` from `meta/locales.yaml` until `fa.yaml` ships, or teach the manifest to model metadata-only locales explicitly.

5. **Multiline metadata truncation causes false drift** — `metadata.go:31-45`, `report_drift.go:54-61` [Codex GPT 5.4] (pass 2) (important, regression)

```go
for _, line := range strings.Split(content, "\n") {
    line = strings.TrimSpace(line)
    idx := strings.Index(line, ": ")
    ...
    value := stripYAMLQuotes(line[idx+2:])
    result[key] = value
}
```

`writeMetadata()` emits multiline English strings as YAML block scalars, but `parseMetadata()` line-splits and captures only the first line (e.g., `|-`). When `reportDrift()` compares the full English value against this truncated metadata, untouched multiline keys are always flagged as drifted. Running `drift --locale=de` falsely reports five drifted keys.

Fix: Parse metadata with a YAML decoder instead of line-splitting. Add regression tests for block scalars.

6. **Multiline string formatting corruption in YAML serializer** — `yaml.go:430` [Gemini 3.1 Pro] (important, regression)

```go
trimmed := strings.TrimLeft(line, " ")
```

In `serializeNode`, the serializer strips ALL leading whitespace from multiline block scalar bodies instead of only the baseline indentation added by `yaml.Marshal`. This destroys relative indentation in values like `imageEvents.deleteError` where `{error}` has intentional leading whitespace. The same pattern appears at line 518 in `writeNestedYAML`.

Fix: Use `strings.TrimPrefix(line, baseIndent)` where `baseIndent` is the 2-space indent `yaml.Marshal` adds, or strip a fixed number of leading spaces.

7. **`remove` does not update `meta/*.yaml`, leaving orphaned metadata** — `report_remove.go:43-57` [Codex GPT 5.4] (pass 2) (important, regression)

`runRemove()` and `removeStaleKeys()` mutate translation files but skip the `meta/` directory. After running the documented `unused | remove` workflow, metadata files still contain entries for deleted keys. `validate` and `drift` then flag those entries as orphaned, producing an invalid repo state from the tool's own maintenance path.

Fix: After modifying a locale file, remove the same keys from its metadata file or regenerate metadata from the surviving keys.

8. **`setHeader()` clears existing state on partial updates** — `store/page.ts:60-78`, `ContainerInfo.vue:271` [Codex GPT 5.4] (pass 3) (important, regression)

```ts
setHeader({ commit }, args: SetHeaderArgs) {
  if (args.titleKey) {
    commit('SET_TITLE_KEY', args.titleKey);
    commit('SET_TITLE', '');
  } else {
    commit('SET_TITLE', args.title ?? '');
    commit('SET_TITLE_KEY', '');
  }
```

`setHeader()` now treats omitted fields as "clear this field." Existing callers like `ContainerInfo.vue:271`, which dispatch `{ action: null }` to clear only the badge action, now also clear the title and description. Navigating away from the container detail view can leave the page header blank.

Fix: Check whether each field key is present in `args` (via `'titleKey' in args`) before clearing it.

9. **Language picker bypasses Preferences Apply/Cancel staging** — `ApplicationGeneral.vue:56-60` [Codex GPT 5.4] (pass 3) (important, regression)

```ts
onLocaleChange(event: Event) {
  const locale = (event.target as HTMLSelectElement).value;
  this.onChange('application.locale', locale);
  this.$store.dispatch('i18n/switchTo', locale);
}
```

The picker stages `application.locale` via `onChange` but immediately calls `i18n/switchTo`, which persists the locale cookie and switches the live UI before the user clicks Apply. If the user cancels, the saved settings remain unchanged but the UI language and cookie have already been mutated. This breaks the modal's existing staging contract.

Fix: Remove the manual `i18n/switchTo` call. Apply the live locale only after `commitPreferences()` succeeds, or snapshot and restore on cancel.

10. **JSONL parsing drops string-valued LLM content** — `report_merge.go:217-227` [Gemini 3.1 Pro, Claude Opus 4.6] (pass 3) (important, regression)

```go
var blocks []struct {
    Type string `json:"type"`
    Text string `json:"text"`
}
if err := json.Unmarshal(msg.Message.Content, &blocks); err == nil {
    for _, b := range blocks {
        if b.Type == "text" {
```

`extractTranslationText` expects `msg.Message.Content` to unmarshal into a slice of content blocks, but some LLM endpoints return `content` as a plain JSON string. The unmarshal fails silently and the response is skipped entirely, producing empty merge output.

Fix: Add a fallback that parses `Content` as a string when array parsing fails.

11. **`onLocaleChange` callbacks never unregistered — memory leak** — `i18n.ts:87-89`, `tray.ts:203` [Claude Opus 4.6] (pass 2) (important, regression)

```typescript
export function onLocaleChange(callback: () => void): void {
  localeChangeCallbacks.push(callback);
}
```

The `onLocaleChange` API supports registration but not removal. If the Tray is destroyed and recreated (e.g., via `hideNotificationIcon` toggle), the old callback remains, keeping the destroyed Tray instance in memory. The callback array grows unboundedly.

Fix: Return an unregister function from `onLocaleChange` and call it in the Tray destructor.

12. **Multiline string parsing failure in merge** — `report_merge.go:297-305` [Gemini 3.1 Pro] (important, regression)

```go
if idx := strings.Index(trimmed, ": "); idx > 0 {
    candidate := trimmed[:idx]
    if isValidDottedKey(candidate) {
        key = candidate
        value = stripYAMLQuotes(trimmed[idx+2:])
    }
}
```

The line-oriented `parseMergeInput` parser cannot handle multiline values. When the translate command outputs block scalars (4 keys in en-us.yaml use `|-`), the parser captures only the first line and silently discards the rest.

Fix: Use `=` as the delimiter in the translate output format (already used for non-YAML input), or implement multi-line continuation parsing.

13. **`drift` reports false success for missing locale file** — `yaml.go:203-211`, `report_drift.go:43-48` [Codex GPT 5.4] (pass 3) (important, regression)

```go
func loadYAMLDocument(path string) (*yaml.Node, error) {
    data, err := os.ReadFile(path)
    if os.IsNotExist(err) {
        return &yaml.Node{Kind: yaml.DocumentNode, ...}, nil
    }
```

`loadYAMLDocument()` turns ENOENT into an empty document. `reportDrift()` consumes the empty tree, derives zero leaves, and prints "No drift detected" — a false green result for a typo like `--locale=zz` or a deleted locale file.

Fix: Verify that the locale file exists before calling `loadYAMLDocument()` in read-only report commands.

14. **Dynamic pattern wildcard does not match dot notation** — `scan.go:56-69` [Gemini 3.1 Pro] (pass 3) (important, regression)

```go
const segmentWildcard = `[a-zA-Z0-9_-]+`
```

`templateToKeyRegex` converts `prefix.${var}.suffix` into a regex using `segmentWildcard`, which excludes dots. If `var` resolves to a multi-segment key like `nested.category.item`, the regex will not match. This causes `unused` to erroneously flag keys and `references` to miss usages.

Fix: Expand to `[a-zA-Z0-9_.-]+`.

15. **Inconsistent YAML serialization between merge and remove** — `report_merge.go:133-134` vs `report_remove.go:174-177` [Gemini 3.1 Pro, Claude Opus 4.6] (pass 2) (important, regression)

`merge` uses the custom `serializeYAMLNode` while `remove` uses `yaml.NewEncoder`. The two serializers produce different formatting, causing noisy diffs when both operations touch the same file.

Fix: Use `serializeYAMLNode` consistently across all mutation commands.

16. **Duplicated `t()` implementation across test files** — `__mocks__/i18n.ts`, `UpdateStatus.spec.ts:19-46` [Claude Opus 4.6] (important, regression)

The `resolveKey` and `tFn` functions in `UpdateStatus.spec.ts` are exact duplicates of `getByPath` and `t` in `__mocks__/i18n.ts`. The helper functions should be shared via a test utility.

17. **`onWarningClick` method duplicated identically** — `BodyKubernetes.vue:78-83`, `ContainerEngineGeneral.vue:41-46` [Claude Opus 4.6] (important, regression)

Both components use identical click-delegation logic. Extract into a shared mixin or helper function.

18. **Deployment profile errors thrown before i18n initialization** — `deploymentProfiles.ts:89-91` [Gemini 3.1 Pro] (pass 3) (important, gap)

```typescript
throw new DeploymentProfileError(
  t('deploymentProfile.noVersion', { path: fullDefaultPath, version: ... }));
```

`readDeploymentProfiles()` runs at `background.ts:237`, before `initMainI18n()` at line 287. If a profile is invalid, the error dialog will always show the English fallback regardless of the user's locale.

Fix: Defer `readDeploymentProfiles()` until after `initMainI18n()`, or accept that early-startup errors are always English.

## Suggestions

1. **XSS risk from `v-html` rendering of translation strings** — `BodyKubernetes.vue:206`, `ContainerEngineGeneral.vue:95`, `KubernetesError.vue:39-41`, `General.vue:5-6` [Claude Opus 4.6] (pass 2) (suggestion, regression)

Translation values rendered via `v-html` could inject arbitrary HTML. The YAML files are generated by AI agents and merged without HTML sanitization. In Electron with `nodeIntegration: true`, XSS is equivalent to arbitrary code execution. Practically, the attack requires compromising translation files that are built into the app at compile time, so the risk is low. Defense-in-depth: add a validation check in `report_validate.go` that rejects `<script`, `on[a-z]+=`, and `javascript:` in values rendered via `v-html`.

2. **Nested ICU validation gap** — `report_validate.go:185-208` [Gemini 3.1 Pro] (suggestion, gap)

`extractPlaceholderNames` only extracts placeholders at brace `depth == 0`, missing nested placeholders inside ICU branches (e.g., `{name}` inside `{count, plural, other {items for {name}}}`). Document the limitation or replace with an AST-based ICU parser.

3. **ICU branch parser ignores quoted braces** — `report_validate.go:420-435` [Gemini 3.1 Pro] (pass 2) (suggestion, gap)

The brace-counting logic does not account for ICU literals wrapped in single quotes (e.g., `'{'`), causing depth tracking to desync on strings with literal braces.

4. **Iterative string replacement allows placeholder cross-contamination** — `i18n.ts:114-118` [Gemini 3.1 Pro] (pass 2) (suggestion, gap)

Sequential `replaceAll` calls mean an earlier argument's value could contain a later argument's placeholder pattern. Unlikely in practice, but a single regex `replace` pass would be safer.

5. **Renderer eagerly loads ALL locale YAML files at startup** — `store/i18n.js:148-154` [Claude Opus 4.6] (pass 2) (suggestion, regression)

The `init` action loads every locale file so `availableLocales` can display native names. With 4 locales, this blocks the initial render with ~20,000 lines of YAML parsing — for a feature-gated language picker most users cannot access.

Fix: Lazy-load non-selected locales when the language picker renders, or extract locale names into lightweight metadata.

6. **`settings-read` race: locale flash on startup** — `store/i18n.js:174-183` [Claude Opus 4.6] (pass 2) (suggestion, regression)

The `init` action dispatches `switchTo` with the cookie value, then sends an async `settings-read` IPC message. When the response arrives with a different locale, the UI flashes. Guard `switchTo` with serialization similar to `pendingLocale` in `main/i18n.ts`.

7. **`intlCache` eviction could leak entries** — `store/i18n.js:215-221` [Claude Opus 4.6] (suggestion, regression)

Switching from locale A to B evicts only A's entries. Switching A → B → A leaves B's entries. A full `Object.keys(intlCache).forEach(k => delete intlCache[k])` on switch would be simpler.

8. **`checkNoproxyList()` missed by validator i18n sweep** — `settingsValidator.ts:737-746` [Codex GPT 5.4] (suggestion, regression)

The duplicate/invalid-entry branches still emit hardcoded English while the rest of the validator uses `t(...)`.

9. **Redundant store dispatch in `ApplicationGeneral.vue`** — `ApplicationGeneral.vue:53-60` [Gemini 3.1 Pro] (suggestion, enhancement)

The manual `$store.dispatch('i18n/switchTo', locale)` duplicates the `settings-update` IPC handler that already dispatches `switchTo`. (Also interacts with finding I-9 above.)

10. **`driftEntry` results not sorted** — `report_drift.go:66-82` [Claude Opus 4.6] (pass 2) (suggestion, regression)

Iteration over `localeLeaves` (a map) produces non-deterministic output, complicating diff-based workflows.

11. **`manifest --cross-validate` regex too strict** — `report_manifest.go:43-56` [Codex GPT 5.4] (pass 2) (suggestion, regression)

`apiLocaleEnumRe` requires `enum:` immediately after `type: string`, but `command-api.yaml` now has comment lines between them. Parse the OpenAPI YAML instead of scraping with a line-shape regex.

12. **Hardcoded punctuation concatenated to translated phrases** — `UpdateStatus.vue:157-174` [Gemini 3.1 Pro] (pass 2) (suggestion, gap)

Punctuation marks (`.` and `; `) are hardcoded in the script. Some locales use different punctuation or clause ordering.

13. **Stale TODO comment** — `transientSettings.ts:2-3` [Gemini 3.1 Pro] (pass 3) (suggestion, regression)

The TODO about translating `navItemNames` was resolved by this branch via `labelKey` in `preferenceConstants.ts`. Remove the stale comment.

14. **Remaining hardcoded UI strings in touched files** — `PortForwarding.vue:29`, `ContainerInfo.vue:171`, `volumes/files/_name.vue:7,120` [Codex GPT 5.4] (pass 4) (suggestion, gap)

Several components touched by this branch still have hardcoded English strings (e.g., "Include Kubernetes services", "Available", "Not Found"). These leave visible English islands when a non-English locale is active.

## Design Observations

### Concerns

- **HTML in translation strings is fragile** (in-scope) [Claude Opus 4.6]: Five translation keys embed `<a>` tags with `data-navigate` or `data-action` attributes that application code relies on for click handling. Translators can easily break navigation by altering these attributes. Using named slots or structured placeholders (e.g., `{link_start}WebAssembly{link_end}`) would let the code own the HTML while translators handle only text, and would eliminate the `v-html` usage.

- **Main-process `t()` used in API error messages** (future) [Claude Opus 4.6]: Validation errors from `settingsValidator.ts` flow through the HTTP command API. Translating API error messages means `rdctl` output language depends on the server's locale, not the CLI user's preference. Consider keeping API-facing errors in English.

- **Three coupled artifacts per locale** (in-scope) [Codex GPT 5.4]: The tool now maintains locale YAML, metadata YAML, and manifest registration per locale. Only `merge` updates locale state holistically; `remove`, `meta`, `validate`, `check`, and `manifest` each touch a subset. A shared "load/mutate/rewrite locale+metadata/validate manifest" layer would remove this split-brain state.

- **Dual interpolation systems** (in-scope) [Claude Opus 4.6]: The main process uses `{variable}` while the renderer uses ICU MessageFormat. Translation strings must use different syntax depending on which process renders them. The `validate` command does not distinguish between the two schemes.

### Strengths

- The feature gate (`application.locale = 'none'` hiding the selector) is pragmatic — it ships infrastructure without committing to translation quality.
- The `onLocaleChange` callback pattern in `main/i18n.ts` cleanly decouples locale loading from menu/tray rebuilding. The `pendingLocale` guard correctly handles rapid locale switches.
- The Go `i18n-report` tool is well-designed with comprehensive validation (placeholder parity, ICU structure, tag matching, metadata coherence).
- The `titleKey`/`descriptionKey` approach for reactive page headers avoids re-rendering entire pages while keeping titles in sync.
- Using `import.meta.webpackContext` for locale loading keeps the approach consistent with the existing build pipeline.

```diff
-    this.$root.navigate(target.getAttribute('data-navigate'));
```

1. **Locked-field regex will break under non-English locales** — `settingsValidator.ts:287-290`, `commandLineOptions.ts:142`, `backendHelper.ts:171` (important, gap)


## Design Observations

**Concerns**

- **HTML in translation strings is fragile** (in-scope): Five translation keys embed `<a>` tags with `data-navigate` or `data-action` attributes that application code relies on for click handling. Translators can easily break navigation by altering or omitting these attributes. The README (`README.md:157-160`) already acknowledges this. A safer approach: use named slots or structured placeholders (e.g., `{link_start}WebAssembly{link_end}`) so the code owns the HTML and translators only handle text. ### [Gemini] — Pass 1

### Executive Summary

This PR introduces comprehensive internationalization (i18n) support across Rancher Desktop, including a lightweight main-process translation engine, locale state management in the Vue renderer, and a new Go-based CLI tool (`i18n-report`) for translation maintenance. While the frontend and Electron main-process architectures are solidly implemented, the custom YAML parser and serializer in the new Go tooling contain critical data-loss bugs that corrupt multiline block scalars (e.g., Markdown-formatted strings or indented code blocks) on both reading and writing.

### Findings

#### Critical Issues

1. **Multiline string formatting corruption (write)** — `src/go/i18n-report/yaml.go:377-388` `(critical, regression)`

```go
               scalar := yamlScalar(valNode.Value)
               if strings.Contains(scalar, "\n") {
                       lines := strings.Split(scalar, "\n")
// ...
                       for _, line := range lines[1:] {
                               trimmed := strings.TrimLeft(line, " ") // <--- Strips ALL leading spaces
                               if trimmed == "" {
```

When `yaml.Marshal` serializes a multiline block scalar, it indents the body by a fixed number of spaces (typically 2). The custom YAML serializer in `serializeNode` (and similarly in `writeNestedYAML`) splits the block scalar and calls `strings.TrimLeft(line, " ")`. This incorrectly strips **all** leading whitespace, completely destroying the relative indentation of multiline translations (such as indented code blocks, nested Markdown lists, or procedural spacing).

**Fix**: Strip only the baseline indentation introduced by `yaml.Marshal` rather than a greedy character cutset. Since `yaml.Marshal` indents the body relative to the margin, use `strings.TrimPrefix(line, "  ")` or properly inspect the indentation level.

2. **Multiline string parsing failure (read)** — `src/go/i18n-report/report_merge.go:277-295` `(critical, regression)`

```go
               // Parse key-value pair: try "key: value" then "key=value".
               var key, value string
               if idx := strings.Index(trimmed, ": "); idx > 0 {
                       candidate := trimmed[:idx]
                       if isValidDottedKey(candidate) {
                               key = candidate
                               value = stripYAMLQuotes(trimmed[idx+2:])
                       }
               }
```

The custom line-by-line parser in `parseMergeInput` does not support multiline values. If an agent or human provides a block scalar (e.g., `key: |-\n  multiline\n  text`), the parser processes the first line, assigns `"|-"` to the value, and ignores the subsequent body lines because they lack a `: ` delimiter. This results in severe data loss during merges.

**Fix**: Rather than using a string-splitting heuristic, rely on `yaml.v3` to correctly parse block scalars. If you must extract preceding comments, decode the input into a `yaml.Node` document tree, which natively retains `HeadComment` strings without sacrificing YAML compliance.

The structural validator tracks brace depth and only extracts placeholders at `depth == 0` to avoid incorrectly flagging literal strings inside ICU branch bodies as placeholders. However, this means that real placeholders nested within an ICU branch (e.g., the `{name}` inside `{count, plural, other { items found for {name} }}`) are completely ignored by the validator. #### Suggestions

4. **Redundant store dispatch in preference component** — `pkg/rancher-desktop/components/Preferences/ApplicationGeneral.vue:53-60` `(suggestion, enhancement)`

```typescript

```

Calling `this.$store.dispatch('i18n/switchTo', locale)` is redundant. The `onChange` method sends a settings update to the main process via IPC, which subsequently broadcasts a `settings-update` event back to the renderer. The `i18n` store already listens to `settings-update` (in `store/i18n.js:159`) and dispatches `switchTo` automatically.

**Fix**: Remove the manual dispatch to avoid double-triggering the dynamic locale import and rely strictly on the `settings-update` reactivity loop.

### Design Observations

**Concerns**
- **Custom YAML Serialization (in-scope)**: Building a custom YAML serializer and parser just to retain `@reason` comments is inherently brittle. The `gopkg.in/yaml.v3` library natively supports reading, modifying, and writing AST trees (`yaml.Node`) while perfectly preserving `HeadComment` and `LineComment` values. Removing the custom text-manipulation logic in `yaml.go` and `report_merge.go` in favor of standard `yaml.Node` mutations would eliminate the parsing and formatting bugs entirely.

**Strengths**
- **Clean Event-Driven Reactivity**: The `onLocaleChange` pattern in `main/i18n.ts` is excellently executed. It allows independent modules (like the Tray and Application Menu) to hook into locale updates without tight coupling, sidestepping messy race conditions during the initial settings fetch.
- **Architectural Fallbacks**: The renderer store correctly handles unmapped native language names by defensively checking if the translated name is missing, preventing UI rendering errors when an incomplete translation is loaded.

## Executive Summary

This branch adds comprehensive i18n support to Rancher Desktop, including a Go CLI maintenance tool, main-process and renderer-process translation infrastructure, and translations for ~40 components across 3 locales. ## Findings

### Critical Issues

1. **XSS via v-html rendering of translation strings** — `BodyKubernetes.vue:206`, `ContainerEngineGeneral.vue:95`, `KubernetesError.vue:39-41`, `General.vue:5-6` (critical, regression)

```vue
<!-- eslint-disable-next-line vue/no-v-html -->
<banner
  color="warning"
  @click.prevent="onWarningClick"
  v-html="t('kubernetesPrefs.options.spinkubeWarning', {}, true)"
/>
```

Translation values are rendered as raw HTML via `v-html`. While the current English strings in `en-us.yaml` are controlled, the YAML translation files (`de.yaml`, `zh-hans.yaml`) are generated by AI translation agents and merged by the `i18n-report merge` tool. A malicious or compromised translation string could inject arbitrary HTML/script into the renderer process. The `merge` tool at `report_merge.go:297-305` (the `parseMergeInput` function) does no HTML sanitization on input values — it passes raw text directly to the YAML file. The `validate` command (`report_validate.go`) checks placeholder parity and ICU structure but does not check for HTML injection in values that the codebase renders via `v-html`.

This is an Electron app with `nodeIntegration: true` and `contextIsolation: false` (visible at `preferences.ts:32-33`), so XSS in the renderer grants full Node.js access — it is equivalent to arbitrary code execution.

Fix: Either (a) sanitize HTML in `v-html`-bound translation values (e.g., strip `<script>`, event handlers) at the point of use, or (b) add a validation check in `report_validate.go` that rejects translation values containing `<script`, `on[a-z]+=`, `javascript:`, etc. for keys known to be rendered via `v-html`. Option (b) is defense-in-depth; option (a) is the primary mitigation.

### Important Issues

2. **`onLocaleChange` callbacks are never unregistered — memory leak** — `i18n.ts:87-89`, `tray.ts:203`, `mainmenu.ts:24` (important, regression)

```typescript
export function onLocaleChange(callback: () => void): void {
  localeChangeCallbacks.push(callback);
}
```

The `onLocaleChange` API in `main/i18n.ts` (line 87) only supports registration, not removal. The README at line 167-169 acknowledges this: _"Each tray hide/show cycle leaks one callback."_ However, `mainmenu.ts:24` registers at module scope (called once), and `tray.ts:203` registers in the `Tray` constructor. If the Tray is destroyed and recreated (e.g., `hideNotificationIcon` toggle), the old callback remains. While `updateMenu()` at `tray.ts:326` checks `isDestroyed()`, the callback array grows unboundedly. For the mainmenu case, `buildApplicationMenu()` is safe since it operates on global Electron state, but the Tray closure captures `this`, keeping destroyed Tray instances in memory.

Fix: Return an unregister function from `onLocaleChange` and call it in the Tray destructor:

```diff
-export function onLocaleChange(callback: () => void): void {
-  localeChangeCallbacks.push(callback);
-}
+export function onLocaleChange(callback: () => void): () => void {
+  localeChangeCallbacks.push(callback);
+  return () => {
+    const idx = localeChangeCallbacks.indexOf(callback);
+    if (idx >= 0) localeChangeCallbacks.splice(idx, 1);
+  };
+}
```

3. **Renderer eagerly loads ALL locale YAML files at startup** — `store/i18n.js:148-154` (important, regression)

```javascript

    // Load all translation files so availableLocales can show native names.
    await Promise.all(
      state.available
        .filter(locale => !state.translations[locale])
        .map(locale => dispatch('load', locale)),
    );
```

The `init` action now awaits loading every locale file (line 148-154) so the `availableLocales` getter at line 28-44 can display native locale names. With 4 locale files currently (~5000 lines each), this blocks the initial render while parsing ~20,000 lines of YAML. As more locales are added, this will scale linearly. The `availableLocales` getter is only used by the language picker in `ApplicationGeneral.vue`, which is feature-gated behind `locale !== 'none'`. For the vast majority of users (those who haven't enabled the locale feature), all this work is wasted.

Fix: Lazy-load non-selected locales only when the language picker is actually rendered, or extract only the `locale.*` keys into a lightweight metadata file that can be loaded cheaply.

4. **Renderer `settings-read` race: locale could override cookie-based selection** — `store/i18n.js:174-183` (important, regression)

```javascript

  const raw = settings?.application?.locale;
  const locale = (!raw || raw === 'none') ? state.default : raw;
  if ( locale !== state.selected ) {
    dispatch('switchTo', locale);
  }

ipcRenderer.send('settings-read');
```

The `init` action at line 185 dispatches `switchTo` with the cookie value, then sends an async `settings-read` IPC message (line 183). When the response arrives (line 175-182), if the persisted locale differs from the cookie, it dispatches another `switchTo`. This causes a visible flash: the UI renders in one locale, then switches. Also, the `settings-update` listener at line 165-172 does not await `switchTo`, so rapid locale changes from `rdctl` could cause multiple concurrent `switchTo` dispatches. Since `switchTo` is async (it may call `load`), interleaving is possible.

Fix: Await the `settings-read` response before the first render, or remove the cookie-based fast path when `settings-read` is available. At minimum, guard `switchTo` with a serialization mechanism similar to `pendingLocale` in `main/i18n.ts`.

5. **`availableLocales` getter accesses unloaded translations before init completes** — `store/i18n.js:32` (important, regression)

## Design Observations

### Concerns

**Dual i18n systems with divergent interpolation** (in-scope) — The main process uses simple `{variable}` interpolation (`main/i18n.ts:74`) while the renderer uses ICU MessageFormat (`store/i18n.js:68`). This means translation strings must use different syntax depending on which process renders them — `{count}` works in the main process but would need `{count, number}` for ICU in the renderer. The `validate` command checks placeholder parity but doesn't distinguish between the two interpolation schemes. If a key is currently main-process-only but later moves to the renderer, its format strings may silently break. A comment in the translation README or validation logic could flag keys used in both contexts.

**Eager locale loading for an unreachable feature** (in-scope) — The language picker is feature-gated behind `locale !== 'none'` (the default), meaning most users cannot access it. Yet the renderer loads all locale files at startup (finding #3). The startup cost is borne by everyone while the feature benefits no one by default.

### Strengths

- The `pendingLocale` guard in `main/i18n.ts:114-119` correctly handles rapid locale switches by discarding stale loads. This is a well-designed concurrency pattern.
- The `page` store's `titleKey`/`descriptionKey` approach for reactive locale switching in page headers is clean — it avoids re-rendering entire pages while keeping titles in sync.
- The `i18n-report` Go CLI is comprehensive and well-decomposed. The subcommand architecture, batch support for parallel translation, and drift detection via metadata are production-quality tooling.
- Using `import.meta.webpackContext` for both main and renderer processes keeps locale loading consistent with the existing build pipeline.

### [Codex] — Pass 2

**Executive Summary**

This branch adds a full i18n workflow to Rancher Desktop: runtime translation support, a new locale setting, translated UI/main-process strings, and a substantial `i18n-report` maintenance tool. The highest-signal new issues are all in the Go tooling: the committed locale manifest is invalid and breaks manifest-aware commands, multiline metadata cannot round-trip so drift detection produces false positives, and `remove` leaves metadata orphaned so the documented cleanup workflow invalidates the repo state.

**Critical Issues**

None.

**Important Issues**

1. **`meta/locales.yaml` registers a locale that does not ship, so manifest-aware commands fail for every locale** — `pkg/rancher-desktop/assets/translations/meta/locales.yaml:7-12`, `src/go/i18n-report/manifest.go:89-97` (important, regression)

```yaml
locales:
  en-us:
    status: source
  de:
    status: experimental
  fa:
    status: experimental
```

[`meta/locales.yaml`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/translations/meta/locales.yaml#L7) adds `fa`, but [`Manifest.validate()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/manifest.go#L89) rejects any non-source locale without a matching `pkg/rancher-desktop/assets/translations/<locale>.yaml` file at line 95. There is no `fa.yaml` in this branch, so `loadManifest()` fails globally; on the current tree, `cd src/go/i18n-report && go run . validate --locale=de` exits with `manifest: locale "fa" listed but .../translations/fa.yaml does not exist`. That means `manifest`, `validate`, `check --policy`, and `meta --locale=all` are all broken for unrelated locales as soon as commit `48c5999169` lands.

Fix: either remove `fa` from [`meta/locales.yaml`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/translations/meta/locales.yaml) until `fa.yaml` is actually shipped, or add the missing locale file in the same change. If metadata-only locales are intentional, change [`Manifest.validate()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/manifest.go#L60) and its callers to model that state explicitly instead of treating the manifest as “all locales are buildable now”.

2. **Multiline metadata is written in YAML form but read back as only `|` / `|-`, causing false drift** — `src/go/i18n-report/metadata.go:31-45`, `src/go/i18n-report/metadata.go:65-73`, `src/go/i18n-report/report_drift.go:54-61` (important, regression)

```go
for _, line := range strings.Split(content, "\n") {
	line = strings.TrimSpace(line)
	idx := strings.Index(line, ": ")
	...
	value := stripYAMLQuotes(line[idx+2:])
	result[key] = value
}
```

[`writeMetadata()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/metadata.go#L65) emits multiline English strings with [`yamlScalar()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/metadata.go#L72), so keys like `imageEvents.deleteError` are stored as YAML block scalars in [`meta/de.yaml`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/translations/meta/de.yaml#L185). But [`parseMetadata()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/metadata.go#L31) line-splits the file and records only the first `key: |` / `key: |-` line at line 43, discarding the indented body lines entirely. When [`reportDrift()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_drift.go#L54) compares `enValue` against that truncated metadata, untouched multiline keys are always reported as drifted. On the current tree, `cd src/go/i18n-report && go run . drift --locale=de` falsely reports five drifted keys, including `imageEvents.deleteError`, `imageEvents.pushError`, and `sortableTable.paging.generic`.

Fix: stop hand-parsing metadata lines. Read metadata with a real YAML decoder into `map[string]string` or reuse [`loadYAMLFlat()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/yaml.go). Add regression tests that write/read block scalars and verify `drift`, `translate --mode=drift`, `check`, and `validate` all agree on multiline keys.

3. **`remove` updates locale files but never updates `meta/*.yaml`, so the documented cleanup workflow leaves orphaned metadata** — `src/go/i18n-report/report_remove.go:43-57`, `src/go/i18n-report/report_remove.go:71-103`, `pkg/rancher-desktop/assets/translations/README.md:131-138` (important, regression)

```go
for _, path := range targets {
	removed, err := removeKeysFromFile(path, keySet)
	...
}
```

[`runRemove()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_remove.go#L43) and [`removeStaleKeys()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_remove.go#L71) only mutate translation YAML files returned by [`findTranslationFiles()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_remove.go#L123), and `findTranslationFiles()` deliberately excludes the `meta/` directory at lines 125-137. The new workflow in [`translations/README.md`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/translations/README.md#L131) explicitly tells maintainers to run `unused | remove` and `remove --stale`, but after either command the per-locale metadata files still contain entries for deleted keys. [`reportValidate()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_validate.go#L93) and [`reportDrift()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_drift.go#L89) then flag those entries as orphaned, so the tool’s own maintenance path produces an invalid repo state.

Fix: whenever a locale file is changed by `remove`, remove the same keys from its metadata file or regenerate metadata from the surviving locale keys before returning. The simplest safe path is to collect affected locales, reload their post-removal YAML, and call [`writeMetadata()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/metadata.go#L51) for each one.

**Suggestions**

1. **`manifest --cross-validate` is too regex-dependent to parse the current schema block** — `src/go/i18n-report/report_manifest.go:43-56`, `pkg/rancher-desktop/assets/specs/command-api.yaml:598-602` (suggestion, regression)

```go
var apiLocaleEnumRe = regexp.MustCompile(
	`locale:\s*\n\s+type:\s+string\s*\n\s+enum:\s*\[([^\]]+)\]`)
```

[`apiLocaleEnumRe`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_manifest.go#L43) requires `enum:` to appear immediately after `type: string`, but [`command-api.yaml`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/specs/command-api.yaml#L598) now has two comment lines at lines 600-601 between them. So even after finding 1 is fixed, [`crossValidateManifest()`](/private/tmp/review-worktree-ntn9Tl/src/go/i18n-report/report_manifest.go#L54) will still return `could not find locale enum in command-api.yaml` on the current file layout.

Fix: parse the OpenAPI YAML instead of scraping it with a line-shape regex, or at minimum relax the matcher to tolerate comments and extra properties between `type` and `enum`. Add a test that uses a commented schema block like the one currently committed.

**Design Observations**

**Concerns**

- `(in-scope)` The tool now has three coupled artifacts per locale: locale YAML, metadata YAML, and manifest registration. Only `merge` updates locale state holistically; `remove`, `meta`, `validate`, `check`, and `manifest` each touch or validate a subset. A small shared “load locale / mutate locale / rewrite locale+metadata / validate manifest” layer would remove this split-brain state and eliminate classes of bugs like findings 2 and 3.

**Strengths**

- The split between renderer ICU formatting and simpler main-process interpolation is pragmatic; it avoids dragging `intl-messageformat` into Electron main while still keeping a single translation source.
- The new validation surface in `i18n-report` is directionally good. Placeholder/tag/manifest checks are the right kind of guardrails for a translation-heavy workflow.

**Testing Assessment**

`cd src/go/i18n-report && go test ./...` passes, but the missed cases are meaningful:

1. No test covers a manifest entry whose locale file is absent, so `meta/locales.yaml` can brick `manifest`/`validate`/`check --policy` without any failure.
2. No test round-trips multiline metadata through `writeMetadata()` and `loadMetadata()`, even though multiline English strings already exist in the shipped metadata files.
3. No test verifies that `remove` or `remove --stale` also keeps `meta/<locale>.yaml` coherent.
4. No test exercises `manifest --cross-validate` against a `command-api.yaml` block containing comments between `type` and `enum`.

**Documentation Assessment**

[`pkg/rancher-desktop/assets/translations/README.md`](/private/tmp/review-worktree-ntn9Tl/pkg/rancher-desktop/assets/translations/README.md) needs two follow-ups: the maintenance workflow at lines 131-138 should mention metadata regeneration/removal, and the `command-api.yaml` comment at lines 600-601 says “run i18n-report manifest to cross-validate” even though the actual flag is `manifest --cross-validate`.

### [Gemini] — Pass 2

### Executive Summary

This change introduces comprehensive internationalization (i18n) support across Rancher Desktop, including a robust Go-based CLI tool (`i18n-report`) for managing translations, main-process locale injection, and broad translation of UI text. While the implementation successfully covers an extensive surface area and establishes a solid foundation for i18n workflows, a few significant edge cases remain. Specifically, there is a regression where the main process fails to revert to the default locale when the user clears the setting, a formatting destruction bug when removing keys, and parsing inaccuracies in the ICU structure validator. 

### Findings

#### Critical Issues

1. **Main process fails to revert to default locale when set to 'none'** — `pkg/rancher-desktop/main/i18n.ts:125-130` (critical, regression)

```typescript
export async function onSettingsUpdate(settings: RecursivePartial<Settings>) {
  const locale = settings?.application?.locale;

  if (locale && locale !== 'none') {
    await switchTo(locale);
  }
}
```

When a user switches their application locale back to "Language (None / System Default)", `application.locale` is set to `'none'`. The renderer process correctly handles this by reverting to `'en-us'` (`store/i18n.js`), but the main-process `onSettingsUpdate` handler simply ignores `'none'` entirely. This leads to split-brain behavior: the main process (including tray menus and system dialogs) remains stuck in the previously selected language until the application is fully restarted.

Fix: Fall back to the default language ('en-us') when the locale is missing or `'none'`.

```diff
 export async function onSettingsUpdate(settings: RecursivePartial<Settings>) {
-  const locale = settings?.application?.locale;
+  const raw = settings?.application?.locale;
+  const locale = (!raw || raw === 'none') ? 'en-us' : raw;
 
-  if (locale && locale !== 'none') {
+  if (locale !== currentLocale) {
     await switchTo(locale);
   }
 }
```

#### Important Issues

2. **ICU branch parser incorrectly aborts on quoted braces** — `src/go/i18n-report/report_validate.go:420-435` (important, gap)

```go
		// Skip the branch body by matching braces.
		depth := 0
		j := 0
		for j < len(rest) {
			if rest[j] == '{' {
				depth++
			} else if rest[j] == '}' {
				depth--
				if depth == 0 {
					j++
					break
				}
			}
			j++
		}
```

The ICU syntax parser in `report_validate.go` blindly counts `{` and `}` to skip branch bodies. It fails to account for ICU literals (text wrapped in single quotes, e.g., `one {'{'}`). If a translation branch contains a quoted brace literal, the parser misinterprets it as a structural brace, incorrectly tracking depth and causing subsequent branches in the same construct to fail validation.

Fix: Add logic within the loop to track whether the parser is inside a single-quoted string literal (`'...'`) and suppress depth changes while inside it.

3. **Inconsistent YAML serialization strips formatting and contradicts README** — `src/go/i18n-report/report_merge.go:133-134` and `src/go/i18n-report/report_remove.go:174-177` (important, regression)

```go
// In report_merge.go:
	// Serialize and write.
	var buf strings.Builder
	serializeYAMLNode(&buf, doc)

// In report_remove.go:
	var buf bytes.Buffer
	enc := yaml.NewEncoder(&buf)
	enc.SetIndent(2)
	if err := enc.Encode(&doc); err != nil {
```

The `merge` tool uses `serializeYAMLNode` which directly serializes the `yaml.Node` tree but strips all blank lines separating top-level groups, breaking the structure of `en-us.yaml` and contradicting its own README. Additionally, the `remove` command uses `yaml.NewEncoder`, which uses a completely different formatting strategy that completely reflows the file, leading to massive formatting diffs when removing a single key.

Fix: Standardize formatting across all mutation commands. Update `serializeYAMLNode` to preserve or recreate blank lines between top-level nodes, or update both `merge` and `remove` to build the final list of leaves and use `yaml.go:writeNestedYAML` to ensure strict, uniformly sorted YAML output.

4. **Iterative string replacement causes placeholder cross-contamination** — `pkg/rancher-desktop/main/i18n.ts:114-118` (important, gap)

```typescript
  if (args) {
    for (const [name, value] of Object.entries(args)) {
      msg = msg.replaceAll(`{${ name }}`, String(value));
    }
  }
```

The main process `t()` function performs sequential string replacements based on the arguments' `Object.entries`. If an earlier argument's injected `value` string happens to contain a substring matching a later argument's key (e.g. `{count}`), the later iteration will incorrectly expand it. This is a classic vulnerability for untrusted or dynamic input (like container names or error details).

Fix: Use a single regex `replace` pass instead of iterative `replaceAll`.

```diff
   if (args) {
-    for (const [name, value] of Object.entries(args)) {
-      msg = msg.replaceAll(`{${ name }}`, String(value));
-    }
+    msg = msg.replace(/\{(\w+)\}/g, (match, name) => {
+      return args[name] !== undefined ? String(args[name]) : match;
+    });
   }
```

#### Suggestions

5. **Hardcoded punctuation concatenated to translated phrases** — `pkg/rancher-desktop/components/UpdateStatus.vue:157-174` (suggestion, gap)

```typescript
      const prefix = this.t('updateStatus.available', { version: info.version });

      if (!progress) {
        return `${ prefix }.`;
      }
// ...
      return `${ prefix }; ${ this.t('updateStatus.downloading', { percent: String(percent), speed }) }`;
```

Punctuation marks (`.` and `; `) are hardcoded directly into the script instead of being handled by the translation map. Some locales (like Chinese) do not use ASCII periods or semicolons, and sentence structures might dictate a different ordering of clauses.

Fix: Move the punctuation and structure into the translation YAML (e.g. `updateStatus.statusWithProgress: '{prefix}; {suffix}'`).

2. **Dynamic pattern wildcard does not match dot notation** — `src/go/i18n-report/scan.go:56-69`

```go
// segmentWildcard matches a single key segment produced by an interpolation.
const segmentWildcard = `[a-zA-Z0-9_-]+`
```

`templateToKeyRegex` converts a template literal (e.g., `prefix.${var}.suffix`) into a regular expression by replacing the interpolation with `segmentWildcard`. Because `[a-zA-Z0-9_-]+` does not include a dot (`.`), interpolations that resolve to nested key segments (e.g., `var` = `nested.category.item`) will not match the regex. This causes the `unused` command to erroneously flag keys as unused and the `references` command to miss usages.

**Fix**: Expand the wildcard character class to allow dots so that interpolations spanning multiple segments match correctly.

```diff
-const segmentWildcard = `[a-zA-Z0-9_-]+`
+const segmentWildcard = `[a-zA-Z0-9_.-]+`
```

**Important Issues**

1. **`noproxy` validation still returns hardcoded English errors after the validator was localized** — [`settingsValidator.ts`](\/private\/tmp\/review-worktree-ntn9Tl\/pkg\/rancher-desktop\/main\/commandServer\/settingsValidator.ts#L737) (important, regression)

```ts
if (duplicateValues.length > 0) {
  duplicateValues.sort(Intl.Collator().compare);
  errors.push(`field "${ fqname }" has duplicate entries: "${ duplicateValues.join('", "') }"`);
}
...
if (invalidEntries.length > 0) {
  errors.push(`field "${ fqname }" has invalid entries (must be IP addresses, CIDR subnets, or domain names): "${ invalidEntries.join('", "') }"`);
}
```

The i18n sweep converted nearly all `SettingsValidator` diagnostics to `t(...)`, but the `checkNoproxyList()` branches at line 739 and line 746 still emit raw English. That means a user on a non-English locale gets mixed-language API / `rdctl` validation output specifically for `experimental.virtualMachine.proxy.noproxy`, even though this file’s stated goal is to localize validator errors. `git blame` shows line 739 came from commit `505269b1f5` and line 746 from `86191e3f37`, both on this branch, so this is introduced here rather than a pre-existing gap.

Fix: route both branches through translation keys, e.g. reuse `validation.duplicateEntries` for the duplicate case and add a dedicated `validation.invalidNoProxyEntries` key for the invalid-entry case. Extend [`settingsValidator.spec.ts`](\/private\/tmp\/review-worktree-ntn9Tl\/pkg\/rancher-desktop\/main\/commandServer\/__tests__\/settingsValidator.spec.ts) with coverage for duplicate and invalid `noproxy` lists so future validator-localization sweeps can’t miss these branches again.
