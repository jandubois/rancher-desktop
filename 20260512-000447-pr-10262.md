## Executive Summary

The three commits address a real production hang: the `/proc/net` scanner appends `PREROUTING -j DNAT --to-destination 127.0.0.1:<port>` for bridge-network ports whose `CNI-HOSTPORT-DNAT` chain already manages traffic, shadowing the engine's rule on `iptables-nft` kernels and breaking `docker run -p 127.0.0.1:N:N` end-to-end. The transient listener procnet catches turns out to be nerdctl's own OCI `createRuntime` hook — not the CNI portmap plugin the PR description names — holding `127.0.0.1:<hostPort>` while it wires up host port forwarding, before CNI installs its own iptables rules. The fix stacks three mechanisms in order of effectiveness: a two-tick stability gate (`seenLastScan` / `added`) as the **primary** filter — the hook never lives long enough to be seen twice on a normal-load system, so it never reaches `tracker.Add`; an engine-chain precheck (`engineChainManagesPort` + `dnatChainContainsPort`) as **defence-in-depth** for the rare case where the hook lingers across two scans on a loaded box and CNI has by then written its rule; and a `"proxy already running"` delegation path that suppresses the resulting `tracker.Add` retry storm without touching correctness (the rogue iptables Append was already skipped because the failing `tracker.Add` short-circuited the helper).

Structure: the diff is well-scoped to one package, the commits map onto the actual debugging arc, and the new code carries unusually thorough block comments. One Important finding remains: `engineChainManagesPort` receives only the port string, so a UDP engine rule on port N suppresses a TCP host-network DNAT for port N (and vice versa) — the same shadow-DNAT failure mode the PR fixes, flipped to the engine side. Both new iptables probes (`engineChainManagesPort` and `preroutingHasLoopbackRule`) also collapse every command failure to "not present" — Codex's original Important — but the stability gate narrows the failure window enough that this is now a Suggestion rather than a regression; on the append side the joint case (hook lingers, probe transiently loses the `nat` lock) is rare though correlated under load, and on the delete side a probe miss leaks a single PREROUTING rule rather than reintroducing the shadow class.

## Important Issues

I1. **Engine-chain check ignores protocol — UDP rule can suppress TCP DNAT and vice versa** — `src/go/guestagent/pkg/procnet/scanner_linux.go:222`, `260-277`, `285-288` [Claude Opus 4.7, Codex GPT 5.5] (important, regression)

```go
if action == Append && engineChainManagesPort(p.context, binding.HostPort) {
    log.Debugf(
        "skipping PREROUTING DNAT for %s/%s: port already managed by container engine chain",
        portProto.Proto(), binding.HostPort)
    continue
}
```

```go
func engineChainManagesPort(ctx context.Context, port string) bool {
    for _, chain := range []string{"CNI-HOSTPORT-DNAT", "DOCKER"} {
        cmd := exec.CommandContext(ctx,
            "iptables", "--table", "nat", "--list", chain, "--numeric")
        out, err := cmd.Output()
        if err != nil { continue }
        if dnatChainContainsPort(string(out), port) { return true }
    }
    return false
}
```

`execLoopbackIPtablesRule` already knows the protocol (`portProto.Proto()`), but only the port string is passed to `engineChainManagesPort`, and `dnatChainContainsPort`'s regex anchors on `dpt:` / `dports ` without anchoring on the `prot` column. `iptables -L --numeric` prints both `tcp dpt:N` and `udp dpt:N` in the same chain. The existing engine writers preserve protocol explicitly (`pkg/containerd/events_linux.go:333-335`, `pkg/docker/events.go:247-252`), so the new detection has lost a distinction the rest of the code respects.

Concrete failure: a user runs a bridge container that publishes UDP 53 (a DNS-style workload) and, separately, a `--network=host` container that listens on TCP `127.0.0.1:53`. The host TCP listener now stops being reachable from the Windows host: `engineChainManagesPort("53")` matches the UDP rule, procnet skips the TCP `PREROUTING` DNAT, and the host listener has no path back. This is the exact shadow-DNAT failure mode the PR exists to fix, flipped to the opposite side.

Fix: thread the protocol through and anchor the regex on it. The existing fixtures all use `tcp`; add a `udp dpt:N` fixture to lock the new behavior in.

## Suggestions

S1. **iptables probe errors silently treated as absent state** — `src/go/guestagent/pkg/procnet/scanner_linux.go:267-277`, `293-305` [Codex GPT 5.5] (downgraded from Important — see Reconciliation)

```go
out, err := cmd.Output()
if err != nil {
    // Chain absent (engine has not created it) or iptables errored: treat as not managed.
    continue
}
```

```go
func preroutingHasLoopbackRule(ctx context.Context, protocol, port string) bool {
    cmd := exec.CommandContext(ctx, "iptables", "--table", "nat", "--check", "PREROUTING", ...)
    return cmd.Run() == nil
}
```

Both probes collapse every `iptables` failure to "absent". With the stability gate as the primary defense, the append-side failure window narrows to a joint case: the hook listener must survive two scans *and* the engine-chain probe must transiently lose the `nat` table lock on the same tick. The two conditions are correlated under engine startup load (the same shape that makes the hook lingering more likely also makes lock contention more likely), so the window is not as small as independent rates suggest, but it is much narrower than "every probe failure". The codebase already treats `exit status 4` as a transient resource condition at `pkg/iptables/iptables.go:76-83` ("something else is running iptables"); the new helpers should follow that precedent for consistency.

The delete side has a different shape: `preroutingHasLoopbackRule` returns `false` on any error, `execLoopbackIPtablesRule` skips the Delete, and `ForwardPorts` then `delete(added, port)`s the entry — so a real PREROUTING rule we wrote leaks when the precheck transiently fails. Not a correctness regression (the leak self-cleans on RD restart), but it is a long-lived artifact.

Fix: distinguish "rule/chain does not exist" from "iptables errored". Treat absent as the only "false" path; return other errors so the caller can skip this scan without dropping `added` state.

```diff
-func preroutingHasLoopbackRule(ctx context.Context, protocol, port string) bool {
+func preroutingHasLoopbackRule(ctx context.Context, protocol, port string) (bool, error) {
     cmd := exec.CommandContext(ctx, "iptables", "--table", "nat", "--check", "PREROUTING", ...)
-    return cmd.Run() == nil
+    err := cmd.Run()
+    if err == nil { return true, nil }
+    if isIptablesRuleAbsent(err) { return false, nil }
+    return false, err
 }
```

An `isIptablesRuleAbsent` helper inspecting the error string (`"No chain/target/match by that name"` for the rule-absent case) plus a special case for `exit status 4` lines the new probes up with `pkg/iptables/iptables.go`.

S2. **Regex accepts `dport ` (singular) which iptables never emits** — `src/go/guestagent/pkg/procnet/scanner_linux.go:286` [Claude Opus 4.7]

```go
re := regexp.MustCompile(`(?:dpt:|dports? )(?:[0-9]+,)*` + regexp.QuoteMeta(port) + `\b`)
```

`iptables -L --numeric` emits `tcp dpt:N` for single ports and `multiport dports N[,N…]` for the multiport match — never `dport ` (singular with space). The `dports?` optional `s` only opens the door to false positives from chain comments or rule annotations that happen to contain the literal text `dport N`. The docstring at lines 279-284 already documents only `dpt:` and `dports`, so the optional `s` is also inconsistent with the stated contract.

Fix:

```diff
-re := regexp.MustCompile(`(?:dpt:|dports? )(?:[0-9]+,)*` + regexp.QuoteMeta(port) + `\b`)
+re := regexp.MustCompile(`(?:dpt:|dports )(?:[0-9]+,)*` + regexp.QuoteMeta(port) + `\b`)
```

S3. **Port-range form `dpts:M:N` is not detected** — `src/go/guestagent/pkg/procnet/scanner_linux.go:285-288` [Claude Opus 4.7] (gap)

`iptables -L --numeric` prints a port range as `tcp dpts:8000:9000` (plural `dpts:` with `:`-separated bounds). The current regex matches only `dpt:` (singular) and `dports ` (multiport list). A Docker container started with `-p 8000-9000:8000-9000` produces a `dpts:` rule the regex will not see, so a `--network=host` listener on any port in that range would still get a shadowing PREROUTING DNAT.

Not the typical bug path (most ports are published individually), but worth either a regex extension that also handles the range form (with `lo <= port <= hi` parsed out separately) or an explicit `// limitation:` comment on the helper documenting the gap.

S4. **Delegated-port cleanup is safe-by-accident, not safe-by-design** — `src/go/guestagent/pkg/procnet/scanner_linux.go:143-152`, `164-180` [Gemini 3.1 Pro] (downgraded from critical — see Reconciliation)

```go
if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
    log.Debugf("/proc/net scanner port %s already exposed elsewhere, delegating", port)
    added[port] = bindings
    continue
}
...
// Remove ports we exposed once their listener disappears.
for port, previousBindings := range added {
    if _, exists := newPortMap[port]; exists { continue }
    log.Infof("/proc/net scanner removed port: %s -> %+v", port, previousBindings)
    err := p.tracker.Remove(utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port())))
    ...
    if err = p.execLoopbackIPtablesRule(previousBindings, port, Delete); err != nil { ... }
    delete(added, port)
}
```

When a port is delegated (the engine's events handler already exposed it), procnet writes the binding into `added` to suppress retries but performs no iptables append. Later, when the engine's reservation socket disappears from `/proc/net/tcp`, the Remove loop runs `tracker.Remove` and `execLoopbackIPtablesRule(Delete)` for that port without distinguishing delegated from owned ports.

In the current code this is harmless because two unrelated invariants line up: (1) procnet's `Remove` key is `utils.GenerateID("tcp/N")` while the events handler stored the port under the real container ID, so `APITracker.Remove` ends up walking an empty `portStorage` entry and calls neither `apiForwarder.Unexpose` nor `wslProxyForwarder.Send`; (2) `preroutingHasLoopbackRule` returns false because procnet never wrote a rule, so the iptables Delete is also a no-op. The misleading `Infof("removed port ...")` log line is the only visible artifact.

The risk is that both invariants are implicit and easy to break: any future refactor that makes `Tracker.Remove` do a best-effort search across port mappings, or that drops the `preroutingHasLoopbackRule` precheck, would silently turn this into the bug Gemini originally described. Make the state explicit so the cleanup encodes its own correctness:

```diff
-added := make(nat.PortMap)
+type portState struct {
+    bindings  []nat.PortBinding
+    delegated bool
+}
+added := make(map[nat.Port]portState)
```

…and `if state.delegated { delete(added, port); continue }` at the top of the Remove loop.

S6. **Compile `dnatChainContainsPort` regex once at package level** — `src/go/guestagent/pkg/procnet/scanner_linux.go:285-288` [Gemini 3.1 Pro] (enhancement)

```go
func dnatChainContainsPort(iptablesListOutput, port string) bool {
    re := regexp.MustCompile(`(?:dpt:|dports? )(?:[0-9]+,)*` + regexp.QuoteMeta(port) + `\b`)
    return re.MatchString(iptablesListOutput)
}
```

`regexp.MustCompile` runs on every call. Not a hot loop (it fires only when a port passes the two-tick gate), so the performance cost is negligible, but it is the Go idiom to declare regex variables once at package scope. A natural refactor is to compile a port-list extractor once and check membership against the captured group:

```go
var dnatPortMatcher = regexp.MustCompile(`(?:dpt:|dports )([0-9,]+)\b`)

func dnatChainContainsPort(out, port string) bool {
    for _, m := range dnatPortMatcher.FindAllStringSubmatch(out, -1) {
        for _, p := range strings.Split(m[1], ",") {
            if p == port { return true }
        }
    }
    return false
}
```

This pairs naturally with the protocol fix in I1 (anchor the package-level regex on `prot` too).

The PR description states "CNI's portmap plugin briefly opens a port-reservation socket on `127.0.0.1:<port>` during container start." ## Design Observations

### Strengths

- *(in-scope)* The decision to key the Delete path off `added` rather than `seenLastScan` — and to write delegated ports into `added` — is what prevents a retry-storm of `"proxy already running"` log spam every tick. The inline comment at lines 104-108 captures the *why* well. [Claude Opus 4.7]
- *(in-scope)* Splitting `dnatChainContainsPort` out as a pure function with a table-driven test is exactly the right shape; the fixtures pull in the realistic iptables variants (single, multiport, IP-octet collision) instead of stubbing the regex against synthetic input. [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]
- *(in-scope)* The "alternative considered" block at lines 94-102 documenting the rejected pid/cgroup/inode classification path is unusual and valuable. It records the analysis behind the chosen approach so a future "we should classify by pid" suggestion can land on the prior reasoning rather than restart it. [Claude Opus 4.7]
- *(in-scope)* The two-scan stability gate is a narrow, stateless fix for the OCI-hook reservation-socket race — no `/proc/<pid>/comm`-scraping, no cgroup inspection, no inode-to-pid resolution. The cost (one tick of latency for `--network=host` listeners) is explicitly documented. [Codex GPT 5.5, Gemini 3.1 Pro]

### Concerns

- *(in-scope)* `ForwardPorts` now coordinates three maps (`seenLastScan`, `added`, `newPortMap`) and two transition rules (the gate; the delegation marker), and the loop has no unit-test coverage — only the regex helper is tested. The state-machine invariants the S1 fix relies on (Delete-side preserves `added` on error, Append-side preserves `added` when the probe fails) are exactly the shape that benefits from a `portStateTracker` extraction with a fake `tracker.Tracker` and an injectable port-map source. Not a merge blocker, but the next bug here will be easier to land if the loop is decomposed. [Claude Opus 4.7]

## Testing Assessment

The new `TestDnatChainContainsPort` table is solid for the regex it covers: single-port, multiport, suffix/prefix-of-port, IP-octet-collision pitfalls. Gaps, ranked by risk:

1. **Protocol-mismatch matrix** (covers I1) — no fixture exercises `udp dpt:N` alongside a TCP probe (and vice versa). One representative row: `{output: "...udp dpt:8080...", protocol: "tcp", port: "8080", want: false}` directly anchors the I1 fix.
2. **iptables-probe error handling** (covers S1) — neither `engineChainManagesPort` nor `preroutingHasLoopbackRule` is unit-testable today because they shell out directly. Once that lands, add a fixture for "chain absent (expected)" and one for "exit status 4 (transient)" to lock the distinction in.
3. **State machine in `ForwardPorts`** — no unit tests exercise the two-scan gate, the delegation path, or the divergence between `added` and `seenLastScan`. A fake tracker plus an injectable port-source would make this testable and would protect the S4 invariant against future refactors.
4. **Port-range fixture** (covers S3) — `tcp dpts:8000:9000` is not in test fixtures. Either lock the current behavior (`want: false`) and document the gap, or extend the matcher to handle ranges.
5. **Cross-platform verification** — the investigation that motivated this PR was Windows-only (WSL2 kernel `6.6.87.2`, iptables-nft 1.8.11). The same `published-ports.bats` scenario on macOS and Linux against current alpine-lima kernels has not been verified to either reproduce the original hang or accept the fix. If it passes there today the fix is still right (the rogue rule should not exist for bridge-network ports on any platform), but a one-shot manual check rules out a separate kernel-version interaction.

The PR's manual test plan correctly identifies the BATS scenario (`bats/tests/containers/published-ports.bats`) and the two manual smoke tests (bridge-network published port from Windows; `--network=host` loopback). Those were not run from this review.

## Review Process Notes

### Skill improvements

- **When an agent flags a bug centered on a call to a stored interface (a tracker, registry, cache, port map), verify the keying assumption before trusting the severity.** Re-check the agent's argument by tracing the actual lookup key the caller passes against the keys the storage was populated with — different ID schemes (synthetic vs. natural) can quietly neutralize a "this destructively modifies shared state" claim. Asymmetric IDs are a common pattern when two writers share a tracker, and the resulting "safe by accident" code reads exactly like a bug at a quick scan. 	re := regexp.MustCompile(`(?:dpt:|dports? )(?:[0-9]+,)*` + regexp.QuoteMeta(port) + `\b`)
```go
	return re.MatchString(iptablesListOutput)
}
```
(suggestion, enhancement)

**Problem**: `regexp.MustCompile` is executed dynamically on every invocation. While this is not a tight hot loop (it runs only when a port passes the 3-second gate), compiling regexes per function call is an anti-pattern that creates unnecessary allocations and CPU overhead.

**Fix**: Since `QuoteMeta(port)` changes the pattern dynamically, you can extract the regex compilation globally and match the extracted port numbers instead of templating the target port into the regex itself. This handles single and comma-separated multiports perfectly while only compiling once:

```go
var dnatPortMatcher = regexp.MustCompile(`(?:dpt:|dports? )([0-9,]+)\b`)

func dnatChainContainsPort(iptablesListOutput, port string) bool {
	matches := dnatPortMatcher.FindAllStringSubmatch(iptablesListOutput, -1)
	for _, match := range matches {
		if len(match) > 1 {
			for _, p := range strings.Split(match[1], ",") {
				if p == port {
					return true
				}
			}
		}
	}
	return false
}
```

### Design Observations

**Strengths**
- The stability gate (`seenLastScan`) is an elegant, stateless way to filter out the short-lived OCI hook listeners without resorting to expensive PID resolution or cgroup scraping. 
- The robust unit tests for `dnatChainContainsPort` excellently anticipate the quirks of `iptables` outputs, including multi-port lists and the risk of matching IP octets.

### Testing Assessment
1. **Delegation cleanup lifecycle**: The state transitions for a port that triggers `"proxy already running"` and later closes its reservation socket are untested. This gap hides the `CRITICAL` bug (C1) where `tracker.Remove` is actively called on an engine-owned port. Adding a unit test covering the `ForwardPorts` loop logic—mocking `tracker.Add` to return the already-exposed error, simulating the listener disappearing, and asserting `tracker.Remove` is skipped—would secure this lifecycle.
2. **Scanner flap resilience**: The core stability gate logic (`seenLastScan`) relies entirely on E2E integration tests. A localized test of the state machine (feeding it a sequence of mock port maps and asserting on the resultant `tracker` calls) would prevent future regressions in the loop mechanics.
