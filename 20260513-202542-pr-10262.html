Executive Summary ¶

Round 2's resolution code closed three Important findings (engine-managed-claim, Append retry asymmetry, partial-failure leak) but the new partial-failure plumbing introduces three new race windows. I1 is the load-bearing finding — independently raised by all three agents — and it negates the PR's stated guard #3: when wslProxyForwarder.Send fails transiently the first time, the resume-ownership branch marks the port delegated:false, appendFailed:false and continues, after which the alreadyAdded fast-path consumes every subsequent tick and neither the iptables Append nor the wsl-proxy retry ever fires. The inline comment at line 234 and the PR description both claim "the wsl-proxy retry will come on the next loop iteration"; the code at lines 182-196 only honors appendFailed. The bug is the exact failure mode procnet's PREROUTING DNAT exists to handle: a --network=host loopback listener stays unreachable from the Windows host for the listener's lifetime.

I2 is the symmetric gap in the Append retry path: when xtables-lock contention causes the first Append to fail, the retry at line 187 calls execLoopbackIPtablesRule again without re-probing the engine chain. preroutingHasLoopbackRule only checks for procnet's own exact rule; it does not look at CNI-HOSTPORT-DNAT or DOCKER. If the engine's events handler installed the chain rule between the failed tick and the retry, the retry installs the rogue PREROUTING DNAT after the authoritative engine rule exists — recreating the original shadow-DNAT bug. I3 is the cleanup-side deadlock: tracker.Remove clears portStorage via a leading defer regardless of whether the unexpose/wsl-proxy calls succeeded, so the PR's "retain added entry to retry the iptables Delete" pattern silently no-ops after the first tick, and if the listener reappears while the port is still in added, the Add loop's alreadyAdded fast-path locks the port out of all recovery.

The architectural recommendation converges across agents: merge this PR with I1-I3 fixed, then plan an explicit-marker (nonce) approach for the next release. The heuristic stack has reached the point where every round closes one race window and opens another in the new state-machine surface; a nonce eliminates the inference layer by construction but introduces coordination cost with nerdctl/CNI/docker that is bigger than this PR's scope. None of the three agents recommend blocking this PR pending a nonce design.

func engineChainManagesPort(ctx context.Context, protocol, port string) (bool, error) {
	for _, chain := range []string{"CNI-HOSTPORT-DNAT", "DOCKER"} {
		cmd := exec.CommandContext(ctx,
			"iptables",

Design Observations ¶

Concerns ¶

O1. The alreadyAdded fast-path is a chokepoint for all future-tick recovery (in-scope) Claude Codex. Lines 182-196 are the single integration point for retries from previous ticks, but the branch only acts when state.appendFailed is set. Anything else that wants future-tick attention (wsl-proxy retry, engine-chain re-probe, partial-failure cleanup) has to invent a new state flag and a new retry branch. I1, I2, and I3 are all instances of one half of the state machine completing and the next guard reading a different world. Folding appendFailed, delegated, and any future flags into a single state action enum (or extracting portStateTracker with an injectable tracker.Tracker — the author's own suggestion at line 99-101) would make the transitions explicit and unit-testable.

O2. The state machine is documented in prose but tested only end-to-end via BATS (future) Claude. The portState comment (lines 61-101) describes a non-trivial state machine, none of whose transitions is unit-tested. Promoting the portStateTracker extraction to a follow-up issue is worth it: regressions like I1, I2, and I3 are exactly what a state-machine unit test would catch on the first review pass instead of the third.

Strengths ¶

  • The two-scan stability gate is a small, well-documented heuristic with one acknowledged failure mode (one-tick delay for genuine listeners). Cheap and effective.
  • isIptablesRuleAbsent cleanly separates "rule doesn't exist" (an answer) from "iptables errored" (defer). Right shape for a retry-aware caller, table-driven test pins the stderr classifier across the known exit modes.
  • dnatChainContainsPort is a pure function with 23 well-chosen fixtures — IP-octet, cross-protocol, prefix/suffix, intentional range-form miss. The tests pin behavior so any future range-matching change must be deliberate.
  • TestPortAlreadyExposedSubstringIsReachable pins the substring contract across the verifyResponseBody → ErrExposeAPI wrap chain. The right way to defend against silent upstream breakage in gvisor-tap-vsock.
  • The Append idempotency probe via preroutingHasLoopbackRule and the symmetric Delete-side retain-on-error are good defenses against duplicate rules and rule leaks — even though I3 shows the Delete retain is not load-bearing for the failure mode it documents.
  • The explicit delegated flag corrects Round 1's safe-by-accident cleanup behavior. Future refactors can see that engine-owned ports must not go through procnet cleanup.

Testing Assessment ¶

The pure helpers (dnatChainContainsPort, isIptablesRuleAbsent) and the substring contract are well-covered. The state machine in ForwardPorts has zero unit-level coverage; every transition is exercised only by published-ports.bats end-to-end. Untested scenarios, ranked by risk:

  1. Partial-failure resume path (I1): Tick 1 returns wsl-proxy.Send error → Tick 2 returns "proxy already running" → assert iptables Append is present.
  2. Append-retry post engine-chain (I2): Tick 1: tracker.Add succeeds, Append fails → Tick 2: engine chain now has rule → assert procnet does not install PREROUTING DNAT.
  3. Cleanup deadlock (I3): Listener disappears → Delete fails → listener reappears → assert procnet re-adds.
  4. Two-scan stability gate: First sighting defers; second sighting acts; one-tick listener never reaches tracker.Add.
  5. Engine-chain delegation: engineManagesLoopbackBinding returns true → neither tracker.Add nor execLoopbackIPtablesRule is called; cleanup drops the local marker on listener disappearance.
  6. addErrorLogged log-once classification: First transient tracker.Add failure logs Error; subsequent failures log Debug; clearing on success or listener disappearance.

A portStateTracker extraction with an injectable tracker.Tracker (mocked) is the natural fixture for items 1-3 and 5-6.

Architectural Recommendation — Nonce vs. Heuristic ¶

All three agents converge: merge this PR with I1-I3 fixed, plan an explicit-marker (nonce) approach for the next release, do not block this PR on a nonce design.

1. Failure modes the heuristic stack still carries (post-I1/I2/I3) ¶

  • Partial-failure plumbing brittleness. Every round has introduced new retry/recovery paths; every round has missed a transition. The state machine has six paths through alreadyAdded / engine-probe / tracker.Add / cleanup, gated on three booleans (delegated, appendFailed, the implicit addErrorLogged), and no unit tests pin the transitions.
  • Two-scan gate latency. Three-second delay for genuine --network=host listeners. Acknowledged tradeoff, not a bug, but a real UX cost.
  • Hard-coded engine chain set {CNI-HOSTPORT-DNAT, DOCKER}. Adding a third engine silently regresses to the shadow-DNAT bug. Docstring names the risk; nothing prevents it.
  • Substring-match dependency on "proxy already running". Defended by TestPortAlreadyExposedSubstringIsReachable, but an upstream rewording silently flips behavior from "delegate" to "log error and never retry."
  • Kernel-version assumption: DNAT in PREROUTING terminates the chain on iptables-nft. Comment names this; a kernel default change reverses the bug class.
  • Regex parsing of iptables --list output. Documented range-form misses; works for shipped engines, fragile against hand-rolled iptables setups or upstream output-format changes.

2. Failure modes a nonce scheme would introduce or shift ¶

A concrete nonce — say, the engine's OCI hook writes /run/rancher-desktop/procnet/<proto>-<port> (with container ID, expiry, and engine identifier) before opening the reservation socket — replaces inference with declaration. New surface:

  • Coordinated change with nerdctl, containerd, CNI portmap, and the Docker OCI hook surface. None of these are owned by RD. Either ship a forked hook (maintenance drift) or upstream the feature and wait a release cycle.
  • Fallback when the nonce is absent — older nerdctl, sideloaded engines, third-party CNI plugins. Procnet keeps the heuristic guards for these cases; the heuristic stack does not disappear, it becomes the fallback path.
  • Stale-marker cleanup. /run markers need TTL or container-identity reconciliation; a crashed engine leaves orphans.
  • RD update model must tolerate mixed guestagent/engine versions — guestagent reads markers older engines don't write, and vice versa.
  • Test surface multiplies. Instead of "verify procnet correctly ignores the reservation socket," reviewers must verify nerdctl-with-marker × docker-with-marker × bridge × host across all engine combinations.

3. Weighing one more round vs. the nonce ¶

Nonce scope in this repo: changes in src/go/guestagent/pkg/procnet/scanner_linux.go (read the marker), a new oci-hook component (RD-owned), nerdctl/containerd config to invoke the hook in createRuntime, and the package/ISO build to deliver the hook. Realistic scope: 4-6 weeks for one engineer including end-to-end validation, plus an ongoing upstream conversation if RD wants the change in nerdctl proper.

One more round of heuristic refinement: I1/I2/I3 are ~50 lines of code total (Claude's inline Append fix, Codex's rollback on Append failure, Gemini's delete(added, port) simplification). The state-machine refactor (portStateTracker extraction with injectable tracker.Tracker) is another ~200 lines and makes the next regression catchable at unit-test time. Total: ~1 week.

4. Recommendation: merge this PR with I1-I3 fixed, plan the nonce for the next release ¶

  • I1, I2, and I3 are real correctness gaps, but each fix is small and self-contained. Blocking the PR forfeits the gains of guards 1 and 2, which by themselves close the original shadow-DNAT bug for the common case.
  • The residual failure modes after I1-I3 are narrow (partial-failure retries with transient downstream failures, third-engine support, kernel default changes). All self-heal across container restarts or RD restart.
  • A nonce eliminates inference but introduces coordination cost RD does not currently bear. The right time is when the heuristic surface becomes more expensive to maintain than the heuristic-vs-nonce tradeoff justifies — by all three agents' assessment, that point is the next refinement round, not this one.
  • The state-machine refactor (O1, O2) is the highest-leverage next investment regardless of the nonce decision. It makes the heuristic surface explicit and unit-testable, and gives a future nonce reader a clean integration point.

The signal across rounds is clear: each refinement closes one race window and exposes another in the new state-machine surface. Round 4 should stop adding heuristic layers and start designing the marker.
