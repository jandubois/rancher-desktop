Executive Summary ¶

This is Round 2. Round 1's two Important findings and three Suggestions all landed: retireDisappeared now continues after a failed tracker.Remove (keeping the loopback rule), retryAppend's engine-takeover branch now issues the idempotency-gated Delete, the exposeNew engine-probe failure routes through the addErrorLogged log-once guard, and the two stale doc comments are corrected. The test suite added TestRemoveFailureRetainsRuleForReappearance and TestAppendRetryEngineTakeoverDeletesCommittedRule as the I1/I2 regression tests, and fakeIptables now models installed-rule state. All Round 1 resolutions verify against the current code.

The new finding this round sits in the same retryAppend engine-takeover branch Round 1's I2 touched, but from a different angle. I1: that branch treats "an engine iptables chain references the port" as proof the engine owns the host-switch proxy, then calls tracker.Remove on procnet's synthetic entry. The two facts are independent — the CNI-HOSTPORT-DNAT / DOCKER chain rule is installed during container network setup, while portTracker.Add runs later in the events handler and can fail (when it races procnet's own proxy it gets proxy already running and never populates portStorage). In the bad interleaving, retryAppend unexposes the only host-switch proxy and marks the port delegated, so the published port is unreachable from the Windows host until RD restart. The probability is low — it needs the OCI reservation socket to survive the two-scan gate, a transient Append failure, and the engine's Add to lose the proxy race — but the impact matches Round 1's I1/I2, and the retryAppend comment actively claims an "now-authoritative" invariant the code does not establish.

The remaining four findings are a const-vs-literal inconsistency, two stale or overstated comments, and an iptables-error-propagation gap that drops stderr context from transient-failure logs. All are small.

In the bad interleaving — engine Add runs and fails first, then procnet's retryAppend fires — the port ends with the engine's iptables chain present, no host-switch proxy at all, and procnet marked delegated so it never retries. A published port (the common non-admin case maps both procnet's and the engine's bindings to 127.0.0.1:<port>, so the collision is real) is unreachable from the Windows host until RD restart.

Reachability is narrow: the OCI createRuntime reservation socket must survive the two-scan stability gate (the documented heavy-load limitation), the exposeNew Append must fail transiently, and the engine's Add must lose the proxy race. But every piece is individually plausible, the impact matches Round 1's I1/I2, and the retryAppend comment overstates the invariant ("the engine's now-authoritative chain") — the code never establishes that the engine owns the proxy.

Reconciliation: the raising agent labeled this a regression; reclassified as a gap. port_state_tracker_linux.go is net-new (+366 −0) and the merge-base ForwardPorts had no append-retry path, so this is an incompletely-closed corner of new code, not a behavior the PR broke — the same reconciliation Round 1 applied to I2, which is the same branch.

Fix: do not release the synthetic tracker entry on the strength of the engine chain alone. Two directions, both larger than a Round 1-style three-line patch:

  1. Delete only procnet's PREROUTING DNAT (the actual shadowing artifact) and keep the synthetic proxy as the owner; do not mark delegated. procnet's proxy keeps the port reachable, and the engine's later Add is a benign proxy already running. The cost is that procnet's proxy now outlives the reservation socket, so cleanup must key off the listener disappearing rather than off delegated.
  2. Add an explicit ownership-transfer probe so the synthetic entry is removed only once the engine's portStorage entry is known to exist.

At minimum, if the branch stays as-is, the comment must stop claiming the engine is authoritative and the limitation must be documented in the portState doc comment alongside the other acknowledged partial-failure gaps.

Design Observations ¶

Concerns ¶

  • (future) Residual netns-blind heuristic race — port_state_tracker_linux.go:138-205 Claude. The scanner reads /proc/net in the main netns and cannot distinguish a --network=host loopback listener (which it should expose) from a bridge container's OCI reservation socket (which it must not). The two-scan gate plus the exposeNew engine probe close the common windows, but if the reservation socket survives two scans, CNI-HOSTPORT-DNAT/DOCKER does not yet reference the port at the second-sighting probe, and procnet's tracker.Add wins the race, procnet installs a stable owned PREROUTING DNAT — and only appendFailed ports ever re-probe the engine chain, so an owned-and-stable port never re-checks once the CNI rule lands. Same root cause as I1 (the heuristic is netns-blind), different failure mode (a shadowing rule that persists rather than a removed proxy). The PR explicitly frames itself as accepting heuristic limits; worth a tracking note and a comment in exposeNew acknowledging the "engine rule lands after our second sighting" window.
  • (future) Same-port-number collision between a host-net listener and a bridge publish — port_state_tracker_linux.go:154-176 Claude. If a --network=host container listens on 127.0.0.1:8080 while a bridge container has published 8080, engineChainManagesPort returns true and procnet delegates the host-network listener — which then never gets the PREROUTING DNAT it actually needs. A genuine ambiguity (two things want the same host port), rare, but a behavior change from the pre-PR code, which always added the rule. Worth a comment, not code.
  • (in-scope) The new CI target gates on never-run pre-existing tests — package.json Claude. test:unit:guestagent runs go test ./..., which subjects apitracker_test.go, iptables_test.go, and servicewatcher_linux_test.go to CI for the first time. iptables/iptables_test.go:TestForwardPorts synchronizes through unbuffered channels with time.After(5 * time.Second) deadlines — a classic CI-flakiness shape under load. Worth confirming these pass reliably (not just once) before relying on the gate, since flakiness here would block unrelated PRs.
  • (in-scope) The closed engine-chain list is now load-bearing — scanner_linux.go:194-198 Codex. The chain list (CNI-HOSTPORT-DNAT, DOCKER) is documented as closed, but with this PR a missed future engine chain silently reintroduces the shadow-DNAT bug. The "revisit when adding a third engine" comment is correct; the consequence of forgetting is now more severe.
  • (future, pre-existing) %w in a log.Errorf call — scanner_linux.go:119 Claude. log.Errorf("failed to create portMapping for entry: %w", err) — %w is meaningful only in fmt.Errorf and renders as %!w(...). Confirmed present in the merge-base (scanner_linux.go:84 there), so pre-existing, not a regression; one line from changed code and a one-character fix (%w → %s). Also flagged in Round 1.

Acknowledged Limitations ¶

  • wsl-proxy re-notify not retried on the resume-ownership path — port_state_tracker_linux.go:53-59,222-235. The resume path installs the iptables rule the partial-failure path skipped but does not re-send the port mapping to wsl-proxy; Tracker exposes no idempotent re-notify hook. Documented in the portState doc comment.
  • Delete-side iptables leak on transient probe failure — scanner_linux.go:273-276. retireDisappeared drops the local marker unconditionally, so a transient preroutingHasLoopbackRule failure during cleanup leaks the rule until RD restart. The comment describes the leak but not that a leaked PREROUTING DNAT could re-trigger the shadow bug if the same port number is later reused — a borderline understatement, narrow enough that expanding the comment is the right response.
  • Stability gate bypass under heavy load — scanner_linux.go:88-99. A reservation socket living longer than two scan intervals (~6 s) bypasses the gate. This is the precondition for both I1 and the residual-race design observation.
  • Port-range iptables forms intentionally not matched — scanner_linux.go:236-247. dpts:LO:HI and multiport-embedded ranges are unmatched, with locked-in test fixtures; theoretical for the engines RD ships.
  • Engine chain list closed to two engines — scanner_linux.go:194-198. Documented, with an explicit "revisit when adding a third engine" note.
  • Tick aliases newPortMap into seenLastScan — port_state_tracker_linux.go:132-135. Documented invariant that the caller must pass a freshly-built map and not mutate it; ForwardPorts honors it.

Review Process Notes ¶

Skill improvements ¶

  • When a code path releases or transfers ownership of a resource because a correlated signal appeared (an iptables chain, a file, a lock file, a registry entry), check whether that signal is installed by the same code that establishes the ownership the path is transferring to. A signal and the ownership state it implies are often written by different components at different times; treating "signal present" as "the other owner is ready" creates a window where the releasing path hands off to nobody. Flag ownership-transfer branches that key off an observable side effect rather than off the destination owner's own state.
  • When reviewing a follow-up round, verify each prior-round fix against the current code before trusting the resolution table, and then look hardest at the code the fixes touched — a corner closed in round N often exposes an adjacent corner of the same branch in round N+1. Re-read the modified branch end to end rather than diffing only the fix.

Repo context updates ¶

  • [repo] In the guestagent port-forwarding code, an engine's iptables chain (CNI-HOSTPORT-DNAT, DOCKER, the per-container CNI-DN-* chains) is installed during container network setup, independently of and earlier than the engine's portTracker.Add call in the events handler (containerd/events_linux.go /tasks/start, docker/events.go ActionStart). Both handlers only log on an Add failure — neither retries. So "an engine chain references the port" does not imply "the engine owns the host-switch proxy / has a portStorage entry." When reviewing code that probes engine chains to decide procnet/tracker ownership, check that it does not conflate the two.
