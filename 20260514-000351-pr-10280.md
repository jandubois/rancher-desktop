landed: `retireDisappeared` now `continue`s after a failed `tracker.Remove`
(keeping the loopback rule), `retryAppend`'s engine-takeover branch now issues

resolutions verify against the current code.

The new finding this round sits in the same `retryAppend` engine-takeover
branch Round 1's I2 touched, but from a different angle. **I1**: that branch
treats "an engine iptables chain references the port" as proof the engine owns
the host-switch proxy, then calls `tracker.Remove` on procnet's synthetic
entry. The two facts are independent — the `CNI-HOSTPORT-DNAT` / `DOCKER` chain
rule is installed during container network setup, while `portTracker.Add` runs

interleaving, `retryAppend` unexposes the only host-switch proxy and marks the
port `delegated`, so the published port is unreachable from the Windows host

to survive the two-scan gate, a transient Append failure, and the engine's
`Add` to lose the proxy race — but the impact matches Round 1's I1/I2, and the
`retryAppend` comment actively claims an "now-authoritative" invariant the code
does not establish.

The remaining four findings are a const-vs-literal inconsistency, two stale or
overstated comments, and an iptables-error-propagation gap that drops `stderr`
context from transient-failure logs. All are small.

## Important Issues

I1. **`retryAppend` engine-takeover removes the only host-switch proxy** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:246-267` [Codex GPT 5.5]

```go
func (pst *portStateTracker) retryAppend(port nat.Port, state portState) {
	managed, err := pst.anyEngineBinding(port, state.bindings)
	...
	if managed {
		log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
		if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
			log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
		}
		// ... Delete the (maybe-committed) loopback rule ...
		pst.added[port] = portState{bindings: state.bindings, delegated: true}
		return
	}
```

`retryAppend` runs only for a port already in `appendFailed=true` state, which
means `exposeNew`'s `tracker.Add` succeeded — **procnet owns the host-switch
proxy for `127.0.0.1:<port>`**. When the engine chain probe then returns true,
this branch unexposes that proxy and marks the port `delegated`, betting the
engine owns an equivalent proxy. That bet is not backed by the code.

The engine chain and the engine's host-switch proxy are installed by different
code at different times. The `CNI-HOSTPORT-DNAT` / `DOCKER` chain rule is
installed during container network setup; `portTracker.Add` runs afterward in
the events handler.

For containerd, `MonitorPorts`'s `/tasks/start` handler runs `execIptablesRules`
then `portTracker.Add`, and on an `Add` error it only logs and `continue`s — no
retry (`src/go/guestagent/pkg/containerd/events_linux.go:118-128`).

For docker, `ActionStart` calls `portTracker.Add` and only logs on error
(`src/go/guestagent/pkg/docker/events.go:100-105`); the `DOCKER` chain rule
already exists from dockerd's own container start.

When `portTracker.Add` runs while procnet's synthetic proxy is bound on the
same `127.0.0.1:<port>`, `apiForwarder.Expose` returns `proxy already running`;
`APITracker.Add` appends to `errs`, skips `portStorage.add` (it is guarded by
`len(successfullyForwarded) != 0`), and returns `ErrExposeAPI`
(`src/go/guestagent/pkg/tracker/apitracker.go:109-138`). The engine therefore
has no `portStorage` entry — it does not own the proxy.

In the bad interleaving — engine `Add` runs and fails first, then procnet's
`retryAppend` fires — the port ends with the engine's iptables chain present,
**no host-switch proxy at all**, and procnet marked `delegated` so it never
retries. A published port (the common non-admin case maps both procnet's and
the engine's bindings to `127.0.0.1:<port>`, so the collision is real) is
unreachable from the Windows host until RD restart.

Reachability is narrow: the OCI `createRuntime` reservation socket must survive
the two-scan stability gate (the documented heavy-load limitation), the
`exposeNew` Append must fail transiently, and the engine's `Add` must lose the
proxy race. But every piece is individually plausible, the impact matches Round
1's I1/I2, and the `retryAppend` comment overstates the invariant ("the engine's
now-authoritative chain") — the code never establishes that the engine owns the
proxy.

Reconciliation: the raising agent labeled this a regression; reclassified as a
gap. `port_state_tracker_linux.go` is net-new (+366 −0) and the merge-base
`ForwardPorts` had no append-retry path, so this is an incompletely-closed
corner of new code, not a behavior the PR broke — the same reconciliation Round
1 applied to I2, which is the same branch.

Fix: do not release the synthetic tracker entry on the strength of the engine
chain alone. Two directions, both larger than a Round 1-style three-line patch:

1. Delete only procnet's PREROUTING DNAT (the actual shadowing artifact) and
   keep the synthetic proxy as the owner; do not mark `delegated`. procnet's
   proxy keeps the port reachable, and the engine's later `Add` is a benign
   `proxy already running`. The cost is that procnet's proxy now outlives the
   reservation socket, so cleanup must key off the listener disappearing
   rather than off `delegated`.
2. Add an explicit ownership-transfer probe so the synthetic entry is removed
   only once the engine's `portStorage` entry is known to exist.

At minimum, if the branch stays as-is, the comment must stop claiming the
engine is authoritative and the limitation must be documented in the `portState`
doc comment alongside the other acknowledged partial-failure gaps.

## Suggestions

S1. **`loopbackIP` const exists but the iptables helpers still hardcode the literal** — `src/go/guestagent/pkg/procnet/scanner_linux.go:181,286` [Claude Opus 4.7]

```go
// line 181, in applyLoopbackIPtablesRule:
		"--to-destination", fmt.Sprintf("127.0.0.1:%s", hostPort),
```

This commit introduces `const loopbackIP = "127.0.0.1"` (`scanner_linux.go:53`)
and uses it in `port_state_tracker_linux.go` (`binding.HostIP != loopbackIP`).
The two iptables helpers in the same file, both rewritten in this commit, still

```diff

+		"--to-destination", fmt.Sprintf("%s:%s", loopbackIP, hostPort),
```

and likewise at line 286. The example in the `iptablesCmd` comment at line 171

S2. **`portAlreadyExposedSubstring` doc comment names the wrong function and overstates "nothing to do"** — `src/go/guestagent/pkg/procnet/scanner_linux.go:55-62` [Claude Opus 4.7]

```go
// ... the tracker's wrapping. ForwardPorts treats this case as success: the
// engine has wired up the authoritative CNI rule, leaving procnet
// nothing to do.
```

After this PR `ForwardPorts` is a thin parse-and-`Tick` loop; the
"proxy already running" case is handled in `handleAlreadyExposed`. And "leaving
procnet nothing to do" is true only on the delegate branch — the
resume-ownership branch actively installs the iptables rule the partial-failure
path skipped (`port_state_tracker_linux.go:222-235`). The comment predates the
extraction.

Fix: reword to name `handleAlreadyExposed` and describe both outcomes (delegate
vs. resume-and-install).

S3. **`TestPortAlreadyExposedSubstringIsReachable` comment claims it catches an upstream rename — it cannot** — `src/go/guestagent/pkg/procnet/scanner_linux_test.go:178-204` [Claude Opus 4.7]

```go
// ... If
// either wrap layer ever drops the body, or upstream renames the
// message, this test fails before the procnet scanner silently regresses.
```

The test's own mock handler hardcodes the string (`scanner_linux_test.go:189`).
If gvisor-tap-vsock renames the message — it lives there as a bare
`errors.New("proxy already running")`, not an exported symbol — production
breaks but this test keeps passing against its hardcoded copy. The test does
legitimately pin the `verifyResponseBody` → `APITracker.Add` wrap chain; it just
cannot detect an upstream rename.

Fix: drop the "or upstream renames the message" clause; state the real scope —
the test pins that the wrap chain preserves the response body verbatim, and an
upstream rename is an inherent unguarded risk because the substring is not an
exported upstream symbol.

S4. **iptables error wrapping drops `stderr` context from transient-failure logs** — `src/go/guestagent/pkg/procnet/scanner_linux.go:183,221,293` [Gemini 3.1 Pro]

When iptables fails for a reason other than the `isIptablesRuleAbsent` cases
(xtables lock contention, fork/exec failure), the surfaced error is just
`exit status N`:

- `applyLoopbackIPtablesRule` (line 183) uses `iptablesCmd.Run()` with no
  `Stderr` set, so the child's stderr is discarded entirely.
- `engineChainManagesPort` (line 221) returns `fmt.Errorf("iptables --list %s: %w", chain, err)`;
  `%w` wraps the `*exec.ExitError`, whose `Error()` is `exit status N` — the
  `ExitError.Stderr` that `Output()` captured is never surfaced.
- `preroutingHasLoopbackRule` (line 293) returns `err` directly, same loss.

The caller logs `err.Error()`, so a transient failure shows `exit status 4`
instead of `Another app is currently holding the xtables lock...`, which makes
the deferral-causing failures hard to diagnose. No correctness impact.

Fix: for `Run()`, capture and append `stderr`; for `Output()`, extract
`ExitError.Stderr` into the wrapped error.

Reconciliation: the raising agent rated this Important; downgraded to Suggestion
— it is a diagnosability gap, not a correctness defect, and the `Run()` pattern
in `applyLoopbackIPtablesRule` is carried unchanged from the merge-base's
`execLoopbackIPtablesRule` (the two `Output()` helpers are net-new).

## Design Observations

### Concerns

- *(future)* **Residual netns-blind heuristic race** — `port_state_tracker_linux.go:138-205` [Claude Opus 4.7]. The scanner reads `/proc/net` in the main netns and cannot distinguish a `--network=host` loopback listener (which it should expose) from a bridge container's OCI reservation socket (which it must not). The two-scan gate plus the `exposeNew` engine probe close the common windows, but if the reservation socket survives two scans, `CNI-HOSTPORT-DNAT`/`DOCKER` does not yet reference the port at the second-sighting probe, and procnet's `tracker.Add` wins the race, procnet installs a stable *owned* PREROUTING DNAT — and only `appendFailed` ports ever re-probe the engine chain, so an owned-and-stable port never re-checks once the CNI rule lands. Same root cause as I1 (the heuristic is netns-blind), different failure mode (a shadowing rule that persists rather than a removed proxy). The PR explicitly frames itself as accepting heuristic limits; worth a tracking note and a comment in `exposeNew` acknowledging the "engine rule lands after our second sighting" window.
- *(future)* **Same-port-number collision between a host-net listener and a bridge publish** — `port_state_tracker_linux.go:154-176` [Claude Opus 4.7]. If a `--network=host` container listens on `127.0.0.1:8080` while a bridge container has published `8080`, `engineChainManagesPort` returns true and procnet delegates the host-network listener — which then never gets the PREROUTING DNAT it actually needs. A genuine ambiguity (two things want the same host port), rare, but a behavior change from the pre-PR code, which always added the rule. Worth a comment, not code.
- *(in-scope)* **The new CI target gates on never-run pre-existing tests** — `package.json` [Claude Opus 4.7]. `test:unit:guestagent` runs `go test ./...`, which subjects `apitracker_test.go`, `iptables_test.go`, and `servicewatcher_linux_test.go` to CI for the first time. `iptables/iptables_test.go:TestForwardPorts` synchronizes through unbuffered channels with `time.After(5 * time.Second)` deadlines — a classic CI-flakiness shape under load. Worth confirming these pass reliably (not just once) before relying on the gate, since flakiness here would block unrelated PRs.
- *(in-scope)* **The closed engine-chain list is now load-bearing** — `scanner_linux.go:194-198` [Codex GPT 5.5]. The chain list (`CNI-HOSTPORT-DNAT`, `DOCKER`) is documented as closed, but with this PR a missed future engine chain silently reintroduces the shadow-DNAT bug. The "revisit when adding a third engine" comment is correct; the consequence of forgetting is now more severe.
- *(future, pre-existing)* **`%w` in a `log.Errorf` call** — `scanner_linux.go:119` [Claude Opus 4.7]. `log.Errorf("failed to create portMapping for entry: %w", err)` — `%w` is meaningful only in `fmt.Errorf` and renders as `%!w(...)`. Confirmed present in the merge-base (`scanner_linux.go:84` there), so pre-existing, not a regression; one line from changed code and a one-character fix (`%w` → `%s`). Also flagged in Round 1.

## Review Process Notes

### Skill improvements

- When a code path releases or transfers ownership of a resource because a *correlated signal* appeared (an iptables chain, a file, a lock file, a registry entry), check whether that signal is installed by the *same* code that establishes the ownership the path is transferring to. A signal and the ownership state it implies are often written by different components at different times; treating "signal present" as "the other owner is ready" creates a window where the releasing path hands off to nobody. Flag ownership-transfer branches that key off an observable side effect rather than off the destination owner's own state.
- When reviewing a follow-up round, verify each prior-round fix against the current code before trusting the resolution table, and then look hardest at the code the fixes *touched* — a corner closed in round N often exposes an adjacent corner of the same branch in round N+1. Re-read the modified branch end to end rather than diffing only the fix.

### Repo context updates

- [repo] In the `guestagent` port-forwarding code, an engine's iptables chain (`CNI-HOSTPORT-DNAT`, `DOCKER`, the per-container `CNI-DN-*` chains) is installed during container network setup, independently of and earlier than the engine's `portTracker.Add` call in the events handler (`containerd/events_linux.go` `/tasks/start`, `docker/events.go` `ActionStart`). Both handlers only log on an `Add` failure — neither retries. So "an engine chain references the port" does not imply "the engine owns the host-switch proxy / has a `portStorage` entry." When reviewing code that probes engine chains to decide procnet/tracker ownership, check that it does not conflate the two.

## Design Observations

### Concerns

**Residual netns-blind heuristic race (future).** The scanner reads `/proc/net` in the main netns and cannot distinguish a `--network=host` container's loopback listener (which it *should* expose) from the OCI `createRuntime` reservation socket of a bridge container (which it must not). The two-scan stability gate plus the engine-chain probe in `exposeNew` (lines 161-176) close the common windows, but a residual race remains: if the reservation socket survives two scan intervals (~6 s) *and* `CNI-HOSTPORT-DNAT`/`DOCKER` does not yet reference the port at the second-sighting probe *and* procnet's `tracker.Add` wins the race against the engine's events handler, procnet installs a stable owned PREROUTING DNAT that shadows the CNI rule once it lands — and only `appendFailed` ports ever re-probe the engine chain, so an owned-and-stable port never re-checks. The PR explicitly frames itself as accepting heuristic limits ("rather than a fourth round of heuristic refinement"), and a fully robust fix (querying the engine for which containers are `--network=host` and exposing only those ports) is a larger architectural change. Worth a tracking note rather than a blocker, but the `exposeNew` comment could acknowledge the "engine rule lands after our second sighting" window explicitly.

**Same-port-number collision between a host-net loopback listener and a bridge publish (future).** If a `--network=host` container listens on `127.0.0.1:8080` while a bridge container has published `8080`, `engineChainManagesPort` returns true and procnet *delegates* the host-network listener — so it never gets the PREROUTING DNAT it actually needs. This is a genuine ambiguity (two things want the same host port) and rare, but it is a behavior change from the pre-PR code, which always added the rule. Not worth code now; worth a comment.

**CI target newly gates on never-run pre-existing tests (in-scope sanity check).** `test:unit:guestagent` runs `go test ./...`, which now subjects `apitracker_test.go`, `iptables_test.go`, and `servicewatcher_linux_test.go` to CI for the first time. `iptables/iptables_test.go:TestForwardPorts` synchronizes through unbuffered channels with `time.After(5 * time.Second)` deadlines — a classic CI-flakiness shape under load. The PR author's test plan checkbox covers a clean local run; it would be worth confirming these pass reliably (not just once) before relying on this gate, since flakiness here would now block unrelated PRs.

### Strengths

- Extracting the state machine behind `iptablesRunner` + `tracker.Tracker` is the right call, not premature abstraction: the bug took three review rounds precisely because the logic was untestable inlined, and each interface has a real production impl plus a faithful fake.
