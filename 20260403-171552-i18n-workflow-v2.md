## Executive Summary

This branch adds comprehensive i18n support to Rancher Desktop: a Go-based `i18n-report` CLI tool for translation management, main-process locale switching, translated UI strings across ~55 Vue components, and German/Farsi locale translations. The architecture is well-designed — the Go tool's YAML-node-level merge preserves comments and structure, the `onLocaleChange` callback pattern cleanly decouples locale changes from menu/tray rebuilds, and the `titleKey`/`descriptionKey` approach makes page titles reactive without modifying every component.

Three review passes uncovered 1 critical issue, 17 important issues, and 14 suggestions. The critical bug (`locales.yaml` treated as a selectable locale) and several important regressions (hidden language picker, missing translation keys, broken `LockedFieldError` detection) must be fixed before merge. The Go tool has strong test coverage, but the TypeScript side lacks tests for locale switching, callback lifecycle, and placeholder parity.

## Critical Issues

1. **`locales.yaml` manifest treated as a selectable locale** — `pkg/rancher-desktop/main/i18n.ts:12-13`, `pkg/rancher-desktop/store/i18n.js:9`, `pkg/rancher-desktop/main/__mocks__/i18n.ts:23-25` [Claude Opus 4.6, Codex GPT 5.4] (critical, regression)

```ts
const translationContext = import.meta.webpackContext(
  '@pkg/assets/translations', { recursive: false, regExp: /\.yaml$/ },
);
```

All three i18n implementations scan `assets/translations/` for `*.yaml` files and derive locale codes from filenames. The webpack context regex `/\.yaml$/` (line 12-13 of `i18n.ts`, line 9 of `i18n.js`) matches the newly added `locales.yaml` manifest, producing `"locales"` as an entry in `availableLocales`. Consequences: the language picker renders "locales" as a selectable option; `settingsValidator.ts:102` accepts it via `checkEnum('none', ...availableLocales)`; selecting it loads the manifest's structure as translation data, breaking every `t()` call.

Fix: Filter the manifest from `availableLocales`, or change the regex to exclude it:

```diff
- { recursive: false, regExp: /\.yaml$/ },
+ { recursive: false, regExp: /^\.\/[a-z][\w-]+\.yaml$/ },
```

Apply the same fix in `store/i18n.js:9` and `__mocks__/i18n.ts:23-25`.

4. **Localized validator messages break `LockedFieldError` detection** — `pkg/rancher-desktop/main/commandServer/settingsValidator.ts:287`, `pkg/rancher-desktop/config/commandLineOptions.ts:142`, `pkg/rancher-desktop/backend/backendHelper.ts:171` [Codex GPT 5.4] (pass 2) (important, regression)

```ts
// commandLineOptions.ts:142

  throw new LockedFieldError(errorString);

```

The validator now emits translated error strings, but `commandLineOptions.ts:142` and `backendHelper.ts:171` detect locked-field violations by regex-matching the English text `field "..." is locked`. When the locale is not English, the regex fails, so locked-field violations degrade into generic errors. 8. **HTML in translation strings creates fragile coupling** — `pkg/rancher-desktop/assets/translations/en-us.yaml`, `pkg/rancher-desktop/pages/KubernetesError.vue`, `pkg/rancher-desktop/components/Preferences/ContainerEngineGeneral.vue` [Claude Opus 4.6] (important, regression)

9. **Spinkube warning sentence fragments freeze word order** — `pkg/rancher-desktop/components/Preferences/BodyKubernetes.vue:193-198` [Claude Opus 4.6, Codex GPT 5.4] (pass 2) (important, regression)

10. **`onLocaleChange` callbacks fire without error isolation** — `pkg/rancher-desktop/main/i18n.ts:112-114` [Claude Opus 4.6] (important, gap)


If any registered callback throws, subsequent callbacks are skipped. Currently `mainmenu.ts` and `tray.ts` register callbacks. If the menu rebuild throws, the tray never updates.

11. **Concurrent locale changes can interleave** — `pkg/rancher-desktop/main/i18n.ts:105-116` [Claude Opus 4.6, Gemini 3.1 Pro] (pass 2) (important, regression)

```ts

  // ...

    await loadLocale(locale);
    currentLocale = locale;
```

If two `settings-update` events arrive quickly, both pass the `locale !== currentLocale` guard before either completes. The final `currentLocale` is set by whichever `loadLocale` resolves last, which may not match the most recent user intent.

Fix: Track a `pendingLocale` and check it after `await loadLocale()` to discard superseded changes.

12. **JSONL parsing fails on leading blank lines** — `src/go/i18n-report/report_merge.go:183-190` [Gemini 3.1 Pro] (important, regression)

```go

    firstLine = strings.TrimSpace(content[:idx])

if len(firstLine) > 0 && firstLine[0] == '{' {
```

The JSONL detection takes the substring before the first `\n` and trims it. If the content starts with a blank line (e.g., `\n{"message"...}`), `content[:0]` is empty, so JSONL detection fails and the entire output is skipped.

Fix: Iterate lines to find the first non-empty line instead of assuming it precedes the first `\n`.

## Suggestions

1. **`onLocaleChange` callbacks leak on tray hide/show** — `pkg/rancher-desktop/main/tray.ts:203,257-270` [Claude Opus 4.6] (pass 2) (suggestion, regression)

The `onLocaleChange` API supports only `push()` — no removal. `Tray.hide()` removes all `mainEvents` listeners but cannot unregister the locale callback. Each hide/show cycle leaks one callback. The leaked callbacks call `updateMenu()` on a destroyed tray (guarded by `isDestroyed()`), so there is no crash, but callbacks accumulate.

Fix: Add `offLocaleChange` to `main/i18n.ts` and call it from `Tray.hide()`.

2. **`reportValidateQuiet` diverges from `reportValidate`** — `src/go/i18n-report/report_check.go:220-265`, `src/go/i18n-report/report_validate.go:106-126` [Claude Opus 4.6] (pass 2) (suggestion, regression)

`reportCheckPolicy` calls `reportValidateQuiet` for the "validate passes" check, but `reportValidateQuiet` omits the metadata coherence checks that `reportValidate` includes. The policy check has its own metadata check, so the practical impact is limited, but the two paths will diverge as new checks are added.

Fix: Extract shared validation logic into a function both callers use.

3. **`loadLocale` silently swallows errors** — `pkg/rancher-desktop/main/i18n.ts:45-54` [Claude Opus 4.6] (pass 2) (suggestion, regression)

```ts
try {
  translations[locale] = await translationContext(`./${ locale }.yaml`);
} catch {
  console.error(`i18n: failed to load locale "${ locale }"`);
}
```

If loading fails, the catch logs an error but returns successfully. The caller then sets `currentLocale` to a locale that never loaded. Future `settings-update` events with the same locale skip loading because `locale !== currentLocale` is false.

Fix: Return a boolean or throw so the caller avoids setting `currentLocale` on failure.

4. **Eager loading of all YAML files at startup** — `pkg/rancher-desktop/store/i18n.js:149-154` [Claude Opus 4.6] (suggestion, regression)

The `init` action loads every translation file so `availableLocales` can display native locale names. With four locales this adds ~170 KB of YAML parsing. The tradeoff is acceptable for a small locale count — document the intent with a code comment.

5. **`command-api.yaml` locale enum is hardcoded** — `pkg/rancher-desktop/assets/specs/command-api.yaml:598-601` [Claude Opus 4.6] (suggestion, gap)

The API spec hardcodes `[none, de, en-us, fa, zh-hans]`. Adding a locale requires updating this file alongside the manifest, validator spec, and locale files. The `crossValidateManifest` tool catches mismatches but only when run manually. Add it to CI.

6. **Duplicated `t()` implementation across mock, module, and test** — `pkg/rancher-desktop/main/__mocks__/i18n.ts:27-53`, `pkg/rancher-desktop/main/i18n.ts:29-75`, `pkg/rancher-desktop/components/__tests__/UpdateStatus.spec.ts:7-53` [Claude Opus 4.6] (suggestion, regression)

The `getByPath` and interpolation logic is reimplemented in three places. Extract the shared logic into a test utility.

7. **No settings migration entry for version 19** — `pkg/rancher-desktop/config/settings.ts:9` [Claude Opus 4.6] (suggestion, gap)

`CURRENT_SETTINGS_VERSION` was bumped from 18 to 19, but no `18:` entry was added to `updateTable`. This works because `_.defaultsDeep` fills the new `locale: 'none'` default, but a no-op entry would document the version bump reason.

8. **README documents the buggy discovery model** — `pkg/rancher-desktop/assets/translations/README.md:24-25` [Codex GPT 5.4] (suggestion, regression)

The README states "Webpack auto-discovers YAML files in this directory. Adding a new file here makes it available as a locale without code changes." This is no longer true now that the directory contains non-locale YAML. Update to reflect manifest-driven registration.

9. **Non-atomic writes to locale and metadata files** — `src/go/i18n-report/report_merge.go:142`, `src/go/i18n-report/metadata.go:76` [Gemini 3.1 Pro] (suggestion, gap)

`os.WriteFile` truncates before writing. An interrupted process could corrupt the file. Consider write-to-temp-then-rename for atomic updates.

10. **`nodeRemoveLeaf` is dead code** — `src/go/i18n-report/yaml.go:331-351` [Gemini 3.1 Pro] (suggestion, enhancement)

The function is unused in the application. The `remove` command uses `removeKeyFromNode` instead. Consider removing it.

11. **Preferences window title not refreshed on locale change** — `pkg/rancher-desktop/window/preferences.ts:21` [Codex GPT 5.4] (pass 2) (suggestion, regression)

The native window title is set at creation with `t('preferencesWindow.title')` but not refreshed when the locale changes in-place. The title stays in the previous language until the window is closed and reopened.

Fix: Register an `onLocaleChange` handler that calls `getWindow('preferences')?.setTitle(t('preferencesWindow.title'))`.

12. **`yamlScalar` uses `TrimRight` instead of `TrimSuffix`** — `src/go/i18n-report/yaml.go:182` [Gemini 3.1 Pro] (pass 2) (suggestion, regression)

```go
return strings.TrimRight(string(data), "\n")
```

`yaml.Marshal` appends a single document-ending newline. `TrimRight` strips all trailing newlines, so a value that intentionally ends with `\n\n` would lose trailing newlines. Use `TrimSuffix` to remove only the document-ending newline.

13. **`scanSourceFiles` skips `__tests__` directories** — `src/go/i18n-report/scan.go:121` [Claude Opus 4.6] (pass 2) (suggestion, regression)

The `__tests__` skip prevents detection of test-only translation key usage. This is intentional (test keys should not keep unused keys alive), but the `references` report also omits test usages, which can confuse developers investigating "unused" keys.

14. **ICU nested placeholder extraction is depth-limited** — `src/go/i18n-report/report_validate.go:132-151` [Gemini 3.1 Pro] (pass 3) (suggestion, gap)

`extractPlaceholderNames` only detects placeholders at `depth == 0`. Placeholders nested inside ICU plural or select branches (e.g., `{count, plural, other {Folder is {folder}}}`) are ignored, so the validator cannot detect missing placeholders in nested constructs. The current translation strings do not use nested ICU, so this is not yet a practical issue.

## Design Observations

### Concerns

1. **Three separate locale registries** — The branch has three independent locale discovery mechanisms: directory scanning in `main/i18n.ts`, directory scanning in `store/i18n.js`, and the explicit manifest in `locales.yaml`. Consolidating all executable code onto the manifest would eliminate the "workflow file becomes locale" bug class and make CLI/UI validation naturally consistent. **(in-scope)** [Claude Opus 4.6, Codex GPT 5.4]

2. **Divergent interpolation engines** — `main/i18n.ts` uses simple `{variable}` replacement while `store/i18n.js` uses ICU MessageFormat via `intl-messageformat`. The same translation string can behave differently in the main process vs. renderer. An ICU plural form works in the renderer but renders literally in the main process. Document which strings can use ICU features and which cannot. **(in-scope)** [Claude Opus 4.6]

3. **Localized error strings as API contracts** — Localizing validator messages (finding Important-4) creates a subtle API contract change. External tooling and internal code paths that parse error text will break for non-English locales. Consider keeping API-facing error messages in English regardless of locale, or switching to structured error codes. **(in-scope)** [Claude Opus 4.6, Codex GPT 5.4]

### Strengths

- The `onLocaleChange` callback pattern in `main/i18n.ts` decouples locale changes from menu/tray rebuilds without introducing a full event emitter. [Claude Opus 4.6]
- The `page` store's `titleKey`/`descriptionKey` approach makes page titles reactive to locale changes without modifying every page component's lifecycle. [Claude Opus 4.6]
- The YAML-node merge path in `yaml.go` preserves comments, structure, and `@override` annotations across updates — a significant quality improvement over unmarshal/remarshal cycles. [Codex GPT 5.4, Gemini 3.1 Pro]
- The `@override` annotation system solves the real problem of intentional translation divergence in a maintainable way. [Claude Opus 4.6]
- The main-process `t()` initializes with English built-in, so all calls return sensible output even before `initMainI18n()` runs. [Claude Opus 4.6]
- Source-hash drift detection provides an excellent workflow improvement for isolating translations that need attention because the English source text changed. [Gemini 3.1 Pro]

## Testing Assessment

1. **No test for locale switching in the renderer store** — The `store/i18n.js` changes (cache invalidation, eager loading, IPC sync) have no unit tests. The cache invalidation at lines 215-220 is particularly worth testing.
2. **No test for placeholder parity** — The `invalidTabName` mismatch (Important-7) and missing keys (Important-5, Important-6) would have been caught by a test that verifies TypeScript `t()` arguments match the English YAML placeholders.
3. **No test for `onLocaleChange` callback lifecycle** — The tray and mainmenu locale refresh paths are untested. A test that fires locale changes and verifies callbacks would catch the leak (Suggestion-1) and error isolation gap (Important-10).
4. **No test for `LockedFieldError` with translated messages** — The regex-based detection (Important-4) would have been caught by running the validators with non-English locale.
5. **Go tool tests are thorough** — The `i18n-report` tool has comprehensive test coverage for merge, validate, translate, scan, remove, metadata, drift, and manifest operations.

## Findings

### Critical Issues

1. **`locales.yaml` manifest is treated as a locale** — `pkg/rancher-desktop/assets/translations/locales.yaml`, `pkg/rancher-desktop/main/i18n.ts:12-13`, `pkg/rancher-desktop/store/i18n.js:9`, `pkg/rancher-desktop/main/__mocks__/i18n.ts:23-25` (critical, regression)

```js
// main/i18n.ts:12-13
const translationContext = import.meta.webpackContext(
  '@pkg/assets/translations', { recursive: false, regExp: /\.yaml$/ },
);
```

The `locales.yaml` manifest file (added in commit `6da368ed2`) is placed in the same directory as locale files. All three i18n implementations scan this directory for `.yaml` files: the main-process module at line 12-13, the renderer store at line 9, and the Jest mock at lines 23-25. Each extracts a locale code from the filename, producing `"locales"` as an entry in `availableLocales`.

Consequences: (1) the language picker shows "locales" as a selectable language; (2) the settings validator at `settingsValidator.ts:102` accepts `"locales"` as a valid `application.locale` value via `this.checkEnum('none', ...availableLocales)`; (3) selecting it would load the manifest's structure (`locales.en-us.status: source`) as translation data, producing broken UI text.

Fix: Either move `locales.yaml` outside the translations directory (e.g., into a `meta/` subdirectory alongside the metadata files), or make the webpack context regex exclude it:

```diff
- { recursive: false, regExp: /\.yaml$/ },
+ { recursive: false, regExp: /^\.\/[a-z][\w-]+\.yaml$/ },
```

And the mock similarly:
```diff
- .filter(f => f.endsWith('.yaml'))
+ .filter(f => f.endsWith('.yaml') && f !== 'locales.yaml')
```

### Important Issues

1. **HTML in translation strings creates fragile coupling** — `pkg/rancher-desktop/assets/translations/en-us.yaml:777`, `pkg/rancher-desktop/pages/KubernetesError.vue`, `pkg/rancher-desktop/components/Preferences/ContainerEngineGeneral.vue` (important, regression)

```yaml
# en-us.yaml:777
logLines: 'Some recent <a href="#" data-action="showLogs">logfile</a> lines:'
```

Several translation strings embed HTML with `data-action` or `data-navigate` attributes that application code relies on. In `KubernetesError.vue`, the `onLogLinesClick` handler (lines 118-121) checks `event.target?.getAttribute('data-action') === 'showLogs'`. In `ContainerEngineGeneral.vue`, `onWarningClick` (lines 41-46) reads `data-navigate` from the clicked element.

Translators must preserve exact HTML attribute names and values or the UI breaks silently. If a translator omits the `data-action="showLogs"` attribute, clicking the link does nothing with no error. This is an XSS-adjacent pattern — `v-html` renders arbitrary HTML from translation files.

Fix: Use Vue component slots or structured i18n (e.g., `{beforeLink}logfile{afterLink}`) to separate the translatable text from the application's event-handling markup. At minimum, add prominent comments in en-us.yaml warning translators that these attributes are required and validated.

2. **`onLocaleChange` callbacks fire without error isolation** — `pkg/rancher-desktop/main/i18n.ts:112-114` (important, gap)

```ts
for (const callback of localeChangeCallbacks) {
  callback();
}
```

If any registered `onLocaleChange` callback throws, subsequent callbacks are not invoked. Currently `mainmenu.ts` (line 24) and `tray.ts` (line 203) register callbacks. If the menu rebuild throws (e.g., a missing translation key for a new menu item), the tray never updates.

Fix: Wrap each callback in a try/catch:
```diff
 for (const callback of localeChangeCallbacks) {
-  callback();
+  try {
+    callback();
+  } catch (err) {
+    console.error('Locale change callback failed:', err);
+  }
 }
```

3. **`i18n.js` store loads all YAML files eagerly at startup** — `pkg/rancher-desktop/store/i18n.js:149-154` (important, regression)

```js
async init({ state, commit, dispatch }) {
  // Load all translation files so availableLocales can show native names.
  await Promise.all(
    state.available
      .filter(locale => !state.translations[locale])
      .map(locale => dispatch('load', locale)),
  );
```

The `init` action now loads every translation file upfront so that `availableLocales` can display native locale names. With four locales this adds ~170KB of YAML parsing at startup. As more locales are added, this cost grows linearly. The pre-existing code only loaded the selected locale.

Fix: Load only `locale.{code}` keys from each file (or extract native names into `locales.yaml`), or accept the tradeoff and add a code comment documenting the intentional eager loading.

4. **`command-api.yaml` locale enum is hardcoded** — `pkg/rancher-desktop/assets/specs/command-api.yaml:598-601` (important, gap)

```yaml
locale:
  type: string
  enum: [none, de, en-us, fa, zh-hans]
  x-rd-usage: set the UI language
```

The API spec hardcodes the locale list. Adding a new locale requires updating this file in addition to the manifest, validator spec, and locale files. The `crossValidateManifest` tool at `report_manifest.go:47` catches mismatches but only when explicitly run. A CI check should enforce this.

Fix: Add an `i18n-report manifest --cross-validate` step to CI, or generate the enum from the manifest at `yarn postinstall` time.

## Design Observations

### Concerns

**(in-scope)** The split between `main/i18n.ts` (simple `{variable}` interpolation) and `store/i18n.js` (full ICU MessageFormat via `intl-messageformat`) means the same translation string can behave differently in the main process vs. renderer. For example, an ICU plural form like `{count, plural, one {# image} other {# images}}` works in the renderer but would be rendered literally in the main process. This divergence could cause subtle bugs as more strings are translated. Consider either using the same interpolation engine everywhere, or documenting clearly which strings can use ICU features and which cannot.

### Strengths

- The `onLocaleChange` callback pattern in `main/i18n.ts` elegantly decouples locale changes from menu/tray rebuilds without introducing a full event emitter.
- The `page` store's `titleKey`/`descriptionKey` approach is a clean way to make page titles reactive to locale changes without modifying every page component's lifecycle.
- The Go-based `i18n-report` tool is well-designed with comprehensive subcommands, batch support for parallel translation, and the metadata/drift detection system provides a solid foundation for maintaining translation quality.
- The `@override` annotation system for translations that intentionally differ from the English source is a thoughtful solution to a real problem in translation maintenance.

#### Critical Issues

1. **JSONL parsing fails on leading blank lines** — `src/go/i18n-report/report_merge.go:183-187` (critical, regression)

```go

	firstLine := content
	if idx := strings.IndexByte(content, '\n'); idx >= 0 {
		firstLine = strings.TrimSpace(content[:idx])
	}
	if len(firstLine) > 0 && firstLine[0] == '{' {
```

The `extractTranslationText` function claims to detect JSONL by checking if the first non-empty line starts with `{`. However, `strings.IndexByte(content, '\n')` strictly captures the string *before* the first newline. If the LLM agent output begins with a blank line (e.g., `\n{"message"...`), `content[:idx]` is whitespace, `firstLine` becomes an empty string, and `len(firstLine) > 0` evaluates to false. This causes the merge tool to completely skip parsing valid JSONL output.

Fix: Iterate over the lines to find the actual first non-empty line instead of assuming it's the very first string before `\n`.
```diff
-	firstLine := content
-	if idx := strings.IndexByte(content, '\n'); idx >= 0 {
-		firstLine = strings.TrimSpace(content[:idx])
-	}
+	firstLine := ""
+	for _, line := range strings.Split(content, "\n") {
+		trimmed := strings.TrimSpace(line)
+		if trimmed != "" {
+			firstLine = trimmed
+			break
+		}
+	}
```

## Executive Summary

This change introduces a comprehensive i18n infrastructure for Rancher Desktop, including a main-process translation module, reactive locale switching in Vue components, a Go-based translation management CLI, and German/Farsi locale translations. The key new findings focus on cross-component race conditions in locale switching, callback leak in the tray lifecycle, divergent validation logic between check and validate commands, and inconsistent handling of the spinkube warning's translated sentence fragments.

## Findings

### Critical Issues

None.

### Important Issues

1. **Concurrent locale changes can interleave, producing inconsistent UI state** — `main/i18n.ts:105-116` (important, regression)

```ts
mainEvents.on('settings-update', async(settings) => {
    const raw = settings?.application?.locale;
    const locale = (!raw || raw === 'none') ? 'en-us' : raw;

    if (locale !== currentLocale) {
      await loadLocale(locale);
      currentLocale = locale;
      for (const callback of localeChangeCallbacks) {
        callback();
      }
    }
  });
```

If two `settings-update` events arrive in quick succession (user toggles locale rapidly, or rdctl scripts apply two changes), both pass the `locale !== currentLocale` guard at line 109 before either completes. Both `await loadLocale()` at line 110 run concurrently. The final `currentLocale` at line 111 is set by whichever resolves last, which may not be the most recent user intent. Additionally, callbacks at line 112-114 fire for both transitions, so the tray and application menu get rebuilt twice with potentially stale intermediate state.

Fix: Guard with a sequence counter or compare locale against `currentLocale` again after the await:

```diff
+ let pendingLocale: string | null = null;
  mainEvents.on('settings-update', async(settings) => {
    const raw = settings?.application?.locale;
    const locale = (!raw || raw === 'none') ? 'en-us' : raw;
+   pendingLocale = locale;
    if (locale !== currentLocale) {
      await loadLocale(locale);
+     if (pendingLocale !== locale) {
+       return; // superseded by a newer change
+     }
      currentLocale = locale;
```

2. **onLocaleChange callbacks leak on tray hide/show cycle** — `main/tray.ts:203`, `main/tray.ts:257-270` (important, regression)

```ts
// Constructor (line 203):
onLocaleChange(() => this.updateMenu());

// hide() (line 257-270):
public hide() {
  this.trayMenu.destroy();
  mainEvents.off('k8s-check-state', this.k8sStateChangedEvent);
  mainEvents.off('settings-update', this.settingsUpdateEvent);
  // ... no onLocaleChange cleanup
}
```

The `onLocaleChange` API at `main/i18n.ts:82-84` only supports `push()` — there is no `removeCallback` or `offLocaleChange`. When `hide()` is called, it carefully removes all `mainEvents` listeners (lines 259-261), but the `onLocaleChange` callback registered at line 203 stays in the `localeChangeCallbacks` array permanently. When `show()` at line 277-279 creates a new Tray instance, a second callback is registered. Each hide/show cycle leaks one callback. Each leaked callback calls `this.updateMenu()` on a destroyed tray, and while `updateMenu` at line 326 guards against this with `isDestroyed()`, the callbacks still accumulate.

Similarly, `mainmenu.ts:24` registers `onLocaleChange(() => buildApplicationMenu())` at module load time — this is fine since it's module-scoped and called once, but the pattern lacks cleanup ergonomics.

Fix: Add an `offLocaleChange` function to `main/i18n.ts`:

```ts
export function offLocaleChange(callback: () => void): void {
  const idx = localeChangeCallbacks.indexOf(callback);
  if (idx >= 0) {
    localeChangeCallbacks.splice(idx, 1);
  }
}
```

Then call it from `Tray.hide()`.

3. **reportValidateQuiet omits metadata coherence checks that reportValidate includes** — `report_check.go:220-265`, `report_validate.go:106-126` (important, regression)

```go
// reportValidateQuiet runs validate and returns an error summary without
// printing individual errors.
func reportValidateQuiet(root, locale string) error {
    // ... checks placeholders, ICU, tags, override placement
    // MISSING: metadata coherence (lines 106-126 in reportValidate)
```

`reportCheckPolicy` at line 152 calls `reportValidateQuiet` for the "validate passes" check in policy mode. But `reportValidateQuiet` at line 220 doesn't include the metadata coherence checks that `reportValidate` at lines 106-126 performs (checking that every translated key has a metadata entry and vice versa). This means `check --policy=experimental` can report "validate passes: OK" while `validate --locale=X` reports metadata errors.

The policy check at lines 155-181 does have its own separate metadata coherence check, so the practical impact is limited — but the two "validate" code paths will diverge as new checks are added to `reportValidate`.

Fix: Extract the validation logic into a shared function that both callers use, returning the error list. `reportValidate` prints them, `reportValidateQuiet` just counts them.

4. **Translated sentence fragments create untranslatable coupling** — `BodyKubernetes.vue:193-198`, `en-us.yaml:786-787` (important, regression)

```vue
{{ t('kubernetesPrefs.options.spinkubeWarningPrefix') }}<a
  href="#"
  @click.prevent="$root.navigate('Container Engine', 'general')"
>{{ t('kubernetesPrefs.options.spinkubeWarningLink') }}</a>{{ t('kubernetesPrefs.options.spinkubeWarningSuffix') }}
```

```yaml
spinkubeWarningLink: WebAssembly
spinkubeWarningPrefix: 'Spin operator requires '
spinkubeWarningSuffix: ' to be enabled.'
```

The sentence "Spin operator requires WebAssembly to be enabled" is split into three translation keys: a prefix, a link word, and a suffix. This forces every locale to match the same word order (subject-verb-link-suffix), which doesn't work for languages with different syntax. German would need "Spin-Operator erfordert aktiviertes **WebAssembly**" (adjective order differs). Farsi is even more constrained with SOV word order.

In contrast, the same PR correctly uses the v-html + data-action/data-navigate pattern for `KubernetesError.vue:logLines` and `ContainerEngineGeneral.vue:webAssembly.warning`. The spinkube warning should follow the same pattern.

Fix: Use a single key with embedded markup, like the other cases in this PR:

```yaml
spinkubeWarning: 'Spin operator requires <a href="#" data-navigate="Container Engine,general">WebAssembly</a> to be enabled.'
```

### Suggestions

5. **`reportCheckPolicy` duplicates validation logic instead of reusing `reportValidate`** — `report_check.go:107-215` (suggestion, regression)

`reportCheckPolicy` at lines 107-215 reimplements stale-key counting, metadata coherence checking, and missing-key counting — all of which exist in other subcommands (`report_stale.go`, `report_validate.go`, `report_missing.go`). The duplication means bugs fixed in one place won't be fixed in the other. Consider extracting shared "count stale/missing/metadata-incoherent" helper functions that both the individual subcommands and the policy check can use.

6. **`loadLocale` silently swallows load errors, leaving `currentLocale` pointing to an unloaded locale** — `main/i18n.ts:45-54` (suggestion, regression)

```ts
async function loadLocale(locale: string): Promise<void> {
  if (translations[locale]) {
    return;
  }
  try {
    translations[locale] = await translationContext(`./${ locale }.yaml`);
  } catch {
    console.error(`i18n: failed to load locale "${ locale }"`);
  }
}
```

If `translationContext` throws at line 50 (e.g., corrupted webpack chunk), the catch at line 51-53 logs an error but returns successfully. The caller at line 110-111 (`initMainI18n`) then sets `currentLocale = locale` even though `translations[locale]` is undefined. Subsequent `t()` calls will find no translation in `translations[currentLocale]` and fall back to English — which is reasonable — but `currentLocale` remains set to a locale that never loaded, so future `settings-update` events with the same locale will skip loading (guard at line 109: `locale !== currentLocale`).

Fix: Return a boolean or throw from `loadLocale` so the caller can avoid setting `currentLocale` on failure.

7. **UpdateStatus test reimplements t() instead of using the shared mock** — `UpdateStatus.spec.ts:7-53` (suggestion, regression)

```ts
function tFn(key: string, args?: Record<string, string | number>): string {
  let msg = resolveKey(en, key);
  // ... identical implementation to __mocks__/i18n.ts:27-53
```

The test file at lines 7-53 creates its own `resolveKey`/`tFn` functions that duplicate the mock at `main/__mocks__/i18n.ts:27-53`. The mock is already registered in `jest.config.js` for `@pkg/main/i18n` imports, but this test needs `t` as a Vue component mock (not a module import). Consider exporting the helper from the mock or a shared test utility to avoid the duplication.

8. **`scanSourceFiles` skips `__tests__` directories, preventing detection of test-only translation key usage** — `scan.go:121` (suggestion, regression)

```go
if name == "node_modules" || name == ".git" || name == "dist" || name == "vendor" || name == "__tests__" {
    return filepath.SkipDir
}
```

The `__tests__` skip at line 121 means the `unused` report will flag keys that are only referenced in test files as unused. This is probably intentional (test keys shouldn't keep unused keys alive), but it means the `references` report also won't show test usages, which could be confusing when investigating why a key is "unused" that the developer knows is referenced somewhere.

## Design Observations

**Concerns**

- **(in-scope)** The `onLocaleChange` API is append-only with no way to unsubscribe. This is fine for module-scoped singletons (mainmenu.ts) but creates a lifecycle mismatch for components with destroy semantics (Tray). Adding `offLocaleChange` is a two-line change that prevents the leak pattern from spreading.

- **(future)** The spinkube warning's prefix/link/suffix split (Finding 4) was done to avoid v-html, but the same PR uses v-html + data attributes in three other places. The inconsistency suggests this case was missed during the refactoring pass, not a deliberate alternative.

**Strengths**

- The main-process `t()` function initializes with English built-in (import at line 8, default at line 22-23), so all `t()` calls return sensible output even before `initMainI18n()` runs. This is a good defensive design that avoids startup ordering bugs.
- The Vuex i18n store eagerly loads all locale files in `init()` (line 148-154) so that `availableLocales` can display native locale names. This is a correct trade-off for a small number of locales.
- The `nodeSetLeaf` YAML tree manipulation (yaml.go:244-305) correctly preserves existing comments when no new comment is provided, and creates intermediate mapping nodes as needed. This avoids the common pitfall of losing comments during serialization.

An excellent set of changes that brings a robust i18n framework to Rancher Desktop. The Go CLI tool is particularly well-structured for managing translations at scale. ### Critical Issues

1. **Main process i18n fails to unwrap Webpack module default export** — `pkg/rancher-desktop/main/i18n.ts:50`

```typescript
  try {
    translations[locale] = await translationContext(`./${ locale }.yaml`);
  } catch {
```

Webpack's `require.context` returns the ES module object when loading YAML files, meaning the translated data is nested under the `default` property (e.g., `{ default: { tray: { ... } } }`). Because `loadLocale` assigns the raw module to `translations[locale]`, the `getByPath` lookup for translation keys (e.g., `tray.preferences`) will always return `undefined`. As a result, `t()` will always fall back to English for every string in the main process. (Note that `store/i18n.js` correctly handles this via `module.default || module`).

Fix: Unwrap the `default` export when assigning the translation map.
```diff
-    translations[locale] = await translationContext(`./${ locale }.yaml`);
+    const module = await translationContext(`./${ locale }.yaml`);
+    translations[locale] = (module as any).default || module;
```
*(critical, regression)*

### Important Issues

2. **yamlScalar strips all trailing newlines, causing data loss** — `src/go/i18n-report/yaml.go:277`

```go
	data, err := yaml.Marshal(s)
	if err != nil {
		return "'" + strings.ReplaceAll(s, "'", "''") + "'"
	}
	return strings.TrimRight(string(data), "\n")
```

The standard `yaml.Marshal` appends a single trailing newline to close the document. However, `strings.TrimRight(string(data), "\n")` removes *all* consecutive newlines at the end of the string. If a translated multiline string intentionally ends with newlines (e.g., `hello\n\n`), they will be silently permanently stripped out when `i18n-report merge` saves the file.

Fix: Use `strings.TrimSuffix` to remove only the single document-ending newline.
```diff
-	return strings.TrimRight(string(data), "\n")
+	return strings.TrimSuffix(string(data), "\n")
```
*(important, regression)*

3. **Missing newline when concatenating input files** — `src/go/i18n-report/report_merge.go:70`


4. **Race condition in main process locale updates** — `pkg/rancher-desktop/main/i18n.ts:97-101`

```typescript
  mainEvents.on('settings-update', async(settings) => {
    // ...
    if (locale !== currentLocale) {
      await loadLocale(locale);
      currentLocale = locale;
      for (const callback of localeChangeCallbacks) {
        callback();
      }
    }
  });
```

If `settings-update` fires rapidly with different locales, the asynchronous `loadLocale` promises can resolve out of order. A faster network or disk read for an older event might overwrite the state of a slower, more recent event, leaving `currentLocale` and the UI out of sync with the actual settings.

Fix: Track the target locale and ignore the state update if it has been superseded.
```typescript
let targetLocale = currentLocale;

export async function initMainI18n(): Promise<void> {
  // ...
  mainEvents.on('settings-update', async(settings) => {
    const raw = settings?.application?.locale;
    const locale = (!raw || raw === 'none') ? 'en-us' : raw;

    if (locale !== currentLocale && locale !== targetLocale) {
      targetLocale = locale;
      await loadLocale(locale);
      if (targetLocale !== locale) {
        return; // A newer locale change was requested
      }
      currentLocale = locale;
      for (const callback of localeChangeCallbacks) {
        callback();
      }
    }
  });
}
```



2. **macOS Window menu loses automatic window list** — `mainmenu.ts:170-177` (important, regression)

```typescript
// Before (upstream/main):
new MenuItem({ label: '&Window', role: 'windowMenu' }),

// After (this branch):
new MenuItem({
  label:   t('mainMenu.window.label'),
  submenu: [

```

Electron's `role: 'windowMenu'` automatically appends a list of all open application windows at the bottom of the Window menu — a standard macOS UX convention that lets users switch between windows. The replacement explicit submenu at lines 170-177 drops this `role`, losing the automatic window list. Introduced in commit `0482bcf0fa`.
```diff
  new MenuItem({
+     role:    'windowMenu',
      label:   t('mainMenu.window.label'),
      submenu: [
        { role: 'minimize', label: t('mainMenu.window.minimize') },
        { role: 'zoom', label: t('mainMenu.window.zoom') },
        { type: 'separator' },
        { role: 'front', label: t('mainMenu.window.front') },
      ],
  }),
```

Note: when both `role` and `submenu` are specified, Electron may ignore the role. Testing is needed to find the right approach — likely the submenu items with their roles are sufficient and Electron will still append the window list if the menu has the `windowMenu` role at the top level.

## Design Observations

**Concerns**

- **(in-scope)** The localized validator messages (findings #10 from prior pass, #1 here) create a subtle API contract change. The `rdctl` CLI and external tooling may parse error messages to detect specific conditions (e.g., the regex `/field ".*" is locked/` in `commandLineOptions.ts:142`). Localizing these strings means external parsers will break when the locale is not English. Consider keeping API-facing error messages in English regardless of locale, or switching to structured error codes alongside human-readable messages.

**Strengths**

- The `onLocaleChange` callback pattern in `main/i18n.ts` neatly decouples locale-dependent components from the settings system, avoiding direct IPC listeners that would race against locale file loading.
- The `nodeSetLeaf`/`nodeGetLeaf` YAML tree manipulation in `yaml.go` preserves comments and ordering — a significant quality improvement over naive unmarshal/remarshal cycles.
- The dual-path translation approach (ICU MessageFormat in renderer via `intl-messageformat`, simple `{variable}` interpolation in main process) is pragmatic — it avoids pulling a heavy ICU library into the main process for strings that rarely need pluralization.

### Executive Summary

This PR introduces comprehensive i18n support across the main process, CLI (`rdctl`), and UI. The integration and Go tooling are well structured, but there are critical gaps in translation completeness and interpolation handling, as well as important gaps in the validation AST parser. Specifically, the parser fails to inspect placeholders or parity on nested ICU structures, while deployment profile errors and settings validation have missing or incorrect parameters leading to raw templates being rendered to the user.

### Critical Issues

1. **Missing `deploymentProfile.noVersion` translations hide startup errors** — `pkg/rancher-desktop/main/deploymentProfiles.ts:89,95,236,244`

```typescript
      throw new DeploymentProfileError(t('deploymentProfile.noVersion', { path: fullDefaultPath, version: settings.CURRENT_SETTINGS_VERSION }));
// ...
      throw new DeploymentProfileError(t('deploymentProfile.noVersionRegistry', { path: registryPath, version: settings.CURRENT_SETTINGS_VERSION }));
```

The `deploymentProfile.noVersion` and `deploymentProfile.noVersionRegistry` keys are completely missing from `en-us.yaml` (and all other locales). When a user provides an invalid deployment profile, the main process `t()` function falls back to returning the raw key wrapped in percent signs. As a result, the user will see an incomprehensible `%deploymentProfile.noVersion%` error dialog instead of an actionable message that tells them how to fix their deployment profile, making a critical startup failure impossible to diagnose.

Fix: Add the missing keys to `en-us.yaml` and the other locales with appropriate template strings containing `{path}` and `{version}` placeholders.

2. **Missing `{tabName}` interpolation in `validation.invalidTabName`** — `pkg/rancher-desktop/main/commandServer/settingsValidator.ts:849`

```typescript
      if (!navItem?.tabs?.includes(desiredValue[k])) {
        errors.push(t('validation.invalidTabName', { field: fqname, tab: desiredValue[k], page: k }));

        return false;
      }
```

The code passes a variable named `tab` to the `t()` function, but `en-us.yaml` expects `{tabName}` for this specific error string (`'{field}: tab name "{tabName}" is not a valid tab name...'`). Because the interpolation uses simple string replacement in the main process, `{tabName}` is never replaced and remains raw in the error output, losing the actual tab value the user provided.

Fix: Update the argument passed to `t()` in `settingsValidator.ts`:

```diff
-        errors.push(t('validation.invalidTabName', { field: fqname, tab: desiredValue[k], page: k }));
+        errors.push(t('validation.invalidTabName', { field: fqname, tabName: desiredValue[k], page: k }));
```

### Important Issues

3. **Missing validation for placeholders inside ICU constructs** — `src/go/i18n-report/report_validate.go:132-151`

```go
		case '{':
			if depth == 0 {
				// Extract the identifier starting after '{'.
				rest := s[i+1:]
				if m := identRe.FindStringSubmatch(rest); m != nil {
					result[m[1]] = true
				}
			}
			depth++
```

The `extractPlaceholderNames` function only detects placeholders at `depth == 0` (top-level). If a placeholder is nested inside an ICU plural or select branch (e.g., `{count, plural, other {Folder is {folder}}}`), `{folder}` is located at `depth > 0` and is completely ignored. This means the validator cannot detect missing or extra placeholders inside ICU constructs across locales, silently allowing broken interpolations to reach production.

Fix: The placeholder extractor must track keyword usage and extract placeholders accurately at any depth inside branches, skipping only ICU variables and branch labels.

4. **Missing validation for nested ICU structure parity** — `src/go/i18n-report/report_validate.go:210-229`

```go
		case '{':
			if depth == 0 {
				// Try to parse "{var, keyword, branches...}".
				if c, ok := parseICUConstruct(s[i:]); ok {
					results = append(results, c)
				}
			}
			depth++
```

The `extractICUConstructs` function shares the same flawed depth-tracking heuristic. `parseICUConstruct` consumes the text and returns an `icuConstruct`, but the outer loop's `i` index does not advance over the full construct. Instead, `depth` increments, and subsequent characters—including any nested `{plural}` or `{select}` constructs inside the branches—are skipped because `depth > 0`. The validator therefore fails to verify the structural parity of any nested ICU constructs across locales.

Fix: `extractICUConstructs` must recursively parse nested branches or properly advance the index past the closing brace of the parsed AST node to analyze inner branches.

5. **Dead `override` field and ignored `@override` comments in merge tool** — `src/go/i18n-report/report_merge.go:204`

```go
		// Accumulate @reason comments.
		if strings.HasPrefix(trimmed, "# @reason") {
			// ...
		}
		// ...
		// Skip other comment lines.
		if strings.HasPrefix(trimmed, "#") {
			continue
		}
```

The `mergeEntry` struct defines an `override bool` field intended to track if the new translation contains an `@override` comment. However, `parseMergeInput` never populates it, and it ignores all comments that aren't prefixed with `# @reason` (including `# @override`). Consequently, if a translator or agent tool attempts to add an `@override` key via the `i18n-report merge` command, the tool drops the comment entirely. 

Fix: Update `parseMergeInput` to capture `# @override` comments, populate the `override` field, and use that field when determining the node's `HeadComment`.
