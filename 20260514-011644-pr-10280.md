## Suggestions

S1. **A wsl-proxy-only `tracker.Remove` failure orphans a procnet PREROUTING DNAT that can shadow a later bridge publish** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:313-321` [Codex GPT 5.5]

```go
if removeErr != nil {
	// A failed Remove can leave the host-switch proxy bound
	// (the unexpose call did not land). Keep the loopback
	// rule so traffic still reaches the proxy; deleting it
	// would strand the port. A leaked rule self-cleans on
	// RD restart.
	log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
	continue
}
```

`retireDisappeared` keeps the loopback PREROUTING DNAT whenever `tracker.Remove`
returns any error, on the premise stated in the comment: the unexpose call may
not have landed, so the proxy may still be bound. But `APITracker.Remove` runs
`apiForwarder.Unexpose` first and `wslProxyForwarder.Send` after it, and a
`Send` failure returns early without wrapping the unexpose result.

In `src/go/guestagent/pkg/tracker/apitracker.go:167-201`, the `Unexpose` loop
collects failures into `errs`; then `wslProxyForwarder.Send` runs, and on
failure `Remove` returns `"sending port mappings to wsl proxy error: %w"` —
which does **not** wrap `forwarder.ErrUnexposeAPI`. So the wsl-proxy-only shape
(every `Unexpose` succeeded, `Send` failed) yields a non-nil `removeErr` while
the host-switch proxy is already gone. `retireDisappeared` keeps the rule
anyway. The rule now points at a dead proxy.

If a bridge-network container later publishes that same host port, procnet's
scanner can catch the OCI reservation socket, `anyEngineBinding` finds the
engine's `CNI-HOSTPORT-DNAT`/`DOCKER` rule, and `exposeNew` takes the delegate
branch (`port_state_tracker_linux.go:184-189`) — which touches no iptables. The
stale procnet PREROUTING DNAT survives and shadows the engine chain: the exact
bug this PR exists to fix.

The reachability is narrow: it needs a `--network=host` loopback listener whose
`Remove` fails in the wsl-proxy-only shape, then the same host port reused by a
bridge container, then that container's reservation socket surviving the
two-scan gate under heavy load. The host-network reuse case is already covered —
`applyLoopbackIPtablesRule`'s idempotency probe skips a duplicate Append — so
only the bridge-reuse-via-delegation path is exposed.

Reconciliation: the raising agent rated this Important; downgraded to Suggestion.
Round 1's I1 fix rationale already named "retaining the rule after a
wsl-proxy-only `Remove` failure" as the known downside of the keep-the-rule
behavior, and named the fix Codex proposes here — a typed `Remove` result
distinguishing "host-switch unexpose succeeded" from "may still be running" — as
"the larger fix." Keep-the-rule was the deliberate Round 1 decision; the code
change is a `tracker`-package API change deferred by design. What is genuinely
new and in-scope is the inline comment.

Fix: the immediately-actionable item is the comment at lines 313-318 — `"the
unexpose call did not land"` is inaccurate for the wsl-proxy-only shape, where
unexpose *did* land and the proxy *is* gone. Reword it to cover both shapes and
to note that the retained rule can shadow a later same-port bridge publish (the
consequence the `portState` doc comment and `retireDisappeared` comment both
currently understate). The typed-`Remove`-result code fix remains the deferred
larger change.

S2. **`isIptablesRuleAbsent` substring-matches English iptables diagnostics** — `src/go/guestagent/pkg/procnet/scanner_linux.go:317-328` [Claude Opus 4.7]

```go
func isIptablesRuleAbsent(err error) bool {
	...
	stderr := string(exitErr.Stderr)
	return strings.Contains(stderr, "No chain/target/match by that name") ||
		strings.Contains(stderr, "does a matching rule exist in that chain")
}
```

`isIptablesRuleAbsent` classifies an iptables failure as "rule/chain absent" by
substring-matching English diagnostic text. This stderr inspection is new in
this PR — the merge-base `execLoopbackIPtablesRule` did `iptablesCmd.Run()` then
`return err` with no stderr parsing, and `engineChainManagesPort` /
`dnatChainContainsPort` / `isIptablesRuleAbsent` did not exist.

If the guest's iptables ever emits localized messages, a genuinely-absent chain
is misclassified as a transient error: `engineChainManagesPort` returns
`(false, err)` (`scanner_linux.go:228`), `anyEngineBinding` propagates it
(`port_state_tracker_linux.go:350`), and `exposeNew` defers the port every tick
(`port_state_tracker_linux.go:175-183`) — a `--network=host` loopback port
becomes permanently unreachable from the Windows host, the symptom procnet
exists to prevent. Practical likelihood is near zero (RD's Alpine guest runs in
the C locale, and most iptables builds ship no message catalogs), so this is a
latent fragility, not a live bug — but pinning the locale is cheap insurance and
standard practice when parsing tool output.

Fix: pin `LC_ALL=C` on every iptables invocation. A small helper removes the
duplication across the three call sites (`engineChainManagesPort`,
`preroutingHasLoopbackRule`, `applyLoopbackIPtablesRule`):

```go
// iptablesCommand builds an iptables exec.Cmd with the locale pinned to C,
// so isIptablesRuleAbsent's stderr matching is not defeated by a localized
// iptables build.
func iptablesCommand(ctx context.Context, args ...string) *exec.Cmd {
	cmd := exec.CommandContext(ctx, "iptables", args...)
	cmd.Env = append(os.Environ(), "LC_ALL=C")
	return cmd
}
```

(`os` is already imported in `scanner_linux.go`.)

S3. **`addErrorLogged` is shared between the engine-chain probe and `tracker.Add`, so the first `Add` error after a probe failure logs at `Debug`** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:174-204` [Gemini 3.1 Pro]

```go

if err != nil {

		log.Errorf(...); pst.addErrorLogged[port] = true   // line 180
	}

...
if err := pst.tracker.Add(syntheticID, nat.PortMap{port: bindings}); err != nil {
	...
```

A transient engine-chain probe failure sets `addErrorLogged[port]` at line 180.
If the probe succeeds on the next tick but `tracker.Add` then fails, the check

## Design Observations

### Concerns

- *(future)* **`dnatChainContainsPort` parses human-readable `iptables --list` output** — `scanner_linux.go:255-272` [Claude Opus 4.7]. The regex matches the columnar `iptables -L --numeric` rendering, which is presentation output, not a stable interface — column spacing, the `multiport dports` rendering, and the protocol display can drift between iptables versions and the legacy/nft backends. A regression there produces a *false negative*: procnet decides the engine does not manage the port, adds its own DNAT, and silently reinstates the original shadow-DNAT bug. The `engineChainManagesPort` docstring acknowledges only `published-ports.bats` covers this end to end, and that is a single happy-path fixture. `iptables -S <chain>` emits the rule-spec form (`-A CHAIN -p tcp -m tcp --dport 8080 ...`), which is designed to be machine-parsed; worth recording as the fallback if a future iptables bump breaks the regex. Not worth changing now given the bats safety net.
- *(future)* **`exposeNew` does not re-probe the engine chain after a healthy `tracker.Add`** — `port_state_tracker_linux.go:167-218` [Claude Opus 4.7]. `retryAppend` deliberately re-probes the engine chain before re-appending because a full tick elapsed since the original probe; `exposeNew` probes once, then `Add`, then `Append` with no re-probe. The residual gap is the opposite ordering: a port `exposeNew` recorded as healthy-owned, after which an engine chain appears for the same port — `addObserved` no-ops healthy `added` ports, so the procnet rule persists and shadows. This is the same netns-blind / stability-gate-bypass limitation the `portState` doc comment (lines 71-73) and Round 2's "residual netns-blind heuristic race" already call out, restated from the exposeNew/retryAppend asymmetry angle; it self-heals when the transient listener disappears. No action needed.
- *(future, pre-existing)* **`%w` verb in a `log.Errorf` call** — `scanner_linux.go:120` [orchestrator]. `log.Errorf("failed to create portMapping for entry: %w", err)` — `%w` is meaningful only in `fmt.Errorf` and renders as `%!w(...)`. Present in the merge-base, so pre-existing, not a regression; flagged in Rounds 1 and 2 as well. The line sits in a file this PR rewrites heavily, so the one-character fix (`%w` → `%s`) is low-cost to fold in, but it remains out of scope for the procnet change.

## Review Process Notes

### Skill improvements

- When code classifies a subprocess failure by substring-matching its stderr or
  stdout — deciding "file not found" vs "permission denied", "rule absent" vs
  "transient lock contention", and the like — check whether the process locale
  is pinned to `C`/`POSIX`. Unpinned locale-sensitive matching silently
  misclassifies under a localized build of the tool, and the misclassification
  often routes a steady-state "no" into a "retry forever" path. Flag substring
  matching of tool output where the invocation does not pin `LC_ALL=C` (or
  equivalent).

- When a code path keeps or releases a resource (an iptables rule, a lock file,
  a temp directory) based on whether a multi-step operation returned an error,
  check whether that error value distinguishes which step failed. A single
  opaque `error` frequently conflates "step A failed, later steps skipped" with
  "step A succeeded, step B failed" — and the correct cleanup differs between
  them. Flag keep/release decisions that branch only on `err != nil` when the
  operation has multiple side-effecting steps with distinct post-conditions; the
  fix is usually a typed or sentinel-wrapped error, not a change to the cleanup
  logic.

### Repo context updates

- [repo] `APITracker.Remove` (`src/go/guestagent/pkg/tracker/apitracker.go`)
  runs host-switch `apiForwarder.Unexpose` for every binding first, then
  `wslProxyForwarder.Send`. A `Send` failure returns early with `"sending port
  mappings to wsl proxy error: %w"`, which does NOT wrap
  `forwarder.ErrUnexposeAPI`. So a non-nil error from `tracker.Remove` does not
  imply the host-switch proxy is still bound — it can mean every `Unexpose`
  succeeded and only the wsl-proxy notification failed. `APITracker.Add` has the
  same `Expose`-then-`Send` ordering but seeds `portStorage` before `Send`. Code
  that keeps, deletes, or transfers iptables rules or tracker entries based on an
  `Add`/`Remove` error must account for both partial-failure shapes.

## Design Observations

### Concerns

**Parsing human-readable `iptables --list` output (future).** `dnatChainContainsPort` (`scanner_linux.go:255-272`) regex-matches the columnar `iptables -L --numeric` rendering. That format is presentation output, not a stable interface — spacing, the `multiport dports` rendering, and protocol display can drift between iptables versions and the legacy/nft backends. A regression there produces a *false negative*, which silently reinstates the original shadow-DNAT bug. The `engineChainManagesPort` docstring acknowledges this is covered only by `published-ports.bats`, which is a single happy-path fixture. If output drift becomes a concern, `iptables -S <chain>` emits the rule-spec form (`-A CHAIN -p tcp -m tcp --dport 8080 ...`), which is designed to be machine-parsed and round-tripped. Not worth changing now given the bats safety net, but worth recording as the fallback if a future iptables bump breaks the regex.

**`exposeNew` does not re-probe the engine chain after `tracker.Add` succeeds (future).** `retryAppend` (`scanner_linux.go` → `port_state_tracker_linux.go:259-289`) deliberately re-probes the engine chain before re-appending, because a full tick elapsed since the original probe. `exposeNew` (`port_state_tracker_linux.go:167-218`) probes, then `tracker.Add`, then `Append` with no re-probe — correct, because that gap is microseconds. The residual gap is the *opposite* ordering: a port that `exposeNew` exposed as healthy-owned, after which an engine chain appears for the same port. `addObserved` no-ops healthy `added` ports (lines 153-157), so the shadow rule would persist. In practice this requires the engine to bind a host port a `--network=host` container already holds (a port conflict that cannot occur) or the stability gate to be bypassed under heavy load — the limitation the `portState` doc comment at lines 71-73 already calls out — and it self-heals when the transient listener disappears. No action needed; noting it because the asymmetry between `exposeNew` and `retryAppend` invites the question.

### Strengths

- The `iptablesRunner` interface is a genuine test seam, not premature abstraction: the fake `iptablesRunner` models the installed rule set (`fakeIptables.rules`) so tests assert *what iptables state resulted*, not merely that a call happened. Combined with the `addBehavior` queue on `fakeTracker`, this lets every documented state transition — including the partial-failure and commit-but-errored shapes — be exercised deterministically.
- The `portState` doc comment (`port_state_tracker_linux.go:28-84`) is a complete state-machine specification, and each helper's docstring explains *why* (e.g. the `retireDisappeared` rationale for dropping the marker unconditionally, lines 306-318). This is the kind of documentation that survives the next refactor.

S1. **Log suppression mask hides the first `tracker.Add` error** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:167-173`

```go
		if pst.addErrorLogged[port] {
			log.Debugf("/proc/net scanner still failing to add port %s: %s", port, err)
		} else {
			log.Errorf("/proc/net scanner failed to add port: %s", err)

		}
```

The log-suppression marker (`pst.addErrorLogged[port]`) is shared between the engine-chain probe and the `tracker.Add` call. If an engine-chain probe fails transiently (setting the marker to `true`) and succeeds on the next tick, but `tracker.Add` then fails, the first `tracker.Add` error will be logged at `Debugf` instead of `Errorf`. The noise reduction still works, but you lose the loud notification for the distinct failure mode. 

**Fix:** Use an error-specific state marker (e.g., `addErrorLogged` and `probeErrorLogged`) or clear the marker when advancing from the probe phase to the add phase.
