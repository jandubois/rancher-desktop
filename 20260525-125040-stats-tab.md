## Executive Summary

The branch adds a Stats tab with live CPU/memory/network/I/O charts and a `docker top` process table, plumbed through a main-process polling handler modeled on `ContainerExecHandler`. The structure is sound. Two issues block merge: the polling timer outlives a destroyed renderer, and `updateClient` clears every session without telling the renderer, freezing the charts on the last sample. A third gap — the containerd path routes through `nerdctl` but assumes the Docker JSON field names — ships untested.

All three reviewers flagged the `runWithTimeout` subprocess/interval concern; Claude raised the widest set of additional issues.

## Important Issues

I1. **Polling timeout leaks the child process and lets polls stack** — `pkg/rancher-desktop/main/containerStats.ts:50-66, 103` (important, regression) [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```ts
const timeout = new Promise<null>((resolve) => {
  setTimeout(() => resolve(null), FETCH_TIMEOUT_MS);
});
const run = (async() => { ... })();
return Promise.race([run, timeout]);
```

`Promise.race` settles the wrapper after 8 s, but the child process `runClient` spawned (line 57) keeps running until it exits on its own. `runClient` forwards `SpawnOptions.signal` to `spawnFile`, yet the call site passes none. `setInterval` (line 103) fires every `intervalSeconds` without awaiting the previous `poll()`, so a wedged engine accumulates concurrent `stats` and `top` processes per tick. The won-race `setTimeout` stays armed for the full 8 s, pinning its closure.

Fix: thread an `AbortController` through `runWithTimeout`, abort on timeout, and gate `poll()` behind an `inFlight` boolean (or schedule via recursive `setTimeout`).

Gemini classified this Critical; downgraded to Important since `docker stats --no-stream` returns in well under a second against a healthy engine.

I2. **Stats sessions leak when the renderer is destroyed without sending `stop`** — `pkg/rancher-desktop/main/containerStats.ts:68-112` (important, regression) [Claude Opus 4.7, Gemini 3.1 Pro]

```ts
const session: StatsSession = {
  timer:  setInterval(() => { poll().catch(console.error) }, intervalSeconds * 1_000),
  sender: event.sender,
  namespace,
};
this.sessions.set(containerId, session);
```

Only a renderer-sent `'container-stats/stop'` (line 114) ends the session. A hard reload, a renderer crash, or `webContents.close()` skips Vue's `onBeforeUnmount`, so the `setInterval` runs forever, shelling out to `docker stats` and `docker top` each tick; `sendToFrame`'s try/catch then swallows the send errors. `ContainerExecHandler` avoids this by ending its session on `proc.on('exit', ...)` (`containerExec.ts:153`).

Fix: bind session lifetime to the sender — `event.sender.once('destroyed', () => …)` clears the interval. The `sendToFrame` catch can also delete the session when `event.sender.isDestroyed()` returns true.

I3. **`updateClient` drops every active session without notifying the renderer** — `pkg/rancher-desktop/main/containerStats.ts:34-37` (important, regression) [Claude Opus 4.7]

```ts
updateClient(client: ContainerEngineClient) {
  this.client = client;
  this.stopAll();
}
```

`startK8sManager` calls `updateClient` whenever the container engine changes (`background.ts:605`). `stopAll` clears every session map entry but tells the renderer nothing, so the Stats tab keeps its handlers registered and freezes on its last sample. `ContainerExecHandler.updateClient` (`containerExec.ts:42-51`) handles this correctly: `killAll()` kills each child process, which fires `proc.on('exit')` and emits `'container-exec/exit'` to the renderer.

Fix: emit a terminal IPC event (e.g. `'container-stats/stopped'`) before clearing the map, or hook `WebContents.destroyed` per I2 so the renderer's own remount triggers a fresh `start`.

I4. **Stats buffers and process table survive container stop/start** — `pkg/rancher-desktop/components/ContainerStats.vue:490-496` (important, regression) [Codex GPT 5.5]

```ts
watch(() => props.isContainerRunning, (running) => {
  if (running) {
    restartPolling();
  } else {
    stopPolling();
  }
});
```

When `isContainerRunning` toggles false → true while the Stats tab stays mounted, this watcher restarts polling but leaves `labels`, `cpuData`, `memData`, the `prev*` cumulative counters, and `processes` untouched. The user sees prior-run samples mixed into the new container life, and the first delta after restart is computed against a stale `prevNetRx`/`prevBlockR` from minutes earlier. `resetBuffers` (line 434) fires only on `containerId` change (line 484).

Fix: call `resetBuffers` on every transition before starting/stopping.

```diff
 watch(() => props.isContainerRunning, (running) => {
-  if (running) restartPolling(); else stopPolling();
+  stopPolling();
+  resetBuffers();
+  if (running) startPolling();
 });
```

I5. **Containerd path is untested and likely emits a different JSON shape** — `pkg/rancher-desktop/main/containerStats.ts:85-92`, `pkg/rancher-desktop/components/ContainerStats.vue:386-390` (important, gap) [Claude Opus 4.7]

```ts
const [statsRaw, topRaw] = await Promise.all([
  this.runWithTimeout(['stats', '--no-stream', '--format', '{{json .}}', containerId], namespace),
  this.runWithTimeout(['top', containerId], namespace),
]);
```

`client.runClient` dispatches to `nerdctl` when the engine is `containerd` (`nerdctlClient.ts:36`). The renderer's `handleStatsData` (line 372) reads `s.CPUPerc`, `s.MemUsage`, `s.NetIO`, `s.BlockIO` directly off the docker JSON formatter; nerdctl's `--format '{{json .}}'` output does not necessarily share those field names, and rootless containerd omits network counters entirely. Charts then sit silently at zero under containerd. The e2e tests pin `ContainerEngine.MOBY` (`containers.e2e.spec.ts:536`), so this path stays uncovered.

Fix: gate the Stats tab on `currentContainerEngine === MOBY` until an engine-specific parser lands, or push normalization into `ContainerEngineClient` with `moby`/`nerdctl` implementations. Add a containerd e2e variant before re-enabling.

## Suggestions

S1. **Network and Block I/O charts plot per-tick delta, not a rate** — `pkg/rancher-desktop/components/ContainerStats.vue:393-397` (suggestion, enhancement) [Gemini 3.1 Pro]

```ts
push(netRxData.value, Math.max(0, rx - prevNetRx));
push(netTxData.value, Math.max(0, tx - prevNetTx));
push(blockRData.value, Math.max(0, blockR - prevBlockR));
push(blockWData.value, Math.max(0, blockW - prevBlockW));
```

Chart values scale with the refresh interval — a steady 1 MB/s transfer reads as 1 MB at 1 s refresh and 60 MB at 1 min refresh. The y-axis label says "Network I/O" but the units are bytes-per-tick, not B/s.

Fix: divide each delta by `refreshSeconds.value` (or by a real timestamp diff) before pushing, so the y-axis becomes B/s regardless of refresh choice.

S2. **Charts e2e test passes before any stats data arrives** — `e2e/pages/container-stats-page.ts:32-36`, `e2e/containers.e2e.spec.ts:602-605` (suggestion, gap) [Codex GPT 5.5]

```ts
async waitForCharts(timeout = 20_000) {
  await expect(this.cpuChart).toBeVisible({ timeout });
  ...
}
```

`ContainerStats.vue` mounts every chart card unconditionally once `isContainerRunning` is true, so `toBeVisible` passes the moment the component renders — before any `'container-stats/data'` IPC arrives. A handler that crashes or a parser that swallows every sample still satisfies the assertion.

Fix: assert that the rendered Chart.js dataset length is non-zero (read it via `page.evaluate` against the canvas component), or expose a small test-only "samples-received" counter the test can poll.

S3. **Process-row tokeniser breaks when a header column contains spaces** — `pkg/rancher-desktop/components/ContainerStats.vue:417-426` (suggestion, regression) [Claude Opus 4.7]

```ts
const headers = lines[0].trim().split(/\s{2,}/);
const rows = lines.slice(1).map((line) => {
  const parts = line.trim().split(/\s+/);
  const fixed = parts.slice(0, headers.length - 1);
  fixed.push(parts.slice(headers.length - 1).join(' '));
  return fixed;
});
```

Headers split on 2+ spaces (column-aware); rows split on any whitespace (column-unaware). When a header field contains a space (e.g. `START TIME`), the row's tokens overshoot `headers.length` and every value past the multi-space column shifts left. Default `docker top` output uses single-word columns, so this rarely fires; a custom `ps` format would corrupt the table.

Fix: slice rows by header start-column offsets recorded from the header line, rather than re-tokenising.

Claude originally raised this as Important; downgraded since default `docker top` output never hits the case.

S4. **`parsePair` silently drops the second value when no separator is present** — `pkg/rancher-desktop/components/ContainerStats.vue:224-228` (suggestion, regression) [Claude Opus 4.7]

```ts
function parsePair(s: string): [number, number] {
  const [a, b] = s.split('/').map(p => parseBytes(p.trim()));
  return [a ?? 0, b ?? 0];
}
```

If the engine emits a single-value string such as `"648B"` (see I5 for the nerdctl angle), `b` is `undefined` and the `?? 0` at return papers over it. The loss is silent: a NetIO/BlockIO reading with no slash counts TX/Write as 0 forever. When the format then flips mid-stream (engine restart, rootless toggle), the first delta against `prev*=0` is spuriously large.

Fix: log a debug message when `split('/')` returns a single element, and treat that sample as a "missing" delta so the chart skips it instead of inventing a zero.

S5. **Refresh-rate change emits a spurious spike for the first new sample** — `pkg/rancher-desktop/components/ContainerStats.vue:461-464` (suggestion, regression) [Claude Opus 4.7]

```ts
function restartPolling() {
  stopPolling();
  startPolling();
}
```

`restartPolling` fires from the `<select @change>` handler (line 21). The main-process side replaces the timer cleanly, but the renderer keeps `prevNetRx`/`prevBlockR` from the old cadence. Switching 1 s → 30 s emits one delta spanning the ~30 s gap, then the chart shows a one-tick spike before resuming.

Fix: in `restartPolling`, reset the four `prev*` counters (or drop the next delta sample) so the chart resumes cleanly at the new cadence.

S6. **JSON-line guard absorbs every non-`{`-prefixed error** — `pkg/rancher-desktop/components/ContainerStats.vue:376-381` (suggestion, regression) [Claude Opus 4.7]

```ts
const line = statsJson.split('\n').find(l => l.trim().startsWith('{'));

```

The comment explains this as defence against multi-line output when a container name appears more than once. The handler always passes the container ID, not a name (`containerStats.ts:88`), so the multi-line case never reaches here. Meanwhile a CLI banner or warning line silently disappears.

Fix: drop the `find()` loop and `JSON.parse(statsJson)` directly; if a real engine ever needs multi-line handling, that path can land with its own `console.debug` for the skipped lines.

S7. **Memory chart shows used bytes without the limit** — `pkg/rancher-desktop/components/ContainerStats.vue:387, 39-41` (suggestion, enhancement) [Claude Opus 4.7]

```ts
push(memData.value, parseBytes((s.MemUsage ?? '0B / 0B').split('/')[0].trim()));
```

Fix: either rename the card "Memory used" to set expectations, or capture both sides of the pair and plot memory percentage (the panel users coming from `docker stats` and Docker Desktop expect).

## Design Observations

### Concerns

- **Engine-agnostic CLI parsing in the renderer** (in-scope) [Claude Opus 4.7]. The renderer reads `s.CPUPerc`, `s.MemUsage`, `s.NetIO`, `s.BlockIO` directly off the `docker stats` formatter shape (`ContainerStats.vue:386-390`). The main process owns the engine selection (`mobyClient.ts`/`nerdctlClient.ts`); it should own the normalisation too. A `runStats(containerId, namespace): Promise<NormalizedStats>` method on `ContainerEngineClient` (`containerClient/types.ts:165-169`) keeps the renderer engine-agnostic and turns I5 into a single per-engine implementation.
- **Polling architecture vs streaming** (future) [Codex GPT 5.5, Claude Opus 4.7]. `docker stats` without `--no-stream` is itself a streaming API. The current architecture spawns two engine processes per refresh tick per open Stats tab; on macOS that dominates the CPU cost of "watching an idle container." A long-lived `docker stats <id>` stream via the existing `runClient(..., 'stream', ...)` surface (`mobyClient.ts:490`) would cut process churn ~60x at the 1 s default and let the refresh-rate selector down-sample on the renderer side instead of restarting the engine.

### Strengths

- The handler mirrors `ContainerExecHandler`'s session-keyed map and IPC patterns, so anyone who has read the exec implementation can follow this one [Claude Opus 4.7].
- IPC channel namespacing (`container-stats/start`, `container-stats/data`, etc.) follows the established `container-exec/*` convention [Claude Opus 4.7].
- Skipping the first delta sample (`ContainerStats.vue:393`) avoids the classic counter-graph artefact of an enormous initial spike [Claude Opus 4.7].
- Unmounting the stats tab fully suspends polling instead of pushing it to the background [Gemini 3.1 Pro].
- `readColors()` bridges CSS custom properties into Chart.js cleanly, without hard-coding hex values for tick/grid theme tokens [Gemini 3.1 Pro].

## Testing Assessment

E2E coverage is reasonable for a UI feature: it verifies tab placement, chart render on a running container, process-table population, refresh-rate options, and the stopped-container banner. Risk-ranked gaps:

1. **No containerd / nerdctl path coverage** (links to I5). The feature is currently exercised only against `ContainerEngine.MOBY`, so silent-failure on containerd is uncaught.
2. **No assertion that chart data actually arrives over time** (links to S2). `waitForCharts` only checks `toBeVisible`; a chart that mounts and never receives a `data` event passes.
3. **No unit tests for the pure helpers** (links to S3, S4). `parseBytes`, `parsePair`, and the process-row parser are pure functions; trivial to unit-test and exactly where regressions will hide.
4. **No test for the engine-change path** (links to I3). A Jest test against `ContainerStatsHandler` with a stub `IpcMain` and a stub `WebContents` would catch the freeze.
5. **No test for the renderer-destroyed leak** (links to I2). Hard to write as e2e, but the same Jest harness as #4 catches it via `event.sender.destroyed` simulation.
6. **No stop/start coverage for the same container** (links to I4). A short test that stops the running container, waits, restarts it, and asserts the chart buffer reset would prevent regression.

## Review Process Notes

### Skill improvements

For UI features that render async data through IPC or RPC, presence-only assertions (`toBeVisible`, `isPresent`, "the card exists") pass the moment the component mounts, before any data arrives. Flag tests of data-driven UI that stop at presence and never check the rendered dataset — series length, row count, content match. A parser that swallows every sample or a handler that crashes silently still leaves the empty container element visible, so the test passes against the very failure it should catch.

### Repo context updates

[repo] Rancher Desktop ships two container-engine backends behind `ContainerEngineClient` (`mobyClient.ts`, `nerdctlClient.ts`). When a feature invokes `client.runClient(...)` with engine-specific argument shape (`--format` payloads, subcommand structure, output field names), verify that both engines emit the shape the caller assumes and that tests cover both paths. The e2e suites pin one engine per file (`ContainerEngine.MOBY` in `containers.e2e.spec.ts`); engine-specific divergences slip through unless the diff explicitly adds coverage for the other engine.

[repo] `client.runClient` accepts `SpawnOptions` (`containerClient/types.ts:80`), including `signal: AbortSignal`. Any wrapper that imposes a `Promise.race` timeout on `runClient` should thread `signal` through and call `abortController.abort()` when the timeout fires; otherwise the child process outlives the timeout and a polling caller can stack overlapping invocations under engine pressure.

# Deep Review — Stats Tab (branch `stats-tab`)

## Executive Summary

The change adds a Stats tab to the container detail view with live charts (CPU, memory, network, block I/O) and a process table, backed by a main-process polling handler that runs `docker stats --no-stream` / `docker top` per session. The implementation cleanly mirrors the existing `ContainerExecHandler` pattern and includes e2e coverage. The main concerns are an unbounded session leak when the renderer disappears without sending `stop`, a behavior regression where `updateClient` silently wipes active sessions, and a process-output parser whose "join trailing tokens" logic mis-aligns columns when interior fields are blank.

## Findings

### Critical Issues

None.

### Important Issues

I1. **Stats sessions leak when WebContents goes away without sending `stop`** — `pkg/rancher-desktop/main/containerStats.ts:68-112` (important, regression)

```ts
const session: StatsSession = {
  timer:  setInterval(() => { poll().catch(console.error) }, intervalSeconds * 1_000),
  sender: event.sender,
  namespace,
};
this.sessions.set(containerId, session);
```

The session lifecycle relies entirely on the renderer sending `'container-stats/stop'` (line 114). Unlike `ContainerExecHandler`, which is self-cleaning via `proc.on('exit', ...)` (`containerExec.ts:153`), the stats polling timer has no self-terminating event. If the renderer's `WebContents` is destroyed without a clean unmount (window crash, hard reload, `webContents.close()`), the `setInterval` keeps shelling out to `docker stats` / `docker top` once per second forever. `event.sender.send()` calls then silently fail in the `try/catch` at line 78-82, masking the leak.

I3. **Process-output parser corrupts column alignment when an interior field is blank** — `pkg/rancher-desktop/components/ContainerStats.vue:417-426` (important, regression)

```ts

  const parts = line.trim().split(/\s+/);

```

The header is split on runs of 2+ spaces (column-aware), but row tokens are split on any whitespace (column-unaware). When a row has fewer tokens than `headers.length` (an empty interior column, e.g., `WCHAN` empty for a kernel thread, or trailing fields blank), `parts.slice(0, headers.length - 1)` takes the first N-1 tokens regardless of which columns they came from, then jams the remainder into the last column. Every value after the blank is shifted left in the table.

Fix: slice the row by header start-column offsets instead of re-tokenizing, e.g.:

```ts

  .map(m => ({ name: m[0].trim(), start: m.index! }));

const rows = lines.slice(1).map(line =>
  cols.map((c, i) => line.slice(c.start, cols[i + 1]?.start).trim())
);
```

This preserves blanks as empty strings and keeps the CMD column intact (it extends to end-of-line by default).

## Design Observations

**Concerns**

- **Stats source coupling to the Docker CLI shape** (in-scope). The renderer parses `s.CPUPerc`, `s.MemUsage`, `s.NetIO`, `s.BlockIO` directly off the `docker stats` JSON formatter. The main process owns the engine — it should own the normalization too. A `runStats(containerId, namespace): Promise<NormalizedStats>` method on `ContainerEngineClient` (with `moby` and `nerdctl` implementations) keeps the renderer engine-agnostic and gives I4 a single place to fix. See `containerClient/types.ts:165-169` for the existing engine-abstracted surface.
- **Polling architecture vs. streaming** (future). `docker stats` (without `--no-stream`) is itself a streaming API. The current 1 s polling spawn cycle starts a new `docker` process every second per open Stats tab, which on macOS dominates the CPU cost of "watching a container do nothing." A single long-lived `docker stats <id>` stream piped through `runClient(..., 'stream', ...)` (the API already exists per `mobyClient.ts:490`) would cut process churn ~60x at 1 s refresh and let the refresh-rate selector drop client-side samples instead of restarting the engine. Not blocking; the next iteration should consider it.

**Strengths**

- Cleanly mirrors `ContainerExecHandler`'s session-keyed map and IPC patterns, which makes the new code easy to read for anyone who has seen the exec implementation.
- IPC channel namespacing (`container-stats/start`, `container-stats/data`, etc.) follows the established `container-exec/*` convention.
- The "skip the first delta sample" logic (`ContainerStats.vue:393`) avoids the classic counter-graph artifact of an enormous initial spike.

```typescript
      push(netRxData.value, Math.max(0, rx - prevNetRx));
      push(netTxData.value, Math.max(0, tx - prevNetTx));
      push(blockRData.value, Math.max(0, blockR - prevBlockR));
      push(blockWData.value, Math.max(0, blockW - prevBlockW));
```

The network and block I/O charts plot absolute byte differences between the current and previous sample. Because the refresh interval is user-configurable (1s to 60s), the charted values scale visually with the interval (e.g., a constant 1 MB/s transfer will show as 1 MB at 1s refresh, but 60 MB at 60s refresh). This makes the y-axis confusing since it represents "bytes per tick" rather than "bytes per second."

Fix: Divide the delta by the actual elapsed time (or `refreshSeconds.value`) before pushing to the buffer, so the chart displays an intuitive Bytes/sec rate.

```diff
-      push(netRxData.value, Math.max(0, rx - prevNetRx));
+      push(netRxData.value, Math.max(0, rx - prevNetRx) / refreshSeconds.value);
```
