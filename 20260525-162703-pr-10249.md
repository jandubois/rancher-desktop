## Executive Summary

PR #10249 retires the in-process Express dashboard server and the `http-proxy-middleware` dependency; steve 0.1.0-beta9.2 now serves the dashboard assets and proxies the Kubernetes API on a dynamic port reported via stdout. Two correctness issues stand out: `Steve.start()` has no overall timeout once `spawn` succeeds, so a steve binary that hangs mid-init wedges the renderer; and when `Steve.start()` rejects, the dashboard button stays enabled and `openDashboard` loads `http://127.0.0.1:0/`.

All three reviewers converged on the timing and error-handling shape of `pkg/rancher-desktop/backend/steve.ts`. Gemini uniquely flagged the lost browser-level CORS-bypass coverage in `e2e/extensions.e2e.spec.ts`; Codex uniquely caught a `waitForEvent` race in the new dashboard e2e; Claude raised an unhandled-rejection path through the `Latch` and the loss of the explicit `/c/local/explorer` landing URL.

## Important Issues

I1. **Unbounded wait for Steve port** — `pkg/rancher-desktop/backend/steve.ts:103-117` (important, regression) [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```ts
await new Promise<void>((resolve, reject) => {
  this.process?.once('spawn', () => { ... resolve(); });
  this.process?.once('error', (err) => { reject(...); });
  setTimeout(10_000).then(() => reject(new Error('Timed out waiting for Steve to start')));
});
this.#port = await portLatch;
console.debug(`Steve is listening on port: ${ this.#port }`);
await this.waitForReady(this.#port);
```

The 10-second timeout at line 112 guards only the `'spawn'` event. `await portLatch` at line 114 has no timeout: a steve binary that spawns cleanly but never writes the port (or never closes stdout) blocks `start()` indefinitely — the only release path is the `'exit'` handler at line 97. The previous design polled a known port and was bounded by `waitForReady`'s 30 seconds.

A second ordering hazard: `this.#port` is assigned before `waitForReady` succeeds. The public `port` getter therefore exposes a port that may still be 404-ing on `/v1/namespaces`. If `waitForReady` throws, `background.ts:1273` swallows the error and `k8s-check-state` still fires (see I2), leaving callers with a non-ready port in `#port`.

Fix: bound the latch wait and keep `#port` local until health checks pass.

```diff
-this.#port = await portLatch;
-await this.waitForReady(this.#port);
+const port = await Promise.race([
+  portLatch,
+  setTimeout(30_000).then(() => Promise.reject(new Error('Timed out waiting for Steve port'))),
+]);
+await this.waitForReady(port);
+this.#port = port;
```

Also kill `this.process` on the timeout path so the orphaned steve does not linger; drop the `console.debug` and `return this.#port` lines from the original.

I2. **Dashboard button stays enabled after `Steve.start()` failure; `openDashboard` loads `http://127.0.0.1:0/`** — `background.ts:1270-1282`, `pkg/rancher-desktop/window/dashboard.ts:23` (important, regression) [Claude Opus 4.7, Codex GPT 5.5]

```ts
// background.ts

  try {
    await Steve.getInstance().start();
  } catch (ex) {
    console.error('Failed to start Steve:', ex);
  }
}
mainEvents.emit('k8s-check-state', mgr);
window.send('k8s-check-state', state);
```

The catch swallows `Steve.start()` rejections and `k8s-check-state` fires unconditionally with `STARTED`. The tray and renderer then enable the dashboard button. When the user clicks it, `dashboard.ts:23` runs `window.loadURL(\`http://127.0.0.1:${ Steve.getInstance().port }/\`)` with `#port` either zero (start never reached the latch, or `stop()` ran) or pointing at a non-ready Steve. Electron returns `ERR_CONNECTION_REFUSED`. The previous design routed through the fixed `6120` server, which at minimum returned the dashboard shell.

Fix: gate the dashboard button on a steve-ready signal, and refuse to open the window when the port is zero.

```diff

   let window = getDashboardWindow();
+  const port = Steve.getInstance().port;
+  if (!port) {
+    console.log('Refusing to open dashboard: Steve has no listening port');
+    return;
+  }
```

Prefer a dedicated `steve-ready`/`steve-down` main event over re-using `k8s-check-state` so the UI can distinguish "Kubernetes up" from "dashboard reachable."

I3. **E2E `waitForEvent` is registered after the click** — `e2e/dashboard.e2e.spec.ts:25-27` (important, regression) [Codex GPT 5.5]

```ts
await navPage.dashboardButton.click();

dashboardPage = await electronApp.waitForEvent('window', page => page.url().includes('/c/local/explorer'));
```

`waitForEvent` only catches events emitted after it is called. The dashboard window is created in response to the click and can emit `window` before line 27 starts listening. The URL predicate compounds the risk: `window/dashboard.ts:23` loads `http://127.0.0.1:${port}/`, and the dashboard SPA only later redirects to `/c/local/explorer`. The event may fire with a `/` URL, fail the predicate, and be lost.


I5. **`'exit'` handler can produce an unhandled rejection through `Latch`** — `pkg/rancher-desktop/backend/steve.ts:97-101`, `pkg/rancher-desktop/utils/latch.ts:96-101` (important, regression) [Claude Opus 4.7]

```ts
this.process.on('exit', (code, signal) => {
  console.log(`child process exited with code ${ code } and signal ${ signal }`);
  this.isRunning = false;
  portLatch.reject(new Error(`Steve process exited unexpectedly with code ${ code } and signal ${ signal }`));
});
```

If the spawn-phase promise rejects — via `'error'` at line 109 or the 10-second timeout at line 112 — `start()` throws before reaching `await portLatch` at line 114. When the child later emits `'exit'`, `portLatch.reject` settles the latch's inner `state.promise` (`latch.ts:98`) without any `.catch` attached. Node logs `unhandledRejection` and exits under `--unhandled-rejections=strict`. `Latch.reset` installs a defensive catch (`latch.ts:111`); `Latch.reject` does not.

Fix: drain the latch on the error path, or attach a defensive catch inside `Latch.reject`. Minimal change in `steve.ts`:

```diff
-await new Promise<void>((resolve, reject) => { ... });
+try {
+  await new Promise<void>((resolve, reject) => { ... });
+} catch (ex) {
+  portLatch.catch(() => { /* prevent unhandled rejection from late 'exit' */ });
+  throw ex;
+}
 this.#port = await portLatch;
```

A defensive catch inside `Latch.reject` (mirroring `reset()`) would also fix the broader class. See also Design Observations.

S2. **`setTimeout(10_000)` timer keeps the event loop alive after spawn succeeds** — `pkg/rancher-desktop/backend/steve.ts:112` (suggestion, regression) [Claude Opus 4.7, Gemini 3.1 Pro]

`setTimeout` from `timers/promises` returns an uncancelable handle here. Once `spawn` resolves the outer promise, the inner timer still runs for the remaining seconds and pins the event loop. Use an `AbortSignal` so the timer cancels on resolution.

## Design Observations

**Concerns**

- **Steve health is no longer separable from Kubernetes state (in-scope)** [Claude Opus 4.7, Codex GPT 5.5]. The renderer takes its "open the dashboard" cue from `k8s-check-state`, but `Steve.start()` is now the only path that produces a usable dashboard URL. A dedicated `steve-ready`/`steve-down` main-process event would decouple "Kubernetes is up" from "dashboard is reachable" and avoid the port-0 navigation in I2.
- **`Steve.getInstance().port` is read synchronously across a stop/start cycle (in-scope)** [Claude Opus 4.7]. `stop()` resets `#port` to zero. Any caller that reads `port` between `STOPPING` and the next ready-state crossing sees the transient zero. A `getPortReady(): Promise<number>` accessor that awaits an internal latch would close both the I2 zero-port window and the I1 ordering hazard.

**Strengths**

- Retiring the in-process Express server and the bespoke `proxyUtils.ts` (especially the `onProxyReqWs` path, a known source of bugs) removes a meaningful chunk of attack surface and IPC complexity [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro].
- The new `getChecksums` reads a single upstream `steve.sha512sum` and removes the hardcoded platform matrix; future steve releases that add a platform no longer need a code change here [Gemini 3.1 Pro].
- Reusing `Latch` for "wait until child reports its port" matches the established pattern in `K3sHelper` and reads cleanly once the unhandled-rejection edge (I5) is closed [Claude Opus 4.7].
- The new dashboard e2e is the first dashboard smoke test in the repo and exercises a real cluster path end-to-end [Claude Opus 4.7].

## Testing Assessment

1. No test covers `Steve.start()` failure paths. Add a unit test (mock `spawn`) for: child never writes the port, child writes garbage, child exits before the latch resolves. These are precisely the inputs that I1 and I5 fail on.
2. `isPortReady`'s switch from `Electron.net.request` to `Electron.net.fetch` is a behavior change with no test. A small unit test that returns `false` on 404 and `true` on 200 would lock the contract in.

5. Steve's stdout protocol (port-on-stdout-then-close) has no integration test. A test fake that emits the port and closes stdout would lock in the contract documented in S3.

## Review Process Notes

**Skill improvements**

- When a file deletes a fixed-port server in favor of a child process that reports its port on stdout, trace the consumers of the port across the kill/start cycle, not only during the initial bring-up. A `stop()` that resets the port to zero combined with a public synchronous getter creates a transient zero-port window that callers can latch onto. Add a dimension cue: "When a singleton's lifecycle changes from "fixed property" to "dynamic, computed at start," enumerate every read of the property across `stop`/`start` boundaries."
- When an `EventEmitter`-driven async handler invokes a method whose internal `isRunning` flag flips only after the first `await`, two concurrent emissions can pass the guard and both proceed to the side-effecting branch. Add to the concurrency-restructure dimension: "Methods guarded by a boolean that is set inside a later async step admit a same-tick race for any caller that does not serialize on a promise."

**Repo context updates**

- Add to `deep-review-context.md`: "`pkg/rancher-desktop/window/dashboard.ts` reads `Steve.getInstance().port` synchronously; `Steve.stop()` resets the port to zero. When reviewing changes that touch either Steve's lifecycle or the dashboard window, confirm the read happens only when Steve is in a ready state."

**I2. Loss of renderer-side CORS bypass test coverage** — `e2e/extensions.e2e.spec.ts:408-433`

The original `bypass CORS` test correctly executed a raw `fetch()` directly within the extension view (`evalInView`) to verify the browser-level CORS override. The new test replaces this with `ddClient.extension.vm.service.get`, which routes through the Docker Desktop extension proxy backend. Because the proxy is a backend service, it naturally bypasses browser CORS policies, leaving the actual renderer-level `fetch` CORS bypass completely untested. 

Fix: Restore the `evalInView` block with a standard `fetch` call and point it at the newly created local `http.createServer` endpoint.

**I3. Concurrent `start()` race condition creates orphaned processes** — `pkg/rancher-desktop/backend/steve.ts:47-66`

```typescript
  public async start(): Promise<number> {
    const { pid } = this.process || { };

    if (this.isRunning && pid) {
      console.debug(`Steve is already running with pid: ${ pid }`);
      return this.#port;
    }
```

The startup guard does not protect against concurrent calls to `start()` because `this.isRunning` is only set to `true` asynchronously when the Node.js `spawn` event fires. If `start()` is invoked twice synchronously or back-to-back in the event loop, both calls bypass the check, spawning two independent Steve processes and overwriting `this.process`. 

Fix: Track the active startup promise in a private class field (`#startPromise`) and return it if a startup is already in progress, or synchronously set a starting flag before spawning.

### Suggestions

**S1. Uncleared timer leaks into the event loop** — `pkg/rancher-desktop/backend/steve.ts:104`

```typescript
      setTimeout(10_000).then(() => reject(new Error('Timed out waiting for Steve to start')));
```

The `setTimeout(10_000)` inside the `spawn` promise executor is never cleared. Even when Steve spawns successfully in a few milliseconds, this timer remains active in the background, keeping the Node.js event loop alive unnecessarily for 10 seconds. 

Fix: Use an `AbortController` to cancel the timeout when the `spawn` or `error` events resolve the promise.

### Design Observations

**Strengths**
- Consolidating the dashboard assets into Steve allows for the complete removal of the internal express-based `DashboardServer` and proxy middlewares, significantly reducing surface area and port management overhead.
- Changing `fetchUpstreamChecksums` to dynamically read and map all components in the upstream `steve.sha512sum` removes the hardcoded platform list and makes dependency updates much more resilient to release changes.

### Testing Assessment

The E2E tests have successfully replaced static port assumptions. However, as noted in **I2**, the testing of the browser CORS bypass rules for extensions was inadvertently dropped when the request tool was swapped from `fetch` to `ddClient`.
