## Executive Summary

The PR fixes a real deadlock: after engine-switch tears down the WSL distro,
residual Hyper-V vsock state can let `vsockHandshake` "succeed" before any
new peer is up, leaving host-switch waiting on a data connection that never
arrives. The fix wraps the handshake in a retry loop that waits for the
peer's data connection on the same listener and treats a missing connection
as a phantom; the first real connection is replayed via a `firstConnListener`
shim so the rest of host-switch is unaffected. Both prior-round comments
from Nino-K were addressed in `d43b3b725`.

The structure is sound and the unit tests cover `firstConnListener` and
`acceptWithTimeout` well. One regression matters: `runHandshakeLoop` is now
an errgroup task whose `serveAccepts` blocks in `vn.AcceptStdio`, and that
call's read loop checks `ctx.Done()` only between reads — so context
cancellation alone does not unblock an idle data connection, and shutdown
can hang. The other findings are minor: a test-only nil-deref, a missing
test for the `ctx.Err()` precedence rule that motivated the fix-up commit,
and several style/placement nits.

## Important Issues

I1. **Context cancellation does not unblock the active data connection** — `src/go/networking/cmd/host/switch_windows.go:178` [Codex GPT 5.5]

```go
// AcceptStdio blocks for the lifetime of the connection, returning when
// the peer goes away.  Returning here lets runHandshakeLoop redo the
// handshake.
err = vn.AcceptStdio(ctx, conn)
```

`runHandshakeLoop` is now an errgroup task (`switch_windows.go:111`), and `groupErrs.Wait()` (`switch_windows.go:126`) waits for it to return. On SIGTERM the signal handler cancels the context (`switch_windows.go:119`), but `serveAccepts` blocks synchronously in `vn.AcceptStdio` → `Switch.Accept` → `rxStream`, whose select on `ctx.Done()` runs only between reads:

```go
// gvisor-tap-vsock@v0.8.7 pkg/tap/switch.go:233
func (e *Switch) rxStream(ctx context.Context, id int, conn net.Conn, sProtocol streamProtocol) error {
    reader := bufio.NewReader(conn)
    ...
    for {
        select {
        case <-ctx.Done():
            break loop
        default:
        }
        _, err := io.ReadFull(reader, sizeBuf)  // blocks until bytes arrive
```

`io.ReadFull` blocks until the peer sends data or the connection is closed. Cancelling `ctx` alone does nothing. Pre-PR, the accept loop ran in a fire-and-forget `go func()` that `Wait()` never tracked, so the leaked goroutine died with the process. Post-PR the loop is in the errgroup, so `Wait()` blocks on it — host-switch can stay alive after intentional shutdown if the data connection is idle and still open. This is a regression from the merge-base behavior.

Fix: close the accepted connection when the context is cancelled, so the blocked read returns:

```diff
 // serveAccepts handles a single connection from ln through vn, returning when
 // the connection ends — because the context is cancelled or because the peer
 // has gone away.  Either way, the caller should redo the handshake.
 func serveAccepts(ctx context.Context, vn *virtualnetwork.VirtualNetwork, ln net.Listener) {
     conn, err := ln.Accept()
     if err != nil {
         logrus.Errorf("failed to accept: %v", err)
         return
     }
+    stopClose := context.AfterFunc(ctx, func() { _ = conn.Close() })
+    defer stopClose()
     // AcceptStdio blocks for the lifetime of the connection, returning when
     // the peer goes away.  Returning here lets runHandshakeLoop redo the
     // handshake.
     err = vn.AcceptStdio(ctx, conn)
```

S1. **Nil-deref risk in test dial helper** — `src/go/networking/cmd/host/switch_windows_test.go:52-55` [Claude Opus 4.7]

```go
go func() {
    c, _ := (&net.Dialer{}).DialContext(t.Context(), "tcp", ln.Addr().String())
    _ = c.Close()
}()
```

If `DialContext` returns `(nil, err)`, `c.Close()` panics on a nil-interface deref and crashes the test goroutine. Two other dial sites in the same file already guard against this — `switch_windows_test.go:92-96` checks `if c != nil`, and `switch_windows_test.go:159-161` checks `if err == nil`. The pattern at line 53 is the inconsistent one.

Fix:
```diff
 go func() {
-    c, _ := (&net.Dialer{}).DialContext(t.Context(), "tcp", ln.Addr().String())
-    _ = c.Close()
+    c, err := (&net.Dialer{}).DialContext(t.Context(), "tcp", ln.Addr().String())
+    if err == nil {
+        _ = c.Close()
+    }
 }()
```

S2. **No test covers `handshakeWithRetry`'s `ctx.Err()` precedence** — `src/go/networking/cmd/host/switch_windows.go:288-298` [Claude Opus 4.7]

The follow-up commit `d43b3b725` exists to fix one specific bug: `acceptWithTimeout`'s timer can win the select race even when the context is already cancelled, so `handshakeWithRetry` must re-check `ctx.Err()` and prefer it over the timeout error:

```go
_ = ln.Close()
if ctx.Err() != nil {
    return nil, ctx.Err()
}
lastErr = err
```

That precedence rule is the entire point of the fix-up commit, but no test exercises it. The existing tests cover only `acceptWithTimeout` directly, where the behavior differs (it has its own `ctx.Done()` case). A small seam — inject `vsockHandshake` via a function variable, or extract the validation step into a testable helper — would let a test feed a never-arriving listener with an already-cancelled context and assert the wrapped error is `context.Canceled`, not `errAcceptTimeout`. Without one, a regression to the bug Nino flagged would not be caught.

S3. **New constants placed away from existing `const` block** — `src/go/networking/cmd/host/switch_windows.go:237,243` [Claude Opus 4.7]

```go
const handshakeValidationTimeout = 10 * time.Second
...
const handshakeMaxAttempts = 5
```

The file already groups package-level tunables in a single `const ( ... )` block at lines 45-50, and this PR added `debugLogInterval` to that block. The two new tunables sit alone next to the function that uses them. Co-locating them improves discoverability for future tuning.

```diff
 const (
     vsockListenPort    = 6656
     vsockHandshakePort = 6669
     timeoutSeconds     = 5 * 60
     debugLogInterval   = 5 * time.Second
+    // handshakeValidationTimeout is how long to wait for a real data
+    // connection from the peer after vsockHandshake reports success.
+    handshakeValidationTimeout = 10 * time.Second
+    // handshakeMaxAttempts caps the retries when the peer never produces
+    // a data connection.
+    handshakeMaxAttempts = 5
 )
```

S4. **`errAcceptTimeout` declaration trails its only consumer** — `src/go/networking/cmd/host/switch_windows.go:340` [Claude Opus 4.7]

```go
var errAcceptTimeout = errors.New("timed out waiting for peer data connection")
```

The variable is referenced inside `acceptWithTimeout` (line 330) but declared after the function. Idiomatic Go places sentinel errors at the top of the file or directly above the function that returns them. Move it next to the related constants.

S5. **Redundant `ln.Close()` after `acceptWithTimeout` returns on timeout/cancellation** — `src/go/networking/cmd/host/switch_windows.go:288-293` [Claude Opus 4.7]

```go
conn, err := acceptWithTimeout(ctx, ln, handshakeValidationTimeout)
if err == nil {
    logrus.Infof("validated handshake on attempt %d", attempt)
    return &firstConnListener{Listener: ln, first: conn}, nil
}
_ = ln.Close()
```

`acceptWithTimeout` already closes `ln` on its `case <-timer.C` and `case <-ctx.Done()` branches (`switch_windows.go:326,332`). The double-close is idempotent on hvsock listeners and harmless, but the responsibility split is muddy: callers should not assume `ln` is half-owned by `acceptWithTimeout` on the error path. Either drop the outer `Close` and document that `acceptWithTimeout` owns `ln` when it returns an error, or move the close out of `acceptWithTimeout` and centralize it in `handshakeWithRetry`.

S6. **Tests use TCP, not hvsock — `Close`-during-`Accept` semantics differ** — `src/go/networking/cmd/host/switch_windows_test.go:24-30` [Claude Opus 4.7, Gemini 3.1 Pro]

`acceptWithTimeout`'s correctness depends on `ln.Close()` unblocking a pending `Accept` so the goroutine drains. TCP listeners on Linux/Darwin honor this contract. `hvsock.Listener` (the production listener) wraps Windows native vsock sockets, where the same guarantee is not part of the package's documented API.

The TCP-based tests therefore validate the helpers' logic but not the production assumption. Add a short comment near `localTCPListener` calling out that production uses hvsock and relies on the same close-unblocks-accept contract, so a future refactor cannot silently invalidate it. Gemini proposed an alternative — drain the channel asynchronously so `acceptWithTimeout` returns on timeout regardless — but that converts a potential leak (if hvsock failed to unblock) into a guaranteed one, so the conservative fix is the comment.

S7. **Use `time.Ticker` instead of `time.After` in `debugLogLoop`** — `src/go/networking/cmd/host/switch_windows.go:188-194` [Gemini 3.1 Pro]

`debugLogLoop` was extracted from the inline goroutine in this PR; the extraction preserved the original `time.After` pattern. `time.After` allocates a fresh timer and channel on every iteration; a single `time.Ticker` is the idiomatic choice for a periodic log loop. The interval is 5 seconds, so the cost is small, but the change costs nothing either:

```diff
 func debugLogLoop(ctx context.Context, vn *virtualnetwork.VirtualNetwork, interval time.Duration) error {
+    ticker := time.NewTicker(interval)
+    defer ticker.Stop()
     for {
         select {
-        case <-time.After(interval):
+        case <-ticker.C:
             logrus.Debugf("%v sent to the VM, %v received from the VM", humanize.Bytes(vn.BytesSent()), humanize.Bytes(vn.BytesReceived()))
         case <-ctx.Done():
             return nil
         }
     }
 }
```

## Design Observations

### Concerns

**Long shutdown latency on context cancellation during a re-handshake** `(future)` [Claude Opus 4.7]

`runHandshakeLoop` now calls `vsockHandshake` repeatedly across host-switch's lifetime. `vsockHandshake` calls `vsock.GetVMGUID` (`pkg/vsock/handshake_windows.go:33`), which uses a 5-minute `bailout` timer (`timeoutSeconds = 5 * 60`); inside, `tryFindGUID` (`pkg/vsock/handshake_windows.go:126-136`) selects only on `found` and `timeout` — never on `ctx.Done()`. If the user shuts down Rancher Desktop while host-switch is mid-re-handshake and no peer is reachable, shutdown can stall up to 5 minutes per attempt before `runHandshakeLoop` propagates the error.

This is pre-existing in `vsock.GetVMGUID`, not introduced by this PR — but the PR turns a one-shot call into a retry loop, so the same latency applies on every engine-switch teardown, not just at startup. With `handshakeMaxAttempts = 5`, the worst case stacks up to 5×5min if WSL is up but the peer is unreachable. Worth tracking as a follow-up to add a `ctx.Done()` case in `tryFindGUID`.

### Strengths

- **Decoupled lifecycle** `(in-scope)` [Gemini 3.1 Pro] — Hoisting `virtualnetwork.New` out of the connection loop into `main` keeps the host-side HTTP server (and its API endpoints) bound and serving across engine switches, so local clients querying port forwards do not see transient errors.
- **`firstConnListener` shim** `(in-scope)` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro] — The shim is the right abstraction: it lets `handshakeWithRetry` validate the data connection without changing the contract the rest of host-switch consumes. The explicit `Close` added in `d43b3b725` cleanly addresses the leak case Nino raised.
- **`acceptWithTimeout` close-then-drain sequencing** `(in-scope)` [Claude Opus 4.7] — `Close` → drain → conditional `Close` of any captured connection is the right pattern for "close-to-unblock" plus leak-free shutdown, and the inline comment at `switch_windows.go:319-321` explains why the drain matters.
- **Forward-looking comment on `runHandshakeLoop`** `(in-scope)` [Claude Opus 4.7] — The comment explaining why a host-only retry was chosen over a nonce-exchange protocol change is exactly the kind of note that helps future maintainers; calling out that the cleaner fix would require coordinating a WSLDistro version bump is especially valuable.

## Review Process Notes

### Repo context updates

- **Errgroup task lifecycle.** When reviewing a Go change that moves a long-blocking call (network read, `Accept`, `io.ReadFull`) into an `errgroup.Group.Go` task, check whether the call honors context cancellation natively. Standard `net.Conn.Read` and most `io.ReadFull(conn, ...)` calls do **not** — they only unblock when the connection is closed. If the call cannot honor the context, the task either needs a `context.AfterFunc` that closes the underlying resource on cancellation, or the change converts a previously fire-and-forget goroutine into one that hangs `errgroup.Wait()` on shutdown. Flag this any time a fire-and-forget goroutine becomes an errgroup task.

- **gvisor-tap-vsock `AcceptStdio` does not unblock on context cancellation.** `virtualnetwork.AcceptStdio` calls into `Switch.Accept` → `rxStream`, whose select on `ctx.Done()` runs only between reads. The blocking `io.ReadFull` only returns when the connection is closed. Any caller in this repo that ties shutdown latency to `AcceptStdio` returning needs a `context.AfterFunc(ctx, conn.Close)` (or equivalent connection-close-on-cancel) to make shutdown deterministic.

- **`vsock.GetVMGUID` ignores its `ctx` parameter.** `pkg/vsock/handshake_windows.go`'s `tryFindGUID` selects only on the `found` channel and a 5-minute `timeout` channel. Passing a context does nothing for shutdown latency. Any caller in the host-side networking code that loops over `GetVMGUID` (the new `handshakeWithRetry` is one) inherits this stall. Reviews of code that calls `GetVMGUID` should ask whether the caller can tolerate up to 5 minutes per call, or whether the call should be guarded by a separate timeout.

`runHandshakeLoop` now calls `vsockHandshake` repeatedly across the host-switch's lifetime. `vsockHandshake` calls `vsock.GetVMGUID` (handshake_windows.go:33), which uses a 5-minute `bailout` timer (handshake_windows.go:217 in the changed file via `timeoutSeconds`) and `tryFindGUID` does **not** select on `ctx.Done()` (handshake_windows.go:126-136). So if the user shuts down Rancher Desktop while host-switch is in the middle of a re-handshake and no Hyper-V VM is up, shutdown can stall up to 5 minutes before `runHandshakeLoop` propagates the error.

This is pre-existing in `vsock.GetVMGUID` — not introduced by this PR — but the PR turns a one-shot path into a loop, so the same latency now applies to every engine-switch teardown, not just to startup. Worth tracking as a follow-up to add a `ctx.Done()` case in `tryFindGUID`.

### Strengths

- The `firstConnListener` shim is the right abstraction: it lets `handshakeWithRetry` validate the data connection without changing the contract the rest of `host-switch` consumes.
- The sequencing in `acceptWithTimeout` (`Close` → drain → conditional `Close` of the captured conn) is the right pattern for "close-to-unblock" + leak-free shutdown.
- The detailed comment on `runHandshakeLoop` explaining *why* a host-only retry was chosen over a nonce-exchange protocol change is exactly the kind of forward-looking note that helps future maintainers; calling out that the cleaner fix requires a coordinated WSLDistro version bump is especially valuable.

S1. **Use `time.Ticker` instead of `time.After` in loop** — `src/go/networking/cmd/host/switch_windows.go:193-195` (suggestion, enhancement)

`time.After` creates a new timer and channel on every iteration of the loop. While the timer will eventually fire and be garbage collected because the interval is short (5s), it is idiomatic to use a single `time.Ticker` for periodic tasks to avoid unnecessary allocations and timer heap churn.

Fix:
```diff

+func debugLogLoop(ctx context.Context, vn *virtualnetwork.VirtualNetwork, interval time.Duration) error {
+	ticker := time.NewTicker(interval)
+	defer ticker.Stop()
 	for {
 		select {
-		case <-time.After(interval):
+		case <-ticker.C:
 			logrus.Debugf("%v sent to the VM, %v received from the VM", humanize.Bytes(vn.BytesSent()), humanize.Bytes(vn.BytesReceived()))
 		case <-ctx.Done():
 			return nil
```

S2. **Prevent potential deadlock if `Accept` fails to unblock** — `src/go/networking/cmd/host/switch_windows.go:323-333` (suggestion, enhancement)

If `timer.C` or `ctx.Done()` fires, `acceptWithTimeout` closes the listener and performs a blocking read `<-ch` to drain the accepted connection. Standard Go `net.Listener` implementations guarantee that `Close()` immediately unblocks `Accept()`. However, if any platform bug in the underlying `virtsock` listener delays or fails to unblock `Accept()`, this blocking read will stall `acceptWithTimeout` forever, recreating a deadlock condition. Draining asynchronously isolates the caller from listener closure bugs.

Fix: Drain the channel in a background goroutine so `acceptWithTimeout` returns immediately upon timeout:

```diff
 	case <-timer.C:
 		_ = ln.Close()
-		if r := <-ch; r.conn != nil {
-			_ = r.conn.Close()
-		}
+		go func() {
+			if r := <-ch; r.conn != nil {
+				_ = r.conn.Close()
+			}
+		}()
 		return nil, errAcceptTimeout
 	case <-ctx.Done():
 		_ = ln.Close()
-		if r := <-ch; r.conn != nil {
-			_ = r.conn.Close()
-		}
+		go func() {
+			if r := <-ch; r.conn != nil {
+				_ = r.conn.Close()
+			}
+		}()
 		return nil, ctx.Err()
```
