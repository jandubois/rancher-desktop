Executive Summary ¶

The branch adds a Stats tab with live CPU/memory/network/I/O charts and a docker top process table, plumbed through a main-process polling handler modeled on ContainerExecHandler. The structure is sound. Two issues block merge: the polling timer outlives a destroyed renderer, and updateClient clears every session without telling the renderer, freezing the charts on the last sample. A third gap — the containerd path routes through nerdctl but assumes the Docker JSON field names — ships untested.

All three reviewers flagged the runWithTimeout subprocess/interval concern; Claude raised the widest set of additional issues.

Design Observations ¶

Concerns ¶

  • Engine-agnostic CLI parsing in the renderer (in-scope) Claude. The renderer reads s.CPUPerc, s.MemUsage, s.NetIO, s.BlockIO directly off the docker stats formatter shape (ContainerStats.vue:386-390). The main process owns the engine selection (mobyClient.ts/nerdctlClient.ts); it should own the normalisation too. A runStats(containerId, namespace): Promise<NormalizedStats> method on ContainerEngineClient (containerClient/types.ts:165-169) keeps the renderer engine-agnostic and turns I5 into a single per-engine implementation.
  • Polling architecture vs streaming (future) Codex Claude. docker stats without --no-stream is itself a streaming API. The current architecture spawns two engine processes per refresh tick per open Stats tab; on macOS that dominates the CPU cost of "watching an idle container." A long-lived docker stats <id> stream via the existing runClient(..., 'stream', ...) surface (mobyClient.ts:490) would cut process churn ~60x at the 1 s default and let the refresh-rate selector down-sample on the renderer side instead of restarting the engine.

Strengths ¶

  • The handler mirrors ContainerExecHandler's session-keyed map and IPC patterns, so anyone who has read the exec implementation can follow this one Claude.

Testing Assessment ¶

E2E coverage is reasonable for a UI feature: it verifies tab placement, chart render on a running container, process-table population, refresh-rate options, and the stopped-container banner. Risk-ranked gaps:

  1. No containerd / nerdctl path coverage (links to I5). The feature is currently exercised only against ContainerEngine.MOBY, so silent-failure on containerd is uncaught.
  2. No assertion that chart data actually arrives over time (links to S2). waitForCharts only checks toBeVisible; a chart that mounts and never receives a data event passes.
  3. No unit tests for the pure helpers (links to S3, S4). parseBytes, parsePair, and the process-row parser are pure functions; trivial to unit-test and exactly where regressions will hide.
  4. No test for the engine-change path (links to I3). A Jest test against ContainerStatsHandler with a stub IpcMain and a stub WebContents would catch the freeze.
  5. No test for the renderer-destroyed leak (links to I2). Hard to write as e2e, but the same Jest harness as #4 catches it via event.sender.destroyed simulation.
  6. No stop/start coverage for the same container (links to I4). A short test that stops the running container, waits, restarts it, and asserts the chart buffer reset would prevent regression.

Review Process Notes ¶

Skill improvements ¶

For UI features that render async data through IPC or RPC, presence-only assertions (toBeVisible, isPresent, "the card exists") pass the moment the component mounts, before any data arrives. Flag tests of data-driven UI that stop at presence and never check the rendered dataset — series length, row count, content match. A parser that swallows every sample or a handler that crashes silently still leaves the empty container element visible, so the test passes against the very failure it should catch.

Repo context updates ¶

[repo] Rancher Desktop ships two container-engine backends behind ContainerEngineClient (mobyClient.ts, nerdctlClient.ts). When a feature invokes client.runClient(...) with engine-specific argument shape (--format payloads, subcommand structure, output field names), verify that both engines emit the shape the caller assumes and that tests cover both paths. The e2e suites pin one engine per file (ContainerEngine.MOBY in containers.e2e.spec.ts); engine-specific divergences slip through unless the diff explicitly adds coverage for the other engine.

[repo] client.runClient accepts SpawnOptions (containerClient/types.ts:80), including signal: AbortSignal. Any wrapper that imposes a Promise.race timeout on runClient should thread signal through and call abortController.abort() when the timeout fires; otherwise the child process outlives the timeout and a polling caller can stack overlapping invocations under engine pressure.
