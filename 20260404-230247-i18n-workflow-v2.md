## Executive Summary

This branch adds a comprehensive i18n system to Rancher Desktop: a Go-based `i18n-report` CLI tool for translation management, main-process and renderer-side translation modules, locale settings integration, and full UI translation into German and Simplified Chinese. The architecture is sound â€” clean separation between lightweight main-process interpolation and renderer-side ICU formatting, reactive locale switching via Vuex, and a robust translation maintenance pipeline with drift detection, validation, and manifest cross-validation.

The review found 16 important issues and ~25 suggestions across 4 passes. The most impactful issues are: (1) translated validator errors breaking locked-field regex detection, (2) missing `raw=true` on two `v-html` + `t()` calls causing escaped HTML in production, (3) IPC listener leak from non-idempotent store initialization, and (4) several merge tool robustness gaps (duplicate keys, unknown keys, non-mapping root handling). No critical bugs were found â€” the feature gate limits runtime exposure.

## Important Issues

1. **Locked-field regex breaks under non-English locales** â€” `settingsValidator.ts:290` [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro] (important, regression)

```typescript
errors.push(t('validation.fieldLocked', { field: `${ prefix }.${ k }` }));
```

The validator emits a translated message, but `commandLineOptions.ts:142` and `backendHelper.ts:171` classify locked-field failures by regex-matching the English text `/field ".+?" is locked/`. A non-English locale causes the regex to miss, downgrading the error from `LockedFieldError` to a generic fatal path. The NOTE comment at lines 287-289 acknowledges this but underestimates exposure: `rdctl set --application.locale=de` bypasses the UI feature gate.

Fix: Return structured error objects with a machine-readable `code` field, or keep `fieldLocked` messages in English until the refactor.

2. **Missing `raw=true` escapes HTML in translated link** â€” `ContainerEngineGeneral.vue:92` [Claude Opus 4.6] (important, regression)

```vue
v-html="t('preferences.containerEngine.webAssembly.warning')"
```

The translation contains `<a href="#" data-navigate="Container Engine,general">`, but `t()` defaults to `raw=false` and calls `escapeHtml()`. The link renders as literal `&lt;a&gt;` text. `BodyKubernetes.vue:205` does it correctly with `t('...', {}, true)`.

Fix: `v-html="t('preferences.containerEngine.webAssembly.warning', {}, true)"`

3. **Missing `raw=true` escapes HTML in log lines link** â€” `KubernetesError.vue:41` [Claude Opus 4.6] (important, regression)

9. **`IntlMessageFormat` constructor can crash renderer** â€” `store/i18n.js:67-68` [Claude Opus 4.6] (pass 2) (important, gap)

```javascript

```

Throws `SyntaxError` on malformed ICU patterns. The crash is cached, so subsequent calls for the same key also fail. Pre-existing code, but risk substantially increased by adding machine-translated locale files.

10. **Hardcoded "Error" in KubernetesError header** â€” `KubernetesError.vue:11` [Claude Opus 4.6] (pass 2) (important, regression)

```html

```

The commit that translated this file left the English word "Error" hardcoded, producing mixed-language headers.

11. **`validate` skips HTML attribute verification** â€” `report_validate.go:210-243` [Codex GPT 5.4] (pass 2) (important, regression)

The tag validator only compares tag-name counts. It accepts translations that keep an `<a>` tag but drop required `data-action`/`data-navigate` attributes, and also accepts malformed open/close pairs. The runtime handlers at `navigateClick.ts:10-14` and `KubernetesError.vue:118-120` depend on these attributes.

Fix: Compare `(tag name, direction, required attributes)` instead of just name counts.

12. **`command-api.yaml` locale enum not validated in CI** â€” `command-api.yaml:600-602` [Claude Opus 4.6] (pass 2) (important, regression)

The locale enum is a hardcoded list that must be manually kept in sync with `meta/locales.yaml` and `settingsValidator.ts`. The `i18n-report manifest` command can cross-validate, but it is not in CI.

Fix: Add `i18n-report manifest` to CI, or generate the enum from `meta/locales.yaml`.

13. **`reportValidateQuiet` diverges from `reportValidate`** â€” `report_check.go:218-265` [Claude Opus 4.6] (pass 3) (important, regression)

`reportValidateQuiet` is a partial copy of `reportValidate` that omits metadata coherence checks. The "validate passes" label from `check --policy` is misleading since `validate --locale=X` runs additional checks.

Fix: Extract shared validation logic into a helper both functions call.

14. **`merge` silently resolves duplicate input keys** â€” `report_merge.go:94-126` [Codex GPT 5.4] (pass 3) (important, regression)

`parseMergeInput()` appends every parsed entry without deduplicating. When two agent output files contain the same key with different translations, the later one silently wins. In multi-batch workflows this is a practical input condition.

Fix: Reject conflicting same-key/different-value entries before mutating the YAML tree.

15. **`merge` writes unknown/hallucinated keys** â€” `report_merge.go:83-126` [Codex GPT 5.4] (pass 3) (important, regression)

The parser only checks syntax with `isValidDottedKey()`; it never verifies that the key exists in `en-us.yaml`. An LLM typo or hallucinated key makes it into the locale file without metadata, undermining the drift machinery.

Fix: Load the source key set before applying entries and reject unknown keys.

16. **`merge` can erase locale file on non-mapping YAML root** â€” `report_merge.go:59-147` [Codex GPT 5.4] (pass 4) (important, regression)

If a locale file has a scalar or sequence root, `nodeSetLeaf()` mutates the non-mapping node, but `serializeYAMLNode()` refuses to emit anything for non-mapping roots. The subsequent `os.WriteFile()` truncates the locale file to empty.

Fix: Reject non-mapping roots before any mutation: `if treeRoot.Kind != yaml.MappingNode { return error }`

## Design Observations

### Concerns

1. **Translated error strings as machine-readable identifiers** (in-scope) [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro] â€” The `fieldLocked` pattern is structural: validation produces translated human-readable strings, and callers regex-match them to determine error types. A `{ code: string, message: string }` return type would eliminate this class of bugs.

2. **Reference scanner as regex inventory** (future) [Codex GPT 5.4] â€” The scanner is a list of "known syntaxes," which is why it missed `v-t` immediately. A syntax table or small parser layer would reduce maintenance burden.

3. **`data-navigate` in translation strings couples navigation to content** (in-scope) [Claude Opus 4.6] â€” Translators must preserve exact page names and tab IDs inside HTML attributes. Extracting navigation targets into component props would eliminate this fragility.

4. **Manifest "source locale" abstraction half-integrated** (future) [Codex GPT 5.4] â€” `manifest.go` models an arbitrary source locale, but most commands hardcode `en-us.yaml`.

### Strengths

1. **`pendingLocale` cancellation pattern** in `main/i18n.ts` elegantly handles rapid locale changes within Node's single-threaded event loop. [Claude Opus 4.6]

2. **`onLocaleChange` callback API** cleanly separates locale loading from menu/tray rebuilding, with proper unregister support. [Claude Opus 4.6]

3. **Test mock design** â€” `testUtils/translations.ts` loads real English YAML and performs real interpolation, avoiding mocks that test themselves. [Claude Opus 4.6]

4. **Reactive locale switching** is well-architected: page titles use `titleKey` via Vuex, preference tabs use `labelKey`, nav items use `routeLabelKeys`. `DiagnosticsBody.vue` headers correctly moved to `computed`. [Claude Opus 4.6]

5. **Manifest + metadata + drift-detection model** turns translation freshness into something the repo can validate instead of relying on manual review. [Codex GPT 5.4]

6. **Two-tier i18n architecture** (simple interpolation in main, ICU MessageFormat in renderer) keeps Electron main free of unnecessary complexity while sharing YAML sources. [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

7. **yaml.Node-based merge** preserves comments and leaf structure directly in the tree, which is the right fit for translation maintenance. [Codex GPT 5.4]

### Executive Summary
This branch introduces a comprehensive i18n system spanning both the main and renderer processes, alongside a dedicated `i18n-report` Go tool for managing translations. The system successfully extracts hardcoded strings and establishes a scalable workflow, but currently introduces several critical runtime regressions in Vue components and architectural gaps regarding consistent ICU syntax support and reactivity.

### Critical Issues

1. **`t` is undefined in `<script setup>`** â€” `pkg/rancher-desktop/pages/containers/ContainerInfo.vue:171` (critical, regression)

```vue
watch(containerName, (name) => {
  store.dispatch('page/setHeader', {
    title:       name || t('containerInfo.fallbackTitle'),
    description: '',
```

The component calls `t(...)` directly inside the `<script setup>` block. Unlike standard Vue components using the options API where `this.t` is injected via a mixin, `t` is not imported, defined, or injected via `useI18n()` in this scope. This will trigger a `ReferenceError` when the watcher executes.

Fix: Import the translation function or retrieve it from the Vuex store (`const t = (key) => store.getters['i18n/t'](key);`) within the setup context.

### Important Issues

2. **Static translation bindings in `data()` break reactivity** â€” `pkg/rancher-desktop/pages/Containers.vue:187-210`, `pkg/rancher-desktop/pages/Volumes.vue:111`, `pkg/rancher-desktop/components/Images.vue:165` (important, regression)

```vue
  data() {
    return {
      headers: [
        {
          name:  'containerState',
          label: this.t('containers.manage.table.header.state'),
        },
```

Table headers and other reactive labels are assigned `this.t(...)` inside the `data()` hook. Because `data()` evaluates exactly once upon component creation, these labels will not update when the user changes the application locale dynamically.

Fix: Move translatable arrays and objects like `headers` from `data()` into `computed` properties so they re-evaluate reactively when the locale state changes.

3. **Validation skips placeholders inside ICU branches** â€” `src/go/i18n-report/report_validate.go:189-191` (important, gap)

```go
		case '{':
			if depth == 0 {
				// Extract the identifier starting after '{'.
```

`extractPlaceholderNames` only scans for placeholders at `depth == 0`. Placeholders nested inside ICU plural or select branches (e.g., `{count, plural, one {item for {name}}}`) exist at `depth > 0` and are completely ignored. If a translator misspells or omits `{name}` in the translated branch, `i18n-report validate` will fail to detect the discrepancy.

Fix: Recursively parse or track placeholder extraction independent of branch depth, or strip ICU keywords to analyze all inner `{placeholder}` tags.

4. **Locale change callbacks array mutated during iteration** â€” `pkg/rancher-desktop/main/i18n.ts:107-109` (important, regression)

```typescript
      for (const callback of localeChangeCallbacks) {
        try {
          callback();
```

The `settings-update` listener iterates over `localeChangeCallbacks` directly. If any callback synchronously invokes its unregister function (calling `localeChangeCallbacks.splice`), the array length decreases mid-iteration, causing the loop to skip the subsequent callback.

Fix: Iterate over a shallow copy of the array: `for (const callback of [...localeChangeCallbacks]) {`.

5. **`invalidK8sVersionMainMessage` is never translated** â€” `pkg/rancher-desktop/backend/backendHelper.ts:188` (important, gap)

```typescript
    const invalidK8sVersionMainMessage = `Requested kubernetes version '${ currentConfigVersionString }' is not a supported version.`;
    // ...

      message,
      detail,
```

Fix: Replace the constant with `t('dialog.invalidK8sVersion.message', { version: currentConfigVersionString })` and add the corresponding keys to the translation files.

### Suggestions

6. **`main/i18n.ts` placeholder replacement is fragile against spacing** â€” `pkg/rancher-desktop/main/i18n.ts:31` (suggestion, gap)

```typescript
      msg = msg.replaceAll(`{${ name }}`, String(value));
```

This naive replacement strictly targets `{name}` without spaces. If translators add spaces for readability (e.g., `{ name }`) â€” which `intl-messageformat` gracefully handles in the renderer â€” the main process string replacement will silently fail and display the raw placeholder.

Fix: Use a regex replacement that allows whitespace: `msg.replace(new RegExp(\`{\\s*${name}\\s*}\`, 'g'), String(value))`.

### Design Observations

**Concerns**
- **Missing ICU MessageFormat support in main process** (in-scope): While the renderer uses `intl-messageformat` via the Vuex store to support plurals and selects, `main/i18n.ts` implements a naive `{variable}` replacement logic. If any main-process label requires pluralization, it will render raw ICU syntax (e.g., `{count, plural, ...}`). Consider using `@formatjs/intl` in `main/i18n.ts` for parity across processes, eliminating this structural divergence.
- **Unlinkable translation arguments for Page Descriptions** (future): `store/page.ts` accepts `titleKey` and `titleArgs`, but lacks `descriptionArgs`. If a dynamic page description needs interpolation, it cannot use the centralized `setHeader` i18n properties.

**Strengths**
- The Go-based `i18n-report` tool is well-structured and uses AST modification (`gopkg.in/yaml.v3`) for merging, properly preserving comments and formatting without relying on fragile regex replacements.
