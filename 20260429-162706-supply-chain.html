Retro: dependency types not explicitly covered ¶

R1. Verifier outside the ±30-line window paired with a downstream script ¶

Where: .github/workflows/smoke-test.yaml:48 (gh release download "$RELEASE_TAG" --pattern '*.dmg' --pattern '*.dmg.sha512sum' …) — the verification loop lives in .github/workflows/smoke-test/smoke-test.sh:50 (for checksum in *.sha512sum; do … sha512sum --check --quiet --strict "$checksum"; done), invoked many steps later from a different file.

Why missed: Phase 2's inline-download check scans ±30 lines in the same file. Cross-file verification is invisible to it.

Proposed change: Extend the inline-downloads check to track *.sha*sum artifact patterns alongside the primary asset; when both are downloaded together (same gh release download or curl invocation), record a "deferred-verifier" hint in inventory.json so Phase 2 can confirm the sister script actually consumes the sums. Alternatively, allow # scan:audited verified-by=<path>:<line> annotations next to the download, then look up the cited line and enforce it carries the verification primitive. The latter is simpler and works for any cross-file pattern.

R2. PowerShell `Start-Job -ScriptBlock { … IWR … | iex }` block scope ¶

Where: scripts/windows-setup.ps1:46-50 — the IWR | iex lives inside a Start-Job -ScriptBlock { … } block, which the script correctly flagged. But the self-fetch finding (raw.githubusercontent.com/.../main/...) at line 74 also lives in a top-level if (! (Get-Command wsl …)) { … } block. The script flags both (good), but the bullet under language_source_fetches[] for line 76 is a near-duplicate of the self_fetch_unpinned[] entry for line 74 (same URL, two-line offset).

Why missed: Two detectors firing on overlapping byte ranges (URL extraction + PowerShell Invoke-WebRequest primitive) produce duplicate-looking entries in different sections.

Proposed change: When self_fetch_unpinned[] already covers a URL, suppress matching language_source_fetches[] entries within ±5 lines whose primitive is the language-level wrapper for the same fetch. The two findings are the same threat described twice.

R3. `gpg --dearmor` of an unpinned upstream key ¶

Where: .github/workflows/smoke-test/install-from-repo.sh:21-23 — curl -s …Release.key | gpg --dearmor > …keyring.gpg. The script's APT-key recipe already exists in the wiki (gpg --show-keys against a hardcoded fingerprint), but gpg --dearmor without a sibling gpg --show-keys looks like a verifier from a regex's perspective ("gpg" is the right primitive) when it actually only reformats the key bytes.

Why missed: The verifier-detection regex matches any gpg invocation in the ±30-line window; gpg --dearmor is non-verifying.

Proposed change: Tighten the GPG verifier regex to require gpg --verify, gpg --show-keys paired with a hardcoded [0-9A-F]{40} fingerprint constant in the same window, or rpm --import paired with a fingerprint check. Plain gpg --dearmor should not satisfy the verifier check.

R4. Custom dependency manifest with documented no-op delegates ¶

Where: pkg/rancher-desktop/assets/dependencies.yaml — three entries (golangci-lint, check-spelling, spinKubePlugin) carry empty checksums: {} blocks because their download() methods return Promise.resolve() and delegate to a separate script. Phase 2 already calls this out as "track the delegate down," but the mechanical inventory cannot tell whether the delegate verifies its bytes.

Why missed: inventory.py records empty_checksum_blocks but does not link to the delegate's source file. Phase 2 had to read scripts/dependencies/tools.ts and grep for Promise.resolve to find the delegate path each time.

Proposed change: When an entry's download() body matches return Promise.resolve() or return Promise.resolve(undefined), scan the comment immediately preceding it for a reference to a sibling script (scripts/spelling.sh, resources/setup-spin, etc.) and surface the path under the manifest entry in inventory.json. Phase 2 then knows exactly which file to apply the inline-download rules to. Alternatively, normalize the convention: every empty-checksums entry must have a scan:delegate=<path> comment in the manifest itself, parsed at scan time.
