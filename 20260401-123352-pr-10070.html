Executive Summary ¶

This PR fixes the broken no-proxy list UI (a missing Array.isArray guard from the Vue 3 migration) and extends no-proxy support to domain names and wildcard patterns via moproxy's --policy routing. The implementation separates IP/CIDR entries (iptables) from domain entries (policy.rules), adds comprehensive input validation in the settings validator, and correctly switches from SIGHUP reload to stop+start because moproxy does not reload its policy file on SIGHUP.

All three agents approve the change. No critical or important issues were found. Three suggestions address minor gaps: inconsistent || true guards on ip6tables loop commands, and two duplicate-detection edge cases in the validator.

Suggestions ¶

1. ip6tables loop commands lack || true guards (suggestion, regression) Claude Gemini ¶

The script guards standalone ip6tables commands with || true throughout (lines 95, 101, 102, 114, 140) but omits the guard inside the loops in add_noproxy_rules (line 65) and enable_redirection_to_moproxy_chain (line 76). If create_moproxy_chain fails for ip6tables (tolerated at line 95), these loop commands attempt operations on a nonexistent chain, producing stderr noise in the service log.

pkg/rancher-desktop/assets/scripts/moproxy.initd
add_noproxy_rules() {
	for i in ${noproxy_rules//,/ }
	do
		add_accept_rule iptables "$i"
	done
	for i in ${noproxy6_rules//,/ }
	do
		add_accept_rule ip6tables "$i"
	done
	return 0
}

The same issue exists in enable_redirection_to_moproxy_chain, where both iptables and ip6tables are iterated in a single loop:

pkg/rancher-desktop/assets/scripts/moproxy.initd
enable_redirection_to_moproxy_chain() {
	local chain="$1"
	local cmd
	for cmd in iptables ip6tables; do
		if ! redirect_to_moproxy_chain "$cmd" check "$chain" &> /dev/null; then
			redirect_to_moproxy_chain "$cmd" append "$chain"
		fi
	done
	return 0
}

For context, create_moproxy_chain already guards the ip6tables call:

pkg/rancher-desktop/assets/scripts/moproxy.initd
create_moproxy_chain() {
	iptables --table nat --new MOPROXY
	ip6tables --table nat --new MOPROXY || true
}

Gemini rated this IMPORTANT, arguing it could abort the script under set -e. OpenRC init scripts do not run under set -e, and both functions end with return 0, so the script continues. Modern WSL2 kernels support ip6tables, making the failure path unlikely. Rated SUGGESTION for practical likelihood.

Fix:

 for i in ${noproxy6_rules//,/ }
 do
-    add_accept_rule ip6tables "$i"
+    add_accept_rule ip6tables "$i" || true
 done
2. Semantically equivalent noproxy entries bypass duplicate detection (suggestion, gap) Claude ¶

findDuplicates compares raw strings. *.example.com, .example.com, and example.com all produce identical moproxy policy rules via stripNoproxyPrefix in wsl.ts:794, but pass as three separate entries. A user could add all three, producing redundant rules. No functional harm — just confusing.

pkg/rancher-desktop/main/commandServer/settingsValidator.ts
  protected checkNoproxyList<S>(mergedSettings: S, currentValue: string[], desiredValue: string[], errors: string[], fqname: string): boolean {
    if (!Array.isArray(desiredValue) || desiredValue.some(s => typeof (s) !== 'string')) {
      errors.push(this.invalidSettingMessage(fqname, desiredValue));

      return false;
    }
    const duplicateValues = this.findDuplicates(desiredValue);

    if (duplicateValues.length > 0) {
      duplicateValues.sort(Intl.Collator().compare);
      errors.push(`field "${ fqname }" has duplicate entries: "${ duplicateValues.join('", "') }"`);

      return false;
    }
    const invalidEntries = desiredValue.filter(entry => !SettingsValidator.isIPAddressOrCIDR(entry) && !SettingsValidator.isDomainName(entry));

Fix: Normalize domain entries before duplicate comparison. The added complexity may not justify the low practical risk.

3. Domain duplicate detection is case-sensitive (suggestion, gap) Claude ¶

DNS names are case-insensitive, but Example.COM and example.com pass as separate entries because findDuplicates uses exact string comparison. Low practical risk since the UI normalizes input, but rdctl could submit mixed-case entries.

Same code location as Finding 2 (settingsValidator.ts:720).

Fix: Case-normalize domain entries before comparing. Same trade-off as finding 2.

Design Observations ¶

Concerns ¶
  • The no-proxy parsing and classification logic now lives in two places: validation in settingsValidator.ts and runtime routing in wsl.ts. They are aligned today, but if the accepted syntax grows again, extracting a shared classifier would reduce drift risk. (future) Codex
Strengths ¶
  • Clean separation of entry classification (wsl.ts), validation (settingsValidator.ts), and routing (moproxy.initd + policy.rules). Each layer has a clear responsibility. Claude Gemini
  • Validation lives in settingsValidator.ts, protecting both the UI path and rdctl/API callers instead of relying on renderer-only checks. Codex Claude
  • The backend correctly uses stop+start instead of SIGHUP reload, matching the documented moproxy limitation. The reload() function still sends SIGHUP for the proxy list, and the inline comment at moproxy.initd:135-136 explains the constraint. Codex Claude
  • isIPAddressOrCIDR delegates to Node's net.isIP() instead of reimplementing IP validation with regex — simpler and more correct. Claude Gemini
  • The MOPROXY_NOPROXY- (no colon) vs MOPROXY_NOPROXY6:- (with colon) distinction at moproxy.initd:29-30 ensures IPv4 rules are genuinely empty when all entries are domains, while IPv6 retains safe defaults. The inline comments explain the rationale. Claude

Testing Assessment ¶

The 202 lines of new tests in settingsValidator.spec.ts cover validation thoroughly: valid entries (IPv4, IPv6, CIDR, domains, wildcards, leading-dot, subdomains, single-label hostnames, mixed types), invalid entries (bare wildcards, hyphen-prefixed domains, out-of-range CIDR prefixes, whitespace), and edge cases (empty list, non-array input, boundary lengths at 253/254 and 63/64 characters, platform gating). The stripNoproxyPrefix tests in networks.spec.ts cover all three prefix forms plus edge cases.

Untested scenarios, ranked by risk:

  1. The wsl.ts runtime split (IP/CIDR to environment variables, domains to policy.rules) has no automated test. A unit test that mocks this.writeFile would verify the classification logic and generated policy content.
  2. The OpenRC script's behavior when noproxy lists are empty or contain only domains has no direct test coverage. This is typical for shell scripts in the project.
  3. IPv4-mapped IPv6 addresses like ::ffff:192.168.1.1 — classified as IPv6 by net.isIP(), routed to ip6tables. Functionally correct but untested.

Documentation Assessment ¶

The command-api.yaml x-rd-usage update at line 759 and the en-us.yaml placeholder text at line 433 accurately describe the expanded entry types. Inline comments in moproxy.initd (lines 24-28, 135-136) and wsl.ts (lines 793, 1719-1720) explain non-obvious design decisions. No documentation gaps.

Acknowledged Limitations ¶

  1. moproxy.initd:135-136: "SIGHUP reloads the proxy server list only, not the --policy file. Domain noproxy rules require a full stop+start cycle." The backend uses stop+start for settings updates, so this limitation affects only manual rc-service moproxy reload invocations — rare in practice. Claude Codex Gemini
  2. moproxy.initd:27-28: "IPv6 uses :-: no UI adds IPv6 entries, so empty should fall back to sane defaults (loopback, ULA, link-local)." When a user clears the noproxy list entirely, IPv6 still retains default bypass rules. Intentional and documented. Gemini
