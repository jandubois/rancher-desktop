## Executive Summary

Round 2 of the userspace-forwarder rewrite. Round 1's critical and four important findings are resolved in `2725f96b0`; the design holds, and the implementation is materially simpler than #10280. This round surfaces three new important issues in the scanner's reconciliation loop: binding-shape changes after publish never reconcile, the publish/rollback path floods logs every 3 seconds on persistent failure, and unpublish's "removed port" Info log fires even when `tracker.Remove` returned an error.

All three reviewers converged on the binding-shape and retry-storm findings; Claude alone caught the unpublish log inconsistency. Codex raised one false positive (UDP outbound dial sockets cannot reach the scanner — they sit in `state=0x1`, not `UDPEstablished=0x7`), and one over-severe finding (the tap-IP filter dropping legitimate tap-IP listeners is real but requires a container to bind to the namespace-internal tap address, which no realistic engine does).

## Important Issues

I1. **`Tick` never reconciles binding-shape changes on published ports** — `src/go/guestagent/pkg/procnet/scanner_linux.go:109-115` [Claude Opus 4.7, Gemini 3.1 Pro] (important, gap)

```go
for port, bindings := range p.published {
    if _, ok := scanned[port]; ok {
        continue
    }
    p.unpublish(port, bindings)
    delete(p.published, port)
}
```

The loop checks key membership alone; it never compares `scanned[port]` against the stored `p.published[port]` bindings. A container restart with a new bind — typically `0.0.0.0:8009 → 127.0.0.1:8009` — keeps the port present, so `Tick` skips unpublish, the published map keeps the stale `0.0.0.0` binding, and loop 3 skips because the port is in published. The forwarder, which was never started for the wildcard, is also never started for the new loopback. Windows-side traffic to `bindIP:8009` finds no listener until the upstream truly vanishes. The reverse transition is normally blocked by `EADDRINUSE` against the forwarder's lingering `bindIP:8009`, so the realistic failure mode is wildcard-to-loopback.

Fix: when the key is present but bindings differ, run the full unpublish/republish cycle.

```diff
 for port, bindings := range p.published {
-    if _, ok := scanned[port]; ok {
+    if newBindings, ok := scanned[port]; ok && bindingsEqual(bindings, newBindings) {
         continue
     }
     p.unpublish(port, bindings)
     delete(p.published, port)
 }
```

I2. **`publish` rollback path floods logs every 3 seconds on persistent failure** — `src/go/guestagent/pkg/procnet/scanner_linux.go:143-176` [Claude Opus 4.7, Gemini 3.1 Pro] (important, gap)

```go
if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
    log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
    if removeErr := p.tracker.Remove(id); removeErr != nil {
        log.Errorf("/proc/net scanner rollback after tracker.Add failure for %s: %s", port, removeErr)
    }
    return err
}
```

When wsl-proxy or host-switch is persistently down, `tracker.Add` and the rollback `tracker.Remove` both fail every tick. Each failed port emits two Error lines per `scanInterval` (~3 s). With multiple ports stuck, the log fills quickly. #10280 specifically added an `addErrorLogged` map to demote follow-up failures to Debug; this redesign drops that surface. The error class is preserved, but the diagnostic guardrail is not.

Fix: keep a per-port "first failure logged" flag and demote subsequent occurrences to `Debugf`; reset on success. See S2 below — `EADDRINUSE` on the forwarder is the most likely persistent trigger.

I3. **`unpublish` Info log fires even when `tracker.Remove` failed** — `src/go/guestagent/pkg/procnet/scanner_linux.go:179-184` [Claude Opus 4.7] (important, gap)

```go
if err := p.tracker.Remove(id); err != nil {
    log.Errorf("/proc/net scanner failed to remove %s: %s", port, err)
}
log.Infof("/proc/net scanner removed port: %s -> %+v", port, bindings)
```

The Info log is unconditional. An operator scanning logs sees "removed port" and concludes wsl-proxy was notified, even when `wslProxyForwarder.Send` returned an error (per the `APITracker.Remove` contract — see `tracker/apitracker.go:189`). The contradiction sits on consecutive log lines. The pre-PR code had the same shape (Info before Remove, unconditional), so this is a pre-existing observability gap, not a regression — but the Round 1 refactor was the natural point to fix it.

Fix:

```diff
-if err := p.tracker.Remove(id); err != nil {
-    log.Errorf("/proc/net scanner failed to remove %s: %s", port, err)
-}
-log.Infof("/proc/net scanner removed port: %s -> %+v", port, bindings)
+if err := p.tracker.Remove(id); err != nil {
+    log.Errorf("/proc/net scanner failed to remove %s: %s", port, err)
+} else {
+    log.Infof("/proc/net scanner removed port: %s -> %+v", port, bindings)
+}
```

## Suggestions

S1. **`entriesToPortMap` filter drops legitimate tap-IP listeners** — `src/go/guestagent/pkg/procnet/scanner_linux.go:216-227` [Codex GPT 5.5, Gemini 3.1 Pro] (suggestion, regression)

```go
for _, entry := range entries {
    if entry.IP.Equal(p.bindIP) {
        continue
    }
```

The filter targets the forwarder's own listener, but the predicate is address-only. A container that explicitly binds to the tap-interface IP (`192.168.127.2`) instead of `0.0.0.0` or `127.0.0.1` is dropped before reaching the publish loop, even when no forwarder owns that proto/port. The pre-PR code published the port. Realistic engines (nerdctl, docker) do not bind to the tap IP — it is a namespace-internal artifact — so the practical surface is small. Worth tightening when `procnettcp` exposes inode-level ownership; until then, document the trade-off in the package comment.

Fix: replace the address-only filter with an ownership check against the forwarder's `tcp`/`udp` maps, or document the limitation.

S2. **`forwarder.Add` retries every tick on `EADDRINUSE` against an unrelated bind** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:77-79` [Claude Opus 4.7] (suggestion, regression)

```go
lis, err := net.ListenTCP("tcp", &net.TCPAddr{IP: f.bindIP, Port: int(port)})
if err != nil {
    return fmt.Errorf("listen %s: %w", k, err)
}
```

If any other process inside the engine namespace already holds `bindIP:<port>` — most realistically a `--network=host` container with both a `127.0.0.1` and a `0.0.0.0` listener on the same port — `ListenTCP` returns `EADDRINUSE`. `publish` rolls back `tracker.Add` and retries every tick indefinitely. Combined with I2, this is the most likely source of a real retry storm. The DNAT path had no such failure mode.

Fix: detect `EADDRINUSE` in `loopbackForwarder.Add`, return a typed error the scanner can recognise, and skip the forwarder for that port (the conflicting wildcard listener already exposes it via eth0).

S3. **`pipeTCP` drain deadline applies only to reads, not writes** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:205-208` [Codex GPT 5.5] (suggestion, regression)

```go
<-done
drainDeadline := time.Now().Add(halfCloseDrainTimeout)
_ = in.SetReadDeadline(drainDeadline)
_ = out.SetReadDeadline(drainDeadline)
<-done
```

The drain unblocks the still-running copy when its peer goes silent on the read side, but a copy stuck in `Write` (TCP backpressure from a slow consumer) is not bounded. The 30 s window is short and the upstream is local loopback, so the practical likelihood is low; the fix is also small.

Fix:

```diff
 _ = in.SetReadDeadline(drainDeadline)
 _ = out.SetReadDeadline(drainDeadline)
+_ = in.SetWriteDeadline(drainDeadline)
+_ = out.SetWriteDeadline(drainDeadline)
```

S4. **No end-to-end test wires a real `loopbackForwarder` through `Tick`** — `src/go/guestagent/pkg/procnet/scanner_linux_test.go` [Claude Opus 4.7] (suggestion, gap)

The forwarder tests bind on `127.0.0.99` and call `Add`/`Remove` directly; the scanner tests use a `fakeForwarder`. The full path (scan snapshot → `Tick` → `publish` → real forwarder → real upstream listener → dial round-trip) has no Go-side coverage. The bats test in #10262/#10280 covers it on Windows but runs out of the unit-test loop. The components are individually well-tested, so this is a coverage seam, not an active gap.

Fix: add a `_linux` integration test that constructs `ProcNetScanner` with a real `newLoopbackForwarder(bindIP)`, opens an upstream on `127.0.0.1:<ephemeral>`, drives `Tick`, dials through a `127.0.0.x` alias, and asserts payload round-trip. Optionally guard with `testing.Short()`.

S5. **`tcp6`/`udp6` listeners are silently ignored** — `src/go/guestagent/pkg/procnet/scanner_linux.go:229-241` [Claude Opus 4.7] (suggestion, gap)

```go
switch entry.Kind {
case procnettcp.TCP:
    if entry.State == procnettcp.TCPListen {
        return addEntryToPortMap(entry, portMap)
    }
case procnettcp.UDP:
    if entry.State == procnettcp.UDPEstablished {
        return addEntryToPortMap(entry, portMap)
    }
}
```

The switch handles `TCP` and `UDP` only. A `--network=host` container that listens exclusively on `[::1]:<port>` is invisible to the scanner. This is pre-existing behaviour, but the new package doc (which advertises the userspace forwarder as the cure for the iptables minefield) leaves the IPv6 gap undocumented. The user-visible symptom is "host-network IPv6-only listener not reachable from Windows."

Fix: extend the switch to handle `TCP6`/`UDP6` (dialing `::1`), or document the limitation in the package comment.

S6. **UDP forwarder dial captures the per-`Add` ctx for indefinite-lifetime flows** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:93-95` [Claude Opus 4.7] (suggestion, enhancement)

```go
proxy, err := forwarder.NewUDPProxy(pc, func() (net.Conn, error) {
    return f.dialer.DialContext(ctx, "udp", target)
})
```

`UDPProxy` invokes the dial closure for every new client flow, including flows that arrive long after `Add` returned. The captured `ctx` is the caller's; in production this is `p.ctx` (scanner lifetime), so the behaviour is correct today. A future caller passing a request-scoped or per-tick ctx would silently break new-flow dialing because the ctx would already be cancelled.

Fix: thread a `ctx` field through `loopbackForwarder` (set in `newLoopbackForwarder` from a context derived in `main`, cancelled in `Close`) and use that in the dial closure, decoupling the dial lifetime from any caller's `Add` ctx.

S7. **Exported scanner symbols lack godoc** — `src/go/guestagent/pkg/procnet/scanner_linux.go:54, 65, 83` [Claude Opus 4.7] (suggestion, gap)

`ProcNetScanner` (struct), `NewProcNetScanner` (constructor), and `ForwardPorts` (driver loop) have no godoc comments. `Tick` is well-commented; the others should match. The new `bindIP` parameter on `NewProcNetScanner` is undocumented — a future caller would need to read the package doc to learn what address to pass.

Suggested:

```go
// NewProcNetScanner constructs a /proc/net scanner that publishes
// loopback listeners through tracker t and bridges traffic from
// bindIP (the namespace's tap interface IP) into 127.0.0.1 via a
// userspace forwarder. scanInterval controls the poll cadence.
```

## Design Observations

### Concerns

- **The stability gate covers presence but not shape** *(in-scope)* [Claude Opus 4.7, Gemini 3.1 Pro]

  I1 covers the symptom. The deeper observation: the two-scan gate is now the only correctness barrier on the publish side, and it answers only "is the port still here." Bindings are stored once at promotion and never re-checked. A small structural addition — compare bindings on every tick, or store a generation counter and re-publish on mismatch — closes the class.

- **`publish` collapses transient and permanent failures into one retry loop** *(in-scope)* [Claude Opus 4.7]

  I2 and S2 describe two failure modes — wsl-proxy down and `EADDRINUSE` on the forwarder bind — that share one rollback-then-retry path with no rate limit. #10280's `addErrorLogged` was the explicit guard against this class. The redesign drops the guard along with the state machine; the underlying failure semantics need it as much as before.

- **Scanner and forwarder lack a Go-side integration seam** *(future)* [Claude Opus 4.7]

  S4 covers the symptom. The structural point: the redesign trades a 665-line state machine with table-driven unit tests for two cleanly-separated components with focused unit tests on each side. The seam between them — the actual scan → publish → forward → upstream loop — only meets in production. The bats test is the existing integration check; one Go-level integration test would catch a future seam regression earlier.

### Strengths

- **Net simplification of ~1850 lines.** Eliminating `portStateTracker`, the engine-chain probing helpers, and the `route_localnet` sysctl in exchange for a 209-line userspace forwarder is the central design win. The justification in `loopback_forwarder_linux.go:9-25` reads cleanly.
- **`Tick`'s three-phase structure** (unpublish-vanished → promote-pending → rebuild-pending) makes the invariants legible and unit-testable.
- **`bindIP` self-filter plus its dedicated test** (`TestEntriesToPortMapSkipsForwarderBindIP`) is the minimal fix for Round 1's C1 and has a regression pin.
- **TCP `accept` loop uses bounded exponential backoff** rather than tight-looping on transient `EMFILE`/`ENOBUFS`. The pattern proactively addresses a class of CPU-pegging bugs.
- **Half-close + drain deadline** (Round 1's I4 fix) correctly preserves the client-CloseWrite → upstream-response shape that the DNAT path handled at the IP layer.

# Deep Review — PR #10357: procnet userspace forwarder

## Executive Summary

This change replaces the rogue `PREROUTING DNAT` rule with a userspace TCP/UDP forwarder bound on the namespace tap IP, eliminating the iptables-coordination minefield that #10262 and #10280 chased. The design is materially simpler and the equivalence reasoning is sound. The main concerns are around partial-failure recovery in `publish` (a binding-shape change after publishing is never reconciled), a retry-storm potential when the wsl-proxy is down (no log-once), and an end-to-end coverage gap: no test drives a real `loopbackForwarder` through the scanner's `Tick` loop.

## Findings

### Critical Issues

None.

### Important Issues

**I1. Binding-shape changes after publish are never reconciled** — `src/go/guestagent/pkg/procnet/scanner_linux.go:109-127`

```go
for port, bindings := range p.published {
    if _, ok := scanned[port]; ok {
        continue
    }
    p.unpublish(port, bindings)
    delete(p.published, port)
}
```

`Tick` only reacts when a published port disappears from `scanned`; it never compares the new `scanned[port]` bindings against the stored `published[port]` bindings. If an upstream listener changes shape — e.g. tick N saw `127.0.0.1:8009` (loopback forwarder created), tick N+1 sees `0.0.0.0:8009` (additional wildcard binding) — the new wildcard is never exposed via `tracker.Add`, and the published bindings (used by `unpublish` later for `Unexpose`) stay stale. The scanner test `TestBindingShapeChangePublishesCurrent` only covers the case where pending (not yet published) changes shape; the published case has no coverage. (severity: important, classification: gap)

Fix: when `port` is still in `scanned` but the bindings slice differs, treat it as an unpublish-then-republish (or diff the bindings and Add/Remove the delta).

```diff
 for port, bindings := range p.published {
-   if _, ok := scanned[port]; ok {
-       continue
+   if newBindings, ok := scanned[port]; ok {
+       if bindingsEqual(bindings, newBindings) {
+           continue
+       }
+       // shape changed: full unpublish, then fall through to pending
+       p.unpublish(port, bindings)
+       delete(p.published, port)
+       p.pending[port] = newBindings
+       continue
    }
    p.unpublish(port, bindings)
    delete(p.published, port)
 }
```

**I2. publish retry-storm when wsl-proxy is persistently down** — `src/go/guestagent/pkg/procnet/scanner_linux.go:143-176`

```go
if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
    log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
    if removeErr := p.tracker.Remove(id); removeErr != nil {
```

#10280 explicitly added a log-once behavior (`addErrorLogged`) to prevent a 3-second retry-storm flood when host-switch or wsl-proxy stayed down. This PR drops that surface entirely. With the new design, every tick that finds a still-down wsl-proxy emits two `Errorf` lines (Add failure + rollback Send failure), repeating at `scanInterval` (~3 s) for the life of every failed port. **I3. `unpublish` "removed port" Info log fires even when `tracker.Remove` failed** — `src/go/guestagent/pkg/procnet/scanner_linux.go:179-198`

## Design Observations

### Concerns

- **(in-scope) Stability gate is binary; binding-shape changes are not modelled.** The two-scan gate is now the only correctness barrier on the publish side, and it covers only the "is the port present" axis. **I1. Missing log-once guard causes 3-second retry storms** — `src/go/guestagent/pkg/procnet/scanner_linux.go:121-125`
(important, regression)

```go
	if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
		log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
		if removeErr := p.tracker.Remove(id); removeErr != nil {
```

Previous iterations of this logic (#10280) included a local state map (`addErrorLogged`) to prevent log floods on persistent `tracker.Add` failures. By removing the state machine and using unconditional `log.Errorf` in `publish`, any persistent failure (e.g., `wsl-proxy` being down, or a proxy-already-running collision) results in a log flood every `scanInterval` (3 seconds). Because failed ports remain unpromoted, `Tick` leaves them in `p.pending` and retries them infinitely.

**Fix**: Reintroduce a local state map (e.g., `addErrorLogged map[nat.Port]bool`) to downgrade subsequent identical errors for a port to `Debugf`, or apply an exponential retry backoff within `Tick`.

**I2. `Tick` ignores binding IP changes for already-published ports** — `src/go/guestagent/pkg/procnet/scanner_linux.go:94-100`
(important, gap)

```go
	for port, bindings := range p.published {
		if _, ok := scanned[port]; ok {
			continue
		}
		p.unpublish(port, bindings)
		delete(p.published, port)
	}
```

`Tick` keys exclusively on the `nat.Port` (protocol and port number) to determine if a port is still present. If a container restarts and changes its bind address (e.g., from `127.0.0.1` to `0.0.0.0`) on the same port, `scanned[port]` remains truthy. `Tick` skips `unpublish` and later skips `publish` because the port is already in `p.published`. Consequently, `p.published` retains the old bindings. If it changed to `0.0.0.0`, the loopback forwarder leaks indefinitely; if it changed to `127.0.0.1`, the forwarder is never started, rendering the port unreachable.

**Fix**: Compare the actual bindings in `scanned[port]` against `p.published[port]`.

```diff
 	for port, bindings := range p.published {
-		if _, ok := scanned[port]; ok {
+		if newBindings, ok := scanned[port]; ok && equalBindings(bindings, newBindings) {
 			continue
 		}
 		p.unpublish(port, bindings)
```
*(You will need to implement a simple `equalBindings` helper to compare the `HostIP` and `HostPort` fields).*
