Executive Summary ¶

This branch adds a comprehensive i18n system to Rancher Desktop: a Go-based CLI tool (i18n-report) for translation management, main-process and renderer-side locale support with settings integration, and translation files for English, German, and Simplified Chinese. The implementation is well-architected with clean separation between main process (simple {variable} interpolation) and renderer (ICU MessageFormat via intl-messageformat), proper race-condition handling for locale switching, and a feature gate for gradual rollout. Four review passes across three agents found no critical issues, one important stale-comment issue, and 20 suggestions concentrated in the Go tooling's edge cases. The code is ready to merge after fixing the misleading comment.

S1. stripYAMLQuotes double-quote unescaping order Claude ¶

Replacing \\ before \" corrupts strings containing both sequences. Input "foo\\\"bar" yields foo"bar instead of foo\"bar. Practical impact is negligible — translation strings never contain literal backslash-quote sequences — but the fix is trivial.

src/go/i18n-report/yaml.go
// stripYAMLQuotes removes outer YAML quotes from a value string.
func stripYAMLQuotes(s string) string {
	if len(s) >= 2 && s[0] == '\'' && s[len(s)-1] == '\'' {
		inner := s[1 : len(s)-1]
		return strings.ReplaceAll(inner, "''", "'")
	}
	if len(s) >= 2 && s[0] == '"' && s[len(s)-1] == '"' {
		inner := s[1 : len(s)-1]
		inner = strings.ReplaceAll(inner, `\\`, `\`)
		inner = strings.ReplaceAll(inner, `\"`, `"`)
		return inner
	}
	return s
}

Design Observations ¶

Concerns ¶

Translated error messages in HTTP API responses (future) — The settings validator now produces translated error messages via t(). External tools calling the API would receive localized errors. The hasLockedFieldError flag handles the known parsing case, but the broader pattern of returning translated error strings in API responses may need revisiting when the feature gate is removed. Consider returning structured error codes alongside human-readable messages. Codex

Manifest is not the runtime source of truth (future) — main/i18n.ts, store/i18n.js, and settingsValidator.ts all discover locales by enumerating YAML files, while manifest status is only enforced by the Go tooling. Making meta/locales.yaml authoritative at runtime would prevent an accidentally-added partial locale file from becoming selectable. Codex

Locale-loading state machine duplicated (in-scope) — The locale-switching logic exists in both main/i18n.ts and store/i18n.js with subtly different race-condition handling. A shared "last request wins" pattern would remove a class of ordering bugs. Codex

Strengths ¶

  • Two-tier i18n architecture — Main process uses lightweight {variable} interpolation; renderer uses full ICU MessageFormat. The right tradeoff for their respective needs. Claude
  • Race-condition handling in i18n.ts — The pendingLocale mechanism correctly handles most rapid locale switches. The callback array is copied before iteration to handle self-unregistering callbacks. Claude
  • Feature gate design — Hiding the locale selector when locale === 'none' provides a clean rollout path enforced at validation, UI, and both i18n modules. Claude
  • AST-level YAML manipulation — Using gopkg.in/yaml.v3 node trees in nodeSetLeaf and removeKeyFromNode preserves whitespace, comments, and structural integrity during merges. Gemini
  • Comprehensive Go tooling — The i18n-report CLI covers the full translation lifecycle with 14 subcommands, significantly reducing the chance of translation regressions. Claude
  • Defensive intlCache invalidation — The store clears the entire formatter cache before setting the new locale, preventing stale cached results. Claude

Testing Assessment ¶

Test coverage is solid overall. The Go tool has comprehensive test suites (3,000+ lines across 12 test files). The settings validator has new test cases for locale handling, and the UpdateStatus component test was improved to use real translations instead of mocks.

Untested scenarios ranked by risk:

  1. Main-process locale switching — initMainI18n(), loadLocale(), and the onLocaleChange callback mechanism have no direct unit tests. The rapid A→B→A race guard and callback error handling are complex enough to warrant coverage.
  2. Tray menu label refresh — The onLocaleChange → updateMenu() path is untested.
  3. Renderer i18n store fallback — The fallback to en-us when locale loading fails (store/i18n.js lines 222–228) is untested.
  4. handleNavigateClick — The new shared utility has no unit tests despite being used by two components.
  5. E2E tests with non-English locale — All e2e/bats tests run with default English locale; none exercise localized error messages through rdctl.

Acknowledged Limitations ¶

  1. store/i18n.js:158-165 — "Acceptable overhead with a small number of locales; revisit if locale count grows significantly." Currently 3 locales; would need attention at ~10+.
  2. main/i18n.ts:64-66 — "Placeholders are replaced iteratively; in theory a value containing '{other}' could match a later placeholder, but no current usage does."
  3. main/i18n.ts:126-128 — "Rapid A->B->A switches may be dropped because the final locale equals currentLocale. Acceptable: locales are pre-loaded, so the race window is negligible." See S14 for a more precise characterization.
  4. main/deploymentProfiles.ts:13-15 — "t() uses English fallback here because deployment profiles are read during early startup, before initMainI18n() has loaded the user's locale."
  5. report_validate.go:379 — "Known limitation: does not handle ICU single-quote escaping (e.g., '{')."
  6. command-api.yaml:600-604 — Hardcoded locale enum must be kept in sync with meta/locales.yaml and settingsValidator.ts. The i18n-report manifest cross-validates, but CI integration is not yet done.
  7. Known scanner gaps — The i18n-report scanner does not detect showErrorBox calls or port forwarding errors in backend/kube/client.ts.
