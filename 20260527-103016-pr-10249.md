## Important Issues

I1. **Existing dashboard window keeps the old Steve port after restart** — `pkg/rancher-desktop/window/dashboard.ts:9-14` (important, regression) [Codex GPT 5.5]

```ts
let window = getDashboardWindow();
if (restoreWindow(window)) {
  return window;
}
const { port } = Steve.getInstance();
if (!port) {
  return;
}
```

`openDashboard()` restores an existing dashboard window at line 12 before reading the current Steve port at line 16 or building the new URL at line 29. On `K8s.State.STOPPING`, `background.ts:1285` calls `Steve.getInstance().stop()`, which kills the child and clears `#port` (`steve.ts:240-246`). When Kubernetes restarts, Steve picks a fresh port; clicking the dashboard button returns the old window still pointed at the dead port — the dashboard webapp is a separate `BrowserWindow` and does not observe the renderer's `steve-port` IPC. The Round 1 fix gates the button on `port > 0`, but the restore path bypasses that check.

Fix: read the current port first; reload stale dashboard windows on restore.

```diff
+  const { port } = Steve.getInstance();
+  if (!port) return;
+  const url = `http://127.0.0.1:${ port }/c/local/explorer`;
+  const window = getDashboardWindow();
+  if (window && window.webContents.getURL() !== url) {
+    window.loadURL(url);
+  }
   if (restoreWindow(window)) return window;
```

S1. **`server.listen` Promise wrappers swallow `'error'` events** — `e2e/extensions.e2e.spec.ts:295,449` (suggestion, regression) [Gemini 3.1 Pro]

```ts
await new Promise<void>(resolve => server.listen(0, '127.0.0.1', resolve));
```

`server.listen()` emits `'error'` asynchronously on failure (e.g., `EADDRINUSE` in the very rare case `0` cannot bind, or a permission error) and never invokes the listening callback in that path. The Promise then hangs until the test's outer timeout. Vanishingly unlikely with port `0`, but the fix is one line and the same pattern appears in both new test bodies.

Fix:

```diff
- await new Promise<void>(resolve => server.listen(0, '127.0.0.1', resolve));
+ await new Promise<void>((resolve, reject) => {
+   server.once('error', reject);
+   server.listen(0, '127.0.0.1', resolve);
+ });
```

## Design Observations

**Concerns**

- **Single-channel request/response over IPC for the steve port (future)** [Claude Opus 4.7]. `pkg/rancher-desktop/store/steve.ts:24-32` and `pkg/rancher-desktop/backend/steve.ts:30-35` use one `'steve-port'` channel for both directions: the renderer "asks" by sending it with no args, the main process broadcasts the current port back on the same channel. The contract — "send-with-no-args means request; reply arrives via the broadcast" — is implicit and uncommented. Two channels (`steve-port-get` for the request, `steve-port` for the broadcast) would make the direction explicit and match the existing `IpcMainEvents` / `IpcRendererEvents` type split.
- **Dashboard window lifecycle is independent of Steve lifecycle (in-scope)** [Codex GPT 5.5]. I1 is the symptom of a missing rule: nothing closes or reloads the dashboard `BrowserWindow` when Steve stops. A `Steve.on('stop')`-style hook that closes (or marks-stale) the dashboard window would close the class of bugs that the per-call URL-check fix only patches one site of.

**Strengths**

- Retiring the in-process Express server and the bespoke `proxyUtils.ts` removes a known-difficult `onProxyReqWs` path and a non-trivial chunk of IPC surface [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro].
- The Round 1 fix package — `pendingStart` guard, port-not-stored-until-ready, dashboard button gated on `port > 0`, e2e `waitForEvent` registered before click — addresses every Round 1 important finding with a clean, narrowly-scoped change [Claude Opus 4.7, Codex GPT 5.5].
- `Latch<number>` + port-on-stdout + Vuex broadcast is a clean, debuggable replacement for the prior fixed-port `/api/steve-port` HTTP route [Claude Opus 4.7, Gemini 3.1 Pro].

## Review Process Notes

**Skill improvements**

- When a previous round's fix replaces an explicit enumeration with a filter (regex, predicate, allow-list expressed declaratively), check whether the filter still enforces the *completeness* the enumeration did. A filter accepts what matches; an enumeration asserts what must be present. Add to the Correctness dimension's "Enumeration symmetry" pattern: a regression that converts "iterate required items, check each exists" into "iterate available items, keep matches" silently weakens the required-set guarantee even when every individual filter is correct.

- When a PR introduces a long-lived window or surface that depends on a backend resource (port, socket, certificate) whose value can change across a stop/start cycle, enumerate every read of that resource across the window's lifecycle — including the restore/reuse path. The bug shape is reliable: the cold-open path picks up the new value, the restore path replays the cached one. Add to the Correctness dimension: "When a singleton's lifecycle is dynamic (start/stop cycle resets a value), trace every read across reuse paths — focus, restore, re-show — not just the create path."

**Repo context updates**

- Add to `deep-review-context.md`: "When reviewing changes that affect the dashboard `BrowserWindow` or `Steve` lifecycle, verify that the dashboard window is reloaded or closed on every Steve stop/restart. The dashboard webapp runs in its own `BrowserWindow` that does NOT observe the renderer's Vuex `steve-port` IPC — the cold-open path reads the current port, but the `restoreWindow` path in `pkg/rancher-desktop/window/dashboard.ts` reuses the cached URL. Treat the dashboard window as a sink for stop-cycle invalidation, not a passive observer of Steve state."

**I2. `pendingStart` reset can authorize a duplicate Steve spawn during overlapping calls** — `pkg/rancher-desktop/backend/steve.ts:55-64`

## Design Observations

### Concerns

**Single-channel request/response over IPC for port discovery** (future, in-scope) — `pkg/rancher-desktop/store/steve.ts:24-32` and `pkg/rancher-desktop/backend/steve.ts:30-35`. The renderer "requests" the port by sending the same `steve-port` channel it later receives broadcasts on. Two channels (`steve-port` for broadcasts, `steve-port-get` for the request) would be clearer and would match the existing IPC-typing pattern of one channel = one message direction. The current code requires readers to follow the convention "send with no args = ask; main responds via the broadcast channel" — that contract is implicit, with no comment.

### Strengths

- `Latch<number>` plus the port-on-stdout protocol gives a clean, debuggable start path. Replacing the prior HTTPS-with-cert-bypass health check with a plain HTTP probe simplifies the certificate-error handler considerably.
- Hoisting the Steve port into a Vuex store (rather than fetching it via a one-off HTTP `/api/steve-port` route as the old dashboard server did) lets the dashboard button react to Steve restarts without polling.
- `rddepman`'s `RD_DEPMAN_LOCAL_CHANGES` escape hatch is the minimal change needed to test the workflow locally and is appropriately env-var gated.
