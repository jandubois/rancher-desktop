## Executive Summary

After verification, three Important findings remain, all sharing one theme: when an engine chain takes over a port, the code leaves prior procnet ownership unreconciled — a stale iptables rule from a prior session, a `tracker.Remove` error class, or a partial-Add `portStorage` entry. The bug class the PR targets (a stale `PREROUTING` DNAT shadowing CNI) can therefore reappear through three rare-but-real cleanup gaps.

Gemini found nothing new; Codex raised one Important; Claude raised three Important plus seven Suggestions. The state-machine extraction itself remains solid and well-tested. The gaps live at the boundary between the state machine and the surviving iptables and tracker state it inherits.

## Important Issues

I1. **Stale procnet rule from a prior session shadows the engine chain** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:202-207` [Claude Opus 4.7]

```go
if managed {
    pst.added[port] = portState{bindings: bindings, delegated: true}
    delete(pst.addErrorLogged, port)
    return
}
```

`portStateTracker` starts every `ForwardPorts` call with an empty `added` map (`scanner_linux.go:105`), but procnet's `PREROUTING` DNAT rules in the WSL2 distro persist across guestagent restarts (only a full distro shutdown clears them). When the engine republishes a port that procnet wrote a rule for in the previous session, the delegation branch above marks the port `delegated` and leaves the leftover rule in place. On iptables-nft (the default on supported WSL2 kernels) the rule still terminates `PREROUTING` and shadows `CNI-HOSTPORT-DNAT`/`DOCKER`, recreating the exact bug this PR fixes until the next distro reboot.

The acknowledged in-session leak (a Delete failure that has no retry, `scanner_linux.go:289`) feeds the same gap whenever the engine later republishes the port.

Fix: at scanner startup, enumerate procnet-shaped DNAT rules in `PREROUTING`, then reconcile each against the current engine-chain probe — delete if the engine owns it, leave alone otherwise.

I2. **`retryAppend` engine-takeover marks `delegated` regardless of the `tracker.Remove` error class** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:296-301` [Claude Opus 4.7]

```go
if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
    log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
}
pst.added[port] = portState{bindings: state.bindings, delegated: true}
```

`retryAppend`'s engine-takeover branch logs any `Remove` error and unconditionally marks the port `delegated`. The mirror path in `retireDisappeared` (`:336-352`) classifies the same error: a wsl-proxy-only failure means the gvisor proxy is gone (safe to drop the rule); any other shape means the host-switch unexpose did not land and the proxy may still be bound. This branch omits that classification. Once the port is `delegated`, the listener's eventual disappearance hits the delegated branch, which never retries `tracker.Remove`. The leaked gvisor proxy at `(tcp, 127.0.0.1:N)` then blocks every future engine `Add` with `proxy already running` until the next RD restart.

Fix: reuse the `retireDisappeared` classification. On a host-switch-failure shape, leave `appendFailed=true` so the next tick retries `tracker.Remove` instead of abandoning the proxy.

I3. **`exposeNew` engine-delegation does not release procnet's prior partial Add** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:202-207`, `:209-221` [Codex GPT 5.5]

```go
if managed {
    pst.added[port] = portState{bindings: bindings, delegated: true}
    delete(pst.addErrorLogged, port)
    return
}
```

`exposeNew`'s `tracker.Add` error branch (`:209-221`, not shown) records nothing in `pst.added` when the error does not contain `portAlreadyExposedSubstring`. APITracker's partial-failure shape (`apitracker.go:130-140`) — `apiForwarder.Expose` succeeds, `portStorage[syntheticID]` is populated, then `wslProxyForwarder.Send` fails — produces exactly such an error. On the next tick `exposeNew` runs again because `pst.added[port]` is still absent. If the CNI/DOCKER chain has appeared in the meantime, line 202 delegates immediately without checking `tracker.Get(syntheticID)` — bypassing the `handleAlreadyExposed` disambiguation at `:253` that exists precisely for this case. The synthetic `portStorage` entry and the bound gvisor proxy then survive until RD restart, blocking the engine's later `Add` with `proxy already running`.

Fix: probe `tracker.Get(syntheticID)` before the `if managed` branch. When procnet owns the entry, release it using the same error classification I2 needs; share a helper between both call sites.

## Suggestions

S1. **`engineChainManagesPort` chain list is hardcoded** — `src/go/guestagent/pkg/procnet/scanner_linux.go:225-244` [Claude Opus 4.7]

```go
for _, chain := range []string{"CNI-HOSTPORT-DNAT", "DOCKER"} {
```

The closed list is documented (lines 211-215), but a future contributor adding a third engine — or chasing an upstream chain rename — has no signal at compile time or runtime. The shadow-DNAT bug then returns silently.

Fix: hoist the list into a package-level `var` next to the related types, or emit a one-line startup log enumerating the chains procnet will probe. The log makes divergence visible in support bundles.

S2. **`fakeIptables`'s `commitOnError` does not model the real-iptables idempotency probe** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux_test.go:146-159` [Claude Opus 4.7]

The real `applyLoopbackIPtablesRule` (`scanner_linux.go:175-205`) calls `preroutingHasLoopbackRule` first and short-circuits an Append when the rule already exists. The fake honours every Append. No test exercises an Append-on-present sequence today, so the gap is dormant; a future test added without noticing the divergence would assume real-iptables idempotency, pass on the fake, and break in production.

Fix: add a comment on `fakeIptables` calling out the divergence, or model the rule-presence probe in the fake.

S3. **Extract the `tracker.Remove` error-classification predicate to a shared helper** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:337` [Claude Opus 4.7]

```go
case errors.Is(removeErr, tracker.ErrWSLProxy) && !errors.Is(removeErr, forwarder.ErrUnexposeAPI):
```

I2 and I3 each need the same predicate. Extracting it gives every call site the same answer by construction and documents the invariant once.

Fix:

```go
func removeReportedProxyDestroyed(err error) bool {
    return errors.Is(err, tracker.ErrWSLProxy) && !errors.Is(err, forwarder.ErrUnexposeAPI)
}
```

S4. **`/proc/net scanner failed to remove port` logs the wsl-proxy-only and host-switch-failed branches with identical text** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:343, 350` [Claude Opus 4.7]

Both `case` branches emit the same `Errorf` message even though their post-conditions differ: the wsl-proxy-only branch deletes the rule (proxy gone, rule gone); the default branch retains it (proxy may still be bound, rule kept). When a stuck port lands in the log, the message alone cannot identify which path ran.

Fix: split the lines per branch with a short suffix — `…removed port (wsl-proxy notification failed; loopback rule deleted): %s` vs `…removed port (host-switch unexpose may have left proxy bound; loopback rule retained): %s`. Optionally downgrade the wsl-proxy-only line to `Warnf`.

S5. **`retryAppend` success log omits the bindings that peer lifecycle logs include** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:309` [Claude Opus 4.7]

`exposeNew`'s "added port" log (`:223`) and `retireDisappeared`'s "removed port" log (`:325`) both format `%s -> %+v` (port and bindings). The retry-success log shows only the port, breaking single-port grep through a lifecycle.

Fix:

```diff
-log.Infof("/proc/net scanner installed deferred iptables rule for port: %s", port)
+log.Infof("/proc/net scanner installed deferred iptables rule for port: %s -> %+v", port, state.bindings)
```

S6. **`test:unit:guestagent` silently skips procnet tests on non-Linux dev hosts** — `package.json:39` [Claude Opus 4.7]

```json
"test:unit:guestagent": "cd ./src/go/guestagent/ && go test ./...",
```

Procnet tests live in `_linux.go` files. On macOS or Windows, `go test ./...` reports `[no test files]` for `pkg/procnet/` with no warning that the headline tests for this change did not actually run. CI on Ubuntu catches this; the local-dev silence will bite the next procnet change.

Fix: append a one-line warning when `GOOS != linux`, or document the platform requirement at the top of each test file.

## Design Observations

### Concerns

- *(future)* **Engine-managed state is inferred from an iptables snapshot, not from a shared event source** — `port_state_tracker_linux.go:180-207`, `scanner_linux.go:207-244` [Claude Opus 4.7]. `engineChainManagesPort` shells out to iptables every probe and matches against `dnatRuleRe`, then `portStateTracker` decides whether the engine "owns" the port. CNI portmap rules and the engine `portTracker.Add` are written by independent code paths at different times (per the repo context), so the chain-presence signal can drift from `portStorage`. The two-scan stability gate plus the "proxy already running" classifier paper over the drift well enough for the shipped engines, but a procnet that subscribed to the engine's container-start/stop events directly would be simpler and more robust. Out of scope; flag for the next time procnet opens up.

- *(future)* **`portStateTracker` mixes scan-cadence I/O with state-machine policy** — `port_state_tracker_linux.go` [Claude Opus 4.7]. The package separates `Tick` from the iptables runner, but the state machine still calls `pst.tracker.Add`/`Get`/`Remove` and `pst.iptables.*` from inside methods that also encode the defer / probe / delegate / retry policy. Splitting policy from side effects — e.g. `Tick` returns a list of `action{Add, Append, Remove, Delete}` decisions that the caller applies — would let the tests run the full state machine without mocks and reduce S4's classification gap to a unit test over decision shapes. Not worth doing now.

## Review Process Notes

### Skill improvements

- **For round N ≥ 2 reviews, surface partial-failure → next-tick-takeover sequences as a dedicated dimension.** Pattern: when a state machine handles error A by not recording state, and error B by recording it, a follow-up tick can reach a branch that assumes no prior state exists. Reviewers should enumerate, for each "error path that returns without recording state," every reachable branch on the next tick — including ones that touch external resources the failed attempt may have reserved. Add to the dimensions: "For each state-machine method that returns without recording state on error, list the branches that can run on the next tick and verify each one accounts for state the failed attempt may have left in external systems (proxies, files, iptables, sockets)."

- **For PRs that refactor across an extracted boundary (state → object, function → service), add a dimension on inherited external state.** Pattern: code that previously ran continuously now starts and stops; external state it wrote may outlive its in-memory state. Add: "When a process boundary is introduced or repositioned (a struct's lifetime changes, a service is extracted), enumerate every piece of external state the code writes (files, kernel tables, sockets, remote resources) and verify the new boundary either reconciles or cleanly owns each one. A regression in this dimension reintroduces bugs the code fixed in prior sessions."

## Executive Summary

This change extracts the procnet per-port state machine into a `portStateTracker` with an injectable `iptablesRunner` and tracker, fixing three race-window classes uncovered during #10262's review and adding unit-test coverage for every transition. The implementation is solid; findings are scoped to a few correctness corners and clarity polish.

## Findings

### Critical Issues

### Important Issues

I1. **Stale procnet rule survives guestagent restart and shadows engine chains** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:180-207`, `scanner_linux.go:175-205`

```go
func (pst *portStateTracker) exposeNew(port nat.Port, bindings []nat.PortBinding) {
    managed, err := pst.anyEngineBinding(port, bindings)
    ...
    if managed {
        pst.added[port] = portState{bindings: bindings, delegated: true}
        ...
        return
    }
```

`portStateTracker` initialises with an empty `added` map every time `ForwardPorts` starts (line 105 of `scanner_linux.go`), but iptables state in the WSL2 distro persists across guestagent restarts (only WSL2 reboot clears it). After a restart, a port that procnet wrote a PREROUTING DNAT for in the previous session — then is republished by the engine in the current session — takes the delegation path (lines 202-207) without removing the surviving procnet rule. Because PREROUTING DNAT in iptables-nft terminates the chain, the leftover rule shadows the engine's `CNI-HOSTPORT-DNAT`/`DOCKER` chain — the exact bug this PR fixes — until the next full RD restart.

Fix: at startup (e.g. in `NewProcNetScanner` after `enableLocalnetRouting`), flush procnet-authored PREROUTING DNAT rules, or recognise them during the first scan and reconcile against the current engine-chain state. A predicate `iptables -t nat -L PREROUTING -n | grep -E '^DNAT .* tcp dpt:[0-9]+ .* to:127.0.0.1:'` would identify candidates.

`(important, regression)`

I2. **`retryAppend` engine-takeover may release the tracker entry with the host-switch proxy still alive** — `port_state_tracker_linux.go:283-301`

```go
if managed {
    if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
        log.Debugf(...)
        return
    }
    if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
        log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
    }
    pst.added[port] = portState{bindings: state.bindings, delegated: true}
```

The retry-side engine-takeover at line 296-300 fires `tracker.Remove` and then unconditionally marks the port `delegated`, regardless of `removeErr`'s shape. The `retireDisappeared` path two screens away (line 337) is careful to keep the loopback rule when `removeErr` does not look like a clean unexpose, because the gvisor proxy may still be bound. Here the same `removeErr` is logged and ignored: the rule has already been deleted (line 292) and the port is marked delegated, so when the listener later disappears the delegated branch never runs unexpose. A leaked gvisor proxy at `(tcp, 127.0.0.1:8080)` then blocks any future engine `Add` with `proxy already running` until RD restart.

Fix: apply the same `errors.Is(..., tracker.ErrWSLProxy) && !errors.Is(..., forwarder.ErrUnexposeAPI)` classification used in `retireDisappeared`. On the "may still be bound" branch, defer the delegation: leave `appendFailed=true` for another retry, or keep `delegated=false` so the next listener disappearance triggers a real `tracker.Remove`.

`(important, regression)`

I3. **`/proc/net scanner failed to remove port` log line loses the wsl-proxy-vs-unexpose distinction** — `port_state_tracker_linux.go:342, 349`

```go
case errors.Is(removeErr, tracker.ErrWSLProxy) && !errors.Is(removeErr, forwarder.ErrUnexposeAPI):
    log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
default:
    log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
    continue
```

Both branches emit the identical message at Error, but the operator-relevant detail is that the wsl-proxy-only case proceeds to delete the rule (post-condition: proxy gone, rule gone) while the default case retains it (post-condition: proxy may still be bound, rule kept). When debugging a sticky port from a single log line, you cannot tell which branch ran without re-reading the source. The wsl-proxy-only case also self-classifies as a less severe failure (proxy gone is the desired end state) and arguably warrants Warn rather than Error.

Fix: split the log lines per branch with a short suffix, e.g. `…removed port (wsl-proxy notification failed; loopback rule deleted): %s` vs `…removed port (host-switch unexpose may have left proxy bound; loopback rule retained): %s`. Optionally downgrade the wsl-proxy-only line to `log.Warnf`.

`(important, gap)`

### Suggestions

err := f.applyErrors[key]
if err == nil || f.commitOnError[key] {

`(suggestion, gap)`

S3. **Duplicated `errors.Join`-aware classification not extracted** — `port_state_tracker_linux.go:337`

The classification `errors.Is(err, tracker.ErrWSLProxy) && !errors.Is(err, forwarder.ErrUnexposeAPI)` is the single cleanup-decision predicate but it lives inline. ## Design Observations

### Concerns

- **Engine-managed state is inferred via an out-of-band iptables snapshot, not a shared event source** `(future)`. `engineChainManagesPort` shells out to iptables every probe and matches against `dnatRuleRe`, then `portStateTracker` decides whether the engine "owns" the port. The CNI portmap rules and the engine `portTracker.Add` are written by independent code paths at different times (per the repo context), so the chain-presence signal can drift from `portStorage` state. The two-scan stability gate plus the "proxy already running" classifier paper over the drift well enough for shipping engines, but the design would be simpler and more robust if procnet subscribed to the engine's container-start/stop events directly (the same source the engine's own portTracker uses) instead of probing iptables every tick. Out of scope for this PR, but worth recording for the next time the procnet design opens up.

- **`portStateTracker` mixes scan-cadence I/O with pure state-machine logic** `(future)`. The package has good separation between `Tick` and the iptables runner, but the state machine still calls `pst.tracker.Add`/`Get`/`Remove` and `pst.iptables.*` from within methods that also encode the policy (defer/probe/delegate/retry). Splitting the policy from the side effects — e.g. `Tick` returns a list of `action{Add, Append, Remove, Delete}` decisions that the caller applies — would let the tests run the full state machine without any mocks at all, and make the I3 fix from this PR (cleanup-side error classification) reduce to a unit test over decision shapes. Not worth doing now; flag for next time.

### Strengths

- **Test coverage of the regression cases is exemplary.** Each of the three review-discovered race windows from #10262 (I1 resume, I2 engine-takeover, I3 cleanup-marker-leak) has a dedicated regression test whose name calls out the round/scenario it pins, and the `TestPortAlreadyExposedSubstringIsReachable` test pins the upstream substring contract through both wrap layers.

- **State-machine docstring is the right level of detail.** The comment at lines 30-92 of `port_state_tracker_linux.go` reads like a small spec: every transition is enumerated with its rationale, and the "Append retry" / "Listener disappears" subsections explicitly document the order of side effects (Delete before Remove, drop marker before iptables Delete) and why. Future maintainers have a real map.

- **`APITracker.Add`/`Remove` error joining is a strict improvement.** The old code returned a single failure mode and silently dropped the other; the new `errors.Join` with distinct sentinels lets every caller (procnet today, others tomorrow) make a state-correct decision, and the `TestRemoveWSLProxyFailureIsDistinguishable` test pins the contract.

## Testing Assessment

The state-machine unit tests are thorough. Specific scenarios still not covered, ranked by risk:

1. **Startup with surviving procnet rules from a previous guestagent session** (links to I1). No test confirms that a procnet PREROUTING DNAT installed before a restart is reconciled against the engine chain on the next first-sighting tick.

2. **`retryAppend` engine-takeover with a `tracker.Remove` error that classifies as "proxy may still be bound"** (links to I2). The existing `TestAppendRetryEngineTakeoverRetriesFailedDelete` covers the Delete-side failure; no test covers a Remove-side error of mixed shape on the retry path.

The `applyLoopbackIPtablesRule` idempotency gate (scanner_linux.go:175-205) only runs against real iptables; published-ports.bats covers it.
