## Executive Summary

The PR drops procnet's PREROUTING DNAT writes in favor of a userspace TCP/UDP proxy bound on the tap-interface IP, deleting the 665+1187-line engine-chain probing surface #10280 carried. The architectural shift is sound. One critical issue: the new forwarder's own tap-IP listener appears in `/proc/net/tcp`, so the scanner never observes the original loopback listener "disappear" and never unpublishes it — every `--network=host` loopback container leaves a stuck forwarder, host-switch entry, and accept goroutine behind on exit.

The structure: Codex caught the scanner self-reference and the TCP half-close gap; Claude and Gemini flagged the `Accept`/`ReadFrom` error-handling termination; Gemini and Claude both caught the partial-failure path; all three agents independently spotted the stale-pending-bindings issue. The non-critical work splits into reconciliation-loop fixes (I2/I3) and forwarder hardening (I1/I4/S1–S5). Test coverage gaps are concentrated in UDP and error-injection paths.

## Critical Issues

C1. **Forwarder's own tap-IP listener prevents the scanner from ever unpublishing a vanished loopback port** — `src/go/guestagent/pkg/procnet/scanner_linux.go:108-114` [Codex GPT 5.5]

```go
for port, bindings := range p.published {
    if _, ok := scanned[port]; ok {
        continue
    }
    p.unpublish(port, bindings)
    delete(p.published, port)
}
```

When the scanner publishes a 127.0.0.1 listener, `forwarder.Add` opens a socket on `tapIfaceIP:<port>` (`loopback_forwarder_linux.go:74`). `scanListeners` sees both that socket and the upstream's. `addEntryToPortMap` coerces the tap-IP entry to `0.0.0.0` and appends (line 224); after the container exits, the key stays in `scanned` — only the binding list shrinks. The first `Tick` loop checks key membership alone, takes `continue`, and never calls `unpublish`. `tracker.Remove`, `forwarder.Remove`, and the accept goroutine all leak; host-switch keeps routing Windows-side traffic to a forwarder whose 127.0.0.1 dial now returns connection refused. The pre-PR DNAT had no self-reference because iptables rules never appear in `/proc/net`.

Fix: filter the scanner's own listeners out of the snapshot before reconciliation, or compare bindings rather than keys. The first variant is more direct — the forwarder already owns the bindIP/port pairs in `f.tcp` / `f.udp`:

```diff
 func scanListeners() (nat.PortMap, error) {
     entries, err := procnettcp.ParseFiles()
     ...
 }

+func (p *ProcNetScanner) excludeForwarderEntries(scanned nat.PortMap) {
+    // drop bindings the forwarder itself opened on bindIP, then drop the
+    // map entry if no other bindings remain.
+    ...
+}
```

Then call it from `Tick(scanned)` before the published/pending loops, after `scanListeners` returns.

## Important Issues

I1. **TCP `Accept` and UDP `ReadFrom` errors permanently terminate forwarding for the port** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:134-146, 184-188` [Claude Opus 4.7, Gemini 3.1 Pro]

```go
conn, err := lis.Accept()
if err != nil {
    if errors.Is(err, net.ErrClosed) {
        return
    }
    log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
    return
}
```

Any non-`ErrClosed` error (transient `EMFILE`/`ENFILE`/`ENOBUFS` from FD pressure, transient kernel state) returns from the goroutine. The listener stays in `f.tcp[k]` so subsequent `forwarder.Add` calls short-circuit at `loopback_forwarder_linux.go:71-73` and never reinstall an acceptor. Combined with C1, the scanner cannot recover via re-publish either: the port stays in `p.published`, the FD stays open, and no new traffic is processed until the guestagent process restarts. `relayUDP` has the same shape at line 184-188 — any `ReadFrom` error returns and closes the per-flow map via the deferred cleanup. The pre-PR DNAT had no equivalent because iptables rules are stateless.

Fix: on a non-`ErrClosed` error, remove the listener from the map and close it so the next stable scan reopens it; or retry with bounded backoff.

```diff
     log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
-    return
+    _ = f.Remove("tcp", port)
+    return
 }
```

I2. **Two-scan stability gate publishes bindings captured from the first scan rather than the current one** — `src/go/guestagent/pkg/procnet/scanner_linux.go:116-122` [Codex GPT 5.5, Gemini 3.1 Pro, Claude Opus 4.7]

```go
for port, bindings := range p.pending {
    if _, ok := scanned[port]; !ok {
        continue
    }
    p.publish(port, bindings)
    p.published[port] = bindings
}
```

The gate verifies the same proto/port appears on the second scan but publishes the bindings captured from the *first* scan at line 120-121. If the first scan saw a transient 127.0.0.1 reservation and the second saw a real listener on `0.0.0.0` (or vice versa), procnet publishes the wrong binding. The wrong direction (loopback → wildcard) opens a tap-IP forwarder where none is needed; the more harmful direction (wildcard → loopback) skips the forwarder the real listener requires and leaves Windows-side traffic refused. Each scenario requires a same-port listener replacement inside ~3 s, but the gate was added specifically to filter transient sockets, so its correctness against transients is load-bearing.

Fix: require the same binding set in both scans and publish the current snapshot.

```diff
-for port, bindings := range p.pending {
-    if _, ok := scanned[port]; !ok {
+for port := range p.pending {
+    current, ok := scanned[port]
+    if !ok {
         continue
     }
-    p.publish(port, bindings); p.published[port] = bindings
+    p.publish(port, current); p.published[port] = current
```

I3. **`publish` partial failure leaves the port stuck in `p.published` with no retry** — `src/go/guestagent/pkg/procnet/scanner_linux.go:133-153` [Gemini 3.1 Pro, Claude Opus 4.7]

```go
if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
    log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
    return
}
...
if err := p.forwarder.Add(p.ctx, port.Proto(), uint16(portNum)); err != nil {
    log.Errorf("/proc/net scanner: loopback forwarder %s/%s: %s", port.Proto(), b.HostPort, err)
}
```

Two failure shapes converge on the same dead-end. `tracker.Add` errors after partial side effects (per the package's documented `APITracker` contract — `Expose`→`portStorage.add`→`wslProxyForwarder.Send`, errors mid-sequence leave state behind) cause the early return at line 137 and skip the forwarder; `Tick` still records `p.published[port] = bindings` (line 121). `forwarder.Add` errors (e.g. `EADDRINUSE` from a stale listener or `EADDRNOTAVAIL` from a tap-interface flap) are logged but execution continues, with the same `p.published[port] = bindings` outcome. In both cases the next tick sees the port in published, takes the early-return at line 109, and never retries. Host-switch routes Windows-side traffic to a missing or incomplete forwarder until the upstream listener disappears.

Fix: have `publish` return an error and let `Tick` keep the port in `pending` (not `published`) on failure so the next tick retries. Roll back `tracker.Remove(id)` when `forwarder.Add` fails after `tracker.Add` succeeded.

I4. **TCP half-close from the client drops the upstream response** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:148-161` [Codex GPT 5.5]

```go
done := make(chan struct{}, 2)
go func() { _, _ = io.Copy(out, in); done <- struct{}{} }()
go func() { _, _ = io.Copy(in, out); done <- struct{}{} }()
<-done
```

The function returns after the first copy completes; `defer in.Close()` and `defer out.Close()` (lines 149, 156) then forcibly close both sockets. A client that calls `CloseWrite` after sending a request — the standard signal for end-of-request in HTTP/1.x with `Connection: close`, FTP control, and several custom protocols — makes the `in → out` copy finish first; the proxy tears down the upstream connection before the response can drain. The pre-PR DNAT preserved TCP half-close because the kernel proxied at the IP layer.

Fix: half-close per direction and wait for both copies before tearing down.

```go
copyDone := func(dst, src net.Conn) {
    _, _ = io.Copy(dst, src)
    if tcp, ok := dst.(*net.TCPConn); ok {
        _ = tcp.CloseWrite()
    }
    done <- struct{}{}
}
go copyDone(out, in)
go copyDone(in, out)
<-done; <-done
```

## Suggestions

S1. **Reuse gvisor-tap-vsock's `UDPProxy` instead of hand-rolling flow demux** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:167-231` [Claude Opus 4.7]

```go
func (f *loopbackForwarder) relayUDP(ctx context.Context, pc net.PacketConn, port uint16) {
    target := net.JoinHostPort("127.0.0.1", strconv.Itoa(int(port)))
    var (
        mu    sync.Mutex
        flows = make(map[string]net.Conn)
    )
    ...
```

`github.com/containers/gvisor-tap-vsock/pkg/services/forwarder` is already a direct dependency of `src/go/guestagent` (`go.mod:10`). `forwarder.NewUDPProxy` implements the same multiplexing — connection-tracking key by source IP+port, per-flow reply loop with `SetReadDeadline`, idle timeout. Lima's `pkg/portfwd/client.go` (which the file docstring cites at line 23) uses `forwarder.NewUDPProxy` for the UDP path; only the TCP path structurally mirrors Lima's listener. Adopting it removes ~55 lines plus the mu/map/reply-goroutine pairing and eliminates S2/S3/S4 below.

S2. **UDP idle timeout 30s is shorter than the gvisor-tap-vsock default (and undocumented)** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:210, 218` [Claude Opus 4.7]

```go
const udpIdleTimeout = 30 * time.Second
...
_ = out.SetReadDeadline(time.Now().Add(udpIdleTimeout))
```

30 s is half the Linux conntrack `nf_conntrack_udp_timeout_stream` default and a third of gvisor-tap-vsock's `UDPProxy` default. Long-lived UDP services with infrequent traffic (RTP keepalives, mDNS responders, audio streams with >30 s gaps) lose their outbound socket; the next datagram from the same client opens a new socket with a new source port. Anything that pins a session to the source port breaks every 30 s. Document the choice with the failure modes it accepts, or align with gvisor-tap-vsock — moot if S1 is adopted.

S3. **`relayUDP` accepts `ctx` but never reads `ctx.Done()`** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:167-208` [Claude Opus 4.7]

```go
func (f *loopbackForwarder) relayUDP(ctx context.Context, pc net.PacketConn, port uint16) {
```

`ctx` is forwarded only to `f.dialer.DialContext` on the first datagram of each flow (line 194). Once flows exist, cancellation does nothing — the goroutine exits only when `pc` is closed. This works because `Close`/`Remove` close `pc`, but the `ctx` parameter is misleading. Either drop it or wire `context.AfterFunc(ctx, func() { _ = pc.Close() })`.

S4. **UDP `out.Write` runs outside the lock; concurrent flow cleanup can drop datagrams silently** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:191-206` [Claude Opus 4.7]

```go
mu.Lock()
out, ok := flows[srcKey]
if !ok { ... }
mu.Unlock()
_, _ = out.Write(buf[:n])
```

If `readUDPReplies` times out and closes `out` between the unlock at line 204 and the write at line 206, the datagram is silently dropped (line 206 ignores the error). Combined with S2's 30 s timeout, low-traffic UDP flows have a small window for spurious packet loss across the natural idle boundary. UDP semantics make this acceptable in the general case; mention in a comment, or replace with S1's `UDPProxy`.

S5. **Test coverage gaps: UDP, mixed protocols on the same port, error returns, gate restart** — `src/go/guestagent/pkg/procnet/scanner_linux_test.go`, `loopback_forwarder_linux_test.go` [Claude Opus 4.7, Gemini 3.1 Pro]

The five scanner tests exercise TCP only. The fake tracker/forwarder are already in place; each gap below is a few lines:

- TCP and UDP listeners on the same port number (distinct `nat.Port` keys).
- `tracker.Add` and `forwarder.Add` returning errors — does the port retry next scan? (I3)
- Port appears in pending, gone on tick 2, reappears on tick 3 — does the gate restart?
- Listener binding shape changes between two ticks (loopback ↔ wildcard). (I2)

The forwarder end-to-end tests bind on `127.0.0.99`, which requires a loopback alias. They `t.Skipf` when the dial fails, silently no-opping on hosts without the alias — either document the precondition or wire `ip addr add 127.0.0.99/8 dev lo` in CI.

## Design Observations

### Concerns

**Hand-rolled UDP demultiplexer where the in-tree gvisor-tap-vsock library covers it** *(in-scope)* [Claude Opus 4.7]

S1 covers the code-level swap. As a structural observation, the PR replaces 665+1187 lines of state machine and tests with a 231-line forwarder plus 244 lines of tests. The remaining novelty is concentrated in the UDP flow table — the per-flow goroutine, the mutex passed by pointer to `readUDPReplies`, the connection-tracking map, the deferred cleanup. `forwarder.UDPProxy` is the same code Lima uses and is exercised in the gvisor-tap-vsock host-switch process this PR already routes through. Adopting it shrinks the new package by another ~20% and folds S2/S3/S4 into upstream.

**Scanner sees its own forwarder; reconciliation needs ownership filtering** *(in-scope)* [Codex GPT 5.5]

C1 covers the symptom. The underlying issue: the new forwarder creates sockets in the same namespace the scanner monitors, so procnet results no longer represent purely external state. Modeling published bindings separately from observed sockets — or filtering forwarder-owned tap-IP sockets out of the scan before the published/pending loops run — eliminates C1 and makes I2 harder to reintroduce. The forwarder already holds the authoritative bindIP/port set in `f.tcp` and `f.udp`.

**`publish` treats `tracker.Add` and `forwarder.Add` as independent when they form a unit** *(in-scope)* [Claude Opus 4.7, Gemini 3.1 Pro]

I3 covers the symptom. Without the forwarder, the tracker entry misleads host-switch into routing traffic to a missing listener. A `publish` that returns success only when both steps succeed (with rollback on the second-step failure) lets `Tick` keep the port in `pending` and retry next scan. The current "log and continue" pattern is the right shape for stateless rules and the wrong shape for stateful resources.

### Strengths

- Abandoning engine-chain probing entirely (rather than another refinement pass on #10280's heuristic stack) removes the coordination surface that motivated the bug class — a strict reduction in the failure-mode surface area.
- The `Tick(scanned)` factoring with injected `tracker.Tracker` and `loopbackController` makes the reconciliation logic directly unit-testable in plain Go without iptables or network plumbing.
- Adding `test:unit:guestagent` to `package.json` so the package's tests actually run in CI closes a gap that #10262 and #10280 both rediscovered. Several of the issues above would have been caught by the test cases already present in the package once they ran.
- Binding the userspace listener on the tap IP integrates cleanly with the existing gvisor-tap-vsock routing — no new path to add or remove from host-switch's perspective.

## Testing Assessment

Coverage of the happy paths in the stability gate is good (transient suppression, persistent promotion, immediate removal, wildcard skip, mixed loopback/wildcard). Gaps in risk order:

1. C1 / scanner self-reference: publish a loopback port, then call `Tick` with only the tap-IP/wildcard binding the forwarder would create, and assert `unpublish` runs.
2. I2 / gate uses stale bindings: tick 1 has `{127.0.0.1:port}`, tick 2 has `{0.0.0.0:port}`; the scanner must not publish the stale loopback binding.
3. I3 / publish partial failure: inject a fake tracker that returns an error after partial state; assert the port returns to pending (not published) and retries.
4. I4 / TCP half-close: client `CloseWrite` after request; assert the upstream response still arrives.
5. UDP listener through the scanner (only TCP is exercised).
6. TCP and UDP listeners on the same port number.
7. Port reappearance after one-tick absence: does the two-scan gate restart counting?
8. I1 / `Accept` and `ReadFrom` non-`ErrClosed` errors: inject a listener wrapper returning a transient error; assert the listener is cleaned up rather than silently leaked.
9. Forwarder UDP idle-timeout cleanup is asserted only via the absence of flow leaks; no test verifies that a flow that idles for 30 s actually closes its outbound socket.

The forwarder end-to-end tests `t.Skipf` when `127.0.0.99` is not configured as a loopback alias, which means they silently no-op on hosts without the alias. Either pin the precondition in CI (`ip addr add 127.0.0.99/8 dev lo`) or document the skip behavior in the test file.

## Review Process Notes

**Repo context updates**

- [repo] When reviewing changes to `src/go/guestagent/pkg/procnet/scanner_linux.go`, check whether the diff introduces any new in-namespace listener (TCP or UDP) opened by the guestagent process itself. `procnettcp.ParseFiles` reads `/proc/net/{tcp,udp}` for the entire namespace, so any socket the guestagent opens — userspace forwarders, debug listeners, health checks — appears in the scanned set and will short-circuit the published-port reconciliation loops unless filtered out before `Tick`. The forwarder's authoritative bindIP/port set is the natural filter source; review against that. Rationale: this is the exact category that produced PR #10357's C1 leak and the bug pattern recurs for any future scanner that observes its own outputs.

**Skill improvements**

- When a change introduces a new in-process listener or actor whose outputs are observed by the same component's monitoring loop, trace from the monitor's input source to the new actor's emission point and confirm the actor's outputs are either filtered or distinguishable. State observers that include their own emissions in the observation produce reconciliation loops that never converge for that key. The principle: any change that turns a one-way external observation into a closed-loop self-observation needs an explicit ownership filter, regardless of whether the loop is symptomatic in the test suite.

```go
            if errors.Is(err, net.ErrClosed) {
                return
            }
            log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
            return
        }
```

On any non-`ErrClosed` `Accept` error (`EMFILE`, `ENFILE`, transient kernel errors), the goroutine returns but the listener stays in `f.tcp[k]`. Subsequent `forwarder.Add` calls for the same key short-circuit at line 71-73, so the listener remains FD-open with no acceptor. The scanner has no way to notice (the upstream `127.0.0.1` listener is still in `/proc/net/tcp`, so `Remove` is never called) and connections from Windows accumulate in the kernel accept queue forever. The pre-PR DNAT had no such failure mode — iptables rules either apply or they don't.

Fix: either close the listener and delete the map entry before returning, so the next stable scan re-installs it, or retry with bounded backoff:

```diff
 conn, err := lis.Accept()
 if err != nil {
     if errors.Is(err, net.ErrClosed) {
         return
     }
     log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
-    return
+    f.Remove("tcp", port)
+    return
 }
```

#### Suggestions

**S1. Reuse gvisor-tap-vsock's UDPProxy instead of hand-rolling flow demux** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:167-231` (suggestion, enhancement)

`github.com/containers/gvisor-tap-vsock/pkg/services/forwarder` is already a direct dependency of `src/go/guestagent` (`go.mod:10`). `forwarder.NewUDPProxy` (`udp_proxy.go:61`) implements exactly this multiplexing — `connTrackKey` keyed by source IP+port, per-flow `replyLoop` with `SetReadDeadline`, 90s idle timeout, fewer goroutine races. Lima's `pkg/portfwd/client.go:51-63` (which the docstring claims to mirror) uses `forwarder.NewUDPProxy`; only the TCP path is structurally similar. Adopting it removes ~55 lines plus the bespoke mu/map/reply-goroutine pairing.

**S2. Hand-rolled UDP idle timeout is shorter than the gvisor-tap-vsock default and undocumented** — `loopback_forwarder_linux.go:210, 218` (suggestion, gap)

30s is half the Linux conntrack `nf_conntrack_udp_timeout_stream` (default 120s) and a third of gvisor-tap-vsock's 90s. Long-lived UDP services (a chatty mDNS responder, an audio stream with > 30s gaps, RTP keepalives at >30s intervals) will see their flow torn down and the next datagram from the same client open a new outbound socket; from the upstream service's perspective the source-port shifts every 30s, which breaks anything that pins a session to source-port. Document the choice or align with gvisor-tap-vsock. If S1 is taken, this disappears.

**S3. Bindings stored in `pending` aren't refreshed against the second scan** — `src/go/guestagent/pkg/procnet/scanner_linux.go:116-130` (suggestion, gap)

When a port is promoted from pending to published on tick N, `bindings` come from the tick-(N-1) snapshot. If the listener's `HostIP` changed between scans (e.g., loopback → wildcard, or vice versa), publish opens the forwarder against stale bindings. Functionally this usually works because nat.Port keys by proto/port and host-switch always routes to `tapIP:port`, but the forwarder may be opened where it isn't needed, or skipped where it is.

**S4. Permanent stuck state when `forwarder.Add` fails after `tracker.Add` succeeded** — `scanner_linux.go:133-153` (suggestion, gap)

`forwarder.Add` failure (e.g., `EADDRINUSE` from a stale listener, `EADDRNOTAVAIL` from a brief tap-interface flap) is logged but the port is still inserted into `p.published`. The next Tick sees the port still present, takes the early-return at line 109, never retries. Host-switch routes to `tapIP:HostPort` and sees no listener until the upstream listener finally disappears. Either roll back `tracker.Remove(id)` and skip the `p.published[port] = ...` assignment in `Tick`, so the next tick retries; or have publish return an error so Tick can decide.

**S5. Hand-rolled UDP race window between Tick removal and concurrent client datagrams** — `loopback_forwarder_linux.go:191-206` (suggestion, gap)

`out.Write` runs outside the lock. If `readUDPReplies` times out and closes `out` between the unlock and the write, the datagram is silently dropped. Combined with S2 (aggressive 30s timeout), this creates a window for spurious packet loss under low-traffic UDP flows. gvisor-tap-vsock's UDPProxy (S1) writes while holding nothing but doesn't share `out` between cleanup paths in the same way. Acceptable for the connectionless protocol, worth a comment if not fixed.

**S6. Scanner does not test UDP, mixed protocols on same port, or forwarder/tracker error returns** — `scanner_linux_test.go` (suggestion, gap)

All five scanner tests use TCP only. No test for a UDP listener (`procnettcp.UDP` kind, `UDPEstablished` state); a TCP and UDP listener on the same port number (distinct `nat.Port` keys); `tracker.Add` returning an error (does `pending` retain the entry for retry?); `forwarder.Add` returning an error (S4); a port appearing on tick 1, disappearing on tick 2, reappearing on tick 3 (does the gate restart?). The fake tracker/fake forwarder are already in place; each case is a few lines.

**S8. Whitespace reformat folded into a logic change** — `scanner_linux.go:209-210` (suggestion, regression)

```diff
-            entry.Kind,
-            entry.Port,
-            err)
+            entry.Kind, entry.Port, err)
```

Unrelated formatting change inside the main feature commit. Minor scope creep; pull into a separate cleanup commit or drop.

**S9. `relayUDP` accepts `ctx` but never reads `ctx.Done()`** — `loopback_forwarder_linux.go:167, 525-587` (suggestion, gap)

`ctx` is forwarded only to `f.dialer.DialContext` on the first datagram of each flow. Once flows exist, cancellation has no effect — the goroutine exits only when `pc` is closed. This works because `Close`/`Remove` close `pc`, but it makes the `ctx` parameter misleading. Either drop the parameter or wire `context.AfterFunc(ctx, func() { pc.Close() })`.

### Executive Summary

#### Critical Issues

C1. **Forwarder sockets keep vanished loopback ports published forever** — `src/go/guestagent/pkg/procnet/scanner_linux.go:109` (critical, regression)

```go
done := make(chan struct{}, 2)

<-done
```

The proxy returns as soon as either copy finishes at line 160, and the deferred closes at lines 149 and 156 close both sockets. A client that half-closes its write side to signal end-of-request makes the `in -> out` copy at line 158 finish first; the proxy then closes the upstream connection before the service can send its response. Fix: after each copy, close only the destination write side when possible, and wait for both directions to drain.

```go

	_, _ = io.Copy(dst, src)
	if tcp, ok := dst.(*net.TCPConn); ok {
		_ = tcp.CloseWrite()
	}
	done <- struct{}{}
}
go copyDone(out, in)
go copyDone(in, out)
<-done
<-done
```

### [Gemini] — Pass 1

### Executive Summary

This PR replaces the complex, race-prone iptables `PREROUTING DNAT` rule management with a clean userspace TCP/UDP proxy bound to the tap interface IP. While the architectural shift elegantly resolves the rule-shadowing bugs, the implementation introduces critical regressions in state reconciliation and error handling. Specifically, the scanner ignores IP binding changes for established ports, and an early return during partial exposure failures will permanently orphan the userspace proxy. 

### Findings

#### Critical Issues

**C1. `Tick` ignores IP binding changes for established ports and promotes stale bindings** — `src/go/guestagent/pkg/procnet/scanner_linux.go:88-102`

```go
	for port, bindings := range p.published {
		if _, ok := scanned[port]; ok {
			continue
		}
```

The reconciliation loop identifies listeners solely by protocol and port, ignoring changes to their IP bindings. If an application restarts and changes its bind IP (e.g., from `0.0.0.0` to `127.0.0.1`) without the port disappearing between scans, the first loop `continue`s silently, preserving the stale bindings in `p.published`. 

Furthermore, in the second loop, the scanner promotes ports from `p.pending` using the bindings cached from the *first* scan (`bindings`), rather than the fresh bindings from `scanned[port]`. If a transient listener (like the OCI hook on `127.0.0.1`) is replaced by a permanent proxy on `0.0.0.0` before the second scan, `Tick` incorrectly publishes the stale `127.0.0.1` bindings.

Fix: Compare the actual bindings using `reflect.DeepEqual` to detect changes and ensure fresh bindings are promoted.

```go
import "reflect"

// In Tick:
	for port, bindings := range p.published {
		if newBindings, ok := scanned[port]; ok && reflect.DeepEqual(bindings, newBindings) {
			continue
		}
		p.unpublish(port, bindings)
		delete(p.published, port)
	}

	for port, bindings := range p.pending {
		newBindings, ok := scanned[port]
		if !ok || !reflect.DeepEqual(bindings, newBindings) {
			continue
		}
		p.publish(port, newBindings)
		p.published[port] = newBindings
	}
```

**C2. Early return on `tracker.Add` error permanently orphans the userspace forwarder** — `src/go/guestagent/pkg/procnet/scanner_linux.go:107-110` (critical, regression)

```go
	if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
		log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
		return
	}
```

`APITracker.Add` mutates shared state before returning errors. A partial failure (e.g., in `wslProxyForwarder.Send`) leaves the proxy exposed via `host-switch`, but the early return here skips initializing the `loopbackForwarder`. 

Because the calling `Tick` method unconditionally records the port in `p.published` regardless of `publish`'s success, the scanner will never retry `forwarder.Add`. Traffic arriving from the Windows host will hit a closed port on `tapInterfaceIP` and be refused until the container is recreated.

Fix: Log the error and proceed to set up the forwarder so it is ready to receive traffic from the successfully-exposed proxy.

```diff
 	if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
 		log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
-		return
 	}
 	log.Infof("/proc/net scanner added port: %s -> %+v", port, bindings)
```

#### Important Issues

**I1. `Accept` and `ReadFrom` temporary errors permanently terminate port forwarding** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:142`, `182` (important, regression)

```go
		conn, err := lis.Accept()
		if err != nil {
			if errors.Is(err, net.ErrClosed) {
				return
			}
			log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
			return
		}
```

If `lis.Accept()` or `pc.ReadFrom()` returns a temporary resource error (like `EMFILE` "too many open files", or `ENOBUFS`), logging the error and returning terminates the forwarding loop for that port permanently. The socket remains registered in `f.tcp` / `f.udp`, so subsequent `Add` calls are safely ignored, but no new traffic is processed until the container is completely removed and re-added.

Fix: Apply a short backoff for temporary errors rather than exiting.

```diff
 				return
 			}
 			log.Errorf("loopback forwarder accept tcp/%d: %s", port, err)
-			return
+			time.Sleep(100 * time.Millisecond)
+			continue
 		}
```

**I2. Forwarder hardcodes `127.0.0.1`, breaking non-standard loopback bindings** — `src/go/guestagent/pkg/procnet/scanner_linux.go:114`, `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:149` (important, regression)

```go
// loopback_forwarder_linux.go:149
	addr := net.JoinHostPort("127.0.0.1", strconv.Itoa(int(port)))
```

The previous iptables `PREROUTING DNAT` approach forwarded traffic to whatever loopback IP the container bound to (`binding.HostIP`), which could be `127.0.0.2` or `127.0.0.53`. The new userspace approach explicitly filters for `127.0.0.1` before calling `forwarder.Add` (via `b.HostIP != loopbackIP`), and `pipeTCP`/`relayUDP` hardcode `"127.0.0.1"` as the dial target.
