## Suggestions

S1. **`Moproxy.download` runs `7z e` without `-y`, hanging non-interactive postinstall after a partial-failure retry** — `scripts/dependencies/wsl.ts:40` [Gemini 3.1 Pro]

```ts
// moproxy uses xz with no tar wrapper; just decompress it manually.
// Keep archivePath beside moproxyPath so download() hits its cache on
// subsequent postinstall runs; electron-builder.yml excludes *.xz.
await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
await fs.promises.rename(path.join(context.internalDir, binName), moproxyPath);
```

`7z e` extracts unconditionally on every postinstall (only `download()` is cache-aware in this method, not the extraction). On the happy path, a successful previous run renamed `binName` away from `internalDir`, so the next extraction creates `binName` fresh. If a previous run failed between `7z e` and `rename` — `EBUSY`/`EPERM` overwriting a locked `moproxyPath`, or a Ctrl-C, OOM kill, or any postinstall step after this one that throws — the leftover `binName` survives in `internalDir`. The next run's `7z e` then prompts `file already exists. Overwrite? (Y)es / (N)o /...` and hangs because `simpleSpawn` runs without a TTY. Recovery is to `rm internalDir/moproxy_*.bin` after stopping the consuming process; cheap to prevent.

Fix:

```diff
-    await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
+    await simpleSpawn('7z', ['e', '-y', archivePath], { cwd: context.internalDir });
```

S2. **`MobyOpenAPISpec.getAvailableVersions` returns `commit: ''`, an invalid `MobyOpenAPISpecVersion`** — `scripts/dependencies/moby-openapi.ts:106-127` [Claude Opus 4.7]

```ts
versions.push({ apiVersion: match[1], commit: '' });
…
return versions as MobyOpenAPISpecVersion[];
```

`MobyOpenAPISpecVersion.commit` is documented as "the latest commit that touched [the file]". The empty-string sentinel is structurally invalid against that contract; only `latestVersion` rewrites it with a real SHA on the winner. A future getter or comparator that touches `commit` on a non-winner candidate would silently consume `''`. The `as` cast hides the gap from TypeScript.

Fix: model the placeholder explicitly with a separate candidate type, and resolve `commit` only inside `latestVersion`:

```ts
type MobyOpenAPISpecCandidate = Pick<MobyOpenAPISpecVersion, 'apiVersion'>;
async getAvailableVersions(): Promise<MobyOpenAPISpecCandidate[]> { … }
```

S3. **`getChecksums` paths skip upstream cross-check without an inline comment explaining why** — `scripts/dependencies/wsl.ts:44-49, 69-74`, `scripts/dependencies/tools.ts:560-579, 635-644, 667-676` [Claude Opus 4.7]

`WasmShims` (`tools.ts:605`), `MobyOpenAPISpec` (`moby-openapi.ts:92-94`), and `Wix` (`wix.ts:33-34`) carry one-line comments naming the reason ("upstream does not publish a checksum file"). `ECRCredHelper`, `Moproxy`, `WSLDistro`, `CertManager`, and `SpinOperator` skip the cross-check silently. +    // Upstream does not publish a checksum file; record the sha256 we observe at bump time.
     const archiveName = `moproxy_${ version }_linux_x86_64_musl.bin.xz`;

S4. **`electron-builder.yml`'s recursive exclusions could drop legitimate archives nested under shipped trees** — `packaging/electron-builder.yml:20-23` [Claude Opus 4.7] (round 5 S15 unaddressed)

```yaml
- '!resources/**/*.unsigned'
- '!resources/**/*.tar.gz'
- '!resources/**/*.zip'
- '!resources/**/*.xz'
```

The intent is to exclude the new sibling caches in `resources/{platform}/{bin,internal,...}/`. The glob is broader: any `.tar.gz`, `.zip`, or `.xz` anywhere under `resources/` (including inside `resources/rancher-dashboard/`) drops from the package. None ship today; a future feature that legitimately bundles such an asset would silently lose it.

Fix: pin the exclusions to the directories that actually accumulate cached archives.

S5. **Per-version archive caches accumulate after a bump** — `scripts/dependencies/wix.ts:45`, `scripts/dependencies/lima.ts:49,107`, `scripts/dependencies/wsl.ts:26` [Claude Opus 4.7] (round 5 S10 / round 4 S3 unaddressed)

`Wix`, `Lima`, `Qemu`, and `Moproxy` embed the version in the cache filename (`wix314rtm.zip`, `lima.macos-15.amd64.v1.2.1.rd3.tgz`, `qemu.v9.2.1.tgz`, `moproxy_0.5.1_linux_x86_64_musl.bin.xz`), so each bump leaves the prior archive on disk. The packaged app excludes them via `electron-builder.yml`, so this is developer-disk-only.

Fix: drop the version from the cache filename, or sweep stale archives at postinstall start (keep only the one whose basename matches the current `dependencies.yaml` version).

S6. **`download()`'s EACCES swallow no longer matches the surrounding code** — `scripts/lib/download.ts:95-101` [Claude Opus 4.7]

```ts
} catch (ex: any) {
  // EACCES from the parent directory or the destination itself; either
  // way fall through and let mkdir/rename surface a clearer error.
  if (ex.code !== 'ENOENT' && ex.code !== 'EACCES') {
    throw ex;
  }
}
```

When the cache check fails with EACCES on the parent directory, `mkdir(parent, recursive: true)` is a no-op (the directory already exists), so the error propagates from the later write to `${destPath}.download`. By then the temp path is in the message, not the destination — less informative than rethrowing the original EACCES at the cache-check site.

Fix: rethrow EACCES and only swallow ENOENT.

```diff
-    } catch (ex: any) {
-      // EACCES from the parent directory or the destination itself; either
-      // way fall through and let mkdir/rename surface a clearer error.
-      if (ex.code !== 'ENOENT' && ex.code !== 'EACCES') {
-        throw ex;
-      }
-    }
+    } catch (ex: any) {
+      if (ex.code !== 'ENOENT') throw ex;
+    }
```

S12. **`parseSha256Checksum` lowercases hex without documenting the normalization** — `scripts/lib/dependencies.ts:93-99` [Claude Opus 4.7]

```ts
export function parseSha256Checksum(value: unknown): Sha256Checksum {
  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/i.test(value)) {
    throw new Error(...);
  }
  return value.toLowerCase() as Sha256Checksum;
}
```

`actualChecksum` from `crypto.createHash().digest('hex')` is always lowercase; `expectedChecksum` reaches `download()` only via `lookupChecksum`, which normalises through `parseSha256Checksum`. So `download.ts:78, 121` (`actualChecksum !== expectedChecksum`) is robust today. A test or fixture that constructs `Sha256Checksum` outside `parseSha256Checksum` (the brand allows `sha256:${ string }` literally) could regress this silently.

Fix: document the lowercase normalization in the `Sha256Checksum` JSDoc and the `parseSha256Checksum` body.

S13. **Schema migration shape is undocumented in the commit / release notes** — `pkg/rancher-desktop/assets/dependencies.yaml`, `scripts/lib/dependencies.ts:121-154` [Claude Opus 4.7]

`readDependencyManifest` requires `{version, checksums}` per entry. Anyone who reverts this PR or rebases an old branch on top of `main` sees `readDependencyManifest` throw "missing version field" for keys whose old shape was `name: scalar`. A short note in the commit body or release entry would save the next maintainer fifteen minutes.

## Design Observations

### Concerns

**D1. Unconditional re-extraction trade-off in `downloadTarGZ`/`downloadZip`** *(in-scope)* [Claude Opus 4.7]

The author's choice ("destPath always reflects the current archive") pays a developer-install cost on every run, even when nothing changed. The `RancherDashboard.download` `.source-sha256` stamp pattern is a natural template — short-circuit on stamp match — that would recover the pre-PR skip-on-cache-hit while still detecting partial extractions. The current divergence between `RancherDashboard` (stamps) and `downloadTarGZ`/`downloadZip` (always re-extract) is itself a design smell. The author has accepted this trade-off in this round; documenting it via the new comment is the minimum, but converging the three archive-extraction call sites onto the stamp pattern is the cleaner long-term shape.

**D2. Two parallel manifest types (`DependencyVersions` vs `DependencyManifest`)** *(future)* [Claude Opus 4.7]

mook-as raised this in the PR review and the structure has not changed. The split is ergonomic for the two call sites that only need versions (`lint-go.ts`, `docker-cli-monitor.ts`), but those callers could trivially read `manifest[name].version` themselves. Folding `DependencyVersions` into the per-key version field of `DependencyManifest` would yield a single source of truth.

**D3. Bump-time `getChecksums` materialises every artifact for every platform** *(future)* [carry-over from round 5 D3]

Each `getChecksums()` downloads each artifact, hashes, then deletes. A `--all` rebuild downloads gigabytes. If this becomes a hot path, switch `downloadAndHash` to a streaming hash that pipes the response body straight into `crypto.createHash('sha256')`.

### Strengths

**D4. Round 5 backlog cleared cleanly** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

Round 5's S1, S2, S3, S5, S6, S7, S8, and S11 all landed: 40-char hex commit guard, algorithm-aware sidecar parsing, `.xz` cache retention, codesign error detail, trust-on-bump comment, EACCES comment rewrite, throw-on-zero-entries, codesign filtered out of inner archive download. The compute-checksums-before-branching pattern in `rddepman.ts:291-297` keeps a failed bump from leaving an orphan branch.

**D5. The pinned-commit design for `MobyOpenAPISpec`** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

`apiVersion` for comparison and `commit` for URL immutability is the right separation for a `master`-branched upstream. Resolving the commit only on the winner (`latestVersion` getter) avoids one `listCommits` per historical version. The 40-char hex guard added this round closes the immutability hole.

**D6. `.unsigned` codesign cache is the cleanest option for in-place mutation** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5]

Keeping the verified bytes at `${destPath}.unsigned` and hashing them on cache checks integrates digest verification with `codesign`'s in-place rewrite without re-downloading or trusting the post-signing bytes.

**D7. Schema evolution preserves install-time simplicity** *(in-scope)* [Codex GPT 5.5, Gemini 3.1 Pro]

Storing every digest as `sha256:` even when upstream publishes sha512 confines multi-algorithm parsing to `rddepman` at bump time. Install-time `download()` only handles sha256. The compound entry shape mirrors the on-disk YAML, so the in-memory model and the file stay in sync without translation.

## Testing Assessment

No unit tests cover the new helpers. The build-script directory has minimal test infrastructure (`scripts/lib/__tests__/build-utils.spec.ts` is the only pre-existing spec). Specific scenarios that would catch the issues raised here:

1. `fetchUpstreamChecksums` parser: GNU vs BSD lines, with/without `*` binary marker, path-prefixed entries, mixed sha256/sha512 widths, BSD prefix mismatch (covers S12).
2. `parseSha256Checksum`: rejects empty, wrong-length, non-hex, no-prefix; accepts uppercase and normalises to lowercase (covers S12).
3. `readDependencyManifest`: rejects missing `version`, rejects array `version`, accepts string and object forms, handles missing `checksums` field.
4. `lookupChecksum` artifact-name parity: each dependency's `getChecksums()` must produce filenames that its `download()` later passes to `lookupChecksum` (covers S8).
5. `downloadTarGZ`/`downloadZip` re-extract semantics; `Moproxy.download` `7z e` non-interactive behaviour (covers S1).
6. `MobyOpenAPISpec.download()` rejects non-SHA `commit`; `getAvailableVersions` candidate type round-trip (covers S2).
7. `rddepman` dry-run: `getChecksums()` failure aborts before any git mutation; `updateManifest` preserves the compound YAML shape.

End-to-end, the PR's primary functional test is "rddepman runs and produces a working bump PR" — demonstrated by `9d35ac4cc` and `53aa0796a` in the history (lima 1.2.1.rd2 → 1.2.1.rd3 was generated by this branch).

## Documentation Assessment

The header comment in `dependencies.yaml`, the codesign comment block at `scripts/lib/download.ts:66-69`, the new `MobyOpenAPISpec.getChecksums` trust-on-bump comment, and the `Moproxy.download` cache-retention comment are exemplary — they explain *why*, not *what*. ## Acknowledged Limitations

- **Re-extraction overwrites locked binaries on Windows / Linux** (`scripts/lib/download.ts:215-225, 268-279`, round 5 I1 declined): "*copyFile overwrites destPath unconditionally, so a postinstall run while Rancher Desktop or another shell holds the binary fails with EBUSY/EPERM on Windows or ETXTBSY on Linux; stop the consuming process and retry.*" The author accepted the trade-off this round by adding the comment instead of adopting the `RancherDashboard` stamp pattern. macOS is unaffected (in-place overwrite of mapped binaries is allowed).
- **Stamp absence on crash starts over** (`scripts/dependencies/tools.ts:405-408`): "*A crash between mkdir and tar leaves the stamp absent, so the next run starts over.*"
- **Forcing sha256 over upstream algorithms** (`scripts/dependencies/lima.ts:81` review thread): @mook-as questioned not using upstream sha512; @jandubois explained that uniform sha256 storage simplifies install-time verification by confining multi-algorithm parsing to `rddepman` at bump time. @mook-as accepted ("It's fine to use whatever is easier for you").
- **WiX has no upstream checksum file** (`scripts/dependencies/wix.ts:33-34`): "WiX doesn't publish upstream checksum files; we rely on the digest recorded in dependencies.yaml at bump time."
- **Spin shim has no upstream checksum file** (`scripts/dependencies/tools.ts:605-606`).
- **Docker buildx omits darwin from upstream verification** (`scripts/dependencies/tools.ts:208-210`): upstream `checksums.txt` omits darwin entries (https://github.com/docker/buildx/issues/945).
- **Upstream linux/arm64 not yet published for QEMU** (`scripts/dependencies/lima.ts:96-99, 119`): the `download()` skip and the `getChecksums()` matrix omission mirror each other.
- **MobyOpenAPISpec commit is invisible to upstream security fixes** (`scripts/dependencies/moby-openapi.ts:131-136`): `rcompareVersions` ignores `commit`, so a security fix to `v1.54.yaml` on `master` does not auto-bump. Adopting a newer commit on the same `apiVersion` requires a manual `dependencies.yaml` edit.

## Design Observations

### Concerns

- **Unconditional re-extraction trade-off** *(in-scope)* — see I1. The author optimized for "destPath always reflects the current archive" but pays the cost on every developer install. A sibling-stamp pattern (as `RancherDashboard.download` already does) would cleanly recover the old skip-on-cache-hit behavior while still detecting partial extractions. Recommend converging the three archive-extraction call sites onto the stamp pattern; the current divergence (RancherDashboard: stamps; downloadTarGZ: always re-extracts) is itself a design smell.

- **Two parallel manifest types (`DependencyVersions` vs `DependencyManifest`)** *(future)* — mook-as raised this in the PR review and the structure didn't change. The split is ergonomic for the two call sites that only need versions (`lint-go.ts`, `docker-cli-monitor.ts`), but those callers could trivially call `manifest[name].version` themselves. Folding `DependencyVersions` into the per-key version field of `DependencyManifest` would yield a single source of truth and shrink `lib/dependencies.ts` by ~30 lines. Not a blocker for this PR; worth a follow-up.

### Strengths

- The "compute checksums before branching" change in `rddepman.ts` (line 291–297) is a clean abort-before-side-effects pattern; previous bumps could leave a half-created branch on a verification failure.
- Stripping `codesign` from inner options in `downloadTarGZ`/`downloadZip` (line 216, 272) correctly prevents the archive itself from being codesigned. Good defensive design.
- The `Sha256Checksum` template-literal brand and the per-entry `parseSha256Checksum` validator give strong typing at the boundary; misuse via raw string assignment would TypeScript-error.
- The "preserve mismatched bytes" path in `downloadAndHash` (line 248–258) is excellent for diagnosing CDN-served HTML error pages.

This PR overhauls the dependency management pipeline to securely verify artifact digests at install time. By recording upstream checksums inside the `dependencies.yaml` manifest and hashing every download against these pinned digests, it eliminates the risk of an upstream mirror swapping out an artifact unnoticed. The implementation is highly robust, employing well-isolated schema changes, clean caching semantics, and strong `sha256` type checking.

### Important Issues

I1. **Moproxy extraction hangs on retry if `rename` fails** — `scripts/dependencies/wsl.ts:28-32` (important, gap)

```typescript
    // Keep archivePath beside moproxyPath so download() hits its cache on
    // subsequent postinstall runs; electron-builder.yml excludes *.xz.
    await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
    await fs.promises.rename(path.join(context.internalDir, binName), moproxyPath);
```

By intentionally keeping `archivePath` for caching, `postinstall` will skip the download step on subsequent runs but still execute `7z e` unconditionally. If the `rename` step fails on a given run (e.g., because `moproxyPath` is held open and locked on Windows, throwing `EPERM` or `EBUSY`), the extracted `binName` is left behind in `internalDir`. On the next `postinstall` run, `7z e` will see that `binName` already exists and will prompt for confirmation to overwrite it (`file already exists. Overwrite with ... ? (Y)es / (N)o / ...`). Because this script runs non-interactively, it will hang indefinitely.

Fix: Pass the `-y` flag to `7z e` to assume "yes" on all overwrite prompts.

```diff
-    await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
+    await simpleSpawn('7z', ['e', '-y', archivePath], { cwd: context.internalDir });
```

### Design Observations

**Strengths**
- The `.source-sha256` stamp logic in `RancherDashboard` is a robust approach to caching extracted directories and gracefully recovering from partial or failed extractions.
- Pinned Moby OpenAPI commits cleanly resolve the problem of referencing mutable upstream source files. Overriding `latestVersion` to fetch the real commit allows `rddepman` to securely establish the immutable digest for the build, without burning GitHub API quota on historical versions.
- The `readDependencyManifest` implementation guarantees the in-memory object strictly mirrors the YAML layout, discarding any manual comments cleanly via `yaml.stringify` while ensuring a permanent `MANIFEST_HEADER` comment warns future contributors about the regeneration.

### Testing Assessment

1. **Upstream checksum sidecar parsing:** Consider adding unit tests for `fetchUpstreamChecksums` to guarantee it accurately extracts file paths and hashes from the wide variance of formats observed (GNU `sha256sum`, BSD style, embedded prefixes, and extraneous comments) and rejects malformed ones.
2. **Missing checksum handling:** Assert that missing artifact references inside a `.sha256sum` sidecar file explicitly trigger an upstream checksum mismatch failure, rather than returning a false positive or silently passing an `undefined` digest.

### Documentation Assessment

The rationale behind keeping `.unsigned` cached artifacts, keeping `.xz` archives, and triggering intentional extraction overrides are beautifully documented throughout the scripts directly inline where needed.

### Acknowledged Limitations

- **EBUSY on Windows/Linux during `postinstall`**: The author notes in `scripts/lib/download.ts` that unconditionally extracting the archive on every run causes an `EBUSY/EPERM` (or `ETXTBSY` on Linux) if the binary is currently in use by Rancher Desktop.
  > "so a postinstall run while Rancher Desktop or another shell holds the binary fails with EBUSY/EPERM on Windows or ETXTBSY on Linux; stop the consuming process and retry."
  The author intentionally accepted this limitation (instead of implementing a caching stamp here) to guarantee that `destPath` always strictly reflects the current archive.
