Executive Summary ¶

Round 2 of the userspace-forwarder rewrite. Round 1's critical and four important findings are resolved in 2725f96b0; the design holds, and the implementation is materially simpler than #10280. This round surfaces three new important issues in the scanner's reconciliation loop: binding-shape changes after publish never reconcile, the publish/rollback path floods logs every 3 seconds on persistent failure, and unpublish's "removed port" Info log fires even when tracker.Remove returned an error.

All three reviewers converged on the binding-shape and retry-storm findings; Claude alone caught the unpublish log inconsistency. Codex raised one false positive (UDP outbound dial sockets cannot reach the scanner — they sit in state=0x1, not UDPEstablished=0x7), and one over-severe finding (the tap-IP filter dropping legitimate tap-IP listeners is real but requires a container to bind to the namespace-internal tap address, which no realistic engine does).

I1. Tick never reconciles binding-shape changes on published ports Claude Gemini¶
src/go/guestagent/pkg/procnet/scanner_linux.go

// effect immediately: a vanished listener is unambiguous.
func (p *ProcNetScanner) Tick(scanned nat.PortMap) {

		if _, ok := scanned[port]; ok {
			continue
		}
		p.unpublish(port, bindings)
		delete(p.published, port)
	}

	for port := range p.pending {
		bindings, ok := scanned[port]
		if !ok {

The loop checks key membership alone; it never compares scanned[port] against the stored p.published[port] bindings. A container restart with a new bind — typically 0.0.0.0:8009 → 127.0.0.1:8009 — keeps the port present, so Tick skips unpublish, the published map keeps the stale 0.0.0.0 binding, and loop 3 skips because the port is in published. The forwarder, which was never started for the wildcard, is also never started for the new loopback. Windows-side traffic to bindIP:8009 finds no listener until the upstream truly vanishes. The reverse transition is normally blocked by EADDRINUSE against the forwarder's lingering bindIP:8009, so the realistic failure mode is wildcard-to-loopback.

Fix: when the key is present but bindings differ, run the full unpublish/republish cycle.

+    if newBindings, ok := scanned[port]; ok && bindingsEqual(bindings, newBindings) {

I2. publish rollback path floods logs every 3 seconds on persistent failure Claude Gemini¶
src/go/guestagent/pkg/procnet/scanner_linux.go
// publish reports a new listener to the API tracker and opens a
// userspace forwarder for each loopback binding. It returns an error
// if either step fails after rolling back the tracker entry, so the
// caller can leave the port in pending for next-tick retry instead
// of recording it as published.
func (p *ProcNetScanner) publish(port nat.Port, bindings []nat.PortBinding) error {
	id := utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port()))
	if err := p.tracker.Add(id, nat.PortMap{port: bindings}); err != nil {
		log.Errorf("/proc/net scanner failed to add %s: %s", port, err)
		if removeErr := p.tracker.Remove(id); removeErr != nil {
			log.Errorf("/proc/net scanner rollback after tracker.Add failure for %s: %s", port, removeErr)
		}
		return err
	}
	log.Infof("/proc/net scanner added port: %s -> %+v", port, bindings)

	for _, b := range bindings {
		if b.HostIP != loopbackIP {
			continue
		}
		portNum, err := strconv.ParseUint(b.HostPort, 10, 16)
		if err != nil {
			// b.HostPort is strconv.Itoa of a uint16 (see
			// addEntryToPortMap), so ParseUint always succeeds. If that
			// stops being true, the caller leaks a tracker entry without
			// a matching forwarder; mirror the forwarder.Add rollback
			// above.
			log.Errorf("/proc/net scanner: bad port %q: %s", b.HostPort, err)
			continue
		}
		if err := p.forwarder.Add(p.ctx, port.Proto(), uint16(portNum)); err != nil {
			log.Errorf("/proc/net scanner: loopback forwarder %s/%s: %s", port.Proto(), b.HostPort, err)
			if removeErr := p.tracker.Remove(id); removeErr != nil {
				log.Errorf("/proc/net scanner rollback after forwarder.Add failure for %s: %s", port, removeErr)
			}
			return err
		}
	}
	return nil
}

func (p *ProcNetScanner) unpublish(port nat.Port, bindings []nat.PortBinding) {
	id := utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port()))
	if err := p.tracker.Remove(id); err != nil {

When wsl-proxy or host-switch is persistently down, tracker.Add and the rollback tracker.Remove both fail every tick. Each failed port emits two Error lines per scanInterval (~3 s). With multiple ports stuck, the log fills quickly. #10280 specifically added an addErrorLogged map to demote follow-up failures to Debug; this redesign drops that surface. The error class is preserved, but the diagnostic guardrail is not.

Fix: keep a per-port "first failure logged" flag and demote subsequent occurrences to Debugf; reset on success. See S2 below — EADDRINUSE on the forwarder is the most likely persistent trigger.

S1. entriesToPortMap filter drops legitimate tap-IP listeners Codex Gemini¶
src/go/guestagent/pkg/procnet/scanner_linux.go
// entriesToPortMap converts procnet entries into a port map, dropping
// any entry whose IP matches bindIP. The forwarder opens its own
// socket on bindIP for every loopback port it proxies; leaving those
// entries in the snapshot keeps the proto/port key alive after the
// upstream listener exits and blocks unpublish.
func (p *ProcNetScanner) entriesToPortMap(entries []procnettcp.Entry) nat.PortMap {
	out := make(nat.PortMap)
	for _, entry := range entries {
		if entry.IP.Equal(p.bindIP) {
			continue
		}
		if err := addValidProtoEntryToPortMap(entry, out); err != nil {
			log.Errorf("failed to create portMapping for entry: %s", err)
		}
	}
	return out
}

	case procnettcp.TCP:

The filter targets the forwarder's own listener, but the predicate is address-only. A container that explicitly binds to the tap-interface IP (192.168.127.2) instead of 0.0.0.0 or 127.0.0.1 is dropped before reaching the publish loop, even when no forwarder owns that proto/port. The pre-PR code published the port. Realistic engines (nerdctl, docker) do not bind to the tap IP — it is a namespace-internal artifact — so the practical surface is small. Worth tightening when procnettcp exposes inode-level ownership; until then, document the trade-off in the package comment.

Fix: replace the address-only filter with an ownership check against the forwarder's tcp/udp maps, or document the limitation.

Design Observations ¶

Concerns ¶

  • The stability gate covers presence but not shape (in-scope) Claude Gemini

I1 covers the symptom. The deeper observation: the two-scan gate is now the only correctness barrier on the publish side, and it answers only "is the port still here." Bindings are stored once at promotion and never re-checked. A small structural addition — compare bindings on every tick, or store a generation counter and re-publish on mismatch — closes the class.

  • publish collapses transient and permanent failures into one retry loop (in-scope) Claude

I2 and S2 describe two failure modes — wsl-proxy down and EADDRINUSE on the forwarder bind — that share one rollback-then-retry path with no rate limit. #10280's addErrorLogged was the explicit guard against this class. The redesign drops the guard along with the state machine; the underlying failure semantics need it as much as before.

  • Scanner and forwarder lack a Go-side integration seam (future) Claude

S4 covers the symptom. The structural point: the redesign trades a 665-line state machine with table-driven unit tests for two cleanly-separated components with focused unit tests on each side. The seam between them — the actual scan → publish → forward → upstream loop — only meets in production. The bats test is the existing integration check; one Go-level integration test would catch a future seam regression earlier.

Strengths ¶

  • Net simplification of ~1850 lines. Eliminating portStateTracker, the engine-chain probing helpers, and the route_localnet sysctl in exchange for a 209-line userspace forwarder is the central design win. The justification in loopback_forwarder_linux.go:9-25 reads cleanly.
  • Tick's three-phase structure (unpublish-vanished → promote-pending → rebuild-pending) makes the invariants legible and unit-testable.
  • bindIP self-filter plus its dedicated test (TestEntriesToPortMapSkipsForwarderBindIP) is the minimal fix for Round 1's C1 and has a regression pin.
  • TCP accept loop uses bounded exponential backoff rather than tight-looping on transient EMFILE/ENOBUFS. The pattern proactively addresses a class of CPU-pegging bugs.
  • Half-close + drain deadline (Round 1's I4 fix) correctly preserves the client-CloseWrite → upstream-response shape that the DNAT path handled at the IP layer.

Testing Assessment ¶

Ranked by risk:

  1. Binding-shape change on an already-published port (I1). TestBindingShapeChangePublishesCurrent covers the pending case; the published case is untested.
  2. forwarder.Add failure on EADDRINUSE (S2). fakeForwarder.addErr covers a generic failure; the address-collision scenario (wildcard listener already bound) needs its own test asserting the typed-error path.
  3. Retry behaviour on persistent wsl-proxy failure (I2). No test asserts log-throttle behaviour because none exists.
  4. End-to-end scanner + real forwarder + upstream (S4). The integration this PR is built around is unverified in CI; only the Windows bats test covers it.
  5. Tick timing under jitter. All tests synthesise ticks deterministically; the real ticker can fire late. Low risk but worth a regression guard if practical.

Repo context updates ¶

  • [repo] When reviewing scanner code that processes /proc/net/{tcp,udp} snapshots via lima's procnettcp package, note that procnettcp.UDPEstablished is 0x7 — Linux's TCP_CLOSE constant, which marks unconnected/bound UDP sockets (listeners). Connected UDP sockets — those returned by net.Dial("udp", ...) — are state 0x1 and are excluded from the scanner's snapshot. A cascade finding where outbound UDP dial sockets become "discovered listeners" requires the connected-socket state to match the filter, which it does not. Verify the lima constant value before raising such a finding.
