## Executive Summary

All nine Round 2 suggestions landed in this commit. The schema /
helper plumbing is in good shape: the `Moby` comment was rewritten,
the basename-collision suppression in `fetchUpstreamChecksums`
shipped, the manifest cache is now module-scoped, the
`populate-dependency-checksums.ts` migration script was deleted, the
dead `checksumAlgorithm` plumbing is gone, the `lookupChecksum` error
path now distinguishes "dependency missing" from "artifact missing,"
`RawEntry.checksums` is typed optional, and the
`chartFilename`/`crdsFilename` indentation is back in sync.

The notable new finding is a regression introduced by Round 2's S3
fix itself: `download()` now hashes existing destination files against
the upstream-pinned sha256 before short-circuiting, but call sites
that combine `expectedChecksum` with `codesign` (DockerCLI on darwin
at `scripts/dependencies/tools.ts:160-163`, DockerProvidedCredHelpers
on darwin at `scripts/dependencies/tools.ts:489-495`) now compare the
locally-resigned Mach-O bytes against the upstream pre-signature
digest. The post-codesign hash never matches; every macOS postinstall
re-downloads and re-signs Docker CLI and the Docker credential
helpers, and the postinstall fails offline even when a known-good
signed binary is on disk. I1 below proposes a one-line `codesign`
short-circuit.

The rest of the surface produces refinement-grade suggestions: archive
helpers (`downloadTarGZ` / `downloadZip`) skip the same way Round 2's
S3 explicitly scoped out (S1); `MobyOpenAPISpec.getAvailableVersions`
resolves a commit for every historical apiVersion when only the
winner is used (S2); a small set of `getChecksums` enumerations omit
windows/arm64 while a few peer enumerations include it (S4); the
`parseSha256Checksum` regex rejects uppercase hex from manual edits
(S3); and a handful of polish items (Sha256Checksum return type,
diagnostic preservation in `downloadAndHash`, `readDependencyManifest`
schema validation, GNU regex tightening, MACOS_RUNNER hard-coding,
WiX `v`-prefix asymmetry, in-file no-comments warning).

Verdict: merge with the codesign fix folded in; everything else is
non-blocking.

## Important Issues

I1. **`codesign` callers re-download every postinstall — `expectedChecksum` compared against post-signature bytes** — `scripts/lib/download.ts:67-87` [Codex GPT 5.5]

```ts
if (!overwrite) {
  try {
    await fs.promises.access(destPath, access);
    if (expectedChecksum) {
      const actualChecksum = await hashFile(destPath, 'sha256');

      if (actualChecksum === expectedChecksum) {
        console.log(`${ destPath } already exists with expected checksum, not re-downloading.`);

        return;
      }
      console.log(`${ destPath } exists but sha256 differs ([${ actualChecksum }] vs expected [${ expectedChecksum }]); re-downloading.`);
    } else {
      console.log(`${ destPath } already exists, not re-downloading.`);

      return;
    }
  } catch (ex: any) {
    if (ex.code !== 'ENOENT') {
      throw ex;
    }
  }
}
```

Round 2's S3 fix added the cache-hit checksum check at lines 69-77 so
that a corrupted local file no longer poisons subsequent installs.
That check fires regardless of `options.codesign`, but `download()`
applies an ad-hoc signature to `destPath` after writing it
(lines 126-131 in the same file):

```ts
if (options.codesign) {
  spawnSync(
    'codesign',
    ['--force', '--sign', '-', destPath],
    { stdio: 'inherit' },
  );
}
```

`codesign --force --sign -` rewrites the Mach-O LC_CODE_SIGNATURE load
command in place, mutating the file bytes. The first install passes
the pre-signature checksum check (the file is hashed in
`tempPath` before `chmod`/`rename`/`codesign`, lines 104-115), so the
upstream-pinned sha256 is verified against the as-downloaded artifact
correctly. On the *next* postinstall, however, the file at `destPath`
is the post-signature variant, and the cache-hit path at line 70
hashes it against the same pre-signature `expectedChecksum`. The
hashes never match, so the binary is re-downloaded and re-signed
every time.

Affected call sites:

```ts
// scripts/dependencies/tools.ts:160-163 — DockerCLI.download
const expectedChecksum = lookupChecksum(context, this.name, executableName);
const codesign = process.platform === 'darwin';

await download(dockerURL, destPath, { expectedChecksum, codesign });

// scripts/dependencies/tools.ts:489-495 — DockerProvidedCredHelpers.download
const expectedChecksum = lookupChecksum(context, this.name, fullBinName);
// ...
const codesign = context.platform === 'darwin';

promises.push(download(sourceURL, destPath, { expectedChecksum, codesign } ));
```

Impact: every macOS `yarn postinstall` after the first re-downloads
and re-signs the Docker CLI plus all Docker credential helpers
(`docker-credential-osxkeychain` today; the same pattern applies if
darwin grows more named helpers). Worst case is a developer running
`yarn postinstall` offline after a fresh install: there is a
known-good signed binary on disk, but `download()` rejects it and
fails the network fetch.

Fix: short-circuit when `options.codesign` was applied. The first
install verified the upstream digest in `tempPath` before signing, so
the on-disk file is provably the verified artifact plus a
locally-applied signature; comparing it back to the pre-signature
digest is meaningless for re-installs.

```diff
       await fs.promises.access(destPath, access);
       if (expectedChecksum) {
+        if (options.codesign) {
+          // Codesign rewrites destPath after download, so the on-disk
+          // bytes no longer match the upstream-pinned digest.  Trust
+          // the previously-verified-then-resigned file.
+          console.log(`${ destPath } already exists (locally signed), not re-downloading.`);
+
+          return;
+        }
         const actualChecksum = await hashFile(destPath, 'sha256');
```

Codex flagged this. Verified against the current code at the lead
worktree: `download()` lines 66-88 + 126-131 in `scripts/lib/download.ts`,
and `tools.ts:160-163` and `tools.ts:489-495`. (important, regression)

## Suggestions

S1. **Archive helpers (`downloadTarGZ` / `downloadZip`) skip checksum verification when extracted output exists** — `scripts/lib/download.ts:175-222, 233-268` [Codex GPT 5.5]

```ts
export async function downloadTarGZ(url: string, destPath: string, options: ArchiveDownloadOptions = {}): Promise<string> {
  const overwrite = options.overwrite ?? false;
  const access = options.access ?? fs.constants.X_OK;

  if (!overwrite) {
    try {
      await fs.promises.access(destPath, access);
      console.log(`${ destPath } already exists, not re-downloading.`);

      return destPath;
```

Round 2's S3 fix taught `download()` to re-hash an existing
destination against the pinned digest before short-circuiting, but
the archive helpers retained the pre-existing access-only short
circuit. Helm at `scripts/dependencies/tools.ts:121-124`, Trivy at
`tools.ts:324`, Steve at `tools.ts:391-396`, the WasmShims at
`tools.ts:621-625`, and the SpinCLI archive at `tools.ts:709-718` all
go through `downloadTarGZ` with `expectedChecksum` set, but on
subsequent postinstalls the helper returns at the access check and
never invokes the inner `download()` that would have verified the
archive. The same hole exists in `downloadZip` at lines 233-268
(used by KuberlrAndKubectl on Windows at `tools.ts:50-55`).

Round 2's S3 explicitly scoped this out: *"The archive helpers
(`downloadTarGZ`, `downloadZip`) skip the same way after extraction;
addressing them needs a sidecar stamp file or similar, which is out
of scope for this PR."* That reasoning still holds: the verified
artefact is the archive, not the extracted file, so a hash check on
`destPath` would have to either redownload the archive every run
(wasteful) or persist a stamp recording "we extracted from this
verified source." Track as a follow-up.

Fix: in a follow-up PR, persist a `${ destPath }.checksum` sidecar
after a successful extract, write the pinned `sha256:` and the source
URL, and skip the redownload only when the sidecar matches.

S2. **`MobyOpenAPISpec.getAvailableVersions` makes ~35 GitHub API calls and uses one** — `scripts/dependencies/moby-openapi.ts:95-115` [Claude Opus 4.7]

```ts
async getAvailableVersions(): Promise<MobyOpenAPISpecVersion[]> {
  const args = {
    owner: this.githubOwner, repo: this.githubRepo, path: 'api/docs',
  };
  const response = await getOctokit().rest.repos.getContent(args);
  const fileObjs = response.data as Partial<{ name: string }>[];
  const apiVersions: string[] = [];

  for (const fileObj of fileObjs) {
    const match = fileObj.name?.match(/^v([0-9]+\.[0-9]+)\.yaml$/);

    if (match) {
      apiVersions.push(match[1]);
    }
  }

  return Promise.all(apiVersions.map(async(apiVersion) => ({
    apiVersion,
    commit: await this.latestCommitForApiVersion(apiVersion),
  })));
}
```

`api/docs` lists roughly 35 historical `vX.Y.yaml` files. The map at
line 111-114 spawns one `listCommits` request per entry. Only the
highest `apiVersion` survives `rcompareVersions` at line 121-123, so
the other ~34 commit lookups are discarded. The throttling handler at
`scripts/lib/dependencies.ts:548` caps the damage at delay rather
than failure, but the budget is real (a typical rddepman cron runs
authenticated against a 5000/hr limit shared by every dep).

Fix: defer the commit lookup until after `latestVersion` picks the
winner. One sketch:

```diff
-    return Promise.all(apiVersions.map(async(apiVersion) => ({
-      apiVersion,
-      commit: await this.latestCommitForApiVersion(apiVersion),
-    })));
+    return apiVersions.map(apiVersion => ({ apiVersion, commit: '' }));
   }
+
+  override async latestVersion(): Promise<MobyOpenAPISpecVersion> {
+    const winner = await super.latestVersion();
+    return { ...winner, commit: await this.latestCommitForApiVersion(winner.apiVersion) };
+  }
```

`getChecksums(version)` already trusts the caller's commit, so it
keeps working unchanged. Update `currentVersion` only if the diff
between the cached `commit` and the resolved `commit` matters; with
the design choice that commit-only changes are not auto-bumped
(`rcompareVersions` ignores `commit`), the cached value need not be
re-resolved.

S3. **`parseSha256Checksum` rejects uppercase hex from manual edits** — `scripts/lib/dependencies.ts:91-96` [Gemini 3.1 Pro]

```ts
export function parseSha256Checksum(value: unknown): Sha256Checksum {
  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/.test(value)) {
    throw new Error(`Invalid sha256 checksum ${ JSON.stringify(value) }; expected "sha256:<64 hex chars>"`);
  }

  return value as Sha256Checksum;
}
```

`rddepman` writes lowercase hex consistently, so this is a strictly
manual-edit hazard. A maintainer pasting an upstream hex digest that
came from a generator using uppercase will trip the validator on the
next postinstall. The error message names the file and the offending
value, so the failure is recoverable; cosmetic but worth a one-line
fix.

```diff
-  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/.test(value)) {
+  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/i.test(value)) {
     throw new Error(`Invalid sha256 checksum ${ JSON.stringify(value) }; expected "sha256:<64 hex chars>"`);
   }

-  return value as Sha256Checksum;
+  return value.toLowerCase() as Sha256Checksum;
```

S4. **Several `getChecksums` enumerations omit windows/arm64 while peer dependencies record it** — `scripts/dependencies/tools.ts:59-63, 126-130, 169-174, 212-216, 555-559, 696-700, 371-375` [Gemini 3.1 Pro]

```ts
const platforms: [string, string][] = [
  ['linux', 'amd64'], ['linux', 'arm64'],
  ['darwin', 'amd64'], ['darwin', 'arm64'],
  ['windows', 'amd64'],
];
```

`KuberlrAndKubectl`, `Helm`, `DockerCLI`, `DockerBuildx`,
`ECRCredHelper`, `Steve`, and `SpinCLI` enumerate this five-entry
matrix; `DockerCompose` (line 249-253) and `DockerProvidedCredHelpers`
(line 511-517) instead generate a six-entry matrix that includes
windows/arm64 (or aarch64). The corresponding `download()` methods
*do* handle windows/arm64 — every one uses
`arch = context.isM1 ? 'arm64' : 'amd64'` and constructs a
windows-arm64 filename when `M1=1` is set — so a Windows host running
`yarn postinstall` with `M1=1` would call
`lookupChecksum(context, 'dockerCLI', 'docker-windows-arm64.exe')`
and trip the "No checksum recorded" diagnostic. The official
`.github/workflows/package.yaml` matrix sets `M1=1` only on
`mac/aarch64` (lines 229-231), so no shipping CI build hits this
path; the asymmetry still surfaces for any developer cross-building
on Windows ARM64.

Whether this is a real regression or a no-op depends on what each
upstream actually publishes — `dockerProvidedCredentialHelpers`
publishes `docker-credential-wincred-vX.windows-arm64.exe` (recorded
in `dependencies.yaml`), `dockerCompose` publishes
`docker-compose-windows-aarch64.exe` (recorded). For
`rancher-sandbox/rancher-desktop-docker-cli`, `helm`, `steve`,
`buildx`, `kuberlr`, ECR, and SpinCLI, the upstream may or may not
publish windows-arm64 binaries; a pre-PR `findChecksum` would have
also failed if the artefact was missing from the upstream sidecar.

Fix: confirm the intent. Either drop the `isM1` branch from each
windows-only `download()` (clarifying that windows arm64 is
unsupported and the one-line crash is intentional), or extend the
seven `getChecksums` matrices so every platform/arch combination the
download path can request is recorded. Document the choice once in
`scripts/dependencies/global.ts` so the pattern is consistent.

S5. **`lookupChecksum` returns raw `string`, throwing away the `Sha256Checksum` template-literal type** — `scripts/lib/dependencies.ts:179-201` [Claude Opus 4.7]

```ts
export function lookupChecksum(
  context: DownloadContext,
  name: keyof DependencyVersions,
  artifactName: string,
): string {
  // ...
  return prefixed.slice('sha256:'.length);

```

The function strips the `sha256:` prefix to fit `download()`'s
`expectedChecksum: string` shape, throwing away the type information
that `Sha256Checksum` exists to preserve. Local enhancement: keep the
prefixed value and let `download()` handle the strip, or change

S6. **`downloadAndHash` deletes the temp artefact on logical error** — `scripts/lib/dependencies.ts:209-244` [Claude Opus 4.7]

```ts
} finally {
    await fs.promises.rm(workDir, { recursive: true, maxRetries: 10 });
}
```

The `finally` block deletes `workDir` after every exit, including the
verification-mismatch throws at lines 220-225 and 230-235. The error
message names the URL and the two hashes, but the maintainer has no
way to inspect what was actually served — for a security-relevant
helper, a downloaded HTML error page (200 OK with a benign
`Content-Type`) would be silently discarded.

Fix: on mismatch, move the temp artefact to
`os.tmpdir()/rddepman-mismatch-<basename>-<timestamp>` before
throwing, and include that path in the error message. The `finally`
removes only the empty `workDir` afterward.

S7. **`download()` cache-hit path re-throws `EACCES`** — `scripts/lib/download.ts:66-87` [Claude Opus 4.7]

```ts
} catch (ex: any) {
  if (ex.code !== 'ENOENT') {
    throw ex;
  }
}
```

If `destPath` exists but does not satisfy the access mask (typically
because a previous run wrote it `0644` and this run wants `X_OK`),
`fs.promises.access` throws `EACCES` and the download aborts. Pre-PR
behaviour, but the cache-hit branch this PR added invites a reader
to expect robust handling of stale state. One-line:

```diff
-    if (ex.code !== 'ENOENT') {
+    if (ex.code !== 'ENOENT' && ex.code !== 'EACCES') {
       throw ex;
     }
```

S8. **`MACOS_RUNNER` constant produces an HTTP failure rather than the clear bump message its comment implies** — `scripts/dependencies/lima.ts:24-28` [Claude Opus 4.7]

```ts
// Names the macOS runner that builds the rancher-desktop-lima darwin
// archive.  Both `download()` and `getChecksums()` embed this token in
// artifact filenames, so update it here when upstream bumps the runner
// (e.g. `macos-16`).
const MACOS_RUNNER = 'macos-15';
```

When upstream `rancher-sandbox/rancher-desktop-lima` rolls to
`macos-16`, the next bumped tag will publish
`lima.macos-16.{amd64,arm64}.tar.gz`; `getChecksums()` continues to
request `lima.macos-15.*` and `fetchUpstreamChecksums()` 404s on the
sidecar lookup. The commit's stated workflow ("bump this constant
when upstream updates") still works, but the failure mode is an HTTP
error, not the runner-mismatch message the comment suggests.

Fix (follow-up): list the release assets at bump time and match the
prefix `lima.macos-*.{amd64,arm64}.tar.gz`, falling back to the
constant only at install time.

S9. **`readDependencyManifest` accepts any `unknown` for `version`** — `scripts/lib/dependencies.ts:140-143` [Claude Opus 4.7]

```ts
(manifest as any)[name] = {
  version: entry.version,
  checksums,
};
```

`entry.version` is typed `unknown`. A future YAML edit that puts
`version: [1, 2]` or `version: null` for a string-typed dependency
parses without error; the failure surfaces deep in `download()` as a
`[object Object]` or `undefined`-segmented URL. Adding a narrow guard
here (`typeof === 'string' || (typeof === 'object' && version !==
null)`) names the offending key in the diagnostic.

S10. **`fetchUpstreamChecksums` GNU regex matches any hex-prefixed line** — `scripts/lib/dependencies.ts:259` [Claude Opus 4.7]

S12. **`dependencies.yaml` lacks an in-file no-comments warning** — `pkg/rancher-desktop/assets/dependencies.yaml:1` [Claude Opus 4.7]

```ts
// scripts/lib/dependencies.ts:149-156
/**
 * Writes the manifest to disk in compound-entry form.  Round-tripping through
 * `YAML.parse` and `YAML.stringify` discards any inline comments, so do not
 * add documentation comments to `dependencies.yaml`; the next rddepman bump

```

The constraint is real (`Extension.updateManifest` at
`scripts/lib/extension-data.ts:65-71` deliberately uses string

visible at the point of editing:

```yaml

```

`parseSha256Checksum` constrain the on-disk shape at the type
boundary. Type-system enforcement of the prefix is the correct way
to keep encoding asymmetry (store with prefix, consume without)
local to `lookupChecksum`.

## Executive Summary

This PR converts every entry in `dependencies.yaml` from a bare version string to a compound `{ version, checksums }` record, and rewires both the install-time downloads and the rddepman bump path to use the stored sha256 instead of fetching upstream sidecars at install time. Findings are concentrated around test coverage, a quadratic API usage in MobyOpenAPISpec, and a few polish items.

**I1. `MobyOpenAPISpec.getAvailableVersions` makes N+1 GitHub API calls** — `scripts/dependencies/moby-openapi.ts:95-115`

```typescript
return Promise.all(apiVersions.map(async(apiVersion) => ({
  apiVersion,
  commit: await this.latestCommitForApiVersion(apiVersion),
})));
```

`getAvailableVersions` resolves the latest commit for every API version returned by the directory listing (lines 103-109 push every `vX.Y.yaml` filename). At the time of writing the moby `api/docs` directory contains roughly 35 historical versions; this PR turns one previously-cheap directory listing into one listing plus 35 `listCommits` calls per rddepman run. Only the highest `apiVersion` is needed because `rcompareVersions` (line 121) compares only on `apiVersion` — the commits attached to all the other versions are computed and immediately discarded.

```diff
-    return Promise.all(apiVersions.map(async(apiVersion) => ({
-      apiVersion,
-      commit: await this.latestCommitForApiVersion(apiVersion),
-    })));
+    // Order by apiVersion so latestVersion picks the highest; resolve the
+    // commit lazily via getLatestCommit() so we don't burn an API call per
+    // historical entry.
+    return apiVersions.map(apiVersion => ({
+      apiVersion,
+      commit: '',
+    }));
```

…and resolve the commit inside `getChecksums(version)` and at the top of `latestVersion` only for the chosen entry. (You will also need to adjust `currentVersion` comparison, since the cached current already has a commit and the placeholder doesn't.) An alternative is to override `latestVersion` to pick by `apiVersion` first, then call `latestCommitForApiVersion` once. (important, regression)

**S6. `fetchUpstreamChecksums` GNU regex matches lines that aren't checksum entries** — `scripts/lib/dependencies.ts:259`


## Design Observations

### Strengths

- **Schema migration done in one shot, not behind a flag.** The compound-entry shape is immediately authoritative; the `readDependencyVersions` wrapper at `scripts/lib/dependencies.ts:163-172` lets all the legacy consumers (`lint-go`, `docker-cli-monitor`) keep their existing API. This is the right place to add a translation, since they only need the version field.
- **Pin-the-commit fix for MobyOpenAPISpec is correct.** The previous URL referenced `master` (`scripts/dependencies/moby-openapi.ts` pre-PR), which made digest pinning useless since a force-push or branch rewrite would produce a different file at the same digest-bearing URL. Pinning a commit SHA (`MobyOpenAPISpecVersion.commit`) is the right structural fix and matches the system-prompt rule ("verify the URL itself is immutable").
- **`getChecksums` aborts before any git operations** (`scripts/rddepman.ts:291-297`). The "compute first, branch second" ordering is the right shape, the comment explains why, and an upstream sidecar mismatch produces a single PR-less failure instead of an orphaned branch.
- **Re-download-on-checksum-mismatch in `download()`** (`scripts/lib/download.ts:69-77`) closes a real failure mode: a previous run that wrote a corrupt file (truncated mid-stream, OOM during write) currently keeps poisoning every subsequent install. New behaviour self-heals.

### Concerns

- **Coupling between artifact filename in YAML and code expectation is implicit.** `lookupChecksum` errors out at install time if the call site disagrees with the YAML key by even one character. The error includes a sorted list of available keys, which mitigates it, but a verification step in `postinstall` (or an `eslint-rule`-style check at build time) that walks every `lookupChecksum(context, this.name, X)` and confirms `X` is producible from `getChecksums(currentVersion)` would catch drift before a release. `(future)`

- **Migration script not committed.** The PR description references `scripts/populate-dependency-checksums.ts` ("Used once to seed the data in this commit") but the file is not present in the diff and `git log --diff-filter=A` finds no creation event. If the script will never be needed again (because rddepman's `getChecksums` is the canonical source going forward), document that explicitly in a comment or commit message; otherwise commit the script so the next person who wants to backfill can do so. `(in-scope)`

### Executive Summary

This PR successfully closes a supply-chain vulnerability window by pinning the exact sha256 digests of all dependency artifacts directly into `dependencies.yaml`. Install-time scripts now perform local cryptographic verification without fetching live upstream checksum files. The structural refactoring of `rddepman` correctly handles bump isolation and upstream verification, but there is an important regression where a few dependencies incorrectly drop Windows ARM64 support from their tracked artifact arrays.

#### Important Issues

**I1. Missing Windows ARM64 checksums for multi-platform dependencies** — `scripts/dependencies/tools.ts:60-64` (important, regression)

```typescript
    const platforms: [string, string][] = [
      ['linux', 'amd64'], ['linux', 'arm64'],
      ['darwin', 'amd64'], ['darwin', 'arm64'],
      ['windows', 'amd64'],
    ];
```

The new `getChecksums` methods for `KuberlrAndKubectl`, `DockerCLI`, `DockerBuildx`, and `ECRCredHelper` enumerate a hardcoded matrix of platforms that omits `['windows', 'arm64']`. Previously, `postinstall.ts` dynamically resolved the requested artifact based on the host architecture (via `context.isM1`) and fetched it. Now that `dependencies.yaml` acts as a strict allowlist, omitting the artifact means `lookupChecksum` will crash the build with a "No checksum recorded" error when Rancher Desktop is compiled/installed on a Windows ARM64 host. 

Upstream releases for these dependencies do publish Windows ARM64 assets (e.g., `docker-windows-arm64.exe`). The PR author correctly included `windows/arm64` equivalents for `DockerCompose` (line 248) and `DockerProvidedCredHelpers` (line 507), showing a clear intent to support it across the codebase.

Fix: Add `['windows', 'arm64']` to the `platforms` array in `getChecksums` for `KuberlrAndKubectl`, `DockerCLI`, `DockerBuildx`, and `ECRCredHelper`.

#### Suggestions

**S1. Strict lowercase regex for sha256 checksums rejects manual edits** — `scripts/lib/dependencies.ts:91-95` (suggestion, enhancement)

```typescript
export function parseSha256Checksum(value: unknown): Sha256Checksum {
  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/.test(value)) {
    throw new Error(`Invalid sha256 checksum ${ JSON.stringify(value) }; expected "sha256:<64 hex chars>"`);
  }

  return value as Sha256Checksum;
}
```

The regex strictly enforces lowercase hexadecimal characters. While `rddepman` correctly generates and saves lowercase hashes, a developer manually pinning a new digest in `dependencies.yaml` (e.g., directly copied from an upstream source that provides uppercase hex) will encounter an `Invalid sha256 checksum` crash during `postinstall.ts`.

Fix: Add the `i` flag to the regex to tolerate uppercase letters, and cast the string to lowercase before returning so downstream string comparisons remain strictly case-insensitive.

```diff
-  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/.test(value)) {
+  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/i.test(value)) {
     throw new Error(`Invalid sha256 checksum ${ JSON.stringify(value) }; expected "sha256:<64 hex chars>"`);
   }
 
-  return value as Sha256Checksum;
+  return value.toLowerCase() as Sha256Checksum;
```
