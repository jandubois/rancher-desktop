Executive Summary ¶

The PR adds a default "Info" tab backed by a fetchContainerInspect store action that caches docker inspect output. The architecture is sound, but the new code has lifecycle gaps: a stale-cleanup race in fetchContainers, no retry when the client becomes ready on direct reload, an unconditional spinner that hides cached data on tab return, stale data shown silently on refresh failures, and CLI errors swallowed by an err.message read.

All three reviewers participated in pass 1. Race-condition and lifecycle findings came mostly from Claude and Codex; Gemini flagged the most user-visible UX symptom and the error-shape mismatch. Overlap landed on the CLI error finding and the misleading SCSS comment.

Design Observations ¶

Concerns ¶

  • Inspect data is fetched once per container id and never refreshed while the tab is open (future) Claude. The component only triggers fetchContainerInspect when containerId changes; the store retains the previously-fetched JSON until either fetchContainers observes the container has disappeared or SET_PARAMS clears state. A long-lived Info tab on a container that gets reconfigured (network attach, runtime exec, rename) silently shows stale data forever. Hooking the existing container event subscriber to invalidate the entry on start/update/rename/network-connect events would close the gap without a poll.
  • ContainerInspectData is a hand-rolled partial of moby's types.ContainerJSON (future) Claude.
