Executive Summary ¶

After verification, three Important findings remain, all sharing one theme: when an engine chain takes over a port, the code leaves prior procnet ownership unreconciled — a stale iptables rule from a prior session, a tracker.Remove error class, or a partial-Add portStorage entry. The bug class the PR targets (a stale PREROUTING DNAT shadowing CNI) can therefore reappear through three rare-but-real cleanup gaps.

Gemini found nothing new; Codex raised one Important; Claude raised three Important plus seven Suggestions. The state-machine extraction itself remains solid and well-tested. The gaps live at the boundary between the state machine and the surviving iptables and tracker state it inherits.

S2. fakeIptables's commitOnError does not model the real-iptables idempotency probe Claude¶

The real applyLoopbackIPtablesRule (scanner_linux.go:175-205) calls preroutingHasLoopbackRule first and short-circuits an Append when the rule already exists. The fake honours every Append. No test exercises an Append-on-present sequence today, so the gap is dormant; a future test added without noticing the divergence would assume real-iptables idempotency, pass on the fake, and break in production.

src/go/guestagent/pkg/procnet/scanner_linux.go
//	iptables -t nat -A PREROUTING -p tcp --dport 8009 -j DNAT --to-destination 127.0.0.1:8009
//
// Example iptables rule when act is Delete:
//
//	iptables -t nat -D PREROUTING -p tcp --dport 8009 -j DNAT --to-destination 127.0.0.1:8009
func applyLoopbackIPtablesRule(ctx context.Context, protocol, hostPort string, act action) error {
	exists, err := preroutingHasLoopbackRule(ctx, protocol, hostPort)
	if err != nil {
		return fmt.Errorf("iptables --check for %s/%s: %w", protocol, hostPort, err)
	}
	if act == Delete && !exists {
		log.Debugf("skipping PREROUTING DNAT delete for %s/%s: rule not present", protocol, hostPort)
		return nil
	}
	if act == Append && exists {
		log.Debugf("skipping PREROUTING DNAT append for %s/%s: rule already present", protocol, hostPort)
		return nil
	}
	// iptables -t nat --wait 2 -<A|D> PREROUTING -p tcp --dport 8009 -j DNAT --to-destination 127.0.0.1:8009
	iptablesCmd := iptablesCommand(ctx,
		"--wait", "2",
		"--table", "nat",
		fmt.Sprintf("--%s", act), "PREROUTING",
		"--protocol", protocol,
		"--dport", hostPort,
		"--jump", "DNAT",
		"--to-destination", fmt.Sprintf("%s:%s", loopbackIP, hostPort),
	)
	// Output() rather than Run() so a non-zero exit carries iptables'
	// stderr; the command produces no stdout worth keeping.
	if _, err := iptablesCmd.Output(); err != nil {
		return wrapExitError(err)
	}
	log.Debugf("running the following iptables rule [%s]", iptablesCmd.String())
	return nil
}

S4. /proc/net scanner failed to remove port logs the wsl-proxy-only and host-switch-failed branches with identical text Claude¶

Both case branches emit the same Errorf message even though their post-conditions differ: the wsl-proxy-only branch deletes the rule (proxy gone, rule gone); the default branch retains it (proxy may still be bound, rule kept). When a stuck port lands in the log, the message alone cannot identify which path ran.

Design Observations ¶

Concerns ¶

  • (future) Engine-managed state is inferred from an iptables snapshot, not from a shared event source — port_state_tracker_linux.go:180-207, scanner_linux.go:207-244 Claude. engineChainManagesPort shells out to iptables every probe and matches against dnatRuleRe, then portStateTracker decides whether the engine "owns" the port. CNI portmap rules and the engine portTracker.Add are written by independent code paths at different times (per the repo context), so the chain-presence signal can drift from portStorage. The two-scan stability gate plus the "proxy already running" classifier paper over the drift well enough for the shipped engines, but a procnet that subscribed to the engine's container-start/stop events directly would be simpler and more robust. Out of scope; flag for the next time procnet opens up.
  • (future) portStateTracker mixes scan-cadence I/O with state-machine policy — port_state_tracker_linux.go Claude. The package separates Tick from the iptables runner, but the state machine still calls pst.tracker.Add/Get/Remove and pst.iptables.* from inside methods that also encode the defer / probe / delegate / retry policy. Splitting policy from side effects — e.g. Tick returns a list of action{Add, Append, Remove, Delete} decisions that the caller applies — would let the tests run the full state machine without mocks and reduce S4's classification gap to a unit test over decision shapes. Not worth doing now.

Review Process Notes ¶

Skill improvements ¶

  • For round N ≥ 2 reviews, surface partial-failure → next-tick-takeover sequences as a dedicated dimension. Pattern: when a state machine handles error A by not recording state, and error B by recording it, a follow-up tick can reach a branch that assumes no prior state exists. Reviewers should enumerate, for each "error path that returns without recording state," every reachable branch on the next tick — including ones that touch external resources the failed attempt may have reserved. Add to the dimensions: "For each state-machine method that returns without recording state on error, list the branches that can run on the next tick and verify each one accounts for state the failed attempt may have left in external systems (proxies, files, iptables, sockets)."
  • For PRs that refactor across an extracted boundary (state → object, function → service), add a dimension on inherited external state. Pattern: code that previously ran continuously now starts and stops; external state it wrote may outlive its in-memory state. Add: "When a process boundary is introduced or repositioned (a struct's lifetime changes, a service is extracted), enumerate every piece of external state the code writes (files, kernel tables, sockets, remote resources) and verify the new boundary either reconciles or cleanly owns each one. A regression in this dimension reintroduces bugs the code fixed in prior sessions."
