S2. acceptTCP log throttle resets after every successful Accept Claude¶
src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go
	acceptRetryInitialBackoff = 100 * time.Millisecond
	acceptRetryMaxBackoff     = 5 * time.Second
	halfCloseDrainTimeout     = 30 * time.Second
)

func (f *loopbackForwarder) acceptTCP(ctx context.Context, lis net.Listener, port uint16) {
	backoff := acceptRetryInitialBackoff
	// loggedAcceptError throttles per-listener Accept-error logs the
	// same way logAddFailure throttles publish-failure logs in the
	// scanner. Sustained FD pressure (EMFILE) saturates the loop at
	// the 5 s cap; without the throttle the loop emits one Error
	// every 5 s indefinitely.
	loggedAcceptError := false
	for {
		conn, err := lis.Accept()
		if err != nil {
			if errors.Is(err, net.ErrClosed) {
				return
			}
			// Retry with exponential backoff on transient errors (EMFILE
			// under FD pressure, ENOBUFS). The listener stays registered
			// in f.tcp; once the pressure clears, Accept succeeds.
			if !loggedAcceptError {
				log.Errorf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
				loggedAcceptError = true
			} else {
				log.Debugf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
			}
			select {
			case <-ctx.Done():
				return
			case <-time.After(backoff):
			}
			backoff *= 2
			if backoff > acceptRetryMaxBackoff {
				backoff = acceptRetryMaxBackoff
			}
			continue
		}
		backoff = acceptRetryInitialBackoff
		loggedAcceptError = false
		go f.pipeTCP(ctx, conn, port)
	}
}

func (f *loopbackForwarder) pipeTCP(ctx context.Context, in net.Conn, port uint16) {
	defer in.Close()
	addr := net.JoinHostPort("127.0.0.1", strconv.Itoa(int(port)))

loggedAcceptError resets on every successful Accept. Under intermittent EMFILE pressure where Accept alternates between success and failure (a busy server briefly freeing a few FDs every few seconds), every failure logs at Error. The comment at lines 174-178 frames the throttle as bounding the "sustained" case; the intermittent case still produces log spam. Severity is low because the scanner's addErrorLogged covers publish-level retries, but the forwarder-level loop is independent.

Fix: reset only after N consecutive successes, or use a time-window throttle.

// bindingsEqual reports whether two binding lists hold the same set
// of (HostIP, HostPort) pairs. Order does not matter, but duplicates
// must match in multiplicity so a list with two identical entries
// does not compare equal to one with a single entry.
func bindingsEqual(a, b []nat.PortBinding) bool {
	if len(a) != len(b) {
		return false
	}
	counts := make(map[nat.PortBinding]int, len(a))
	for _, x := range a {
		counts[x]++
	}
	for _, x := range b {
		counts[x]--
		if counts[x] < 0 {
			return false
		}
	}
	return true
}

Design Observations ¶

Concerns ¶

  • gvisor.dev/gvisor enters as an indirect dependency for a single UDP proxy (future) Claude

go.mod now requires gvisor.dev/gvisor v0.0.0-20240916094835-a174eb65023f // indirect. The procnet package uses only the ~150-line udp_proxy.go from github.com/containers/gvisor-tap-vsock/pkg/services/forwarder, which itself has no gvisor.dev/gvisor dependency. Go module resolution still pulls the rest of the package's transitive deps. If dependency surface ever matters (build size, vulnerability scanning, audit burden), inlining udp_proxy.go (Apache 2.0) would cut both gvisor.dev/gvisor and github.com/inetaf/tcpproxy from go.mod. Lima makes the same trade-off in pkg/portfwd/client.go.

  • Two-scan stability gate requires the port to appear twice, not the same shape twice (in-scope) Claude

If tick N sees 127.0.0.1:8009 (pending) and tick N+1 sees 0.0.0.0:8009, the port publishes with the wildcard bindings from N+1, not the loopback bindings of N. TestBindingShapeChangePublishesCurrent pins this as intentional. Worth a one-line comment in Tick so a future reader does not add a shape-equality check thinking it is a bug. (Codex's I2 in this round and Gemini's S2 in Round 3 both reached for the same change; the behavior is intentional but undocumented.)

  • Procnet ingress addresses are split across components (future) Codex

S6 surfaces that procnet knows only the host-switch tap IP, while wsl-proxy independently uses the bridge IP. Passing an explicit list of procnet ingress IPs from the networking setup into guestagent would make the contract visible and testable, even if today's single-IP bind turns out to suffice.

Untested scenarios, ranked by risk:

  2. forwarder.Close while a pipeTCP is mid-stream. The behavior is documented as best-effort, but a test that calls Close() with an active connection would catch a regression that drops drain deadlines.

  5. wsl-proxy ingress IP end-to-end (S6). Pre-merge verification on a real WSL distro answers the open question, and a regression test would prevent re-introduction.

  • UDP dial closure captures the forwarder-lifetime ctx, not per-flow — loopback_forwarder_linux.go:102-110. Tested by TestEndToEndUDPAndRemovalThroughTick.
  • publish ParseUint dead branch — scanner_linux.go:213-221. Deliberately preserved as a future-regression catch.
