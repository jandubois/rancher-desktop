S3. acceptTCP Error-logs every transient Accept failure after backing off Claude¶
src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go
	acceptRetryInitialBackoff = 100 * time.Millisecond
	acceptRetryMaxBackoff     = 5 * time.Second
	halfCloseDrainTimeout     = 30 * time.Second
)

func (f *loopbackForwarder) acceptTCP(ctx context.Context, lis net.Listener, port uint16) {
	backoff := acceptRetryInitialBackoff
	for {
		conn, err := lis.Accept()
		if err != nil {
			if errors.Is(err, net.ErrClosed) {
				return
			}
			// Retry with exponential backoff on transient errors (EMFILE
			// under FD pressure, ENOBUFS). The listener stays registered
			// in f.tcp; once the pressure clears, Accept succeeds.
			log.Errorf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
			select {
			case <-ctx.Done():
				return
			case <-time.After(backoff):
			}
			backoff *= 2
			if backoff > acceptRetryMaxBackoff {
				backoff = acceptRetryMaxBackoff
			}
			continue
		}
		backoff = acceptRetryInitialBackoff
		go f.pipeTCP(ctx, conn, port)
	}
}

func (f *loopbackForwarder) pipeTCP(ctx context.Context, in net.Conn, port uint16) {
	defer in.Close()
	addr := net.JoinHostPort("127.0.0.1", strconv.Itoa(int(port)))
	out, err := f.dialer.DialContext(ctx, "tcp", addr)

Design Observations ¶

Concerns ¶

  • Scanner does not unwind tracker state on agent shutdown (future) Claude

ForwardPorts exits on ctx.Done() and defer p.forwarder.Close() releases sockets, but published is not unwound through tracker.RemoveAll(). host-switch keeps its proxy table populated from the dead agent; on restart the new instance re-issues Expose for the same ports and meets "proxy already running" until something else clears the stale entry. Pre-existing — not introduced by this PR — but the new code structure is cleaner, so a RemoveAll shutdown hook fits naturally. One extra line and an API audit.

Strengths ¶

  • Tick(scanned) extraction makes every state transition unit-testable without sockets. The tests exercise gate promotion, gate restart, shape change before and after publish, throttle hysteresis, rollback symmetry, and end-to-end forwarding through the scanner-to-forwarder seam.
  • Delegating UDP to forwarder.UDPProxy (round 1's S1 fix) eliminates a class of hand-rolled conntrack bugs and matches Lima's own implementation choice.
  • entriesToPortMap bindIP self-filter (scanner_linux.go:289-300) handles forwarder-owned sockets without coupling the scanner to the forwarder's internal state, and the trade-off is documented inline.
  • addErrorLogged throttle sweep (scanner_linux.go:170-178) closes the unbounded-growth seam under listener churn — every port that leaves both pending and published also leaves the throttle map.

Repo context updates ¶

  • [repo] When reviewing scanner code that processes /proc/net/{tcp,udp} snapshots via lima's procnettcp package, note the state constants: TCPListen = 0xA matches TCP listeners; UDPEstablished = 0x7 matches the Linux TCP_CLOSE kernel constant, which is the state for unconnected/bound UDP sockets (listeners). Connected UDP sockets — those returned by net.Dial("udp", ...) — sit in state 0x1 and never reach a scanner that filters by procnettcp.UDPEstablished. Before raising a cascade finding where outbound UDP dial sockets become "discovered listeners," verify the connected-socket state against the filter — it does not match.
