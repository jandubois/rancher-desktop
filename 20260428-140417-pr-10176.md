## Executive Summary

The Round 3 backlog landed cleanly: the codesign + cache-hit interaction
that re-downloaded Docker CLI on every macOS postinstall is fixed by
keeping the verified bytes at `${destPath}.unsigned` and hashing that on
re-runs (Round 3 I1); `MobyOpenAPISpec` overrides `latestVersion` so
only the winner's commit is resolved (Round 3 S2);
`parseSha256Checksum` now accepts case-insensitive hex and normalises
to lowercase (Round 3 S3); `downloadAndHash` preserves the served
bytes to `os.tmpdir()` on mismatch with the path included in the error
(Round 3 S6); `download()` ignores `EACCES` on the cache-hit access
check (Round 3 S7); `readDependencyManifest` rejects array/scalar
`version` values (Round 3 S9); the `gnuLine` regex bounds the hex
prefix (Round 3 S10); and `dependencies.yaml` now carries an in-file
"do not edit comments" header (Round 3 S12).

Two new Important findings surface from the Round 3 → Round 4 changes:

1. The Round 3 S1 follow-up was applied to `downloadTarGZ` and
   `downloadZip` — they now persist the archive next to `destPath` and
   re-extract on every postinstall — but the same skip-on-cache-hit
   pattern still lives in `RancherDashboard.download()` at
   `scripts/dependencies/tools.ts:406`, where an existing
   `resources/rancher-dashboard` directory short-circuits before
   `download()` is ever called (I1, Codex). The PR's stated invariant
   ("every install path verifies the recorded digest") therefore does
   not hold for that one dependency.
2. The "always re-extract" policy `downloadTarGZ`/`downloadZip` now use
   means a developer running `yarn postinstall` on Windows while
   Rancher Desktop is running will fail when `fs.copyFile` tries to
   overwrite a locked `helm.exe` / `docker.exe` (I2, Gemini). Pre-PR
   the helpers skipped extraction entirely on cache hit, so the lock
   was never touched.

The remaining surface is suggestion-grade: bump-time double-hashing in
`downloadAndHash` (every artifact is sha256'd twice when the upstream
sidecar is also sha256), three Round 3 Suggestions still outstanding
(windows/arm64 in `getChecksums` matrices, `MACOS_RUNNER` hard-coding,
`Sha256Checksum` return type round-trip), per-version archive caches
that accumulate after a bump (`wix314rtm.zip`, `lima.*.v1.2.1.rd2.tgz`),
a hand-edit hazard where `{ apiVersion, commit: '' }` produces a
malformed Moby URL, and a few defensive nits.

Verdict: merge with the two Important fixes folded in. Everything else
is non-blocking polish.

## Important Issues

I1. **`RancherDashboard.download()` skips checksum verification on cache hit** — `scripts/dependencies/tools.ts:403-410` [Codex GPT 5.5]

```ts
const expectedChecksum = lookupChecksum(context, this.name, archiveName);
const rancherDashboardDir = path.join(context.resourcesDir, 'rancher-dashboard');

if (fs.existsSync(rancherDashboardDir)) {
  console.log(`${ rancherDashboardDir } already exists, not re-downloading.`);

  return;
}
```

The PR routes `downloadTarGZ` and `downloadZip` through the persistent
archive + cache-hit hash check (`scripts/lib/download.ts:206, 256`), so
`Helm`, `Trivy`, `Steve`, `WasmShims`, and `SpinCLI` now verify the
manifest digest against the on-disk archive on every postinstall.
`RancherDashboard.download()` retained its older custom guard: a bare
`fs.existsSync(rancherDashboardDir)` check that returns before
`download()` is invoked. `expectedChecksum` is computed at line 403 but
never reaches `download()` — it is dead code on the cache-hit path.

A previous extraction left half-complete by a crash between
`fs.promises.mkdir(rancherDashboardDir, …)` (line 420) and the `tar -xf`
that follows therefore satisfies the existence check forever; the
manifest's pinned sha256 is never compared to anything. The same hole
swallows a directory left over from an older version that was never
removed during a downgrade.

Fix: gate the existing-directory short-circuit on a stamp file written
*after* a successful extraction, and tear the directory down when the
stamp is missing or stale:

```diff
 const expectedChecksum = lookupChecksum(context, this.name, archiveName);
 const rancherDashboardDir = path.join(context.resourcesDir, 'rancher-dashboard');
+const stampPath = path.join(rancherDashboardDir, '.source-sha256');

-if (fs.existsSync(rancherDashboardDir)) {
+const stamp = await fs.promises.readFile(stampPath, 'utf-8').catch(() => '');
+if (fs.existsSync(rancherDashboardDir) && stamp === expectedChecksum) {
   console.log(`${ rancherDashboardDir } already exists, not re-downloading.`);

   return;
 }
+await fs.promises.rm(rancherDashboardDir, { recursive: true, force: true, maxRetries: 10 });
```

…and write `stampPath` only after the `tar -xf` resolves. (important, gap)

I2. **`downloadTarGZ`/`downloadZip` overwrite locked binaries on every postinstall** — `scripts/lib/download.ts:208-228, 258-268` [Gemini 3.1 Pro]

```ts
// Re-extract on every postinstall so destPath always reflects the
// current archive, even after a version bump that re-downloaded
// tgzPath above.
const workDir = fs.mkdtempSync(path.join(os.tmpdir(), `rd-${ path.basename(destPath, '.exe') }-`));
const args = ['tar', '-zxf', tgzPath, '--directory', workDir, fileToExtract];

try {
  …
  await simpleSpawn(args[0], args.slice(1));
  await fs.promises.copyFile(path.join(workDir, fileToExtract), destPath);
  await fs.promises.chmod(destPath, mode);
}
```

Round 3 S1 noted that the archive helpers skipped `download()`'s
checksum check on cache hits. Round 4 fixed that by always re-extracting
the (verified) archive. The regression: pre-PR the helpers had an
`fs.promises.access(destPath, …)` early-return that left an existing
`destPath` untouched on cache hits; Round 4 removes the guard, so
`fs.copyFile` now overwrites `destPath` on every postinstall.

A developer running `yarn postinstall` on Windows while Rancher Desktop
is running holds an exclusive lock on `resources/win32/bin/helm.exe`,
`docker.exe`, and the other Round-4-affected binaries; `copyFile` then
fails with `EBUSY` / `EPERM`. The same scenario on macOS / Linux is
benign because the OS allows overwriting an executable while a process
has it mapped.

Fix: skip the extract / copy step when the on-disk binary already
matches the verified archive — for example, compare mtimes:

```diff
   await download(url, tgzPath, { ...options, access: fs.constants.W_OK });
+
+  const statDest = await fs.promises.stat(destPath).catch(() => null);
+  const statTgz = await fs.promises.stat(tgzPath);
+  if (statDest && statDest.mtimeMs >= statTgz.mtimeMs) {
+    return destPath;
+  }
```

mtime compare is cheap and works for both helpers; an alternative is a
sidecar stamp matching the manifest digest, but that is heavier and
duplicates the `tgzPath` cache-hit logic. (important, regression)

S1. **`downloadAndHash` hashes every artifact twice when verifying sha256** — `scripts/lib/dependencies.ts:241, 259` [Claude Opus 4.7, Gemini 3.1 Pro]

```ts

  …
}

…
```

When `options.verify.algorithm === 'sha256'` (the case for kuberlr,
helm, docker, dockerCompose, dockerBuildx, ECR, steve, SpinCLI, the
spelling tool, and a handful of others), the file is hashed once for
verification then re-hashed to produce the recorded digest, even
though the two hashes are identical. Bump-time only — `rddepman`
runs daily under throttle, and individual artifacts run up to ~60 MB
— so the cost is real but small.

Fix: reuse the verified digest when the upstream algorithm is sha256:

```diff
-      const actual = await hashFile(tempPath, options.verify.algorithm);
-
-      if (actual !== options.verify.expected) {
-        …
-      }
-    }
-
-    const sha256 = await hashFile(tempPath, 'sha256');
-
-    return `sha256:${ sha256 }`;
+      const sha256 = await hashFile(tempPath, 'sha256');
+      const actual = options.verify.algorithm === 'sha256'
+        ? sha256
+        : await hashFile(tempPath, options.verify.algorithm);
+
+      if (actual !== options.verify.expected) {
+        …
+      }
+      return `sha256:${ sha256 }`;
+    }
```

S3. **Persistent archives accumulate after a version bump** — `scripts/dependencies/wix.ts:45`, `scripts/dependencies/lima.ts:49`, `scripts/dependencies/lima.ts:107` [Claude Opus 4.7]

```ts
// scripts/dependencies/wix.ts:45

// scripts/dependencies/lima.ts:49

  `lima.${ platform }.v${ context.dependencies.lima.version }.tgz`);

// scripts/dependencies/lima.ts:107

  `qemu.v${ context.dependencies.qemu.version }.tgz`);
```

`Wix`, `Lima`, and `Qemu` invoke `download()` directly with the version
embedded in the cache filename, so a bump from `lima 1.2.1.rd2` to
`1.2.1.rd3` writes `lima.macos-15.amd64.v1.2.1.rd3.tgz` next to the
existing `…rd2.tgz`. The old archive lives forever on the developer's
disk, and a new orphan appears on every bump. The `.unsigned`
sidecars introduced for codesign callers
(`scripts/lib/download.ts:67-94`) compound this — every codesigned
binary now keeps a same-name `.unsigned` peer that the codesign cache
hit re-reads. `electron-builder.yml:20-22` excludes
`*.tar.gz`/`*.zip`/`*.unsigned` from the package, so this is not a
shipping concern, only a developer-disk one.

Fix (any one is enough): drop `v${ version }` from the cache filename
so a bump overwrites the old archive; or sweep
`resources/${ platform }/lima.*.tgz` and `qemu.*.tgz` (and the Wix
`hostDir/*.zip`) at postinstall start, keeping only the archive whose
basename matches the current `dependencies.yaml` version.

```ts
         { stdio: 'inherit' },
```

```diff
+      if (result.status !== 0) {
+        await fs.promises.rm(unsignedPath, { force: true });
+        throw new Error(`codesign failed for ${ destPath } (exit ${ result.status })`);
```

S10. **`fetchUpstreamChecksums` regex admits md5-width hex** — `scripts/lib/dependencies.ts:283` [Claude Opus 4.7]

```ts
const gnuLine = /^([0-9a-fA-F]{32,128})\s+\*?(.+?)\s*$/;
```

Round 3 S10 tightened the prefix from `+` to `{32,128}` to reject
documentation lines starting with a few hex characters. The bound
still admits md5 (32) and sha1 (40) widths; current callers only pass
sha256 / sha512 sidecars, so an md5 sidecar would parse and then
trigger a confusing "Upstream checksum mismatch" rather than a
parse-time rejection. Cosmetic; tightens the diagnostic.

Fix: anchor to the supported widths exactly:

```diff
-  const gnuLine = /^([0-9a-fA-F]{32,128})\s+\*?(.+?)\s*$/;
+  // sha256 = 64, sha512 = 128. Reject md5 / sha1 widths so a
+  // mixed-algorithm sidecar fails parse, not verify.
+  const gnuLine = /^([0-9a-fA-F]{64}|[0-9a-fA-F]{128})\s+\*?(.+?)\s*$/;
```

## Design Observations

### Concerns

**D1. `MobyOpenAPISpec` commit pinning has a soft failure mode for upstream security fixes** (future) [Claude Opus 4.7]

The design correctly notes that pinning `commit` makes the install URL
immutable, and `rcompareVersions` ignoring `commit` means a new commit
on the same `apiVersion` does not auto-bump. Implication: a security
fix to `v1.54.yaml` on `master` is invisible until someone manually
edits `dependencies.yaml`. Worth documenting in the maintenance steps
so the manual-edit case is not lost.

### Strengths

**D2. Round 3 backlog cleared and the codesign repair is well-targeted** (in-scope) [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

Round 3's I1 (codesign + cache-hit) is fixed by keeping the verified
bytes at `${ destPath }.unsigned` and hashing that on cache-hit checks
(`scripts/lib/download.ts:67-94, 130-143`). Round 3 S2/S3/S6/S7/S9/S10/S12

# Deep Review — PR #10176: rddepman: record and verify artifact digests in dependencies.yaml

## Executive Summary

This PR moves install-time integrity verification from on-the-fly upstream checksum fetches to digests recorded in `dependencies.yaml` at bump time, closing a TOCTOU window where a compromised mirror could swap artifacts post-approval. The implementation is broad, careful, and the design is sound; the issues below are mostly polish, with one correctness gap around extension archives that survive postinstall failures.

**I1. Persistent archives accumulate and never get cleaned up after a version bump** — `scripts/lib/download.ts:197,247`

```ts

  const access = options.access ?? fs.constants.X_OK;
  const tgzPath = `${ destPath }.tar.gz`;
  ...
  await download(url, tgzPath, { ...options, access: fs.constants.W_OK });
```

`downloadTarGZ` and `downloadZip` now persist `${destPath}.tar.gz` / `${destPath}.zip` next to the destination so that the next postinstall can short-circuit on a hash check. After a version bump (e.g. helm 4.1.4 → 4.1.5), `download()` writes `helm.tar.gz` (the new archive). But `download()` writes the new archive on top of the same path; the old archive is replaced. So far OK.

The actual leak is in `Wix.download()` (`scripts/dependencies/wix.ts:45`): `archivePath = path.join(hostDir, `${ tagName }.zip`)` — the local filename uses `tagName` (e.g., `wix314rtm.zip`), not the remote `archiveName` (`wix314-binaries.zip`). ```diff

```diff
+    const archivePath = path.join(hostDir, archiveName);
```

### [Gemini] 3.1 Pro

### Executive Summary

This PR correctly transitions the dependency lifecycle to verify install-time downloads against locally stored hashes inside `dependencies.yaml`, rather than dynamically fetching upstream checksum files during `yarn install`. This effectively eliminates the risk of compromised upstream mirrors injecting malicious artifacts between human-approved bumps. The design cleanly maps to an in-memory `DependencyManifest` and smartly limits GitHub API calls. However, `windows-arm64` variants were omitted from the platform iteration matrices for several tools, causing native Windows ARM64 builds to crash. Furthermore, the updated archive extraction logic in `postinstall` introduces Windows file lock (`EBUSY`) crashes on cached archive extraction when Rancher Desktop is running.

### Findings

#### Critical Issues

**C1. Windows ARM64 artifacts omitted from manifest** — `scripts/dependencies/tools.ts:58-61` (critical, regression)

```typescript

      ['linux', 'amd64'], ['linux', 'arm64'],
      ['darwin', 'amd64'], ['darwin', 'arm64'],
      ['windows', 'amd64'],
    ];
```

The statically defined `platforms` array in `getChecksums` for `KuberlrAndKubectl`, `Helm`, `DockerCLI`, `DockerBuildx`, and `ECRCredHelper` omits the `['windows', 'arm64']` variant. During `yarn postinstall` on a Windows ARM64 environment (where `isM1` is truthy, resolving `arch` to `'arm64'`), `download()` attempts to look up the `arm64` artifact.
```diff
     const platforms: [string, string][] = [

-      ['windows', 'amd64'],
+      ['windows', 'amd64'], ['windows', 'arm64'],
```

#### Important Issues

**I1. Archive extraction overwrites locked binaries on cache hits** — `scripts/lib/download.ts:223-228` (important, regression)

```typescript
    await simpleSpawn(args[0], args.slice(1));
    await fs.promises.copyFile(path.join(workDir, fileToExtract), destPath);
    await fs.promises.chmod(destPath, mode);
```

By removing the early `fs.promises.access(destPath)` existence check, `downloadTarGZ` and `downloadZip` now unconditionally re-extract the downloaded archive over `destPath` on every `yarn postinstall`. If a developer runs `yarn install` while Rancher Desktop is running on Windows, overwriting `destPath` (e.g. `docker.exe` or `helm.exe`) will throw an `EPERM` or `EBUSY` error because the executable is locked. We can avoid both the extraction overhead and the file lock error by comparing modification times: if `tgzPath` is older than `destPath`, the archive was a cache hit and is already extracted.

Fix: Use `fs.promises.stat` to skip extraction if the `destPath` is newer than the validated archive.

```diff
   await download(url, tgzPath, { ...options, access: fs.constants.W_OK });
+
+  const statDest = await fs.promises.stat(destPath).catch(() => null);
+  const statTgz = await fs.promises.stat(tgzPath);
+  if (statDest && statDest.mtimeMs >= statTgz.mtimeMs) {
+    return destPath;
+  }

   // Re-extract on every postinstall so destPath always reflects the
   // current archive, even after a version bump that re-downloaded
```
Apply the identical check in `downloadZip()`.

**S1. Redundant file hashing during upstream verification** — `scripts/lib/dependencies.ts:168-170` (suggestion, enhancement)

```typescript
      const actual = await hashFile(tempPath, options.verify.algorithm);
      // ... (mismatch logic) ...
    }

    const sha256 = await hashFile(tempPath, 'sha256');
```
