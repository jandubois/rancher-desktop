Executive Summary ¶

The PR fixes a real deadlock: after engine-switch tears down the WSL distro, residual Hyper-V vsock state can let vsockHandshake "succeed" before any new peer is up, leaving host-switch waiting on a data connection that never arrives. The fix wraps the handshake in a retry loop that waits for the peer's data connection on the same listener and treats a missing connection as a phantom; the first real connection is replayed via a firstConnListener shim so the rest of host-switch is unaffected. Both prior-round comments from Nino-K were addressed in d43b3b725.

The structure is sound and the unit tests cover firstConnListener and acceptWithTimeout well. One regression matters: runHandshakeLoop is now an errgroup task whose serveAccepts blocks in vn.AcceptStdio, and that call's read loop checks ctx.Done() only between reads — so context cancellation alone does not unblock an idle data connection, and shutdown can hang. The other findings are minor: a test-only nil-deref, a missing test for the ctx.Err() precedence rule that motivated the fix-up commit, and several style/placement nits.

S6. Tests use TCP, not hvsock — Close-during-Accept semantics differ Claude Gemini¶

acceptWithTimeout's correctness depends on ln.Close() unblocking a pending Accept so the goroutine drains. TCP listeners on Linux/Darwin honor this contract. hvsock.Listener (the production listener) wraps Windows native vsock sockets, where the same guarantee is not part of the package's documented API.

The TCP-based tests therefore validate the helpers' logic but not the production assumption. Add a short comment near localTCPListener calling out that production uses hvsock and relies on the same close-unblocks-accept contract, so a future refactor cannot silently invalidate it. Gemini proposed an alternative — drain the channel asynchronously so acceptWithTimeout returns on timeout regardless — but that converts a potential leak (if hvsock failed to unblock) into a guaranteed one, so the conservative fix is the comment.

S7. Use time.Ticker instead of time.After in debugLogLoop Gemini¶

debugLogLoop was extracted from the inline goroutine in this PR; the extraction preserved the original time.After pattern. time.After allocates a fresh timer and channel on every iteration; a single time.Ticker is the idiomatic choice for a periodic log loop. Design Observations ¶

Concerns ¶

Long shutdown latency on context cancellation during a re-handshake (future) Claude

runHandshakeLoop now calls vsockHandshake repeatedly across host-switch's lifetime. vsockHandshake calls vsock.GetVMGUID (pkg/vsock/handshake_windows.go:33), which uses a 5-minute bailout timer (timeoutSeconds = 5 * 60); inside, tryFindGUID (pkg/vsock/handshake_windows.go:126-136) selects only on found and timeout — never on ctx.Done(). If the user shuts down Rancher Desktop while host-switch is mid-re-handshake and no peer is reachable, shutdown can stall up to 5 minutes per attempt before runHandshakeLoop propagates the error.

This is pre-existing in vsock.GetVMGUID, not introduced by this PR — but the PR turns a one-shot call into a retry loop, so the same latency applies on every engine-switch teardown, not just at startup. With handshakeMaxAttempts = 5, the worst case stacks up to 5×5min if WSL is up but the peer is unreachable. Worth tracking as a follow-up to add a ctx.Done() case in tryFindGUID.

Strengths ¶

  • Decoupled lifecycle (in-scope) Gemini — Hoisting virtualnetwork.New out of the connection loop into main keeps the host-side HTTP server (and its API endpoints) bound and serving across engine switches, so local clients querying port forwards do not see transient errors.
  • firstConnListener shim (in-scope) Claude Codex Gemini — The shim is the right abstraction: it lets handshakeWithRetry validate the data connection without changing the contract the rest of host-switch consumes. The explicit Close added in d43b3b725 cleanly addresses the leak case Nino raised.
  • acceptWithTimeout close-then-drain sequencing (in-scope) Claude — Close → drain → conditional Close of any captured connection is the right pattern for "close-to-unblock" plus leak-free shutdown, and the inline comment at switch_windows.go:319-321 explains why the drain matters.
  • Forward-looking comment on runHandshakeLoop (in-scope) Claude — The comment explaining why a host-only retry was chosen over a nonce-exchange protocol change is exactly the kind of note that helps future maintainers; calling out that the cleaner fix would require coordinating a WSLDistro version bump is especially valuable.

Testing Assessment ¶

Unit tests cover the new helpers reasonably well: firstConnListener has tests for replay, concurrent-replay-once, and close-before-accept; acceptWithTimeout has tests for happy-path, timeout, and ctx-cancellation. Untested scenarios, by risk:

  1. handshakeWithRetry's ctx.Err() != nil precedence over errAcceptTimeout — the entire reason for the d43b3b725 fix-up commit. See S2.
  2. handshakeWithRetry's 5-attempt cap and the wrapped final error message.
  3. runHandshakeLoop's redo-on-disconnect loop — closest to integration territory; would need a fake virtualnetwork/listener pair.
  4. I1: there is no test that cancels runHandshakeLoop or serveAccepts while the accepted data connection is idle and still open.

Items (1) and (2) are easy with a small interface seam. Items (3) and (4) are integration-style and reasonably skipped given complexity, though (4) becomes attractive once the I1 fix lands.

The author acknowledges that a successful Accept can race with a fired timer/cancellation and have its connection closed. The drain loop is the chosen mitigation. The PR's behavior matches this rationale.

Review Process Notes ¶

Repo context updates ¶

  • Errgroup task lifecycle. When reviewing a Go change that moves a long-blocking call (network read, Accept, io.ReadFull) into an errgroup.Group.Go task, check whether the call honors context cancellation natively. Standard net.Conn.Read and most io.ReadFull(conn, ...) calls do not — they only unblock when the connection is closed. If the call cannot honor the context, the task either needs a context.AfterFunc that closes the underlying resource on cancellation, or the change converts a previously fire-and-forget goroutine into one that hangs errgroup.Wait() on shutdown. Flag this any time a fire-and-forget goroutine becomes an errgroup task.
  • gvisor-tap-vsock AcceptStdio does not unblock on context cancellation. virtualnetwork.AcceptStdio calls into Switch.Accept → rxStream, whose select on ctx.Done() runs only between reads. The blocking io.ReadFull only returns when the connection is closed. Any caller in this repo that ties shutdown latency to AcceptStdio returning needs a context.AfterFunc(ctx, conn.Close) (or equivalent connection-close-on-cancel) to make shutdown deterministic.
  • vsock.GetVMGUID ignores its ctx parameter. pkg/vsock/handshake_windows.go's tryFindGUID selects only on the found channel and a 5-minute timeout channel. Passing a context does nothing for shutdown latency. Any caller in the host-side networking code that loops over GetVMGUID (the new handshakeWithRetry is one) inherits this stall. Reviews of code that calls GetVMGUID should ask whether the caller can tolerate up to 5 minutes per call, or whether the call should be guarded by a separate timeout.
