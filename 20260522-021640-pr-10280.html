Design Observations ¶

Concerns ¶

  • (future) APITracker exposes no idempotent re-notify hook for wsl-proxy.Send failures Gemini. When wslProxyForwarder.Send fails during Add, portStorage is populated and the gvisor proxy is bound, but the Windows-side proxy state never gets the mapping. Procnet's handleAlreadyExposed resume path (lines 323-336) installs the iptables rule but cannot re-trigger the WSL-proxy notification — the only recovery is a full Remove/Add cycle that risks unbinding a working proxy. The author already documents this trade-off at lines 75-77 of port_state_tracker_linux.go. An idempotent Tracker.RetryWSLProxy(containerID) would let callers fully recover from transient wsl-proxy failures. Out of scope for this PR; flag for the next time APITracker is touched.
  • (future) Tracker ownership is still inferred from chain presence, not a shared event source Codex. I1 is symptomatic of using engineChainManagesPort as a proxy for "engine owns the host-switch proxy." A small explicit handoff contract in tracker — either an event the engine emits when its portTracker.Add lands, or an ownership transfer API — would remove several branch-specific race decisions from procnet. The state-machine extraction this PR completes is the right precondition for that next step.

Acknowledged Limitations ¶

  • Stability gate bypassed under heavy load — port_state_tracker_linux.go:62-73. The two-scan gate filters the OCI hook reservation socket under normal load; a slow tick interval can let the socket survive across two ticks. Documented. Claude Codex Gemini
  • In-session Delete-path leak — scanner_linux.go:289-313 and port_state_tracker_linux.go:251-274. A transient Delete probe failure can leak a loopback rule for the rest of the session; the cross-session counterpart is the I1 scenario that the new if-managed reconciliation now addresses. Documented. Claude Codex Gemini
  • retryAppend engine-takeover and exposeNew engine-delegation accept a rare host-switch proxy leak — port_state_tracker_linux.go:226-250, 380-393. When tracker.Remove returns a non-wsl-proxy error, the proxy may still be bound; the code logs and marks the port delegated anyway because APITracker.Remove clears portStorage via a leading defer (apitracker.go:155), so a retry would silently no-op. Documented at lines 380-387; codex's I1 above proposes extending this comment to the third trade-off path. Claude Codex
  • handleAlreadyExposed delegated branch deliberately retains any leftover loopback rule — port_state_tracker_linux.go:338-352. The rule's origin is ambiguous in this branch; the I1 reconciliation in exposeNew's if-managed branch covers the common cross-session case. Codex
  • dnatChainContainsPort does not match port-range syntax — scanner_linux.go:246-263. Theoretical for the shipped engines (nerdctl-CNI and Docker emit per-port rules) but real for hand-rolled iptables chains. Documented. Claude Codex Gemini
  • Design depends on iptables-nft DNAT-in-PREROUTING terminating the chain — scanner_linux.go:155-169. A future WSL2 kernel shipping iptables-legacy defaults would silently invalidate the design's foundation. The doc comment proposes a iptables-save -t nat | grep '^:PREROUTING ACCEPT' probe in the bats setup; not added in this PR. Claude
  • Fake iptables runner does not model preroutingHasLoopbackRule short-circuit — port_state_tracker_linux_test.go:94-101. R5-S2 suggested fixing the fake; the author chose to document the divergence and instruct "Add the probe to the fake before writing such a test." Claude
  • TestPortAlreadyExposedSubstringIsReachable pins the wrap chain but cannot guard an upstream gvisor-tap-vsock rename of proxy already running — scanner_linux_test.go. Documented in the test's comment. Codex
  • No idempotent wsl-proxy re-notify hook in APITracker — port_state_tracker_linux.go:75-77. After a partial Add, the WSL-side proxy state is permanently inconsistent until the next full Remove/Add cycle. Out of scope for this PR; promoted to Design Observation above. Gemini

Review Process Notes ¶

Skill improvements ¶

  • For each finding agents claim to have fixed in a prior round, verify the fix is complete against the original surface. A fix that resolves the narrowest case can leave a residual gap at the original severity. The verification step is: (a) identify the surface the original finding named (a code path, an error class, a callsite set), (b) read the new code at every cited point, (c) walk the original trigger sequence forward through the new code, and (d) note whether each step still hits a recovery path. When the fix is narrower than the surface, raise the residual gap at the original severity unless an in-code comment documents the reduced scope.
  • When a fix introduces new code that runs unconditionally (a new Remove, Delete, or Reset), enumerate the timing windows during which that operation could destroy state another component depends on. The fix that resolves a leak by adding a cleanup call is one of the most common ways to introduce a new bug: the cleanup runs when the other side has not yet (or has already failed to) create its own state. Reviewers should ask, for each new cleanup call: "What if the operation that would replace this state has already attempted and failed silently?" If the consumer (engine event handler, downstream service, retry loop) has no retry, the cleanup may strand the resource.
