## Executive Summary

The PR adds a default "Info" tab backed by a `fetchContainerInspect` store action that caches `docker inspect` output. The architecture is sound, but the new code has lifecycle gaps: a stale-cleanup race in `fetchContainers`, no retry when the client becomes ready on direct reload, an unconditional spinner that hides cached data on tab return, stale data shown silently on refresh failures, and CLI errors swallowed by an `err.message` read.

All three reviewers participated in pass 1. Race-condition and lifecycle findings came mostly from Claude and Codex; Gemini flagged the most user-visible UX symptom and the error-shape mismatch. Overlap landed on the CLI error finding and the misleading SCSS comment.

## Important Issues

I1. **`fetchContainers` cleanup races with `fetchContainerInspect` writes** â€” `pkg/rancher-desktop/store/container-engine.ts:389-396` [Claude Opus 4.7]

```ts
const inspectData = { ...state.inspectData };
for (const id of Object.keys(inspectData)) {
  if (!ids.has(id)) {
    delete inspectData[id];
  }
}
commit('SET_INSPECT_DATA', inspectData);
```

`fetchContainers` reads `state.inspectData` only after the `await client?.docker.listContainers(...)` resolves (line 344). If a `fetchContainerInspect` commits while the list call is in flight, the snapshot taken at line 390 already contains the new entry but `ids` (built from the older list response) does not â€” the cleanup loop then deletes the freshly-fetched inspect. The reverse interleaving (inspect commits after the snapshot) is also bad: `SET_INSPECT_DATA` overwrites the post-inspect state with the snapshot, losing the entry that way instead. Either case forces the user-facing tab to refetch or shows the spinner forever until the next event triggers a new fetch.

Fix: replace the copy-modify-replace pattern with a targeted delete mutation so concurrent writers cannot lose entries.

```ts
const stale = Object.keys(state.inspectData).filter(id => !ids.has(id));
if (stale.length) {
  commit('DELETE_INSPECT_DATA', stale);
}
```

Add a `DELETE_INSPECT_DATA` mutation that iterates the supplied ids and `delete`s each from `state.inspectData` in place.

I2. **Inspect fetch never retries when client or namespace becomes ready** â€” `pkg/rancher-desktop/components/ContainerInspect.vue:280-291` [Codex GPT 5.5]

```ts
await store.dispatch('container-engine/fetchContainerInspect', {
  containerId: props.containerId,
  namespace:   props.namespace,
});
...
watch(() => props.containerId, fetchInspect, { immediate: true });
```

`fetchContainerInspect` throws if `state.client` is null (`container-engine.ts:408-410`). On a direct reload of `/containers/info/:id`, the store starts with `client: null` (`container-engine.ts:236`); `subscribe()` in `ContainerInfo.vue:277` only assigns the client after `settings-read` returns. The watcher only observes `containerId`, so when the client finally becomes available nothing re-triggers the fetch â€” the Info tab shows the error state permanently. The same one-shot behavior bites containerd when `namespace` flips from `undefined` to a real value after settings load (`ContainerInfo.vue:153`).

Fix: watch the readiness inputs as well as the id.

```ts
watch(
  () => [props.containerId, props.namespace, store.state['container-engine'].client] as const,
  ([containerId, _namespace, client]) => {
    if (containerId && client) {
      fetchInspect();
    }
  },
  { immediate: true },
);
```

I3. **Cached inspect data is hidden by the spinner on every Info-tab remount** â€” `pkg/rancher-desktop/components/ContainerInspect.vue:6-15,269,291` [Gemini 3.1 Pro]

```vue

  <i class="icon icon-spinner icon-spin" />
  Loading container detailsâ€¦
</div>

<template v-else-if="data">
```

`ContainerInfo.vue:94-98` wraps `<container-inspect>` in `v-if="containerId && activeTab === 'tab-info'"`, so the component unmounts and remounts on every tab switch. On mount, `loading` is initialised to `true` (line 269) and the eager watcher immediately calls `fetchInspect`, which sets `loading.value = true` again before awaiting the CLI. Because `data` is computed from the cached store entry, the value is available synchronously â€” but `v-if="loading"` wins, so the user sees a spinner each time they return to the tab even when the data is already cached. The 100â€“500 ms CLI round trip becomes a visible flash on every tab return.

Fix: gate the spinner on the absence of cached data.

```vue

```

I4. **Stale inspect data is shown silently when a refresh fails** â€” `pkg/rancher-desktop/components/ContainerInspect.vue:243-250,279-289` [Claude Opus 4.7]

```vue
<template v-else-if="data"> ... </template>
<div v-else class="error-state" data-testid="info-error">
  {{ error || 'An unknown error has occurred' }}
</div>
```

`data` is a computed read of `inspectData[containerId]` (line 266). After a successful fetch, `data` is truthy. If a later `fetchInspect` throws (network blip, container removed, client disconnected), the catch sets `error.value` but does not invalidate `data`, so the `v-else-if="data"` branch keeps rendering the previous snapshot and the dedicated `.error-state` block never shows. The user sees stale state with no indication that the refresh failed â€” and pairs poorly with I3's fix (showing cached data + spinner becomes "showing cached data + nothing" on failure).

Fix: either render an inline error banner above the cached view when both `data` and `error` are set, or clear the cached entry on failure so the dedicated error state fires. The combined model (I3 fix + a small banner overlay) is the cleanest path:

```vue
<template v-if="data">
  <div v-if="error" class="error-banner" data-testid="info-error-banner">
    {{ error }}
  </div>
  <!-- summary + sections -->
</template>
<div v-else-if="loading">â€¦spinnerâ€¦</div>
<div v-else class="error-state" data-testid="info-error">â€¦</div>
```

I5. **CLI error text never reaches the user â€” generic fallback shown instead** â€” `pkg/rancher-desktop/components/ContainerInspect.vue:284-286` [Gemini 3.1 Pro, Claude Opus 4.7]

```ts
} catch (err: any) {
  error.value = err?.message ?? 'Failed to load container details.';
} finally {
```

`client.docker.cli.exec` throws the raw result object on non-zero exit (see `pkg/rancher-desktop/preload/extensions.ts:285-287` â€” `if (result.signal || result.code) { throw result; }`). That object has `signal`, `code`, `stdout`, `stderr`, and the `parseJsonObject`/`lines` helpers â€” but no `message`. So `err?.message` is always `undefined` for the most common failure mode (container deleted, engine error), and the user sees the generic fallback instead of, for example, `Error: No such object: <id>` from nerdctl/dockerd.

Fix: prefer `stderr`, fall back to `message`, then to the generic string.

```ts
} catch (err: any) {
  const detail = (err?.stderr ?? err?.message ?? '').trim();
  error.value = detail || 'Failed to load container details.';
} finally {
```

S4. **`fetchContainerInspect` skips the per-source error wrapping the other store actions use** â€” `pkg/rancher-desktop/store/container-engine.ts:405-421` [Claude Opus 4.7]

```ts
async fetchContainerInspect({ commit, state }, { containerId, namespace }) {
  const { client } = state;
  if (!client) {
    throw new Error('Container client is not available');
  }
  ...
}
```

The sibling actions (`fetchContainers`, `fetchVolumes`, `fetchNamespaces`) wrap failures in `commit('SET_ERROR', { source, error })` and reuse the global error surface. `fetchContainerInspect` propagates raw, requiring every caller to implement its own try/catch (which the Vue component does, but a future caller will not). The deviation is defensible â€” the inspect view wants to render the error inline rather than in the global error banner â€” but it is undocumented, and mook-as's review comment explicitly asked for the `containers` `ErrorSource` to remain in use.

Fix: pick a model and commit to it â€” either route errors through `SET_ERROR`, or add a JSDoc on the action explaining the intentional deviation so the next maintainer does not "fix" it.

## Design Observations

### Concerns

- **Inspect data is fetched once per container id and never refreshed while the tab is open** *(future)* [Claude Opus 4.7]. The component only triggers `fetchContainerInspect` when `containerId` changes; the store retains the previously-fetched JSON until either `fetchContainers` observes the container has disappeared or `SET_PARAMS` clears state. A long-lived Info tab on a container that gets reconfigured (network attach, runtime exec, rename) silently shows stale data forever. Hooking the existing container event subscriber to invalidate the entry on `start`/`update`/`rename`/`network-connect` events would close the gap without a poll.
- **`ContainerInspectData` is a hand-rolled partial of moby's `types.ContainerJSON`** *(future)* [Claude Opus 4.7]. The interface declared in `container-engine.ts:21-48` will drift from the engine's actual shape over time; this round already had one such drift (`NetworkSettings` being marked non-null) caught only at review. Consider importing the upstream TypeScript types if available, or generating from the moby OpenAPI spec, before this surface grows further.

### Strengths

- Keeping inspect data in the `container-engine` store (rather than a per-component ad-hoc fetch) is the right architectural direction, gives a single place to clean up stale entries, and matches the existing store-based subscriber model. [Codex GPT 5.5, Claude Opus 4.7]
- The component uses `:key="containerId"` on `rd-tabbed` (`ContainerInfo.vue:7`) so switching between containers tears down and rebuilds cleanly instead of relying on watchers to scrub leftover state. [Claude Opus 4.7]
- Native `<details>` sections plus computed display helpers keep the first version small and easy to iterate on. The eager `watch(..., { immediate: true })` is the idiomatic way to express "fetch on mount and on prop change" and matches mook-as's review suggestion verbatim. [Codex GPT 5.5, Claude Opus 4.7]

When switching between the "Logs" and "Info" tabs, `ContainerInspect.vue` unmounts and remounts. On mount, `loading` is set to `true` unconditionally, which triggers this `v-if` and completely hides the component's data. Because `data` is loaded from the Vuex store, it is usually available instantly. Unconditionally hiding the DOM defeats the UX benefit of the global cache, causing a jarring layout flicker every time the user returns to the tab.

Fix: Display the loading spinner only if the data is not yet available, allowing the UI to instantly render cached data while a background refresh occurs.

```diff
--- pkg/rancher-desktop/components/ContainerInspect.vue
+++ pkg/rancher-desktop/components/ContainerInspect.vue
@@ -6,7 +6,7 @@
     data-testid="container-inspect"
   >
     <div
-      v-if="loading"
+      v-if="loading && !data"
       class="loading-state"
       data-testid="info-loading"
     >
```
