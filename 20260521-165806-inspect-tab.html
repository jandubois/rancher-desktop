Executive Summary ¶

The branch adds a default Info tab driven by docker inspect. Three reviewers converged on lifecycle gaps: the child fetches before the store's client and namespace land and then never retries; the unsubscribe path leaves stale inspectData; an empty inspect result returns silently; templates render Mounts.length and Args.length without null-safety; and cached previous data masks the next fetch's error. The feature is close to landing once these five issues are addressed.

All three reviewers ran a single pass. Five Important findings cluster around watcher dependencies and template robustness; ten Suggestions cover style, i18n, and CLI hardening. The branch has an open PR (#9989); reconciliation of its prior review comments appears under Acknowledged Limitations and Unresolved Feedback below.

ContainerInfo.vue activates the Info tab by default (activeTab initial = 'tab-info') and mounts the child immediately when route.params.id is present. The child's immediate: true watcher dispatches fetchContainerInspect synchronously, but the parent's subscribe() only sets state.client after the asynchronous settings-read IPC resolves. +watch(

S5. New user-visible strings bypass i18n Claude¶

The component hard-codes every user-visible string ("Name", "ID", "IP Address", "Mounts", "Environment", "Loading container details…", "None", "An unknown error has occurred", etc.). The repo ships an i18n surface at pkg/rancher-desktop/assets/translations/{en-us,de,zh-hans}.yaml and a global t() helper.

Caveat: ContainerLogs.vue and ContainerShell.vue already hard-code their strings, so this PR follows the established (non-i18n) pattern in this folder. The folder is the gap, not this PR — but the gap doubles in surface area here. Worth flagging to the maintainer who owns the i18n migration.

S6. No refetch when container state changes Claude¶

The watcher refetches only when containerId changes. If the container is stopped, started, renamed, or otherwise updated while the Info tab is open (events the parent's subscriber already receives), the inspect view keeps showing the prior Status, StartedAt, IPAddress, etc., until the user navigates away and back.

Fix: re-dispatch when currentContainer.state changes, or hook the inspect refresh into the existing event stream.

Design Observations ¶

Concerns

  • fetchContainerInspect diverges from the surrounding actions in two ways: it does not commit failures to SET_ERROR and it does not classify them against an ErrorSource. The reviewer flagged this in PR comment r3011823415 and suggested reusing 'containers'; the action still surfaces errors only through the local component error ref. (in-scope) Claude
  • commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] }) replaces the whole map on every fetch, which defeats fine-grained reactivity for components observing other keys. A MERGE_INSPECT_DATA(state, { id, data }) mutation would centralise the immutability contract and shorten the call site. (future) Claude

Strengths

  • Stale-data cleanup in fetchContainers (lines 389-396) hooks the new cache into the existing container lifecycle without adding a separate eviction path — the reviewer specifically asked for this reuse and the implementation matches. (in-scope) Claude Codex
  • Gating the inspect dispatch on activeTab === 'tab-info' means Logs and Shell keep their existing lifecycle behaviour while Info is fetched only when its tab is visible. (in-scope) Codex
  • The <!-- rendered in .tab-content so we can mix v-if and v-show --> comment at ContainerInfo.vue:11-15 documents the intent behind the tabbed lifecycle mix — the kind of "why" comment that survives refactors. (in-scope) Claude

This puts the same context PR mode already has in front of every branch-mode agent and removes the coverage dependency on a single agent's tool-use initiative.
