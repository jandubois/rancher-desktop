Executive Summary ¶

The three commits address a real production hang: the /proc/net scanner appends PREROUTING -j DNAT --to-destination 127.0.0.1:<port> for bridge-network ports whose CNI-HOSTPORT-DNAT chain already manages traffic, shadowing the engine's rule on iptables-nft kernels and breaking docker run -p 127.0.0.1:N:N end-to-end. The transient listener procnet catches turns out to be nerdctl's own OCI createRuntime hook — not the CNI portmap plugin the PR description names — holding 127.0.0.1:<hostPort> while it wires up host port forwarding, before CNI installs its own iptables rules. The fix stacks three mechanisms in order of effectiveness: a two-tick stability gate (seenLastScan / added) as the primary filter — the hook never lives long enough to be seen twice on a normal-load system, so it never reaches tracker.Add; an engine-chain precheck (engineChainManagesPort + dnatChainContainsPort) as defence-in-depth for the rare case where the hook lingers across two scans on a loaded box and CNI has by then written its rule; and a "proxy already running" delegation path that suppresses the resulting tracker.Add retry storm without touching correctness (the rogue iptables Append was already skipped because the failing tracker.Add short-circuited the helper).

Structure: the diff is well-scoped to one package, the commits map onto the actual debugging arc, and the new code carries unusually thorough block comments. One Important finding remains: engineChainManagesPort receives only the port string, so a UDP engine rule on port N suppresses a TCP host-network DNAT for port N (and vice versa) — the same shadow-DNAT failure mode the PR fixes, flipped to the engine side. Both new iptables probes (engineChainManagesPort and preroutingHasLoopbackRule) also collapse every command failure to "not present" — Codex's original Important — but the stability gate narrows the failure window enough that this is now a Suggestion rather than a regression; on the append side the joint case (hook lingers, probe transiently loses the nat lock) is rare though correlated under load, and on the delete side a probe miss leaks a single PREROUTING rule rather than reintroducing the shadow class.

S3. Port-range form dpts:M:N is not detected Claude¶

iptables -L --numeric prints a port range as tcp dpts:8000:9000 (plural dpts: with :-separated bounds). The current regex matches only dpt: (singular) and dports (multiport list). A Docker container started with -p 8000-9000:8000-9000 produces a dpts: rule the regex will not see, so a --network=host listener on any port in that range would still get a shadowing PREROUTING DNAT.

Not the typical bug path (most ports are published individually), but worth either a regex extension that also handles the range form (with lo <= port <= hi parsed out separately) or an explicit // limitation: comment on the helper documenting the gap.

When a port is delegated (the engine's events handler already exposed it), procnet writes the binding into added to suppress retries but performs no iptables append. Later, when the engine's reservation socket disappears from /proc/net/tcp, the Remove loop runs tracker.Remove and execLoopbackIPtablesRule(Delete) for that port without distinguishing delegated from owned ports.

In the current code this is harmless because two unrelated invariants line up: (1) procnet's Remove key is utils.GenerateID("tcp/N") while the events handler stored the port under the real container ID, so APITracker.Remove ends up walking an empty portStorage entry and calls neither apiForwarder.Unexpose nor wslProxyForwarder.Send; (2) preroutingHasLoopbackRule returns false because procnet never wrote a rule, so the iptables Delete is also a no-op. S9. PR description names the wrong subsystem for the transient listener¶

The PR description states "CNI's portmap plugin briefly opens a port-reservation socket on 127.0.0.1:<port> during container start." In production the listener is opened by nerdctl's own OCI createRuntime hook — confirmed by walking /proc/<pid>/fd/ to the socket inode and reading /proc/<pid>/comm and cmdline (/usr/local/libexec/nerdctl/nerdctl ... internal oci-hook createRuntime). The in-tree code comment was force-pushed to fix exactly this attribution and now names nerdctl correctly; the PR description was not updated to match.

The distinction matters for future archeology — anyone bisecting a recurrence will reach for CNI portmap code first if the PR description points them there. Update the PR description to match the code comment before merge.

Design Observations ¶

Strengths ¶

  • (in-scope) The decision to key the Delete path off added rather than seenLastScan — and to write delegated ports into added — is what prevents a retry-storm of "proxy already running" log spam every tick. The inline comment at lines 104-108 captures the why well. Claude
  • (in-scope) Splitting dnatChainContainsPort out as a pure function with a table-driven test is exactly the right shape; the fixtures pull in the realistic iptables variants (single, multiport, IP-octet collision) instead of stubbing the regex against synthetic input. Claude Codex Gemini
  • (in-scope) The "alternative considered" block at lines 94-102 documenting the rejected pid/cgroup/inode classification path is unusual and valuable. It records the analysis behind the chosen approach so a future "we should classify by pid" suggestion can land on the prior reasoning rather than restart it. Claude
  • (in-scope) The two-scan stability gate is a narrow, stateless fix for the OCI-hook reservation-socket race — no /proc/<pid>/comm-scraping, no cgroup inspection, no inode-to-pid resolution. The cost (one tick of latency for --network=host listeners) is explicitly documented. Codex Gemini

Concerns ¶

  • (in-scope) ForwardPorts now coordinates three maps (seenLastScan, added, newPortMap) and two transition rules (the gate; the delegation marker), and the loop has no unit-test coverage — only the regex helper is tested. The state-machine invariants the S1 fix relies on (Delete-side preserves added on error, Append-side preserves added when the probe fails) are exactly the shape that benefits from a portStateTracker extraction with a fake tracker.Tracker and an injectable port-map source. Not a merge blocker, but the next bug here will be easier to land if the loop is decomposed. Claude

Testing Assessment ¶

The new TestDnatChainContainsPort table is solid for the regex it covers: single-port, multiport, suffix/prefix-of-port, IP-octet-collision pitfalls. Gaps, ranked by risk:

  1. Protocol-mismatch matrix (covers I1) — no fixture exercises udp dpt:N alongside a TCP probe (and vice versa). One representative row: {output: "...udp dpt:8080...", protocol: "tcp", port: "8080", want: false} directly anchors the I1 fix.
  2. iptables-probe error handling (covers S1) — neither engineChainManagesPort nor preroutingHasLoopbackRule is unit-testable today because they shell out directly. Either inject a func(ctx, args...) ([]byte, error) or a small iptablesRunner interface. Once that lands, add a fixture for "chain absent (expected)" and one for "exit status 4 (transient)" to lock the distinction in.
  3. State machine in ForwardPorts — no unit tests exercise the two-scan gate, the delegation path, or the divergence between added and seenLastScan. A fake tracker plus an injectable port-source would make this testable and would protect the S4 invariant against future refactors.
  4. Port-range fixture (covers S3) — tcp dpts:8000:9000 is not in test fixtures. Either lock the current behavior (want: false) and document the gap, or extend the matcher to handle ranges.
  5. Cross-platform verification — the investigation that motivated this PR was Windows-only (WSL2 kernel 6.6.87.2, iptables-nft 1.8.11). The same published-ports.bats scenario on macOS and Linux against current alpine-lima kernels has not been verified to either reproduce the original hang or accept the fix. If it passes there today the fix is still right (the rogue rule should not exist for bridge-network ports on any platform), but a one-shot manual check rules out a separate kernel-version interaction.

The PR's manual test plan correctly identifies the BATS scenario (bats/tests/containers/published-ports.bats) and the two manual smoke tests (bridge-network published port from Windows; --network=host loopback). Those were not run from this review.

Review Process Notes ¶

Skill improvements ¶

  • When an agent flags a bug centered on a call to a stored interface (a tracker, registry, cache, port map), verify the keying assumption before trusting the severity. Re-check the agent's argument by tracing the actual lookup key the caller passes against the keys the storage was populated with — different ID schemes (synthetic vs. natural) can quietly neutralize a "this destructively modifies shared state" claim. Asymmetric IDs are a common pattern when two writers share a tracker, and the resulting "safe by accident" code reads exactly like a bug at a quick scan. Apply this every time a finding's blast radius depends on a Remove/Delete/Drop call against a tracker, store, or cache.

Repo context updates ¶

  • [repo] Document that APITracker.Remove(key) is a no-op when key is not present in portStorage: the lookup returns nil, the for-loop over the nil map is empty, and the len(portMap) != 0 guard skips the wslProxyForwarder.Send. This invariant matters because callers in the same package use two different ID schemes — engines call Add(container.ID, ...) and procnet calls Add(utils.GenerateID("proto/port"), ...) — and reviewers verifying delegation-cleanup paths or any cross-writer cleanup need to know that mismatched-key removes are benign by design.
