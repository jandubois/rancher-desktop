Executive Summary ¶

This branch adds comprehensive i18n support to Rancher Desktop: a Go-based i18n-report CLI tool for translation management, main-process locale switching, translated UI strings across ~55 Vue components, and German/Farsi locale translations. The architecture is well-designed — the Go tool's YAML-node-level merge preserves comments and structure, the onLocaleChange callback pattern cleanly decouples locale changes from menu/tray rebuilds, and the titleKey/descriptionKey approach makes page titles reactive without modifying every component.

Three review passes uncovered 1 critical issue, 17 important issues, and 14 suggestions. The critical bug (locales.yaml treated as a selectable locale) and several important regressions (hidden language picker, missing translation keys, broken LockedFieldError detection) must be fixed before merge. The Go tool has strong test coverage, but the TypeScript side lacks tests for locale switching, callback lifecycle, and placeholder parity.

Critical Issues ¶

C-1  locales.yaml manifest treated as a selectable locale  ClaudeCodex (critical, regression) ¶

All three i18n implementations scan assets/translations/ for *.yaml files and derive locale codes from filenames. The webpack context regex /\.yaml$/ matches the newly added locales.yaml manifest, producing "locales" as an entry in availableLocales. Consequences: the language picker renders "locales" as a selectable option; settingsValidator.ts accepts it via checkEnum('none', ...availableLocales); selecting it loads the manifest's structure as translation data, breaking every t() call.

		return nil, fmt.Errorf("reading %s: %w", dir, err)

I-12  JSONL parsing fails on leading blank lines  Gemini (important, regression) ¶

The JSONL detection takes the substring before the first \n and trims it. If the content starts with a blank line (e.g., \n{"message"...}), content[:0] is empty, so JSONL detection fails and the entire output is skipped.

src/go/i18n-report/report_merge.go
func extractTranslationText(data []byte) string {
	content := string(data)

	// Detect JSONL (agent output): first non-empty line starts with '{'.
	firstLine := content
	if idx := strings.IndexByte(content, '\n'); idx >= 0 {
		firstLine = strings.TrimSpace(content[:idx])
	}
	if len(firstLine) > 0 && firstLine[0] == '{' {
		var extracted strings.Builder
		for _, line := range strings.Split(content, "\n") {
			line = strings.TrimSpace(line)
			if line == "" || line[0] != '{' {
				continue
			}

Fix: Iterate lines to find the first non-empty line instead of assuming it precedes the first \n.

S-3  loadLocale silently swallows errors  Claude (suggestion, regression) ¶

If loading fails, the catch logs an error but returns successfully. The caller then sets currentLocale to a locale that never loaded. Future settings-update events with the same locale skip loading because locale !== currentLocale is false.

pkg/rancher-desktop/main/i18n.ts
async function loadLocale(locale: string): Promise<void> {
  if (translations[locale]) {
    return;
  }
  try {
    translations[locale] = await translationContext(`./${ locale }.yaml`);
  } catch {
    console.error(`i18n: failed to load locale "${ locale }"`);
  }
}

Fix: Return a boolean or throw so the caller avoids setting currentLocale on failure.

S-4  Eager loading of all YAML files at startup  Claude (suggestion, regression) ¶

The init action loads every translation file so availableLocales can display native locale names. With four locales this adds ~170 KB of YAML parsing. As more locales are added, this cost grows linearly. The tradeoff is acceptable for a small locale count — document the intent with a code comment.

S-5  command-api.yaml locale enum is hardcoded  Claude (suggestion, gap) ¶

The API spec hardcodes [none, de, en-us, fa, zh-hans]. Adding a locale requires updating this file alongside the manifest, validator spec, and locale files. The crossValidateManifest tool catches mismatches but only when run manually. Add it to CI.

S-6  Duplicated t() implementation across mock, module, and test  Claude (suggestion, regression) ¶

The getByPath and interpolation logic is reimplemented in three places: __mocks__/i18n.ts:27-53, i18n.ts:29-75, and UpdateStatus.spec.ts:7-53. Extract the shared logic into a test utility.

S-7  No settings migration entry for version 19  Claude (suggestion, gap) ¶

CURRENT_SETTINGS_VERSION was bumped from 18 to 19, but no 18: entry was added to updateTable. This works because _.defaultsDeep fills the new locale: 'none' default, but a no-op entry would document the version bump reason.

S-8  README documents the buggy discovery model  Codex (suggestion, regression) ¶

The README states "Webpack auto-discovers YAML files in this directory. Adding a new file here makes it available as a locale without code changes." This is no longer true now that the directory contains non-locale YAML. Update to reflect manifest-driven registration.

S-9  Non-atomic writes to locale and metadata files  Gemini (suggestion, gap) ¶

S-10  nodeRemoveLeaf is dead code  Gemini (suggestion, enhancement) ¶

src/go/i18n-report/yaml.go
// nodeRemoveLeaf removes a leaf key from the tree.
// Returns true if the key was found and removed.
func nodeRemoveLeaf(root *yaml.Node, dottedKey string) bool {
	parts := strings.Split(dottedKey, ".")
	current := root
	for i, part := range parts {
		idx := nodeFindKey(current, part)
		if idx < 0 {
			return false
		}
		if i == len(parts)-1 {
			// Remove key-value pair.
			current.Content = append(current.Content[:idx], current.Content[idx+2:]...)
			return true
		}
		valNode := current.Content[idx+1]
		if valNode.Kind != yaml.MappingNode {
			return false
		}
		current = valNode
	}
	return false
}

S-14  ICU nested placeholder extraction is depth-limited  Gemini (suggestion, gap) ¶

extractPlaceholderNames only detects placeholders at depth == 0. Placeholders nested inside ICU plural or select branches (e.g., {count, plural, other {Folder is {folder}}}) are ignored. The current translation strings do not use nested ICU, so this is not yet a practical issue.

src/go/i18n-report/report_validate.go
func extractPlaceholderNames(s string) map[string]bool {
	result := make(map[string]bool)
	depth := 0
	for i := 0; i < len(s); i++ {
		switch s[i] {
		case '{':
			if depth == 0 {
				// Extract the identifier starting after '{'.
				rest := s[i+1:]
				if m := identRe.FindStringSubmatch(rest); m != nil {
					result[m[1]] = true
				}
			}
			depth++
		case '}':
			if depth > 0 {
				depth--
			}
		}
	}
	return result
}

Design Observations ¶

Concerns ¶

D-1  Three separate locale registries  ClaudeCodex (in-scope) ¶

The branch has three independent locale discovery mechanisms: directory scanning in main/i18n.ts, directory scanning in store/i18n.js, and the explicit manifest in locales.yaml. Consolidating all executable code onto the manifest would eliminate the "workflow file becomes locale" bug class and make CLI/UI validation naturally consistent.

D-2  Divergent interpolation engines  Claude (in-scope) ¶

main/i18n.ts uses simple {variable} replacement while store/i18n.js uses ICU MessageFormat via intl-messageformat. The same translation string can behave differently in the main process vs. renderer. An ICU plural form works in the renderer but renders literally in the main process. Document which strings can use ICU features and which cannot.

D-3  Localized error strings as API contracts  ClaudeCodex (in-scope) ¶

Localizing validator messages creates a subtle API contract change. External tooling and internal code paths that parse error text will break for non-English locales. Consider keeping API-facing error messages in English regardless of locale, or switching to structured error codes.

Adding a dimension like "verify that every t() call references a key that exists in the source locale" would catch the class of bugs found in pass 3 (missing deploymentProfile and containerInfo.shell keys).

Repo context updates ¶

  • Validator error messages are now localized via t() calls. Callers that detect specific error conditions by regex-matching error text (e.g., LockedFieldError detection) must use structured error codes instead. Future reviews should flag any code that parses localized error strings.
  • The locales.yaml manifest in assets/translations/ is not a locale file. Any code that scans the translations directory for locale files must exclude it.
