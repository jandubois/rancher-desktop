Executive Summary ¶

The PR pins per-artifact sha256 digests next to each dependency's version in dependencies.yaml, replacing live upstream-checksum lookups during postinstall with a static map populated at bump time. The schema change to a compound { version, checksums } per entry is clean, and the typed sha256:<hex> strings plus single lookupChecksum/downloadAndHash helpers give the new code a small, comprehensible surface. As a strict improvement, several previously-unverified downloads (DockerBuildx-darwin, ECRCredHelper, WasmShims, CertManager, SpinOperator, Moproxy, WSLDistro, Wix, MobyOpenAPISpec) now have a verified digest at install time.

The two important findings are both ergonomic rather than security gaps. I1 points out that pinning a checksum on a URL fetched from the mutable master branch (MobyOpenAPISpec) converts pre-existing silent upstream drift into a hard postinstall failure — an appropriate failure mode, but one that wants an immutable source ref to keep clean installs working between bumps. I2 points out that the bump-time verify mismatch error reports expected sha256:undefined, got <hex> when the upstream sidecar parser misses a line, which misleads the reader into thinking the artifact is corrupt. Suggestions S1–S6 cover code-health issues (parallel dependency lists, parser strictness, hashFile duplication, lost YAML comments, hardcoded macos-15, AlpineLimaISO write-path divergence); none affect correctness today.

S2. fetchUpstreamChecksums is silently strict about sidecar format Claude¶
scripts/lib/dependencies.ts
 * to raw hex checksum.  Ignores lines that do not match `<hex> [* ]<filename>`.
 * Indexes each entry by both its full path and its basename so callers that
 * know an artifact by filename alone still find it in sidecars that embed a
 * path prefix (e.g. `release/foo.tar.gz`).

export async function fetchUpstreamChecksums(url: string): Promise<Record<string, string>> {
  const body = await getResource(url);
  const result: Record<string, string> = {};

  for (const line of body.split(/\r?\n/)) {
    const match = /^([0-9a-f]+)\s+\*?(.+?)\s*$/.exec(line);

    if (match) {
      const [, hex, fullName] = match;
      const basename = fullName.replace(/^.*\//, '');

      result[fullName] = hex;
      result[basename] = hex;
    }
  }

  return result;
}

/**
 * A dependency is some binary that we need to track.  Generally this is some
 * third-party software, but it may also be things we build in an external
 * repository, or some binary we build from them.

Two narrow assumptions:

  1. Hex must be lowercase. The previous findChecksum used split(/\s+/), accepting any case. A single uppercase digit anywhere in the upstream file (e.g. a future move to sha256sum --uppercase) silently drops the line.
  2. Only GNU sha*sum format. BSD lines, SHA256 (file) = hex, never match. None of today's upstreams use BSD format, but a switch by any one of them produces an empty map and I2 takes over.

Fix: match hex case-insensitively ([0-9a-fA-F], then .toLowerCase() before storing), add a second pattern for BSD-style lines, or at minimum log a warning when the file parses to zero entries.

async function hashFile(filePath: string, algorithm: 'sha256' | 'sha512'): Promise<string> {
  const hash = crypto.createHash(algorithm);

  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);

    stream.on('error', reject);
    hash.on('finish', () => resolve());
    stream.pipe(hash);
  });

  return hash.digest('hex');
}

Design Observations ¶

Concerns ¶

D1. Why split versions and checksums into two parallel structures? (in-scope) Claude Gemini

@mook-as flagged this on the PR (discussion_r3149342472). The on-disk format is unified per dependency ({ version, checksums }), but the in-memory model is DownloadContext.versions[name] and DownloadContext.checksums[name] carried separately, with lookupChecksum(context, name, artifact) straddling them. A single shape — for example DownloadContext.dependencies[name] = { version, checksums } — maps 1:1 to disk, eliminates the partial Partial<DependencyVersions> / Partial<DependencyChecksums> reconstruction in readDependencyManifest, and shortens the abstract method to getEntry(). Most consumers today read just versions, so this trades some call-site verbosity for less indirection. Worth at least replying to @mook-as before merge; if the split is intentional, the response justifies leaving it.

D2. Mutable inputs need a first-class model (future) Codex

Most dependencies are release artifacts, but mobyOpenAPISpec is a source file from a branch. Treating both as { version, checksums } hides the extra identity needed for mutable branch content. A proper model — for example a commit field in the version record for branch-backed dependencies — removes I1's failure mode and makes future similar dependencies explicit instead of surprising.

Strengths ¶

D3. Typed Sha256Checksum = \sha256:${ string }\ template-literal Claude

The template-literal type combined with parseSha256Checksum validating at the only ingestion point (readDependencyManifest) is a clean type-safe pattern: the prefix self-documents the algorithm in the YAML, and the slice('sha256:'.length) in lookupChecksum gives download() the raw hex it expects without ceremony.

D4. Computing checksums before git checkout -b in rddepman Claude Gemini

rddepman.ts:318-323 computes fresh checksums before creating the branch. An upstream mismatch aborts before any branch is created or PR opened; no half-mutated git state.

D5. Uniform sha256 install-time path Gemini

Translating diverse upstream digests (sha512 sidecars and others) into unified sha256 checksums in dependencies.yaml simplifies the install-time retrieval layer. Install-time validation needs one consistent mechanism rather than a matrix of upstream-specific hashing rules.

Testing Assessment ¶

No unit tests accompany the new code. Cheap, high-leverage scenarios:

  1. fetchUpstreamChecksums parsing — round-trip a sample sidecar (GNU sha256sum with binary marker, GNU two-space format, possibly BSD style if S2 is addressed) and assert the resulting map. This is the single point where parser bugs silently degrade verification (I2).
  2. parseSha256Checksum — accept/reject the boundary cases in its regex (uppercase, wrong prefix, wrong length, non-hex). Trivial to write and locks in the contract.
  3. readDependencyManifest / writeDependencyManifest round-trip — confirm key order and alpineLimaISO's nested version survive a load/save cycle.
  4. lookupChecksum error — the "Available: …" message is operator-facing; assert it lists the keys when the artifact is missing.

Integration-level: populate-dependency-checksums.ts is the only end-to-end exerciser today, and it requires network access. Acceptable given the scope, but the unit pieces above are cheap.

Documentation Assessment ¶

scripts/lib/dependencies.ts carries thorough JSDoc on the new types and helpers. scripts/populate-dependency-checksums.ts has a clear file-level comment on its one-shot nature and read tolerance. The "no upstream checksum file" cases (wix.ts:33-34, tools.ts:208-210, tools.ts:597-598, lima.ts:116) all carry inline comments explaining the trust-on-first-use trade-off — appropriate.

The PR does not update any developer-facing README to describe the manifest format. The schema is a long-lived artifact; a short note near the manifest reader (or in a contributing guide) would reduce the chance of hand-edits going wrong.

Review Process Notes ¶

Repo context updates

  • [repo] Add to deep-review-context.md: When a PR adds checksum verification or any other failure-on-mismatch check to a download path, check whether the source URL is immutable (a release tag, a specific commit SHA, or a content-addressed reference). Pinning a digest on a mutable URL (master branch, latest tag, a path on the upstream's main branch) converts silent drift into a hard build failure between bumps. Flag any new expectedChecksum paired with a URL that names a moving ref, and suggest recording the resolved immutable reference (commit SHA, release tag) alongside the digest in the manifest.

Why this generalises: the rancher-desktop dependency-bump model records the resolved version in YAML, so the missing piece is just the resolution step at bump time. The pattern applies equally to any future dependency that pulls from raw.githubusercontent.com/.../master/..., charts at "latest", or unsigned mirror URLs.

Skill improvements

  • Add to the review dimensions a sub-bullet under "Security / supply chain": When a PR adds digest pinning to a download path, verify that the URL the digest is pinned against is immutable. A digest on a mutable URL produces silent CI failures between bumps and indicates an incomplete fix. This is the underlying principle behind I1; the prompt currently asks reviewers to think about checksum coverage but not about reference immutability.

Why this generalises: the same anti-pattern shows up in Dockerfiles (FROM image:latest, apt-get install of unpinned versions), CI-pipeline wget calls, vendored mirror URLs, and package-manager lock files that omit hashes. The principle is "checksum + mutable ref = false sense of security."
