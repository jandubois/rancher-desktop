I1. **Package comment misstates which /proc/net files the scanner reads** — `src/go/guestagent/pkg/procnet/scanner_linux.go:25-32` [Claude Opus 4.7] (important, gap)

```

tcp6/udp6 variants. A --network=host container that listens
exclusively on [::1]:port is invisible to the scanner and is not
```

A maintainer extending IPv6 support would look at `scanListeners` instead of the Kind switch.

Fix: name the real filter site.

```diff
-IPv6 limitation: the scanner reads /proc/net/{tcp,udp} only, not the
-tcp6/udp6 variants. A --network=host container that listens
+IPv6 limitation: addValidProtoEntryToPortMap dispatches only on the
+TCP and UDP Kinds, dropping TCP6/UDP6 entries that procnettcp.ParseFiles
+returns. A --network=host container that listens
```

I2. **Guest-agent docs still describe the removed loopback iptables path** — `docs/networking/windows/rancher-desktop-guest-agent.md:171,174` [Claude Opus 4.7] (important, gap)

This PR removes the loopback iptables write entirely; the userspace forwarder replaces it. Fix both arrows together: replace the containerd arrow with `userspace loopback forwarder on tap IP`; drop or rewrite the dockerd arrow.

S2. **`acceptTCP` log throttle resets after every successful Accept** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:172-209` [Claude Opus 4.7] (suggestion, gap)

```go
backoff = acceptRetryInitialBackoff
loggedAcceptError = false
go f.pipeTCP(ctx, conn, port)
```

`loggedAcceptError` resets on every successful `Accept`. Under intermittent EMFILE pressure where Accept alternates between success and failure (a busy server briefly freeing a few FDs every few seconds), every failure logs at Error. The comment at lines 174-178 frames the throttle as bounding the "sustained" case; the intermittent case still produces log spam. Severity is low because the scanner's `addErrorLogged` covers publish-level retries, but the forwarder-level loop is independent.

Fix: reset only after N consecutive successes, or use a time-window throttle.

S3. **Validate `tapIfaceIP` once at runAgent entry** — `src/go/guestagent/main.go:134,241-244` [Claude Opus 4.7] (suggestion, gap)


Fix: move the `net.ParseIP` check to the top of `runAgent` so both consumers fail-fast at the same point.

## Design Observations

### Concerns

- **`gvisor.dev/gvisor` enters as an indirect dependency for a single UDP proxy** *(future)* [Claude Opus 4.7]

  `go.mod` now requires `gvisor.dev/gvisor v0.0.0-20240916094835-a174eb65023f // indirect`. The procnet package uses only the ~150-line `udp_proxy.go` from `github.com/containers/gvisor-tap-vsock/pkg/services/forwarder`, which itself has no `gvisor.dev/gvisor` dependency. Go module resolution still pulls the rest of the package's transitive deps. If dependency surface ever matters (build size, vulnerability scanning, audit burden), inlining `udp_proxy.go` (Apache 2.0) would cut both `gvisor.dev/gvisor` and `github.com/inetaf/tcpproxy` from go.mod. Lima makes the same trade-off in `pkg/portfwd/client.go`.

- **Two-scan stability gate requires the port to appear twice, not the same shape twice** *(in-scope)* [Claude Opus 4.7]

  If tick N sees `127.0.0.1:8009` (pending) and tick N+1 sees `0.0.0.0:8009`, the port publishes with the wildcard bindings from N+1, not the loopback bindings of N. `TestBindingShapeChangePublishesCurrent` pins this as intentional. Worth a one-line comment in `Tick` so a future reader does not add a shape-equality check thinking it is a bug. (Codex's I2 in this round and Gemini's S2 in Round 3 both reached for the same change; the behavior is intentional but undocumented.)

- **Procnet ingress addresses are split across components** *(future)* [Codex GPT 5.5]

  S6 surfaces that procnet knows only the host-switch tap IP, while `wsl-proxy` independently uses the bridge IP. Passing an explicit list of procnet ingress IPs from the networking setup into guestagent would make the contract visible and testable, even if today's single-IP bind turns out to suffice.

### Strengths

- **Injectable `loopbackController` interface and `newScanner` constructor** enable unit testing of reconciliation logic without faking iptables or the host-switch API. The `Tick(scanned)` granularity is exactly right for the test harness. [Claude Opus 4.7]
- **End-to-end tests** (`TestEndToEndForwardingThroughTick`, `TestEndToEndUDPAndRemovalThroughTick`) wire a real forwarder behind the scanner reconciliation and exercise actual traffic, closing the seam pure-mock tests would leave open. [Claude Opus 4.7]
- **Comments at decision sites** — UDP context lifetime, wildcard short-circuit, `addErrorLogged` ordering — explain the *why* exactly where a future maintainer needs it. [Claude Opus 4.7]
- **Net simplification** of ~1850 deleted lines in exchange for a 247-line forwarder plus reorganized scanner is the central design win, and the equivalence reasoning in the PR description holds. [Claude Opus 4.7, Gemini 3.1 Pro]

5. **wsl-proxy ingress IP end-to-end** (S6). Pre-merge verification on a real WSL distro answers the open question, and a regression test would prevent re-introduction.

`loggedAcceptError` resets after every successful `Accept`.
