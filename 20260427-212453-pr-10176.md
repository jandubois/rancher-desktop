## Executive Summary

Round 1's fixups addressed every Round 1 finding: the schema is now a
single compound `DependencyManifest` mirroring the on-disk YAML
(prior D1), `fetchUpstreamChecksums` recognises both GNU and BSD
sidecar formats with case-insensitive hex (prior S2),
`MobyOpenAPISpec` pins an immutable commit SHA alongside the API
version (prior I1), `downloadAndHash` validates `expected` before
hashing with a precise diagnostic (prior I2), `globalDependencies` is
now a single shared list imported by both `rddepman.ts` and
`populate-dependency-checksums.ts` (prior S1), `hashFile` is exported
from `download.ts` and reused (prior S6), the `MACOS_RUNNER` constant
replaces the duplicated platform string (prior S5), and the
`AlpineLimaISO.updateManifest` override is gone (prior S3).

The single point all three agents independently flagged is a
documentation defect introduced by the I1 fix: the comment on
`MobyOpenAPISpec.rcompareVersions` claims that `postinstall` will
"surface such drift loudly through checksum verification" for
same-apiVersion commit changes, but the install URL now pins the
recorded commit, so post-bump drift on `master` cannot reach the
verification path. The behaviour (don't auto-bump for commit-only
changes) matches the author's stated intent; only the comment's
justification is wrong (S1 below). Gemini interpreted the same line
as a behaviour bug and proposed the opposite fix; consolidation
favours the comment-only repair because it preserves the design
intent the author documented.

The remaining suggestions are quality-of-life: a latent basename
collision in `fetchUpstreamChecksums` (consequence of the path/basename
dual-indexing added in the prior S2 fix), a one-shot migration script
that no longer covers the `mobyOpenAPISpec` flat-format path, dead
`checksumAlgorithm` plumbing that survived the algorithm-uniform
refactor, a misleading concurrency comment on a per-subclass static
manifest cache, an error-message disambiguation, a cosmetic
indentation slip, an optional-vs-required type tweak, and the
pre-existing cache-skip in `download()` that survives the new
pinned-checksum claims.

Verdict: merge as-is. The Moby comment fix is worth folding in;
nothing else blocks the merge.

S1. **Misleading drift comment on `MobyOpenAPISpec.rcompareVersions`** — `scripts/dependencies/moby-openapi.ts:117-123` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

```ts
// Compare the API version only.  Two entries that share an apiVersion but
// differ on commit are treated as equal here so install-time drift in a
// released spec does not by itself open a bump PR; postinstall surfaces
// such drift loudly through checksum verification.
rcompareVersions(version1: MobyOpenAPISpecVersion, version2: MobyOpenAPISpecVersion): -1 | 0 | 1 {
  return super.rcompareVersions(version1.apiVersion, version2.apiVersion);
}
```

The comment claims `postinstall` surfaces drift loudly when an
upstream commit moves under a stable apiVersion, but `download()` at
`scripts/dependencies/moby-openapi.ts:30-33` builds the URL from the
pinned `commit`, so the install path fetches byte-identical content
from the recorded commit forever. Drift cannot reach the verification
path. The behaviour — never auto-bump for commit-only changes within
a stable apiVersion — matches the author's stated intent in the
commit message and Round 1 PR discussion; the justification in the
comment, however, describes a mechanism the design no longer
provides.

Gemini read the same lines as a behaviour bug ("Upstream fixes to
existing API versions will be ignored forever") and proposed
treating differing commits as an upgrade. That fix would change the
design — `rddepman` would open a bump PR every time Moby pushes any
commit that touches `api/docs/v${apiVersion}.yaml`, including
trivial annotation cleanups. Claude and Codex both read the lines as
a comment-only defect and that interpretation aligns with the rest of
the change set (the `rcompareVersions` body is deliberately reduced
to `apiVersion`).

Fix: rewrite the comment to describe the actual rationale.

```diff
-  // Compare the API version only.  Two entries that share an apiVersion but
-  // differ on commit are treated as equal here so install-time drift in a
-  // released spec does not by itself open a bump PR; postinstall surfaces
-  // such drift loudly through checksum verification.
+  // Compare the API version only.  The on-disk commit pin makes the
+  // install URL immutable, so a fresh commit on the same apiVersion
+  // cannot drift the install-time content and does not warrant an
+  // auto-bump.  An operator wanting the new commit edits dependencies.yaml.
   rcompareVersions(version1: MobyOpenAPISpecVersion, version2: MobyOpenAPISpecVersion): -1 | 0 | 1 {
     return super.rcompareVersions(version1.apiVersion, version2.apiVersion);
   }
```

S2. **`fetchUpstreamChecksums` basename indexing overwrites on path collision** — `scripts/lib/dependencies.ts:251-286` [Claude Opus 4.7]

```ts
if (hex && fullName) {
  const basename = fullName.replace(/^.*\//, '');

  result[fullName] = hex.toLowerCase();
  result[basename] = hex.toLowerCase();
}
```

The dual-key indexing (introduced by the Round 1 S2 fix that added
basename support for path-prefixed sidecars) silently overwrites the
basename entry when a sidecar lists two entries that share a basename
across different paths (e.g., `release/foo.tar.gz` and
`archive/foo.tar.gz`). Every current caller looks up by basename
because they construct it locally from `archiveName` and pass it to
`fetchUpstreamChecksums(...)[archiveName]`. None of today's upstream
sidecars have such a collision, so the bug is latent. The failure
mode is visible — `downloadAndHash`'s `verify` raises a checksum
mismatch when the wrong hex flows through — but the diagnostic
"expected sha512:X, got sha512:Y" misleads the reader into suspecting
artifact corruption instead of a parser ambiguity.

Fix: only insert a basename alias when the basename is unambiguous
across all parsed entries.

```diff

-      const basename = fullName.replace(/^.*\//, '');
-
-      result[fullName] = hex.toLowerCase();
-      result[basename] = hex.toLowerCase();
-    }
+    if (hex && fullName) {
+      result[fullName] = hex.toLowerCase();
+    }
+  }
+  const basenameCounts: Record<string, number> = {};
+  for (const fullName of Object.keys(result)) {
+    const basename = fullName.replace(/^.*\//, '');
+    basenameCounts[basename] = (basenameCounts[basename] ?? 0) + 1;
+  }
+  for (const fullName of Object.keys(result)) {
+    const basename = fullName.replace(/^.*\//, '');
+    if (basenameCounts[basename] === 1 && !(basename in result)) {
+      result[basename] = result[fullName];
+    }
+  }
```

## Design Observations

### Concerns

None this round.

### Strengths

**D1. The compound `DependencyManifest` mirrors the on-disk YAML 1:1** (in-scope) [Claude Opus 4.7]

Round 1's D1 ("why split versions and checksums?") has landed
cleanly. `DependencyManifest = { [K]: DependencyEntry<K> }` matches
the file shape, so `readDependencyManifest` and
`writeDependencyManifest` carry no translation logic. The
convenience wrapper `readDependencyVersions` keeps the small set of
version-only callers (`lint-go`, `docker-cli-monitor`) unchanged.

**D2. `MobyOpenAPISpec` pins an immutable commit** (in-scope) [Claude Opus 4.7]

The `MobyOpenAPISpecVersion = { apiVersion, commit }` shape plus
`latestCommitForApiVersion()` close Round 1's I1 (mutable Moby
URL). The install URL embeds the recorded commit, so post-bump
drift on `master` cannot reach the verification path; the bump path
records the resolved commit. The accompanying `rcompareVersions`
override (and S1 above) document the deliberate choice not to
auto-bump for commit-only changes within a stable apiVersion.

**D3. Shared `globalDependencies` list** (in-scope) [Codex GPT 5.5]

`scripts/dependencies/global.ts` is now the single declaration of
the global dependency set, imported by `rddepman.ts` and
`populate-dependency-checksums.ts`. Round 1 S1's "keep in sync"
hazard is gone.

**D4. Computing checksums before `git checkout -b`** (in-scope) [Codex GPT 5.5, Gemini 3.1 Pro]

Re-confirmed from Round 1: `rddepman.ts:291-294` resolves fresh
checksums before branching, so an upstream verification failure
aborts the bump before any git state changes.

**D5. Dual GNU + BSD sidecar parsing** (in-scope) [Gemini 3.1 Pro]

`fetchUpstreamChecksums` recognises both `<hex> [* ]<filename>`
and `SHA256 (filename) = <hex>` lines, normalises hex to lowercase
to match what `crypto.createHash` and `hashFile` produce, and warns
when zero entries parse — addressing Round 1 S2 in full. (S2 above
flags the basename-aliasing edge case, but the format coverage is
the strength.)

For archive downloads, persist a sidecar stamp after successful extraction containing the artifact URL and expected checksum; skip only when both the extracted target exists and the stamp matches. If the stamp is missing or mismatched, redownload and replace, then write the stamp last so crashes before stamp creation self-heal on the next run.

### Executive Summary

This PR transitions the `dependencies.yaml` manifest to a compound structure, storing both a pinned version and `sha256` checksums for every upstream artifact. Install-time scripts now perform local cryptographic verification against these pinned digests instead of relying on live upstream sidecars, successfully closing the window in which a compromised mirror could swap an artifact post-bump. The implementation is comprehensive, gracefully mirroring the diverse checksum sidecar formats of all upstream dependencies, and safely isolates state during concurrent dependency bump cycles.

#### Important Issues

I1. **Contradictory `MobyOpenAPISpec` version comparison** — `scripts/dependencies/moby-openapi.ts:117-123`

```typescript
  // Compare the API version only.  Two entries that share an apiVersion but
  // differ on commit are treated as equal here so install-time drift in a
  // released spec does not by itself open a bump PR; postinstall surfaces
  // such drift loudly through checksum verification.
  rcompareVersions(version1: MobyOpenAPISpecVersion, version2: MobyOpenAPISpecVersion): -1 | 0 | 1 {
    return super.rcompareVersions(version1.apiVersion, version2.apiVersion);
  }
```

The logic and comment contain a contradiction that permanently freezes the OpenAPI spec updates. The comment relies on `postinstall` failing checksum verification to "surface such drift loudly." However, the updated `download()` method (lines 30-31) now securely pins the fetch URL to the exact `commit` hash stored in `dependencies.yaml`. Because the fetch URL is immutable, the downloaded content will always match the stored checksum and `postinstall` will never catch any drift. 

Consequently, since `rcompareVersions` ignores commit changes, `rddepman` will never open a bump PR for an updated commit if the `apiVersion` hasn't changed, and `postinstall` will never fail to prompt a manual update. Upstream fixes to existing API versions will be ignored forever.

**Fix**: Update `rcompareVersions` to treat differing commits as an upgrade. This ensures `rddepman` correctly opens bump PRs to adopt upstream spec fixes automatically.

```diff
-  // Compare the API version only.  Two entries that share an apiVersion but
-  // differ on commit are treated as equal here so install-time drift in a
-  // released spec does not by itself open a bump PR; postinstall surfaces
-  // such drift loudly through checksum verification.
+  // Treat differing commits as an upgrade to ensure rddepman opens a bump PR.
+  // Since `download()` uses an immutable commit URL, postinstall cannot
+  // dynamically surface upstream drift.
   rcompareVersions(version1: MobyOpenAPISpecVersion, version2: MobyOpenAPISpecVersion): -1 | 0 | 1 {
-    return super.rcompareVersions(version1.apiVersion, version2.apiVersion);
+    const cmp = super.rcompareVersions(version1.apiVersion, version2.apiVersion);
+    if (cmp !== 0) {
+      return cmp;
+    }
+    return version1.commit === version2.commit ? 0 : 1;
   }
```
