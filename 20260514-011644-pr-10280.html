// Every other failure mode (exit status 4 from xtables lock
// contention, fork/exec failure, invalid-argument errors, etc.)
// returns false so the caller bubbles the error and defers the scan.
// "Absent" here means a steady-state "no" the caller can act on now;
// anything else might succeed on the next tick.
func isIptablesRuleAbsent(err error) bool {
	if err == nil {
		return false
	}
	var exitErr *exec.ExitError
	if !errors.As(err, &exitErr) {
		return false
	}
	stderr := string(exitErr.Stderr)
	return strings.Contains(stderr, "No chain/target/match by that name") ||
		strings.Contains(stderr, "does a matching rule exist in that chain")
}

Design Observations ¶

Concerns ¶

  • (future) dnatChainContainsPort parses human-readable iptables --list output — scanner_linux.go:255-272 Claude. The regex matches the columnar iptables -L --numeric rendering, which is presentation output, not a stable interface — column spacing, the multiport dports rendering, and the protocol display can drift between iptables versions and the legacy/nft backends. A regression there produces a false negative: procnet decides the engine does not manage the port, adds its own DNAT, and silently reinstates the original shadow-DNAT bug. The engineChainManagesPort docstring acknowledges only published-ports.bats covers this end to end, and that is a single happy-path fixture. iptables -S <chain> emits the rule-spec form (-A CHAIN -p tcp -m tcp --dport 8080 ...), which is designed to be machine-parsed; worth recording as the fallback if a future iptables bump breaks the regex. Not worth changing now given the bats safety net.
  • (future) exposeNew does not re-probe the engine chain after a healthy tracker.Add — port_state_tracker_linux.go:167-218 Claude. retryAppend deliberately re-probes the engine chain before re-appending because a full tick elapsed since the original probe; exposeNew probes once, then Add, then Append with no re-probe. The residual gap is the opposite ordering: a port exposeNew recorded as healthy-owned, after which an engine chain appears for the same port — addObserved no-ops healthy added ports, so the procnet rule persists and shadows. This is the same netns-blind / stability-gate-bypass limitation the portState doc comment (lines 71-73) and Round 2's "residual netns-blind heuristic race" already call out, restated from the exposeNew/retryAppend asymmetry angle; it self-heals when the transient listener disappears. No action needed.
  • (future, pre-existing) %w verb in a log.Errorf call — scanner_linux.go:120 [orchestrator]. log.Errorf("failed to create portMapping for entry: %w", err) — %w is meaningful only in fmt.Errorf and renders as %!w(...). Present in the merge-base, so pre-existing, not a regression; flagged in Rounds 1 and 2 as well. The line sits in a file this PR rewrites heavily, so the one-character fix (%w → %s) is low-cost to fold in, but it remains out of scope for the procnet change.

Review Process Notes ¶

Skill improvements ¶

  • When code classifies a subprocess failure by substring-matching its stderr or stdout — deciding "file not found" vs "permission denied", "rule absent" vs "transient lock contention", and the like — check whether the process locale is pinned to C/POSIX. Unpinned locale-sensitive matching silently misclassifies under a localized build of the tool, and the misclassification often routes a steady-state "no" into a "retry forever" path. Flag substring matching of tool output where the invocation does not pin LC_ALL=C (or equivalent).
  • When a code path keeps or releases a resource (an iptables rule, a lock file, a temp directory) based on whether a multi-step operation returned an error, check whether that error value distinguishes which step failed. A single opaque error frequently conflates "step A failed, later steps skipped" with "step A succeeded, step B failed" — and the correct cleanup differs between them. Flag keep/release decisions that branch only on err != nil when the operation has multiple side-effecting steps with distinct post-conditions; the fix is usually a typed or sentinel-wrapped error, not a change to the cleanup logic.

  • [repo] APITracker.Remove (src/go/guestagent/pkg/tracker/apitracker.go) runs host-switch apiForwarder.Unexpose for every binding first, then wslProxyForwarder.Send. A Send failure returns early with "sending port mappings to wsl proxy error: %w", which does NOT wrap forwarder.ErrUnexposeAPI. So a non-nil error from tracker.Remove does not imply the host-switch proxy is still bound — it can mean every Unexpose succeeded and only the wsl-proxy notification failed. APITracker.Add has the same Expose-then-Send ordering but seeds portStorage before Send. Code that keeps, deletes, or transfers iptables rules or tracker entries based on an Add/Remove error must account for both partial-failure shapes.
