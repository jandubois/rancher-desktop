## Executive Summary

This branch adds a comprehensive i18n system to Rancher Desktop: a Go-based CLI tool (`i18n-report`) for translation management, main-process and renderer-side locale support with settings integration, and translation files for English, German, and Simplified Chinese. The implementation is well-architected with clean separation between main process (simple `{variable}` interpolation) and renderer (ICU MessageFormat via `intl-messageformat`), proper race-condition handling for locale switching, and a feature gate for gradual rollout. Four review passes across three agents found no critical issues, one important stale-comment issue, and 20 suggestions concentrated in the Go tooling's edge cases. The code is ready to merge after fixing the misleading comment.

## Suggestions

S1. **`stripYAMLQuotes` double-quote unescaping order** — `src/go/i18n-report/yaml.go:191-195` [Claude Opus 4.6]

```go
inner = strings.ReplaceAll(inner, `\\`, `\`)
inner = strings.ReplaceAll(inner, `\"`, `"`)
```

Replacing `\\` before `\"` corrupts strings containing both sequences. Input `"foo\\\"bar"` yields `foo"bar` instead of `foo\"bar`. Practical impact is negligible — translation strings never contain literal backslash-quote sequences — but the fix is trivial.

Fix: swap the replacement order.

(suggestion, regression)

S2. **`ContainerInfo.vue` local `t()` wrapper drops interpolation args** — `pkg/rancher-desktop/pages/containers/ContainerInfo.vue:124` [Claude Opus 4.6]

```ts

```

This wrapper only passes the key, dropping the `args` parameter the store getter accepts. All currently used keys are plain strings, so this works today but is a latent foot-gun for future parameterized keys.

Fix: `const t = (key: string, args?: Record<string, unknown>) => store.getters['i18n/t'](key, args);`

(suggestion, regression)

## Design Observations

### Concerns

- **Translated error messages in HTTP API responses** (future) — The settings validator now produces translated error messages via `t()`. External tools calling the API would receive localized errors. The `hasLockedFieldError` flag handles the known parsing case, but the broader pattern of returning translated error strings in API responses may need revisiting when the feature gate is removed. Consider returning structured error codes alongside human-readable messages. [Codex GPT 5.4]

- **Manifest is not the runtime source of truth** (future) — `main/i18n.ts`, `store/i18n.js`, and `settingsValidator.ts` all discover locales by enumerating YAML files, while manifest status is only enforced by the Go tooling. Making `meta/locales.yaml` authoritative at runtime would prevent an accidentally-added partial locale file from becoming selectable. [Codex GPT 5.4]

- **Locale-loading state machine duplicated** (in-scope) — The locale-switching logic exists in both `main/i18n.ts` and `store/i18n.js` with subtly different race-condition handling. A shared "last request wins" pattern would remove a class of ordering bugs. [Codex GPT 5.4] (pass 2)

### Strengths

- **Two-tier i18n architecture** — Main process uses lightweight `{variable}` interpolation; renderer uses full ICU MessageFormat. The right tradeoff for their respective needs. [Claude Opus 4.6]

- **Race-condition handling in `i18n.ts`** — The `pendingLocale` mechanism correctly handles most rapid locale switches. The callback array is copied before iteration to handle self-unregistering callbacks. [Claude Opus 4.6]

- **Feature gate design** — Hiding the locale selector when `locale === 'none'` provides a clean rollout path enforced at validation, UI, and both i18n modules. [Claude Opus 4.6]

- **AST-level YAML manipulation** — Using `gopkg.in/yaml.v3` node trees in `nodeSetLeaf` and `removeKeyFromNode` preserves whitespace, comments, and structural integrity during merges. [Gemini 3.1 Pro]

- **Comprehensive Go tooling** — The `i18n-report` CLI covers the full translation lifecycle with 14 subcommands, significantly reducing the chance of translation regressions. [Claude Opus 4.6]

- **Defensive `intlCache` invalidation** — The store clears the entire formatter cache before setting the new locale, preventing stale cached results. [Claude Opus 4.6] (pass 2)

S1. **`stripYAMLQuotes` double-quote unescaping order** — `src/go/i18n-report/yaml.go:191-195` (suggestion, regression)

```go

    inner := s[1 : len(s)-1]

```

Replacing `\\` before `\"` corrupts strings containing both sequences. Practical impact is negligible — translation strings essentially never contain literal backslash-quote sequences — but the function's name implies it handles YAML double-quoted syntax.

Fix:
```diff

- inner = strings.ReplaceAll(inner, `\\`, `\`)
  inner = strings.ReplaceAll(inner, `\"`, `"`)
+ inner = strings.ReplaceAll(inner, `\\`, `\`)
  return inner
```

## Design Observations

### Strengths

- **Two-tier i18n architecture**: The main process uses lightweight `{variable}` interpolation (avoiding a heavy ICU library dependency), while the renderer uses full ICU `MessageFormat` via `intl-messageformat`. This is the right tradeoff — main-process strings are simple labels; renderer strings need pluralization and formatting.

- **Race condition handling in `i18n.ts`**: The `pendingLocale` mechanism (lines 129–135) correctly handles rapid locale switches during async loads by comparing the pending locale after the `await`, dropping superseded changes. The callback array is copied before iteration (line 137) to handle callbacks that unregister themselves.

- **Feature gate design**: Hiding the locale selector when `locale === 'none'` (the default) provides a clean rollout path. The gate is enforced at validation time (`settingsValidator.ts:104`), in the preferences UI (`ApplicationGeneral.vue:41`), and in both i18n modules (main and renderer). Power users can enable it via `rdctl` or `settings.json`.

- **`hasLockedFieldError` flag**: Adding a structured flag (lines 75, 288) to replace regex-matching of error strings is the right approach for the translation of validation messages. The comment at lines 287–290 documents the remaining cleanup needed.

- **Comprehensive Go tooling**: The `i18n-report` CLI covers the full translation lifecycle — unused/missing/stale detection, merge with three input formats, structural validation (placeholders, ICU branches, HTML tags, data attributes), drift detection via metadata, and manifest cross-validation against `command-api.yaml` and `settingsValidator.ts`. This significantly reduces the chance of translation regressions.

### Concerns

- **Translated error messages in HTTP API responses** (future): The settings validator now produces translated error messages via `t()`. External tools calling the API (beyond `rdctl`) would receive localized errors. The `hasLockedFieldError` flag handles the known parsing case, but the broader pattern of returning translated error strings in API responses may need revisiting when the feature gate is removed. Consider returning structured error codes alongside human-readable messages. (future)

I1. **`store/i18n.js` init action races initial render against IPC round-trip** — `pkg/rancher-desktop/store/i18n.js:167-202`

```js
// Use the cookie for fast initial render.
let selected = this.$cookies.get(LOCALE, { parseJSON: false });
// ...
ipcRenderer.once('settings-read', (_, settings) => {
  const raw = settings?.application?.locale;
  const locale = (!raw || raw === 'none') ? state.default : raw;

    dispatch('switchTo', locale);
  }
});
ipcRenderer.send('settings-read');

```

The `init` action dispatches `switchTo` with the cookie value at line 202, then
the `settings-read` IPC response may dispatch `switchTo` again with a different
locale. The comment at lines 189-190 acknowledges this ("May briefly flash if
the cookie and settings disagree"). However, when the locale changes mid-render,
the `intlCache` is cleared (lines 232-234) and all Vue components that use
`t()` re-evaluate. For a user who changes the locale via `rdctl` (which updates
settings.json but not the renderer cookie), every application restart flashes
the old locale then switches.

The cookie is set in `switchTo` at lines 237-242, but only when the renderer's
`switchTo` runs. So the cookie can become permanently stale if the locale is
only changed via `rdctl set` and the app is not restarted between the change
and the next cookie read.

After the `settings-read` handler fires and dispatches `switchTo`, the cookie
does get updated (since `switchTo` always sets it at line 237). So the staleness
self-corrects after one launch. This is a minor UX issue the comment
acknowledges, but the flash is visible: the entire UI renders in the old locale,
then re-renders in the new one. Verify that `rdctl set application.locale=X`
also triggers a `settings-update` IPC event to the renderer while the app is
running, which would update the cookie immediately (avoiding the flash on
*next* launch).

## Design Observations

### Concerns

**Renderer and main process i18n have divergent interpolation semantics** (in-scope)
— The main-process `t()` (`main/i18n.ts:67-81`) uses simple `{variable}`
replacement via `String.replaceAll`, while the renderer (`store/i18n.js:47-98`
and `plugins/i18n.js:6-29`) uses ICU MessageFormat via `intl-messageformat`.
A translation string with ICU plural syntax like
`{count, plural, one {item} other {items}}` works in the renderer but renders
literally in the main process. The comment at `main/i18n.ts:4-5` acknowledges
this. For the current set of main-process strings (tray labels, menu items,
dialog messages), none use ICU constructs. But if a translator adds an ICU
plural to a key used in both contexts, the main process displays raw ICU syntax.
The `validate` command checks ICU structure parity between English and locale
but does not flag ICU usage in keys consumed by the main process. Consider
adding a `validate` check that warns when keys referenced by the main process
contain ICU constructs.

### Suggestions

S1. **`handleNavigateClick` and `onLogLinesClick` check only `event.target`, not ancestors** — `navigateClick.ts:9-10`, `KubernetesError.vue:118-121` (suggestion, regression)

```ts
const target = event.target as HTMLElement;
const nav = target?.getAttribute('data-navigate');
```

Both `handleNavigateClick` (line 9-10 of `navigateClick.ts`) and `onLogLinesClick` (line 119 of `KubernetesError.vue`) check `event.target` directly for `data-navigate` or `data-action` attributes. If a translator wraps link text in a nested element (e.g., `<a data-navigate="..."><em>WebAssembly</em></a>`), the click target would be the `<em>`, which lacks the data attribute, and the handler would silently do nothing. The `validate` command's `checkTags` function (at `report_validate.go:230`) validates tag parity but not nesting structure, so it wouldn't catch this.

Fix: Use `closest()` to walk up the DOM:
```diff
- const target = event.target as HTMLElement;
- const nav = target?.getAttribute('data-navigate');
+ const target = (event.target as HTMLElement)?.closest('[data-navigate]');
+ const nav = target?.getAttribute('data-navigate');
```

And similarly in `KubernetesError.vue`:
```diff
  onLogLinesClick(event: MouseEvent) {
-   if ((event.target as HTMLElement)?.getAttribute('data-action') === 'showLogs') {
+   if ((event.target as HTMLElement)?.closest('[data-action="showLogs"]')) {
      this.showLogs();
    }
  },
```

S2. **Image deletion dialogs use generic "OK"/"Cancel" instead of descriptive action labels** — `Images.vue:324,346` (suggestion, regression)

```ts
buttons:   [this.t('generic.ok'), this.t('generic.cancel')],
```

The old code used `['Yes', 'No']` for both single and batch image deletion confirmations. The new code uses `['OK', 'Cancel']` via `generic.ok` and `generic.cancel`. This is inconsistent with other destructive dialogs in the same codebase: the migration dialog uses `t('dialog.confirmMigration.delete')` ("Delete Workloads"), the preferences reset dialog uses `t('preferences.resetDialog.apply')` ("Apply and reset"), and the close-preferences dialog uses `t('preferencesWindow.close.discardChanges')` ("Discard changes"). Using generic "OK" for a destructive operation (deleting container images) is less informative than a specific action label.

Fix: Add specific translation keys for the image deletion confirm action:
```diff
- buttons:   [this.t('generic.ok'), this.t('generic.cancel')],
+ buttons:   [this.t('images.confirmDelete.confirm'), this.t('generic.cancel')],
```
With a translation like `images.confirmDelete.confirm: "Delete"`.

S3. **`mainmenu.ts` module-level `onLocaleChange` discards cleanup function** — `mainmenu.ts:24` (suggestion, regression)

```ts
onLocaleChange(() => buildApplicationMenu());
```

This module-level call registers a locale-change callback and discards the returned cleanup function. In contrast, `Tray` at line 204 properly stores and calls its cleanup function in `hide()`. While the application menu lives for the process lifetime making cleanup unnecessary in practice, discarding the cleanup handle is inconsistent with the pattern established by the Tray and could become a problem if module-level code is ever restructured.

Fix: Store the cleanup function in a module-level variable for symmetry, even if it's never called:
```ts
const _offLocaleChange = onLocaleChange(() => buildApplicationMenu());
```

## Design Observations

**Strengths**

- The `hasLockedFieldError` flag on `SettingsValidator` is a clean replacement for the previous regex-matching approach. It eliminates coupling between the error message format and the detection logic, making the locale transition safe.

- The `titleKey`/`descriptionKey` extension of the page store (at `store/page.ts:8-14`) is a well-designed approach that keeps page titles reactive to locale changes without requiring each page to re-dispatch when the locale changes. The `TheTitle.vue` computed `displayTitle` at line 24-26 handles both legacy `title` and new `titleKey` seamlessly.

- Feature-gating the locale selector behind `application.locale !== 'none'` (at `ApplicationGeneral.vue:37-39`) is pragmatic — it allows the i18n infrastructure to ship and be tested without exposing incomplete translations to all users.

- The `checkTags` function in `report_validate.go:230-279` validates both HTML tag parity and `data-*` attribute preservation across translations. This catches a category of bugs (broken event delegation in translated HTML) that would otherwise only surface at runtime in non-English locales.

### Executive Summary

This branch adds a broad i18n system across the Electron main process, renderer, settings model, and a new Go-based translation maintenance CLI. The main risks are in the new maintenance workflow: CLI/API validation output is now locale-dependent on paths that were previously stable for automation, and the `translate`/`merge` round-trip loses multiline translation content by design.

I2. **`translate`/`merge` cannot round-trip multiline translations** — [src/go/i18n-report/report_translate.go](/private/tmp/review-codex-1IAa2C/src/go/i18n-report/report_translate.go):11-15 (important, regression)

```go
// normalizeWhitespace collapses newlines and surrounding whitespace into
// single spaces so that multiline values fit on one key=value output line.
func normalizeWhitespace(s string) string {
	return strings.Join(strings.Fields(s), " ")
}
```

`normalizeWhitespace()` at line 11 and `fmt.Printf("%s=%s\n", p.Key, normalizeWhitespace(p.Value))` at line 176 flatten every value before it is handed to translators. On the way back, `parseMergeInput()` in [report_merge.go](/private/tmp/review-codex-1IAa2C/src/go/i18n-report/report_merge.go):335-349 only accepts a single-line `key=value`/`key: value` record, so the original line breaks are unrecoverable. This is already observable in shipped source strings such as [en-us.yaml](/private/tmp/review-codex-1IAa2C/pkg/rancher-desktop/assets/translations/en-us.yaml):79-82 (`imageEvents.deleteError`) and [en-us.yaml](/private/tmp/review-codex-1IAa2C/pkg/rancher-desktop/assets/translations/en-us.yaml):521-525 (`sortableTable.paging.generic`): a translator who runs `translate` and feeds the result back into `merge` will silently rewrite those block scalars into single-line strings. For dialog/error text, that is a visible regression; for more structured ICU content, it also makes translator output harder to review correctly.

Fix: preserve exact string contents in the interchange format instead of whitespace-normalizing them. The least invasive fix is to emit quoted Go/JSON-style strings in text mode and unquote them in `merge`; alternatively, make JSON the only lossless transport for agent workflows and keep text mode for single-line values only.

```diff
- fmt.Printf("%s=%s\n", p.Key, normalizeWhitespace(p.Value))
+ encoded, err := strconv.Quote(p.Value)
+ if err != nil { return err }
+ fmt.Printf("%s=%s\n", p.Key, encoded)
```

Then in `parseMergeInput()`:

```diff
- value = stripYAMLQuotes(trimmed[idx+1:])
+ value = trimmed[idx+1:]
+ if unquoted, err := strconv.Unquote(value); err == nil {
+   value = unquoted
+ } else {
+   value = stripYAMLQuotes(value)
+ }
```

S1. **`merge` still leaves locale and metadata out of sync on write failure** — [src/go/i18n-report/report_merge.go](/private/tmp/review-codex-1IAa2C/src/go/i18n-report/report_merge.go):173-195 (suggestion, regression)

```go
// Serialize locale file and prepare metadata before writing either,
// so a failure in one doesn't leave the other in an inconsistent state.
…
if err := os.WriteFile(localePath, localeData, 0644); err != nil {
	return fmt.Errorf("writing %s: %w", localePath, err)
}
if err := writeMetadata(root, locale, enKeys, localeKeys); err != nil {

```

The comment at line 173 says the code avoids inconsistency, but line 191 writes the locale file before line 194 writes metadata. If the second write fails, the repo is left in the exact half-updated state the comment says it prevents: translated YAML has moved forward, metadata has not, and subsequent `drift`/`validate` results are immediately wrong for that locale.

```diff
-   return fmt.Errorf("writing %s: %w", localePath, err)
- }
- if err := writeMetadata(root, locale, enKeys, localeKeys); err != nil {
-   return err
- }
+ // write locale + metadata to temp files, then rename both into place
+ // so readers never observe a mixed old/new pair
```

### Design Observations

**Concerns**

- `(future)` The manifest is not the runtime source of truth yet. [main/i18n.ts](/private/tmp/review-codex-1IAa2C/pkg/rancher-desktop/main/i18n.ts):18-20, [store/i18n.js](/private/tmp/review-codex-1IAa2C/pkg/rancher-desktop/store/i18n.js):15-23, and [settingsValidator.ts](/private/tmp/review-codex-1IAa2C/pkg/rancher-desktop/main/commandServer/settingsValidator.ts):104 all discover locales by enumerating YAML files, while manifest status is only enforced by the Go tooling. Making `meta/locales.yaml` authoritative at runtime would prevent an accidentally-added partial locale file from immediately becoming a selectable and valid app locale.

**Strengths**

- Using the same YAML catalog for renderer and main-process translations is the right architecture. It removes a whole class of drift bugs that usually appear when tray/menu strings live in a separate catalog.
- The Go merge implementation operates on `yaml.Node` trees instead of plain maps, which is a strong choice for preserving translator comments and `@override` annotations.

### Testing Assessment

1. No integration test flips `application.locale` away from English before asserting CLI/API validation stderr or HTTP bodies, so the locale-dependent contract break in I1 is currently untested.
2. No round-trip test covers multiline/block-scalar values through `translate` -> `merge`, so I2 would not be caught by the current `report_translate_test.go` / `report_merge_test.go` suite.
3. No failure-injection test covers `report_merge()` succeeding on the locale write and failing on metadata write, so the mixed-state path in S1 is unverified.

### Executive Summary

This change introduces a comprehensive i18n system, including a full Go-based `i18n-report` toolset for managing locale files, Vuex/Main process integration, and settings validation. The implementation is robust, with excellent handling of AST-level YAML merging and dynamic discovery. ### Findings

#### Critical Issues
None.

#### Important Issues

I1. **reportCheckBasic fails locale validation for global unused source keys** — `src/go/i18n-report/report_check.go:73-90`

```go
       unusedCount := 0
       for k := range enKeys {
               if _, found := refs[k]; !found {
                       unusedCount++
               }
       }
// ...
       printResult("unused keys", unusedCount)
       // ...
       if passed {
               fmt.Println("All checks passed.")
               return nil
       }
       return fmt.Errorf("checks failed")
```

When `i18n-report check` is run without a `--policy` (e.g., `i18n-report check --locale de`), it invokes `reportCheckBasic`. If any key in the `en-us.yaml` source is unused (`unusedCount > 0`), the command exits with an error and fails the validation for the target locale (`de`). Unused keys are a global, source-level linting concern, while locale validation is specific to the target translation file. Coupling them means that if a developer removes a UI component (leaving a key unused in `en-us.yaml`), translators working on `de.yaml` or `zh-hans.yaml` will see their CI checks fail for reasons entirely outside their control.

**Fix**: Decouple unused key linting from locale validation. Either create a separate `lint-source` subcommand for `unusedCount`, or modify `reportCheckBasic` so that `unusedCount > 0` prints a warning but does not set `passed = false`.

#### Suggestions

S1. **`extractDataAttrs` regex misses single-quoted attributes** — `src/go/i18n-report/report_validate.go:214`

```go
// attrRe matches HTML attributes of the form name="value".
var attrRe = regexp.MustCompile(`(data-\w+)="([^"]*)"`)
```

The attribute extraction heuristic assumes `data-*` attributes are always double-quoted. While double quotes are standard in Rancher Desktop's Vue templates, valid single-quoted HTML attributes (e.g., `data-action='submit'`) will silently bypass validation.

**Fix**: Broaden the regex to capture both quote styles, e.g., ``regexp.MustCompile(`(data-\w+)=(['"])(.*?)\2`)``.

### Design Observations

#### Concerns

**(in-scope) Atomicity of metadata and locale updates**
In several places (`report_merge.go` and `report_remove.go`), the tool modifies the locale `.yaml` file and then attempts to update the corresponding `.yaml` file in `meta/`. If the metadata write fails (e.g., due to directory permissions), the system is left in a partially updated state, which could trigger drift errors later. While acceptable for a developer CLI tool, generating both files in memory and writing them only if both serialize successfully would prevent inconsistent state.
