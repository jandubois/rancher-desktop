## Consolidated Review

### Executive Summary

This PR replaces hardcoded Steve ports (9443/9080) with a probe-and-fallback mechanism: try the defaults, fall back to OS-assigned ephemeral ports. The resolved ports thread through DashboardServer, setupNetworking, and Steve itself. The utility function is clean and well-tested.

The main concern is a TOCTOU gap — ports are probed at app startup but Steve binds them only after Kubernetes reaches STARTED state, potentially minutes later. The practical risk is low (default ports sit outside the OS ephemeral range), but the gap is real and the code's comment misrepresents it.

**Verdict: Merge with fixes.** Address the logging gap and correct the misleading comment. The TOCTOU concern is worth discussing but does not block merging.

**Structure:** 1 commit — adds `findAvailablePort()` utility, wires dynamic ports through Steve, DashboardServer, and certificate-error handling.

### Important Issues

1. **No logging when ports fall back from defaults** — `background.ts:236-239` (important, gap) [Claude Opus 4.6]

```typescript
[steveHttpsPort, steveHttpPort] = await Promise.all([
  findAvailablePort(DEFAULT_STEVE_HTTPS_PORT),
  findAvailablePort(DEFAULT_STEVE_HTTP_PORT),
]);
```

The whole point of this PR is handling port conflicts, yet when a conflict occurs and Steve falls back to an ephemeral port, nothing records what happened. A user troubleshooting connectivity has no way to discover that Steve is on port 52347 instead of 9443.

Fix: Log the resolved ports after the `Promise.all`, at minimum when they differ from defaults:

```typescript
if (steveHttpsPort !== DEFAULT_STEVE_HTTPS_PORT || steveHttpPort !== DEFAULT_STEVE_HTTP_PORT) {
  console.log(`Steve default ports in use; using HTTPS=${ steveHttpsPort } HTTP=${ steveHttpPort }`);
}
```

2. **Ports probed at startup, bound minutes later** — `background.ts:236`, `background.ts:1291` (important, regression) [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

```typescript
// Line 236 — during app.whenReady():
[steveHttpsPort, steveHttpPort] = await Promise.all([...]);

// Line 1291 — after K8s reaches STARTED:
await Steve.getInstance().start(steveHttpsPort, steveHttpPort);
```

`findAvailablePort` releases sockets before returning. Between the probe (line 236) and Steve's bind (line 1291), the app loads settings, reads deployment profiles, initializes the command server, sets up the UI, and waits for Kubernetes — a gap of seconds to minutes. Another process could claim the port in that window.

**Practical risk assessment:** The default ports (9443/9080) sit outside the OS ephemeral range (typically 49152–65535), so when the defaults succeed, nothing can steal them by accident. When both defaults are taken and fallback ports come from port 0 (inside the ephemeral range), the risk grows slightly but remains low — the window would need a process to bind that exact ephemeral port.

Gemini rated this critical; Codex rated it important. I rate it **important** — the race exists but is unlikely to cause real failures. The code's own comment ("the caller binds immediately after launch") is misleading and should be corrected regardless.

Fix (minimal): Correct the comment in `findAvailablePort` to acknowledge the real gap. Fix (thorough): Move port resolution to just before `Steve.getInstance().start()`, and update DashboardServer's proxy target dynamically once Steve reports its actual port.

### Suggestions

1. **TOCTOU comment misrepresents the timing** — `utils/networks.ts:16` (suggestion, regression) [Claude Opus 4.6]

```typescript
 * The port is released before returning, so there is a small race window
 * before the caller actually binds it.  In practice this is negligible
 * because the caller (steve) binds immediately after launch.
```

Steve does not bind "immediately after launch." The comment should state: "In practice the risk is low because the default ports sit outside the OS ephemeral range and are unlikely to be claimed in the interim."

2. **`findAvailablePort(0)` returns 0 instead of the actual port** — `utils/networks.ts:33-35` (suggestion, gap) [Gemini 3.1 Pro]

```typescript
server.listen(preferredPort, '127.0.0.1', () => {
  server.close(() => resolve(preferredPort));
});
```

If someone calls `findAvailablePort(0)` to request an OS-assigned port, the function resolves with 0 instead of the actual bound port. No caller passes 0 today, but the fix is trivial and makes the function's contract honest.

Fix: Resolve with the actual address instead of the input:

```typescript
server.listen(preferredPort, '127.0.0.1', () => {
  const addr = server.address() as net.AddressInfo;
  server.close(() => resolve(addr.port));
});
```

3. **`DashboardServer.init()` silently ignores port on re-calls** — `main/dashboardServer/index.ts:66-71` (suggestion, gap) [Claude Opus 4.6, Gemini 3.1 Pro]

```typescript
public init(stevePort: number) {
  if (this.dashboardApp.address()) {
    console.log(`Dashboard Server is already listening on ${ this.host }:${ this.port }`);
    return;
  }
```

Before this PR the port was hardcoded, so the early-return guard was harmless. Now that `init()` accepts a parameter, a caller passing a different port would silently get stale proxies. This is safe today (init is called once), but the API shape invites misuse.

Fix: Log a warning when the guard fires and the incoming port differs from the current `this.api`.

### Testing Assessment

The `findAvailablePort` utility has solid unit tests covering three scenarios: preferred port available, preferred port occupied, and returned port actually usable. Tests use real sockets, not mocks.

Untested scenarios, ranked by risk:

1. **Integration: Steve starts on a non-default port and DashboardServer proxies to it.** No automated test verifies the end-to-end wiring. Likely covered by manual testing.
2. **Backend restart reuses stale ports.** After Steve stops and the state machine re-triggers `start()`, the module-level port variables are reused without re-probing.
3. **Both default ports occupied simultaneously.** The `Promise.all` path where both probes fall back is untested. Low risk — the OS assigns distinct ephemeral ports.

### Documentation Assessment

No external documentation references port 9443 or 9080 directly. The mock backend (`backend/mock.ts:192`) still hardcodes `desiredPort = 9443`, but this is a test-only default with no production impact.

Support or troubleshooting guides that tell users to connect directly to Steve on 9443 would need updating. Consider logging the resolved ports (see Important Issue #1) so users can discover them.

### Gemini 3.1 Pro

### Executive Summary

This PR successfully introduces dynamic port assignment for the Steve process, falling back to OS-assigned ephemeral ports when defaults (9443, 9080) are occupied. However, it introduces a critical time-of-check to time-of-use (TOCTOU) race condition because ports are queried and released at app startup but not actually bound by Steve until minutes later when Kubernetes finishes starting. The PR should be reworked to reserve the ports by holding the sockets open until Steve is spawned, or by resolving the ports dynamically immediately before Steve starts.

### Critical Issues

1. **Time-of-Check to Time-of-Use (TOCTOU) port hijacking** — `background.ts:236-241` (critical, regression)

```typescript
    [steveHttpsPort, steveHttpPort] = await Promise.all([
      findAvailablePort(DEFAULT_STEVE_HTTPS_PORT),
      findAvailablePort(DEFAULT_STEVE_HTTP_PORT),
    ]);

    DashboardServer.getInstance().init(steveHttpsPort);
```

The application checks for available ports during the initial application startup phase (`Electron.app.whenReady()`). Inside `findAvailablePort` (in `pkg/rancher-desktop/utils/networks.ts`), the ports are bound and then *immediately* closed. However, Steve is not spawned until much later—specifically, inside `newK8sManager()` when the `mgr.on('state-changed')` event reaches `STARTED`.

Because Kubernetes can take several minutes to start, the selected ephemeral ports are completely free during this window. Any other application or system process could claim these ports in the meantime, which would cause the Steve process to crash when it finally tries to bind them. The comment `In practice this is negligible because the caller (steve) binds immediately after launch` is incorrect.

**Fix**:
Reserve the ports by holding the Node.js `net.Server` instances open instead of closing them immediately. Update `findAvailablePort` to return the `net.Server` objects. Then, in `newK8sManager` (right before `Steve.getInstance().start()` is called), close those Node.js servers to free the ports and immediately spawn the Steve child process. (Note: this would require DashboardServer to support dynamic proxy target updates rather than initializing it with a hardcoded port at startup).

### Important Issues

2. **Concurrent port resolution may result in identical ports** — `background.ts:236-239` (important, regression)

```typescript
    [steveHttpsPort, steveHttpPort] = await Promise.all([
      findAvailablePort(DEFAULT_STEVE_HTTPS_PORT),
      findAvailablePort(DEFAULT_STEVE_HTTP_PORT),
    ]);
```

If both default ports (9443 and 9080) are in use, `findAvailablePort` concurrently creates two fallback servers binding to port `0`. While the Node.js event loop *usually* binds these synchronously enough to get different ephemeral ports, the concurrent `Promise.all` execution mixed with the immediate `fallback.close()` creates a subtle race condition. If one socket closes before the second asynchronous binding resolves, the OS could theoretically assign the exact same ephemeral port to both HTTP and HTTPS, causing Steve to crash.

**Fix**:
Resolve the ports sequentially to avoid overlapping socket state transitions:
```typescript
    steveHttpsPort = await findAvailablePort(DEFAULT_STEVE_HTTPS_PORT);
    steveHttpPort = await findAvailablePort(DEFAULT_STEVE_HTTP_PORT);
```

3. **`findAvailablePort` fails to resolve the correct port when `preferredPort = 0`** — `pkg/rancher-desktop/utils/networks.ts:33-35` (important, gap)

```typescript
    server.listen(preferredPort, '127.0.0.1', () => {
      server.close(() => resolve(preferredPort));
    });
```

If a developer uses this utility function and explicitly passes `0` to request an OS-assigned free port (e.g., `findAvailablePort(0)`), the function will successfully bind a random ephemeral port. However, it will resolve `0` instead of the actual assigned port, breaking the contract of the function.

**Fix**:
Always retrieve and resolve the actual bound port from the server address:
```typescript
    server.listen(preferredPort, '127.0.0.1', () => {
      const addr = server.address() as net.AddressInfo;
      server.close(() => resolve(addr.port));
    });
```

### Suggestions

4. **DashboardServer initialization appends duplicate middleware if restarted** — `pkg/rancher-desktop/main/dashboardServer/index.ts:77-80` (suggestion, gap)

```typescript
    ProxyKeys.forEach((key) => {
      this.dashboardServer.use(key, this.proxies[key]);
    });
```

Although `init(stevePort)` returns early if the HTTP server is currently running, if `DashboardServer.stop()` is ever called (which only calls `this.dashboardApp.close()`), a subsequent call to `init()` will re-append the proxy routes and static file handlers to the `this.dashboardServer` Express app. Over time, this duplicates routes in the middleware stack.

**Fix**:
Separate the Express application setup from the HTTP server listening lifecycle. Apply the `.use()` middlewares only once (e.g., in a constructor or behind a boolean flag), or utilize `http-proxy-middleware`'s dynamic `router` option so you don't need to rebuild the proxy middlewares if the Steve port changes.

### Testing Assessment

1. **Missing test for `preferredPort = 0`**: Ensure `findAvailablePort(0)` returns a valid port number > 0.
2. **Missing E2E/Integration coverage for port fallback**: While `networks.spec.ts` covers the standalone utility logic, there are no integration tests ensuring that Dashboard proxy paths successfully route to the dynamically assigned Steve port when the default port (9443) is blocked by another service.

### Documentation Assessment

The PR does not document whether the dynamically assigned Steve ports are surfaced to the user. If Steve defaults to dynamic ports, users attempting to connect CLI tools directly to Steve's API might be confused when it's not on 9443. Consider logging the final assigned ports clearly, writing them to a known state file, or displaying them in the Dashboard settings.
