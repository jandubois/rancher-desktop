## Important Issues

I1. **R5-I3 fix introduces regression: `exposeNew` engine-delegation Remove can tear down the only host-switch proxy** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:226-250` [Codex GPT 5.5]

```go
if len(pst.tracker.Get(syntheticID)) > 0 {
    if err := pst.tracker.Remove(syntheticID); err != nil {
        // ...
    }
}
// ...
pst.added[port] = portState{bindings: bindings, delegated: true}
```

When procnet's partial Add succeeded at gvisor's `Expose(Local=127.0.0.1:8080)` and `wsl-proxy.Send` failed (rare under stability-gate bypass), the engine's later `portTracker.Add` for the same `127.0.0.1` publish gets "proxy already running" from gvisor; the events handler (`containerd/events_linux.go:124-127`, `docker/events.go:101-103`) logs once and does not retry. The new `Remove(syntheticID)` then tears down the only bound proxy; Windows-side traffic to `127.0.0.1:8080` drops until RD restart. The comment at lines 237-241 assumes the engine "will create its own proxy" after Remove, but that holds only when the events handler has not yet attempted Add.

For `HostIP=0.0.0.0` publishes, `Local=0.0.0.0:8080` is a distinct gvisor key and the regression does not manifest. Blast radius is the `127.0.0.1`-publish form under heavy-load stability-gate bypass plus a `wsl-proxy.Send` transient.

Fix: add bounded retry to the engine's events handler on the "proxy already running" condition, or document this third trade-off path and pin the chosen behavior with a regression test.

## Design Observations

### Concerns

- *(future)* **`APITracker` exposes no idempotent re-notify hook for wsl-proxy.Send failures** [Gemini 3.1 Pro]. When `wslProxyForwarder.Send` fails during `Add`, `portStorage` is populated and the gvisor proxy is bound, but the Windows-side proxy state never gets the mapping. Procnet's `handleAlreadyExposed` resume path (lines 323-336) installs the iptables rule but cannot re-trigger the WSL-proxy notification — the only recovery is a full Remove/Add cycle that risks unbinding a working proxy. The author already documents this trade-off at lines 75-77 of `port_state_tracker_linux.go`. An idempotent `Tracker.RetryWSLProxy(containerID)` would let callers fully recover from transient wsl-proxy failures. Out of scope for this PR; flag for the next time `APITracker` is touched.

- *(future)* **Tracker ownership is still inferred from chain presence, not a shared event source** [Codex GPT 5.5]. I1 is symptomatic of using `engineChainManagesPort` as a proxy for "engine owns the host-switch proxy." A small explicit handoff contract in `tracker` — either an event the engine emits when its `portTracker.Add` lands, or an ownership transfer API — would remove several branch-specific race decisions from procnet. The state-machine extraction this PR completes is the right precondition for that next step.

### Strengths

- *(in-scope)* **Each R5-I finding has a directly named regression test** [Claude Opus 4.7, Codex GPT 5.5]. `TestEngineDelegationReconcilesLeftoverProcnetRule` (I1), `TestRemoveReportedProxyDestroyed` (I2's classification helper), `TestEngineDelegationReleasesPriorPartialAdd` (I3). The test names embed the finding ID and the scenario, so a future maintainer reading the test surface can map every test back to a documented failure mode.

- *(in-scope)* **State-machine doc comment at lines 30-114 enumerates every transition with its trade-offs** [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]. The "Append retry" and "Listener disappears" subsections explicitly name the order of side effects (Delete before Remove, drop marker before iptables Delete) and the rationale. The new I1/I3 reconciliation steps are added inline with a "Defer on probe failure — the iptables state is not cleared by any defer, so a next-tick retry recovers" — the kind of explanation that prevents the next contributor from "simplifying" the early return.

- *(in-scope)* **`removeReportedProxyDestroyed` centralizes a previously triplicated classification** [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]. `retireDisappeared`, `retryAppend`, and `exposeNew` all consume the same predicate. The chain list is no longer buried in a function literal; a future contributor adding a third engine sees the var declaration and the doc comment that says exactly what to do. ## Acknowledged Limitations

- **Stability gate bypassed under heavy load** — `port_state_tracker_linux.go:62-73`. The two-scan gate filters the OCI hook reservation socket under normal load; a slow tick interval can let the socket survive across two ticks. Documented. [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]
- **In-session Delete-path leak** — `scanner_linux.go:289-313` and `port_state_tracker_linux.go:251-274`. A transient Delete probe failure can leak a loopback rule for the rest of the session; the cross-session counterpart is the I1 scenario that the new if-managed reconciliation now addresses. Documented. [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]
- **`retryAppend` engine-takeover and `exposeNew` engine-delegation accept a rare host-switch proxy leak** — `port_state_tracker_linux.go:226-250, 380-393`. When `tracker.Remove` returns a non-wsl-proxy error, the proxy may still be bound; the code logs and marks the port `delegated` anyway because `APITracker.Remove` clears `portStorage` via a leading defer (`apitracker.go:155`), so a retry would silently no-op. Documented at lines 380-387; codex's I1 above proposes extending this comment to the third trade-off path. [Claude Opus 4.7, Codex GPT 5.5]
- **`handleAlreadyExposed` delegated branch deliberately retains any leftover loopback rule** — `port_state_tracker_linux.go:338-352`. The rule's origin is ambiguous in this branch; the I1 reconciliation in `exposeNew`'s if-managed branch covers the common cross-session case. [Codex GPT 5.5]
- **`dnatChainContainsPort` does not match port-range syntax** — `scanner_linux.go:246-263`. Theoretical for the shipped engines (nerdctl-CNI and Docker emit per-port rules) but real for hand-rolled iptables chains. Documented. [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]
- **Design depends on iptables-nft DNAT-in-PREROUTING terminating the chain** — `scanner_linux.go:155-169`. A future WSL2 kernel shipping iptables-legacy defaults would silently invalidate the design's foundation. The doc comment proposes a `iptables-save -t nat | grep '^:PREROUTING ACCEPT'` probe in the bats setup; not added in this PR. [Claude Opus 4.7]
- **Fake `iptables` runner does not model `preroutingHasLoopbackRule` short-circuit** — `port_state_tracker_linux_test.go:94-101`. R5-S2 suggested fixing the fake; the author chose to document the divergence and instruct "Add the probe to the fake before writing such a test." [Claude Opus 4.7]
- **`TestPortAlreadyExposedSubstringIsReachable` pins the wrap chain but cannot guard an upstream gvisor-tap-vsock rename of `proxy already running`** — `scanner_linux_test.go`. Documented in the test's comment. [Codex GPT 5.5]
- **No idempotent wsl-proxy re-notify hook in `APITracker`** — `port_state_tracker_linux.go:75-77`. After a partial Add, the WSL-side proxy state is permanently inconsistent until the next full Remove/Add cycle. Out of scope for this PR; promoted to Design Observation above. [Gemini 3.1 Pro]

## Review Process Notes

### Skill improvements

- **For each finding agents claim to have fixed in a prior round, verify the fix is complete against the original surface.** A fix that resolves the narrowest case can leave a residual gap at the original severity. The verification step is: (a) identify the surface the original finding named (a code path, an error class, a callsite set), (b) read the new code at every cited point, (c) walk the original trigger sequence forward through the new code, and (d) note whether each step still hits a recovery path. When the fix is narrower than the surface, raise the residual gap at the original severity unless an in-code comment documents the reduced scope.

- **When a fix introduces new code that runs unconditionally (a new `Remove`, `Delete`, or `Reset`), enumerate the timing windows during which that operation could destroy state another component depends on.** The fix that resolves a leak by adding a cleanup call is one of the most common ways to introduce a new bug: the cleanup runs when the other side has not yet (or has already failed to) create its own state. Reviewers should ask, for each new cleanup call: "What if the operation that would replace this state has already attempted and failed silently?" If the consumer (engine event handler, downstream service, retry loop) has no retry, the cleanup may strand the resource.

None — the existing `deep-review-context.md` bullets on `APITracker`'s partial-failure shape and on the asymmetric ID schemes (engine `container.ID` vs procnet `syntheticIDFor`) were both load-bearing for this round's findings. Keep them.

S1. **Missing WSL proxy retry hook** — `src/go/guestagent/pkg/tracker/apitracker.go:123-146` (suggestion, gap)

When `wslProxyForwarder.Send` fails during `Add`, `APITracker` returns an error but leaves `portStorage` populated.
