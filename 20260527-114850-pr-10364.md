## Executive Summary

The PR introduces a `runningDistros` getter (`wsl --list --running --quiet`) and uses it to filter `sync()` and `getStateForIntegration()` so stopped distros are not started just to query them. The fix is well-scoped, but it leaves three gaps: enabled-but-stopped distros are still started during initial sync when the keepalive proxy cannot run yet (I1), the `verifyAllDistrosKubeConfig` diagnostic still execs into stopped distros (I2), and `listIntegrations` runs `wsl --list --running` once per distro instead of once per call (I3).

## Important Issues

I1. **Enabled stopped distros can still be started without a keepalive proxy** — `pkg/rancher-desktop/integrations/windowsIntegrationManager.ts:202-208` [Codex GPT 5.5] (important, gap)

```ts
.filter(distro => runningDistros.has(distro.name) ||
  this.settings.WSL?.integrations?.[distro.name] === true)
.map(distro => this.syncDistro(distro.name, kubeconfigPath)),
```

The filter treats `settings.WSL.integrations[distro] === true` as enough to include a stopped distro, but that is only safe when the keepalive proxy can start. `syncDistroSocketProxy` (line 403) refuses to start the proxy whenever `dockerSocketProxyReason` is set (lines 383-393: quitting, not enforcing, `!backendReady`, non-Moby). The other `syncDistro*` helpers still call `execCommand({ distro }, ...)`, which runs `wsl --distribution X --exec ...` and starts the distro. With no proxy open, WSL idle-terminates ~30s later and reproduces #10049.

`background.ts:291` calls `integrationManager.enforce()` — which calls `sync()` — *before* `startBackend()` at line 321 sets `backendReady = true`. A user with a stopped, integration-enabled distro hits the binfmt clearing on every cold start.

Fix:

```diff
 const runningDistros = await this.runningDistros;
+const canKeepStoppedDistroAlive = !this.dockerSocketProxyReason;
 ...
-    .filter(distro => runningDistros.has(distro.name) || this.settings.WSL?.integrations?.[distro.name] === true)
+    .filter(distro => runningDistros.has(distro.name) ||
+      (canKeepStoppedDistroAlive && this.settings.WSL?.integrations?.[distro.name] === true))
```

A later `k8s-check-state` event fires another sync once the backend is ready; the enabled stopped distro then gets included and the proxy starts to hold it open.

I2. **`verifyAllDistrosKubeConfig` still execs into stopped distros** — `pkg/rancher-desktop/integrations/windowsIntegrationManager.ts:519-548` [Claude Opus 4.7] (important, gap)

```ts
async verifyAllDistrosKubeConfig() {
  const distros = await this.supportedDistros;

  await Promise.all(distros.map(async(distro) => {
    await this.verifyDistroKubeConfig(distro.name);
  }));
}
```

`verifyDistroKubeConfig` runs `execCommand({ distro }, wslHelper, 'kubeconfig', '--verify')` (line 535) — the same `wsl --exec`-against-foreign-distro pattern that triggers the binfmt teardown. It is invoked by the `VERIFY_WSL_INTEGRATION_KUBECONFIG` diagnostic checker (`pkg/rancher-desktop/main/diagnostics/kubeConfigSymlink.ts:12`), wired up in `pkg/rancher-desktop/main/diagnostics/diagnostics.ts:61`. Whenever diagnostics run, every supported stopped distro is started and idle-terminates, reproducing #10049 outside the integration-sync path. The PR's stated goal is "skip stopped WSL distros," but this sibling call site is left unguarded.

Fix:

```diff

   const distros = await this.supportedDistros;
+  const running = await this.runningDistros;
-  await Promise.all(distros.map(async(distro) => {
-    await this.verifyDistroKubeConfig(distro.name);
-  }));
+  await Promise.all(distros
+    .filter(distro => running.has(distro.name))
+    .map(distro => this.verifyDistroKubeConfig(distro.name)));
 }
```

I3. **`listIntegrations` issues N parallel `wsl --list --running` invocations** — `pkg/rancher-desktop/integrations/windowsIntegrationManager.ts:683-707` [Claude Opus 4.7, Gemini 3.1 Pro] (important, gap)

```ts
async listIntegrations(): Promise<Record<string, boolean | string>> {
  const distros = await this.nonBlacklistedDistros;
  const states = distros.map(d => (async() => [d.name, await this.getStateForIntegration(d)] as const)());
  return Object.fromEntries(await Promise.all(states));
}
```

The new getter `runningDistros` (line 654) has no memoization — every property access spawns a fresh `wsl --list --running --quiet` process. `getStateForIntegration` awaits it once per call (line 703), and `listIntegrations` invokes `getStateForIntegration` in parallel for every non-blacklisted distro. On a host with N distros that is N concurrent `wsl.exe` processes for a single state refresh. `sync()` then calls `listIntegrations` from its `finally` block (line 213), so a single sync incurs N+1 invocations.

Gemini rated this Critical; the impact is wasted CPU and an avoidable race on slow hosts rather than a correctness break, so Important is the right level.

Fix: hoist the query into the caller and pass the snapshot down.

```diff
 async listIntegrations(): Promise<Record<string, boolean | string>> {
-  const distros = await this.nonBlacklistedDistros;
-  const states  = distros.map(d => (async() => [d.name, await this.getStateForIntegration(d)] as const)());
+  const distros = await this.nonBlacklistedDistros;
+  const running = await this.runningDistros;
+  const states  = distros.map(d => (async() => [d.name, await this.getStateForIntegration(d, running)] as const)());
   return Object.fromEntries(await Promise.all(states));
 }
```

Accept the set as an optional second argument to `getStateForIntegration`; `sync()` can pass the same snapshot through to its own `listIntegrations` call.

## Design Observations

**Concerns**

- **Property getters that execute subprocesses** (in-scope) [Gemini 3.1 Pro, Claude Opus 4.7]: `runningDistros` (and the pre-existing `nonBlacklistedDistros`) are property getters that fork `wsl.exe`. Property access reads as cheap and side-effect-free, but each `await this.runningDistros` spawns a process and there is no memoization. The N+1-invocation pattern flagged in I3 is the direct consequence. Refactoring to explicit `async getRunningDistros()` (or a memoize-per-sync wrapper) would make the cost visible at call sites and remove an entire class of accidental fan-out bugs.
- **One of three exec-into-distro call sites left unguarded** (in-scope) [Claude Opus 4.7]: The PR fixes `sync()` and `getStateForIntegration()` but not `verifyAllDistrosKubeConfig` (I2). Closing the whole class in this PR is a few lines and keeps the fix coherent.
- **Race between the running-distros check and the exec** (future): After `runningDistros` resolves, a distro can idle-terminate before `syncDistro` reaches it. The window is small but non-zero. A retry-or-skip path when the subsequent `wsl --exec` discovers the distro stopped would close it.
- **The "trust settings for stopped distros" path is silent** (future) [Claude Opus 4.7]: A `console.debug` breadcrumb would help operators distinguish "marker said X" from "we did not read the marker because the distro was stopped" when triaging integration logs.

**Strengths**

- `wsl --list --running --quiet` is the right primitive — observes state without creating a session in the target distro [Codex GPT 5.5].
- `runningDistros` returns an empty set on error rather than throwing, which keeps the legacy "consider all distros" behavior when the listing itself fails. Defaulting in the safe direction is right (see S5 for the related observability gap) [Claude Opus 4.7].
- The new tests mock by argument matching (`if (args.includes('--running'))`) instead of re-mocking the whole module, which preserves the existing `nonBlacklistedDistros`/`supportedDistros` tests unchanged [Claude Opus 4.7].

### [Gemini 3.1 Pro]

### Executive Summary
This PR successfully mitigates a WSL bug by checking `wsl --list --running` before querying or syncing state, ensuring stopped distros are not accidentally started. However, the implementation introduces a critical concurrency issue where the running distros check is executed multiple times in parallel, which can overwhelm the WSL subsystem.

### Findings

#### Critical Issues

C1. **Concurrent `wsl.exe` executions in `listIntegrations`** — `pkg/rancher-desktop/integrations/windowsIntegrationManager.ts:686` (critical, regression)

```typescript
    const states = distros.map(d => (async() => [d.name, await this.getStateForIntegration(d)] as const)());
```

In `listIntegrations`, the `states` array maps over all `nonBlacklistedDistros` concurrently. Inside `getStateForIntegration` (line 701), the code awaits `this.runningDistros`. Because `runningDistros` is a property getter that returns a new `Promise` containing an I/O subprocess (lines 655-657), this loop spawns a simultaneous `wsl.exe --list --running --quiet` process for every supported distro. Spawning concurrent WSL commands often causes timeouts or process failures, which then trigger the error fallback.

Fix: Change `runningDistros` to an explicit async method. Fetch the running distros once in `listIntegrations` before the loop, and pass the resolved set as an argument to `getStateForIntegration(distro, runningDistros)`.

#### Important Issues

I1. **Opaque error fallback skips integration teardown** — `pkg/rancher-desktop/integrations/windowsIntegrationManager.ts:664-668` (important, gap)

```typescript
      } catch (error) {
        console.error(`Error listing running distros: ${ error }`);

        return new Set<string>();
      }
```

If `captureCommand` fails, it catches the error and silently returns an empty set. In `sync()` (lines 205-207), distros are filtered out if they are not in this set and their UI setting is `false`. If the user just unchecked a running distro, `settings` is `false`, and the empty set makes `has(name)` `false`. The distro is skipped, meaning `syncDistroSocketProxy(distro, false)` is never called, leaking the proxy process indefinitely.

Fix: While returning an empty set prevents accidentally starting distros during an error state, silently dropping teardowns creates zombie proxies. Cache the last known running state as a fallback instead of an empty set, or explicitly log which teardowns were skipped.

#### Suggestions

None.

### Design Observations

**Concerns**

- **Property getters executing subprocesses (in-scope)**: `runningDistros` (like the pre-existing `nonBlacklistedDistros`) is defined as a property getter (`get runningDistros()`) that executes a long-running I/O subprocess. This breaks the expectation that property access is cheap and side-effect-free. It obscures the performance cost at the call site and invites accidental N+1 execution bugs. Refactor these into explicit async methods (e.g., `async getRunningDistros()`) to clearly signal their cost.

### Testing Assessment

1. **Concurrent API access:** `listIntegrations` concurrency is not tested. A test should verify that `wsl --list --running` is only invoked once during a full state refresh.
2. **Error fallback teardown:** The scenario where `runningDistros` fails while a distro has integration disabled is missing from the test matrix.
