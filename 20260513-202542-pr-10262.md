## Executive Summary

Round 2's resolution code closed three Important findings (engine-managed-claim, Append retry asymmetry, partial-failure leak) but the new partial-failure plumbing introduces three new race windows. **I1** is the load-bearing finding â€” independently raised by all three agents â€” and it negates the PR's stated guard #3: when `wslProxyForwarder.Send` fails transiently the first time, the resume-ownership branch marks the port `delegated:false, appendFailed:false` and `continue`s, after which the `alreadyAdded` fast-path consumes every subsequent tick and neither the iptables Append nor the wsl-proxy retry ever fires. The inline comment at line 234 and the PR description both claim "the wsl-proxy retry will come on the next loop iteration"; the code at lines 182-196 only honors `appendFailed`. The bug is the exact failure mode procnet's PREROUTING DNAT exists to handle: a `--network=host` loopback listener stays unreachable from the Windows host for the listener's lifetime.

**I2** is the symmetric gap in the Append retry path: when xtables-lock contention causes the first Append to fail, the retry at line 187 calls `execLoopbackIPtablesRule` again without re-probing the engine chain. `preroutingHasLoopbackRule` only checks for procnet's own exact rule; it does not look at `CNI-HOSTPORT-DNAT` or `DOCKER`. If the engine's events handler installed the chain rule between the failed tick and the retry, the retry installs the rogue PREROUTING DNAT after the authoritative engine rule exists â€” recreating the original shadow-DNAT bug. **I3** is the cleanup-side deadlock: `tracker.Remove` clears `portStorage` via a leading `defer` regardless of whether the unexpose/wsl-proxy calls succeeded, so the PR's "retain `added` entry to retry the iptables Delete" pattern silently no-ops after the first tick, and if the listener reappears while the port is still in `added`, the Add loop's `alreadyAdded` fast-path locks the port out of all recovery.

The architectural recommendation converges across agents: merge this PR with I1-I3 fixed, then plan an explicit-marker (nonce) approach for the next release. The heuristic stack has reached the point where every round closes one race window and opens another in the new state-machine surface; a nonce eliminates the inference layer by construction but introduces coordination cost with nerdctl/CNI/docker that is bigger than this PR's scope. None of the three agents recommend blocking this PR pending a nonce design.

## Important Issues

I1. **Resume-ownership branch silently drops the iptables Append and never retries `wsl-proxy.Send`** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:238-243` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro] (important, regression)

```go

    log.Debugf("/proc/net scanner port %s already added by procnet; resuming ownership after partial-failure retry", port)
    added[port] = portState{bindings: bindings, delegated: false}
    delete(addErrorLogged, port)
    continue
}
```

Trace for a `--network=host` container with a `127.0.0.1:<port>` listener when `wsl-proxy` is momentarily unavailable:

1. **Tick N (second sighting after the stability gate):** `apiForwarder.Expose` succeeds. `portStorage.add` runs at `tracker/apitracker.go:124`. `wslProxyForwarder.Send` returns an error at `apitracker.go:130-133`. `tracker.Add` returns `"sending port mappings to wsl proxy error: â€¦"`. This is *not* `portAlreadyExposedSubstring`, so procnet falls into `addErrorLogged` at line 249, logs once, and `continue`s. The port is **not** added to `added` and the iptables rule is **not** installed.
2. **Tick N+1:** Listener still present, `seenLastScan` still has it. `tracker.Add` calls `apiForwarder.Expose` again, which returns `"proxy already running"` because gvisor-tap-vsock holds the proxy from tick N. The function returns `ErrExposeAPI` wrapping the `"proxy already running"` substring. `portStorage.add` is **not** called this tick and `wslProxyForwarder.Send` is **not** called this tick.
3. procnet sees the substring at line 224, calls `tracker.Get(syntheticID)`, which returns the still-populated entry from tick N (the `defer a.portStorage.remove` fires only on `Remove`, not on `Add`) and takes the resume-ownership branch at lines 238-243: sets `added[port] = portState{delegated: false, appendFailed: false}` and `continue`s.
4. **Tick N+2 and forever after:** `added[port]` exists with `appendFailed=false`. Line 182's `alreadyAdded` branch fires. Line 195's `continue` is the only outcome.

The inline comment at line 234 and the PR description both claim "the wsl-proxy retry will come on the next loop iteration." Lines 182-196 show no such retry â€” the `alreadyAdded` branch only retries the iptables Append, gated on `state.appendFailed`. The resume-ownership branch sets `appendFailed=false`, so nothing fires.

For the lifetime of this listener:

- The PREROUTING DNAT loopback rule is **never** installed.
- `wslProxyForwarder.Send` is **never** re-attempted.

This is the exact failure mode procnet's PREROUTING DNAT exists to solve. External traffic to `127.0.0.1:<port>` at the host's eth0 stays unreachable until the container exits and restarts, or RD restarts.

I2. **Append-retry path can install the rogue rule after the engine chain appears** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:182-188` [Codex GPT 5.5, Claude Opus 4.7] (important, regression)

```go
if state, alreadyAdded := added[port]; alreadyAdded {
    if state.appendFailed {
        if err := p.execLoopbackIPtablesRule(state.bindings, port, Append); err != nil {
            log.Debugf("/proc/net scanner retry of iptables Append for %s still failing: %s", port, err)
```

The first-add path probes engine ownership at line 211 before calling `tracker.Add`. The retry path at line 187 calls `execLoopbackIPtablesRule(..., Append)` without repeating that probe. `execLoopbackIPtablesRule` checks `preroutingHasLoopbackRule` for procnet's own exact rule at lines 379-391; it does not query `CNI-HOSTPORT-DNAT` or `DOCKER`.

The race: procnet observes the transient listener twice, the engine chain is absent at line 211, `tracker.Add` succeeds, the initial append at line 259 fails because CNI/Docker is holding the xtables lock, and by the next scan the engine chain exists. The retry at line 187 installs the rogue PREROUTING DNAT after the authoritative engine rule exists, recreating the shadowing failure this PR fixes.

The window requires three independent transient conditions to align, but the consequence is the exact bug-class the PR exists to close.

Fix: either re-probe `engineManagesLoopbackBinding` in the retry branch, or (Codex's preferred form) roll back the synthetic tracker entry on the first append failure and let the next tick re-run the normal engine probe:

```diff
appendErr := p.execLoopbackIPtablesRule(bindings, port, Append)
if appendErr != nil {
    log.Errorf("/proc/net scanner creating loopback iptable rules for portbinding: %v failed: %s", bindings, appendErr)
+    if removeErr := p.tracker.Remove(syntheticID); removeErr != nil {
+        log.Errorf("/proc/net scanner failed to roll back port after iptables append failure: %s", removeErr)
+        // Retain cleanup state that retries tracker.Remove, not Append.
+        added[port] = portState{bindings: bindings}
+    }
+    continue
}
```

The rollback variant has a smaller blast radius than the re-probe variant; it converts the bug into an extra round-trip of `tracker.Add` (acceptable cost) and inherits the engine-chain re-probe from the normal Add path.

I3. **Cleanup-side retain pattern deadlocks on listener reappearance and silently no-ops on retry** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:288-302` and `tracker/apitracker.go:150-152` [Gemini 3.1 Pro] (important, gap)

The "retain `added` entry to retry the iptables Delete on the next tick" comment promises a recovery path that does not exist. Two interacting problems:

1. **APITracker.Remove wipes portStorage via a leading `defer`** â€” `src/go/guestagent/pkg/tracker/apitracker.go:150-152`:

   ```go
   func (a *APITracker) Remove(containerID string) error {
       portMap := a.portStorage.get(containerID)
       defer a.portStorage.remove(containerID)
       ...
   ```

   The defer runs whether or not the unexpose API or wsl-proxy call succeeded. After the first failed Remove, portStorage is empty. On the next tick procnet calls `tracker.Remove(syntheticID)` again; `portMap := a.portStorage.get(containerID)` returns an empty map at line 151; the for loop runs zero times; `len(portMap) != 0` is false at line 181 so wsl-proxy.Send is not called; the function returns `nil` at line 197. Procnet sees success and the iptables Delete proceeds â€” but the wsl-proxy mapping is leaked.

2. **Listener reappearance deadlocks the entry.** While the port is retained in `added` (because the iptables Delete failed), the Add loop's `alreadyAdded` fast-path at line 182-196 fires for every subsequent sighting. `state.appendFailed` is false (the failure was on the Delete path, not Append), so line 195 `continue`s. The container's reborn listener gets no `tracker.Add`, no engine-chain probe, and no PREROUTING rule until RD restart.

The combined trigger (transient iptables Delete failure plus container restart inside the same tick window) is narrow, but both failure modes are real. The simpler half-fix â€” Gemini's recommendation â€” is to delete from `added` immediately after the successful `tracker.Remove` and accept that an un-deleted PREROUTING rule self-cleans on RD restart (the existing comment already acknowledges this):

```diff

     log.Errorf("/proc/net scanner failed to remove port: %s", err)
     continue
 }
+delete(added, port)
 if err := p.execLoopbackIPtablesRule(state.bindings, port, Delete); err != nil {
     log.Errorf("/proc/net scanner deleting loopback iptable rules for portbinding: %v failed: %s", state.bindings, err)
-    continue
 }
-delete(added, port)
```

The proper fix is to move `defer a.portStorage.remove(containerID)` in `APITracker.Remove` to run only after both `apiForwarder.Unexpose` and `wslProxyForwarder.Send` succeed â€” but that touches a file outside this PR's diff.

## Suggestions

S1. **Cleanup-path errors are not log-once** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:288-300` [Claude Opus 4.7] (suggestion, gap)

```go

    log.Errorf("/proc/net scanner failed to remove port: %s", err)
    continue
}
if err := p.execLoopbackIPtablesRule(state.bindings, port, Delete); err != nil {
    log.Errorf("/proc/net scanner deleting loopback iptable rules for portbinding: %v failed: %s", state.bindings, err)
    continue

```

The Add-side `addErrorLogged` map (introduced in Round 2's S4 resolution) suppresses a 3-second retry-storm log flood when host-switch or wsl-proxy stays down. The Delete-side retry path is symmetric in retry behavior but asymmetric in log behavior: a port whose listener has disappeared and whose iptables Delete keeps failing logs at Errorf every 3 seconds until RD restarts. Apply the same first-error-Error / subsequent-error-Debug treatment. Note this finding is moot if I3's `delete(added, port)` simplification lands â€” the retry path goes away.

S2. **Engine-chain probe forks `iptables --list` twice per port per tick** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:431-451` [Claude Opus 4.7, Gemini 3.1 Pro] (suggestion, enhancement)

```go
for _, chain := range []string{"CNI-HOSTPORT-DNAT", "DOCKER"} {
    cmd := exec.CommandContext(ctx, "iptables", "--wait", "2", "--table", "nat", "--list", chain, "--numeric")
```

Every newly-stable port triggers two `iptables --list` shell-outs. A container publishing ten ports forks twenty `iptables` invocations per scan. Chain contents change at most once per container start/stop, so a per-tick cache (probe both chains once at the top of the add loop, reuse the output for every port observed in that tick) cuts fork count by 2Ă— ports_observed. Gemini's related observation: even a single port with multiple loopback bindings shells out per-binding inside `engineManagesLoopbackBinding`. Worth doing if scaling becomes a concern.

S3. **`addErrorLogged` does not suppress engine-probe spam for persistently-failing ports** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:211-214` [Gemini 3.1 Pro] (suggestion, gap)

```go

if err != nil {
    log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
    continue
}
```

When `tracker.Add` keeps failing (host-switch down), `addErrorLogged` suppresses Errorf log lines but the port never lands in `added`, so every tick re-runs the engine-chain probe (two `iptables --list` shell-outs). Fifty failed ports generate one hundred shell-outs every 3 seconds. The fix lives in the same shape as Round 2's S4 resolution: extend `portState` with an `engineProbeAttempted` bookkeeping field or a parallel sidecar map that lets the engine probe ride the same suppression as the add-error log.

S4. **Reconcile inline comment with code after I1's fix** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:231-237` and PR description [Claude Opus 4.7] (suggestion, regression)

## Design Observations

### Concerns

**O1. The `alreadyAdded` fast-path is a chokepoint for all future-tick recovery (in-scope)** [Claude Opus 4.7, Codex GPT 5.5]. Lines 182-196 are the single integration point for retries from previous ticks, but the branch only acts when `state.appendFailed` is set. Anything else that wants future-tick attention (wsl-proxy retry, engine-chain re-probe, partial-failure cleanup) has to invent a new state flag and a new retry branch. I1, I2, and I3 are all instances of one half of the state machine completing and the next guard reading a different world. Folding `appendFailed`, `delegated`, and any future flags into a single `state action` enum (or extracting `portStateTracker` with an injectable `tracker.Tracker` â€” the author's own suggestion at line 99-101) would make the transitions explicit and unit-testable.

**O2. The state machine is documented in prose but tested only end-to-end via BATS (future)** [Claude Opus 4.7]. The `portState` comment (lines 61-101) describes a non-trivial state machine, none of whose transitions is unit-tested. Promoting the `portStateTracker` extraction to a follow-up issue is worth it: regressions like I1, I2, and I3 are exactly what a state-machine unit test would catch on the first review pass instead of the third.

### Strengths

- The two-scan stability gate is a small, well-documented heuristic with one acknowledged failure mode (one-tick delay for genuine listeners). Cheap and effective.
- `isIptablesRuleAbsent` cleanly separates "rule doesn't exist" (an answer) from "iptables errored" (defer). Right shape for a retry-aware caller, table-driven test pins the stderr classifier across the known exit modes.
- `dnatChainContainsPort` is a pure function with 23 well-chosen fixtures â€” IP-octet, cross-protocol, prefix/suffix, intentional range-form miss. The tests pin behavior so any future range-matching change must be deliberate.
- `TestPortAlreadyExposedSubstringIsReachable` pins the substring contract across the `verifyResponseBody â†’ ErrExposeAPI` wrap chain. The right way to defend against silent upstream breakage in gvisor-tap-vsock.
- The Append idempotency probe via `preroutingHasLoopbackRule` and the symmetric Delete-side retain-on-error are good defenses against duplicate rules and rule leaks â€” even though I3 shows the Delete retain is not load-bearing for the failure mode it documents.
- The explicit `delegated` flag corrects Round 1's safe-by-accident cleanup behavior. Future refactors can see that engine-owned ports must not go through procnet cleanup.

## Testing Assessment

The pure helpers (`dnatChainContainsPort`, `isIptablesRuleAbsent`) and the substring contract are well-covered. The state machine in `ForwardPorts` has zero unit-level coverage; every transition is exercised only by `published-ports.bats` end-to-end. Untested scenarios, ranked by risk:

1. **Partial-failure resume path (I1):** Tick 1 returns wsl-proxy.Send error â†’ Tick 2 returns "proxy already running" â†’ assert iptables Append is present.
2. **Append-retry post engine-chain (I2):** Tick 1: tracker.Add succeeds, Append fails â†’ Tick 2: engine chain now has rule â†’ assert procnet does *not* install PREROUTING DNAT.
3. **Cleanup deadlock (I3):** Listener disappears â†’ Delete fails â†’ listener reappears â†’ assert procnet re-adds.
4. **Two-scan stability gate:** First sighting defers; second sighting acts; one-tick listener never reaches `tracker.Add`.
5. **Engine-chain delegation:** `engineManagesLoopbackBinding` returns true â†’ neither `tracker.Add` nor `execLoopbackIPtablesRule` is called; cleanup drops the local marker on listener disappearance.
6. **addErrorLogged log-once classification:** First transient `tracker.Add` failure logs Error; subsequent failures log Debug; clearing on success or listener disappearance.

A `portStateTracker` extraction with an injectable `tracker.Tracker` (mocked) is the natural fixture for items 1-3 and 5-6.

### Executive Summary

This PR hardens the `procnet` scanner to prevent it from creating rogue PREROUTING DNAT rules that shadow container engine rules, causing connection hangs. The strategy correctly delegates engine-managed ports and introduces a 2-scan stability gate. However, the state machine introduced for partial failures and cleanup retries contains two critical logic bugs where ports can become permanently deadlocked (unforwarded and disconnected). 

### Findings

**Critical Issues**

C1. **Partial-failure recovery skips Append and wslProxy.Send** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:244-250`
(critical, regression)

When `tracker.Add` hits the "proxy already running" partial-failure state, `procnet` resumes ownership by setting `added[port] = portState{..., delegated: false}` and immediately calling `continue`. 

This causes two severe problems:
1. `continue` skips the `execLoopbackIPtablesRule(Append)` execution, so the PREROUTING rule is never created.
2. On all subsequent ticks, the "Add ports" loop hits the `alreadyAdded` fast-path. Because `state.appendFailed` is false (since Append was never attempted), it `continue`s again. `tracker.Add` is never reissued, so the transient `wslProxyForwarder.Send` failure that caused the original error is never recovered. The port is permanently wedged.

Fix: To clear the downstream error, the entry must be completely removed from the tracker to allow a fresh, complete `Add` on the next tick.

```diff
 						if len(p.tracker.Get(syntheticID)) > 0 {
-							log.Debugf("/proc/net scanner port %s already added by procnet; resuming ownership after partial-failure retry", port)
-							added[port] = portState{bindings: bindings, delegated: false}
-							delete(addErrorLogged, port)
+							log.Debugf("/proc/net scanner port %s already added by procnet but failed downstream; cleaning up for full retry", port)
+							_ = p.tracker.Remove(syntheticID)
 							continue
 						}
```

C2. **Zombie state on transient Delete failure** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:285-296`
(critical, regression)

To retry transient iptables errors during cleanup, the "Remove ports" loop retains the port in `added` when `execLoopbackIPtablesRule(Delete)` fails (by bypassing `delete(added, port)`).

If the listener reappears (e.g., the container process restarts) while the port is still stuck in `added`, it creates a deadlock:
- The "Add ports" loop skips the port because it is already in `added`.
- The "Remove ports" loop skips the port because it is present in `newPortMap`.

Because `tracker.Remove` already succeeded before the iptables failure, the wsl-proxy forwarding is gone, but it can never be re-added.

Fix: Do not overload `added` to queue iptables retries, as its presence blocks the Add path. Delete the port from `added` immediately after a successful `tracker.Remove`. (The comment acknowledges that an un-deleted iptables rule is harmless and self-cleans on restart).

**Important Issues**

I1. **APITracker.Remove silently succeeds on retry, defeating PR's error handling** â€” `src/go/guestagent/pkg/tracker/apitracker.go`
(important, gap)

The PR relies on `p.tracker.Remove` returning an error to keep a port in `added` for retry. However, `APITracker.Remove` calls `defer a.portStorage.remove(containerID)` at the very beginning of its function. If the unexpose API or wsl-proxy call fails, it returns an error, but the port is already wiped from memory.

On the next tick, when `procnet` retries `tracker.Remove`, `a.portStorage.get` returns an empty map, and `Remove` silently returns `nil` (success) without actually retrying the downstream unexpose calls. This leaks the `wsl-proxy` mapping.

Fix: Move `a.portStorage.remove(containerID)` to the end of `APITracker.Remove`, executing it only after the `apiForwarder` and `wslProxyForwarder` calls succeed. *(Note: This resides in an out-of-scope file found via tracing, but heavily degrades the reliability of the PR's cleanup logic).*

**Suggestions**

S1. **Redundant engine chain checks for identical ports** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:321-334`
(suggestion, enhancement)

The `engineManagesLoopbackBinding` function loops over all bindings in the `nat.PortMap` slice. Because the map groups bindings by `nat.Port`, `binding.HostPort` is identical to `port.Port()` for every element in the slice. If the first iptables probe returns `false, nil`, subsequent loop iterations will needlessly invoke the exact same `iptables --list` shell commands.

Fix: Move the `engineChainManagesPort` call outside the loop, executing it exactly once per `nat.Port` if at least one loopback binding exists.

S2. **Work-spam on persistent tracker.Add failures** â€” `src/go/guestagent/pkg/procnet/scanner_linux.go:250`
(suggestion, enhancement)

The `addErrorLogged` map successfully suppresses log spam when a port consistently fails `tracker.Add` (e.g., if the host proxy is down). However, because the port is never placed into `added`, every 3-second tick treats it as newly discovered. This forces `engineManagesLoopbackBinding` to shell out to `iptables` twice per port, every tick. 50 failed ports will generate 100 shell-outs every 3 seconds, wasting CPU and creating xtables lock contention.

Fix: If background load from `iptables` under proxy-failure conditions is a concern, consider tracking `addFailed: true` inside `portState` to skip the engine-chain probe on subsequent retries.

### Architectural Recommendation â€” Nonce vs. Heuristic

1. **Specific failure modes this stack still carries:**
   - **Partial-failure race gaps:** If `wslProxy.Send` fails but `Expose` succeeds, `procnet` cannot distinguish an engine's authoritative proxy from its own broken proxy without relying on fragile internal state mapping (`len(p.tracker.Get) > 0`).
   - **Shadow-DNAT race window:** The 3-second stability gate is a probabilistic barrier. If system load stalls the engine for >3 seconds between creating the reservation socket and inserting the iptables rule, `procnet` will still win the race and shadow the authoritative rule.
   - **Brittle text parsing:** Parsing `iptables --list` output via regex will silently fail (re-introducing the shadow-DNAT bug) if upstream engines change chain names, alter comment formats, or use IP port range syntax (`dpts:LO:HI`).

2. **Failure modes a nonce scheme would introduce or shift:**
   - **Deployment and dependency complexity:** It requires patching shipped engines (nerdctl/CNI plugins) or injecting a custom OCI hook to emit the nonce. Upstream drift could easily break the injection.
   - **Fallback behavior:** If the nonce mechanism fails (e.g., the hook fails to run or the host path isn't mounted), `procnet` falls back to blindly exposing the port, recreating the shadow-DNAT bug.
   - **Engine compatibility:** Standard `docker` users (running outside the managed Rancher Desktop lifecycle) would not trigger the hook, leading to inconsistent behavior compared to bundled engines.

3. **Weighing refinement against a nonce:**
   - Heuristic refinement has reached diminishing returns. Every layer added to patch a race condition (stability gates, iptables parsing, retry states) introduces its own state-machine edge cases (like the zombie state and skipped appends found in this review).
   - A nonce scheme is fundamentally deterministic: a file in `/var/run/rancher-desktop/ports/` mapped by container ID immediately communicates intent before iptables rules or CNI networks are even provisioned. It would allow deleting `engineChainManagesPort`, the 3-second stability delay, and the complex `added` state machine entirely.

4. **Recommendation: Merge this PR but plan the nonce for the next release.**
   - The current heuristics, once the bugs identified in this review are fixed, provide a "good enough" mitigation for the immediate user pain. Blocking this PR to design a cross-component nonce leaves the known shadow-DNAT bug active in production. However, further refinement of this scanner is a dead end. The next release should replace the scanner's heuristic guesswork with a deterministic signal (a side-channel file or direct engine API query).
