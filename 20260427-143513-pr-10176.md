## Executive Summary

The PR pins per-artifact sha256 digests next to each dependency's
version in `dependencies.yaml`, replacing live upstream-checksum lookups
during `postinstall` with a static map populated at bump time. The
schema change to a compound `{ version, checksums }` per entry is
clean, and the typed `sha256:<hex>` strings plus single
`lookupChecksum`/`downloadAndHash` helpers give the new code a
small, comprehensible surface. As a strict improvement, several
previously-unverified downloads (DockerBuildx-darwin, ECRCredHelper,
WasmShims, CertManager, SpinOperator, Moproxy, WSLDistro, Wix,
MobyOpenAPISpec) now have a verified digest at install time.

The two important findings are both ergonomic rather than security
gaps. **I1** points out that pinning a checksum on a URL fetched from
the mutable `master` branch (`MobyOpenAPISpec`) converts pre-existing
silent upstream drift into a hard `postinstall` failure — an
appropriate failure mode, but one that wants an immutable source ref
to keep clean installs working between bumps. **I2** points out that
the bump-time `verify` mismatch error reports `expected sha256:undefined,
got <hex>` when the upstream sidecar parser misses a line, which
misleads the reader into thinking the artifact is corrupt. Suggestions
S1–S6 cover code-health issues (parallel dependency lists, parser
strictness, hashFile duplication, lost YAML comments, hardcoded
`macos-15`, AlpineLimaISO write-path divergence); none affect
correctness today.

I1. **Pinned checksum is applied to a mutable Moby source URL** — `scripts/dependencies/moby-openapi.ts:29-36` [Codex GPT 5.5]

The install path now verifies the OpenAPI spec at line 36, but the
URL is still built from `master` at line 29. The bump path records
the digest from the same mutable branch at line 88, so if Moby edits
or regenerates `api/docs/v1.54.yaml` on `master` after this PR lands,
`postinstall` fetches the new bytes and compares them against the old
digest from `dependencies.yaml`, failing every clean install until
another dependency update is merged. The mutable URL pre-dates this
PR — at the merge-base, `download(url, outPath, { access: ... })`
ran without checksum verification, so drift was silent. The new
verification surfaces drift loudly, which is the right direction;
the right *fix* is an immutable source reference.

Fix: resolve the latest commit that touched `api/docs/v${version}.yaml`
during `getChecksums()`, store that commit alongside the API version
in the manifest, and fetch from that commit in `download()`:

```ts

  commit:     string;

const baseUrl = `https://raw.githubusercontent.com/${ this.githubOwner }/${ this.githubRepo }/${ version.commit }/api/docs`;
const fileName = `v${ version.apiVersion }.yaml`;
```

The updater would call GitHub's commits-for-path API and write
`{ apiVersion, commit }` plus the checksum into `dependencies.yaml`.
A release tag is even simpler if there is a reliable mapping from
Docker API version to Moby tag.

## Suggestions

S1. **`populate-dependency-checksums.ts` and `rddepman.ts` keep parallel dependency lists** — `scripts/populate-dependency-checksums.ts:31-58`, `scripts/rddepman.ts:32-59` [Claude Opus 4.7, Gemini 3.1 Pro]

```ts
// scripts/populate-dependency-checksums.ts:30-31
// Every global dependency whose version and checksums live in
// `dependencies.yaml`.  Keep in sync with `scripts/rddepman.ts`.
const globalDependencies: VersionedDependency[] = [
  new tools.KuberlrAndKubectl(),
  ...
];
```

The same 25 entries appear in `rddepman.ts:32-59`. The "keep in sync"
comment is the only thing that prevents drift. If a future PR adds a
dependency only to `rddepman.ts`, the populate script silently leaves
its checksums map empty when re-run, then the next `postinstall`
blows up with `No checksum recorded for X` only after the dep
actually ships.

Fix: extract `globalDependencies` to a single shared module that both
scripts import. Extension entries do not apply (`Extension.getChecksums`
returns `{}`); only the GlobalDependency list matters.

S2. **`fetchUpstreamChecksums` is silently strict about sidecar format** — `scripts/lib/dependencies.ts:244-261` [Claude Opus 4.7]

```ts
for (const line of body.split(/\r?\n/)) {
  const match = /^([0-9a-f]+)\s+\*?(.+?)\s*$/.exec(line);

  if (match) {
    const [, hex, fullName] = match;
    ...
  }
}
```

Two narrow assumptions:

1. **Hex must be lowercase.** The previous `findChecksum` used
   `split(/\s+/)`, accepting any case. A single uppercase digit
   anywhere in the upstream file (e.g. a future move to
   `sha256sum --uppercase`) silently drops the line.
2. **Only GNU `sha*sum` format.** BSD lines, `SHA256 (file) = hex`,
   never match. None of today's upstreams use BSD format, but a
   switch by any one of them produces an empty map and I2 takes over.

Fix: match hex case-insensitively (`[0-9a-fA-F]`, then `.toLowerCase()`
before storing), add a second pattern for BSD-style lines, or at
minimum log a warning when the file parses to zero entries.

S3. **`AlpineLimaISO.updateManifest` re-reads `dependencies.yaml` from disk while the base uses the in-memory cache** — `scripts/dependencies/lima.ts:246-258`, `scripts/lib/dependencies.ts:410-423` [Claude Opus 4.7]


```ts

  const manifest = structuredClone(await GlobalDependency.manifest());  // cache
```

Within today's `rddepman` flow these produce the same file (each
iteration is preceded by `git checkout main`, so the disk and the
cache agree). But if a future caller mutates the cache or runs two
updates between checkouts, the two paths diverge silently — disk-fresh
loses the in-memory edits, cached loses the on-disk edits. The base
override could be parametric in `Version` and the alpineLimaISO
subclass could just call `super.updateManifest(newVersion as any,
newChecksums)`; the only reason for the override today is the
type-narrowing throw.

Fix: drop the `if (name === 'alpineLimaISO')` throw in the base and
let TypeScript carry the version type through, so the override
disappears. Alternatively, mutate the cached manifest the same way the
base does, so both code paths share write semantics.

S4. **Migration of comments in `dependencies.yaml`** — `scripts/lib/dependencies.ts:143-153` [Claude Opus 4.7]

```ts

  const raw: Record<string, RawEntry> = {};
  for (const name of Object.keys(manifest.versions) as (keyof DependencyVersions)[]) {
```

`YAML.parse` plus `YAML.stringify` strips inline comments. The current
file has none, so this is fine today; pre-existing behaviour. Worth
one sentence in a comment so a future contributor does not add
documentation to `dependencies.yaml` and watch it disappear at the
next bump PR.

S5. **Hardcoded `macos-15` platform string in two places** — `scripts/dependencies/lima.ts:37,70` [Claude Opus 4.7]


Pre-existing in `download()`; the new `getChecksums()` clones the

Pulling the value out into a `const MACOS_RUNNER = 'macos-15'` (or

S6. **`hashFile` and `getChecksumForFile` are near-duplicates** — `scripts/lib/dependencies.ts:223-235`, `scripts/lib/download.ts:131-140` [Claude Opus 4.7]

```ts
// dependencies.ts (new)
async function hashFile(filePath: string, algorithm: 'sha256' | 'sha512'): Promise<string> {
  const hash = crypto.createHash(algorithm);
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('error', reject);
    hash.on('finish', () => resolve());
    stream.pipe(hash);
  });
  return hash.digest('hex');
}
```

```ts
// download.ts (existing)
async function getChecksumForFile(inputPath: string, checksumAlgorithm: ChecksumAlgorithm = 'sha256'): Promise<string> {
  const hash = crypto.createHash(checksumAlgorithm);
  await new Promise((resolve) => {
    hash.on('finish', resolve);
    fs.createReadStream(inputPath).pipe(hash);
  });
  return hash.digest('hex');
}
```

The new copy is slightly better — it propagates read errors via
`stream.on('error', reject)` instead of swallowing them. Worth
promoting one to the public surface and deleting the other so future
callers do not pick the weaker version.

Fix: export `hashFile` from `download.ts` (or move to a shared
`lib/hash.ts`), and have `download()`'s internal verification and
`downloadAndHash()` both use it.

## Design Observations

### Concerns

**D1. Why split `versions` and `checksums` into two parallel structures?** (in-scope) [Claude Opus 4.7, Gemini 3.1 Pro]

@mook-as flagged this on the PR
([discussion_r3149342472](https://github.com/rancher-sandbox/rancher-desktop/pull/10176#discussion_r3149342472)).
The on-disk format is unified per dependency
(`{ version, checksums }`), but the in-memory model is
`DownloadContext.versions[name]` and `DownloadContext.checksums[name]`
carried separately, with `lookupChecksum(context, name, artifact)`
straddling them. A single shape — for example
`DownloadContext.dependencies[name] = { version, checksums }` — maps
1:1 to disk, eliminates the partial
`Partial<DependencyVersions>` / `Partial<DependencyChecksums>`
reconstruction in `readDependencyManifest`, and shortens the abstract
method to `getEntry()`. Most consumers today read just `versions`, so
this trades some call-site verbosity for less indirection. Worth at
least replying to @mook-as before merge; if the split is intentional,
the response justifies leaving it.

**D2. Mutable inputs need a first-class model** (future) [Codex GPT 5.5]

Most dependencies are release artifacts, but `mobyOpenAPISpec` is a
source file from a branch. Treating both as `{ version, checksums }`
hides the extra identity needed for mutable branch content. A proper
model — for example a `commit` field in the version record for
branch-backed dependencies — removes I1's failure mode and makes
future similar dependencies explicit instead of surprising.

### Strengths

**D3. Typed `Sha256Checksum = \`sha256:${ string }\`` template-literal** [Claude Opus 4.7]

The template-literal type combined with `parseSha256Checksum`
validating at the only ingestion point (`readDependencyManifest`) is a
clean type-safe pattern: the prefix self-documents the algorithm in
the YAML, and the `slice('sha256:'.length)` in `lookupChecksum` gives
`download()` the raw hex it expects without ceremony.

**D4. Computing checksums before `git checkout -b` in rddepman** [Claude Opus 4.7, Gemini 3.1 Pro]

`rddepman.ts:318-323` computes fresh checksums *before* creating the
branch. An upstream mismatch aborts before any branch is created or
PR opened; no half-mutated git state.

**D5. Uniform sha256 install-time path** [Gemini 3.1 Pro]

Translating diverse upstream digests (sha512 sidecars and others)
into unified sha256 checksums in `dependencies.yaml` simplifies the
install-time retrieval layer. Install-time validation needs one
consistent mechanism rather than a matrix of upstream-specific hashing
rules.

## Testing Assessment

No unit tests accompany the new code. Cheap, high-leverage scenarios:

1. **`fetchUpstreamChecksums` parsing** — round-trip a sample sidecar
   (GNU `sha256sum` with binary marker, GNU two-space format,
   possibly BSD style if S2 is addressed) and assert the resulting
   map. This is the single point where parser bugs silently degrade
   verification (I2).
2. **`parseSha256Checksum`** — accept/reject the boundary cases in
   its regex (uppercase, wrong prefix, wrong length, non-hex). Trivial
   to write and locks in the contract.
3. **`readDependencyManifest` / `writeDependencyManifest` round-trip** —
   confirm key order and `alpineLimaISO`'s nested `version` survive a
   load/save cycle.
4. **`lookupChecksum` error** — the "Available: …" message is
   operator-facing; assert it lists the keys when the artifact is
   missing.

Integration-level: `populate-dependency-checksums.ts` is the only
end-to-end exerciser today, and it requires network access. Acceptable
given the scope, but the unit pieces above are cheap.

## Review Process Notes

**Repo context updates**

- `[repo]` Add to `deep-review-context.md`: When a PR adds checksum
  verification or any other failure-on-mismatch check to a download
  path, check whether the source URL is immutable (a release tag, a
  specific commit SHA, or a content-addressed reference). Pinning a
  digest on a mutable URL (`master` branch, `latest` tag, a path on
  the upstream's main branch) converts silent drift into a hard build
  failure between bumps. Flag any new `expectedChecksum` paired with
  a URL that names a moving ref, and suggest recording the resolved
  immutable reference (commit SHA, release tag) alongside the digest
  in the manifest.

  *Why this generalises:* the rancher-desktop dependency-bump model
  records the resolved version in YAML, so the missing piece is just
  the resolution step at bump time. The pattern applies equally to
  any future dependency that pulls from `raw.githubusercontent.com/.../master/...`,
  charts at "latest", or unsigned mirror URLs.

**Skill improvements**

- Add to the review dimensions a sub-bullet under
  "Security / supply chain": *When a PR adds digest pinning to a
  download path, verify that the URL the digest is pinned against is
  immutable. A digest on a mutable URL produces silent CI failures
  between bumps and indicates an incomplete fix.* This is the
  underlying principle behind I1; the prompt currently asks reviewers
  to think about checksum coverage but not about reference
  immutability.

  *Why this generalises:* the same anti-pattern shows up in
  Dockerfiles (`FROM image:latest`, `apt-get install` of unpinned
  versions), CI-pipeline `wget` calls, vendored mirror URLs, and
  package-manager lock files that omit hashes. The principle is
  "checksum + mutable ref = false sense of security."

## Design Observations

**Concerns**

D1. **Why split `versions` and `checksums` into two parallel structures?** (in-scope) — @mook-as flagged this on the PR (`scripts/lib/dependencies.ts:19`). The on-disk format is unified per dependency, but the in-memory model is `DownloadContext.versions[name]` and `DownloadContext.checksums[name]` carried separately, with `lookupChecksum(context, name, artifact)` straddling them. A single shape — e.g. `DownloadContext.dependencies[name] = { version, checksums }` — would map 1:1 to disk, eliminate the partial `Partial<DependencyVersions>` / `Partial<DependencyChecksums>` reconstruction in `readDependencyManifest`, and shorten the abstract method to `getEntry()`. Most consumers today read just `versions`, so this trades some call-site verbosity for less indirection. Worth at least replying to @mook-as before merge; if the split is intentional, the response justifies leaving it.

**Strengths**

D2. The `Sha256Checksum = \`sha256:${ string }\`` template-literal type combined with `parseSha256Checksum` validating at the only ingestion point (`readDependencyManifest`) is a clean type-safe pattern: the prefix self-documents the algorithm in the YAML, and the `slice('sha256:'.length)` in `lookupChecksum` gives `download()` the raw hex it expects without ceremony.

D3. Computing checksums **before** `git checkout -b` in `rddepman.ts:318-323` is the right place to fail — an upstream mismatch aborts before any branch is created or PR opened.
