// portStateTracker drives the procnet per-port state machine across
// scan ticks. State (added, seenLastScan, addErrorLogged,
// probeErrorLogged) outlives a single Tick call. The struct is not
// safe for concurrent use; callers must serialize Tick.

go vet's containedctx rule (and Go style guidance) prefers passing context.Context per call instead of storing it on a struct: the lifetime of the runner is tied to one specific request/loop. Here the runner lives only for the lifetime of ForwardPorts, so the practical risk is small, but the pattern invites future code to reuse the runner with a stale context. The iptablesRunner interface methods don't currently take a ctx, so adding one threads through every fake.

Fix: extend the interface methods to accept ctx context.Context as the first parameter; drop the struct field. Tests and callers update mechanically. If the noise isn't worth the change at this point in the cycle, a //nolint:containedctx with the rationale comment closes the audit trail.

S3. engineDNATChains is a mutable package var Claude¶
src/go/guestagent/pkg/procnet/scanner_linux.go

// loopbackIP is the host-local address procnet redirects --network=host
// loopback listeners to.

// engineDNATChains is the closed list of container-engine PREROUTING
// chains procnet probes for an existing DNAT to a host port. Adding a
// third engine (e.g. a future containerd-native portmap implementation)
// requires extending this list, or engineChainManagesPort silently

// original shadow-DNAT bug. NewProcNetScanner logs the active list at
// startup so a support-bundle reader sees what procnet considered.
var engineDNATChains = []string{"CNI-HOSTPORT-DNAT", "DOCKER"}

// another component has already exposed the port (typically the
// containerd or docker events handler on /tasks/start). The string
// originates in the /services/forwarder/expose API response and survives

Go does not allow slice constants, so a var is the only option for a multi-element list. The docstring treats this as a closed list and NewProcNetScanner logs the contents at startup as authoritative, but any code in the same package can still mutate it at runtime (e.g., a future test setup using engineDNATChains[0] = "..." would silently affect the rest of the test binary). The risk is low; no current code touches it.

Fix: replace the var with a function engineDNATChainList() []string { return []string{"CNI-HOSTPORT-DNAT", "DOCKER"} } that returns a fresh slice each call, or inline the literal at both call sites. The startup log can fmt.Sprintf from the same literal. Skip the change if the testability of "swap in a fake chain list" is anticipated.

The [^\n]*? clause encodes the invariant "each rule occupies one line." Real iptables --list <chain> --numeric output respects this contract — every rule lives on its own line. The risk is theoretical, but a future iptables version, an unexpected piped paginator, or a malformed exec wrapper that collapses lines would cause one tcp match to consume a rule boundary. Acknowledged Limitations ¶

  • Engine-delegation tracker.Remove tears down the gvisor proxy under a Local-string collision — port_state_tracker_linux.go:240-263 documents the rare regression: when procnet's partial Add bound the proxy first and the engine's later Add lost the race (logged once, didn't retry), procnet's later Remove on engine delegation tears down the only proxy and Windows-side traffic drops until RD restart. The trigger requires stability-gate bypass under heavy load PLUS transient wsl-proxy.Send failure PLUS a HostIP=127.0.0.1 publish (or non-admin install). Documented as an intentional trade-off, with TestEngineDelegationAcceptsLeakWhenRemoveFails locking the behavior.
  • Stability gate bypass under heavy load — the two-scan gate filters the OCI hook reservation socket on normal load only. A heavily-loaded system that takes >3s between procnet polls could see the reservation socket twice and act on it. This is the underlying limitation the retryAppend/engine-takeover paths ride along with.
  • In-session Delete leak — port_state_tracker_linux.go:282-285,305-307 document that the Delete path has no retry, so a transient iptables-Delete failure leaks a procnet rule until RD restart. The I1 reconciliation in exposeNew's if-managed branch self-heals this whenever the engine later republishes the same host port.
  • iptables-nft kernel default assumption — scanner_linux.go:159-169 documents that the entire design depends on iptables-nft's PREROUTING chain-terminating-on-DNAT behavior, with no live assertion of the assumption.
  • engineDNATChains closed list — scanner_linux.go:55-62 and engineChainManagesPort docstring at :227-231 document that adding a third engine without extending the list silently re-introduces the shadow-DNAT bug. NewProcNetScanner logs the active list at startup for diagnosability.
  • Port-range regex gap — scanner_linux.go:269-279 documents that dpts:LO:HI and multiport-embedded ranges are intentionally not matched. RD's shipped engines emit per-port rules, so this is theoretical for shipped engines but real for hand-rolled iptables setups against the same chains.
  • APITracker.Remove wsl-proxy.Send error is not retried — implicit in handleAlreadyExposed's RESUME branch (no idempotent re-notify hook). Documented at port_state_tracker_linux.go:73-77.
  • Engine chain visibility ≠ engine has a host-switch proxy — port_state_tracker_linux.go:93-101 documents that the CNI chain is installed before the engine's portTracker.Add, so a chain-present probe does not prove the engine owns a proxy. Acknowledged in the retryAppend engine-takeover comment.

Review Process Notes ¶

Repo context updates

  • [repo] Repo-wide: when reviewing the procnet state machine, verify symmetry on every state transition's reverse direction. A finding that state X never transitions back to Y when guard G stops holding is a load-bearing architectural shape for this code: prior rounds caught the "engine chain appears mid-Add" direction (R4 design observation), and round 7 caught the reverse "engine chain disappears while delegated" (Codex I1). Reviewers should enumerate, for each state's entry condition, the lifecycle of the input that triggered it — if the input can later flip without the listener disappearing, ask where the re-evaluation happens. The current code re-evaluates appendFailed but not delegated.
  • [repo] Repo-wide: contained-context warnings in src/go/guestagent/ are worth surfacing. The realIptablesRunner.ctx field is the second instance this PR has touched of "struct holds Context for the lifetime of one outer call"; if the project's lint config enables containedctx, the runner pattern will be flagged. The convention guidance is: pass ctx per call where the receiver may outlive a single request; store on the struct only when the receiver's lifetime is intentionally bound to one context (and document it).

Skill improvements

  • Standing rule for re-probing-state design reviews: when a state-machine type holds a flag like delegated, pending, or owned-by-X whose truth depends on an external source the state machine doesn't observe directly (an iptables chain rule, a remote service, a peer process's tracker entry), enumerate each tick's entry path and check whether the flag is re-evaluated against the current external state. A flag that is only set on entry and only cleared on a coarser disappearance event (the parent record being deleted) is structurally vulnerable to stale-belief bugs when the external source's truth flips while the parent record stays. The fix is usually a re-probe on every tick for ports already in that state — symmetric with whatever retry path the design already provides for transient failures. Rationale: state machines that delegate ownership to a peer need a way to take ownership back when the peer disappears; the absence of this path is the canonical "stale delegation" bug class.
