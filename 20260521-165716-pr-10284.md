## Executive Summary

PR adds Vendor, Description, and More information columns to the Installed Extensions table, sourced from existing OCI labels. The default sort falls back to the hidden image ID rather than the displayed title, three new column labels skip the codebase's i18n convention, and the URL regex misclassifies explicitly qualified Docker Hub IDs (`docker.io/...`) and registry-with-port IDs (`localhost:5000/...`). The same URL-fallback logic already lives in `MarketplaceCard.vue`; the PR copies it instead of extracting a shared helper.

Reviewer @mook-as also asked whether `MarketplaceCard.vue` could be adapted for reuse here; the author moved helpers from a separate `metadata.ts` onto `installed.vue` but kept the broader question open. All three reviewers raised the duplication as a design concern.

## Suggestions

S1. **URL regex misclassifies explicit Docker Hub and registry-with-port IDs** — `pkg/rancher-desktop/pages/extensions/installed.vue:127-131` [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro] (suggestion, gap)

```ts
if (!/^[^./]+\//.test(ext.id)) {
  return `https://${ ext.id }`;
}
return `https://hub.docker.com/extensions/${ ext.id }`;
```

The regex treats any first segment containing a dot as a host and any first segment without one as a Docker Hub publisher. Two real cases break:

1. **Explicit Docker Hub prefixes.** `background.ts:1580` passes the raw image to `getExtension()` (`pkg/rancher-desktop/main/extensions/manager.ts:346-358`), so a CLI install of `docker.io/docker/logs-explorer-extension` keeps the prefix. The regex misses it (dot in `docker.io`) and the bare-host branch returns `https://docker.io/...`, which is not Docker Hub's UI.

2. **Registry-with-port.** `localhost:5000/myext` matches the regex because `:` is allowed; the row links to `https://hub.docker.com/extensions/localhost:5000/...` — guaranteed 404.

The same logic lives pre-existing in `MarketplaceCard.vue:130-134`; fix once in the shared helper proposed in D1.

Fix:

```diff
-      if (!/^[^./]+\//.test(ext.id)) {
+      const dockerHubID = ext.id.replace(/^(?:docker\.io|index\.docker\.io)\//, '');
+      if (dockerHubID === ext.id && !/^[^./:]+\//.test(ext.id)) {
         return `https://${ ext.id }`;
       }
-      return `https://hub.docker.com/extensions/${ ext.id }`;
+      return `https://hub.docker.com/extensions/${ dockerHubID }`;
```

S2. **Extension-supplied `more-info` URL bound to `:href` without scheme allow-list** — `pkg/rancher-desktop/pages/extensions/installed.vue:120-125,223-234` [Claude Opus 4.7, Gemini 3.1 Pro] (suggestion, gap)

```ts
extensionLink(ext: ExtensionMetadataSource): string {
  const preferredURL = this.extensionLabel(ext, LABEL_MORE_INFO);

  if (preferredURL) {
    return preferredURL;
  }
```

`row.moreInfo` comes from the extension image's `io.rancherdesktop.extension.more-info` label, fully controlled by whoever publishes the image. Electron's `setWindowOpenHandler` (`pkg/rancher-desktop/window/index.ts:86-94`) intercepts `target="_blank"` clicks and forwards to `shell.openExternal`, which drops most `javascript:` URIs but happily opens `data:`, `file:`, `vscode:`, `mailto:`, and similar schemes via the OS handler. The pre-existing pattern in `MarketplaceCard.vue:22` was tolerable for a curated catalog; the new code applies it to arbitrary installed images.

Fix: validate the scheme before returning:

```ts
try {
  const url = new URL(preferredURL);
  if (url.protocol === 'http:' || url.protocol === 'https:') {
    return preferredURL;
  }
} catch { /* fall through */ }
```

S3. **Test exercises the `methods` bag directly, not via a Vue instance** — `pkg/rancher-desktop/pages/extensions/__tests__/installed.spec.ts:20-21` [Claude Opus 4.7, Gemini 3.1 Pro] (suggestion, gap)

```ts
const { default: InstalledExtensions } = await import('@pkg/pages/extensions/installed.vue');
const methods = (InstalledExtensions as any).methods;

Mount the component with `@vue/test-utils` and assert against `vm.installedExtensionRows` or the rendered cell text — that surface is what users see and the current spec does not cover it.

Fix: replace the methods-bag access with `shallowMount(InstalledExtensions, { ... })` and assert on `vm.installedExtensionRows`.

## Design Observations

### Concerns

**D1. Extract a shared `extensionMetadata` helper for catalog and installed views (in-scope)** — [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

`MarketplaceCard.vue` (catalog) and `installed.vue` (installed) now both consume the same OCI label bag and apply the same fallback rules: title with `id`/`slug` fallback, "more info" URL with Docker Hub fallback, label-trim semantics. Today both files reimplement the URL logic, `MarketplaceCard.vue` reads `extension.labels[...]` directly without the safe-trim wrapper, and only `installed.vue` extracts vendor and description. A small `pkg/rancher-desktop/utils/extensionMetadata.ts` exporting `extensionTitle`, `extensionVendor`, `extensionDescription`, `extensionMoreInfoURL` would: (a) collapse the duplication to a one-line import on each side; (b) make S3's mount-style test redundant — the helpers become pure functions, directly testable; (c) leave a single place to enforce S1's regex and S2's scheme allow-list. This matches @mook-as's review suggestion to make `MarketplaceCard.vue` reusable, while leaving each view in charge of its own rendering.

### Strengths

- Sort key arrays (`['vendor', 'title', 'id']`, etc.) fall through to the next key when the primary is empty, giving stable ordering even when many extensions lack metadata.
- `?.trim() || fallback` treats whitespace-only labels as missing — more forgiving than the pre-existing `?? fallback` (which fell back only on `undefined`).
- `installedExtensionRows` is a computed property, so derived fields update reactively when `installedExtensions` changes.

## Testing Assessment

Untested scenarios, ranked by risk:

1. **No template-level assertion that the rendered table contains the expected cell text.** A regression that drops `description` from the row object or breaks the slot mapping would not fail any current assertion.
2. **`installedExtensionRows` computed property is not exercised.** The spec touches the helpers but not the data shape consumed by `SortableTable`.
3. **Default sort behavior (I1).** The spec would not have caught the displayed-name / sort-by-id mismatch.
4. **URL fallback for `docker.io/...` and `localhost:5000/...` (S1).** No coverage of either.
5. **Whitespace-only labels.** `?.trim() || fallback` is exercised only with the all-empty case, not with `'   '` strings that prove trim is doing work.
6. **Mixed label sets.** An extension that has `vendor` but no `description` is not covered; only "all labels" and "no labels" cases exist.
7. **Sort key fall-through.** Two extensions with empty `vendor` ordering by `title` is not asserted; this is the kind of behavior that subtly breaks when refactoring the sort spec.
8. **Method dispatch via Vue (`this` = component instance, S3).** The helpers happen to be `this`-light today; a future refactor that pulls a value off `this.$store` will break in production but pass the test.

**Skill improvements:**

- [ ] When a Vue/React table component renders a column whose `name` field differs from the column it's sorted by, flag the mismatch as a UX regression. The principle is general — any framework with table abstractions has the same trap when display and sort fields diverge.

**Repo context updates:**

- [repo] **SortableTable default-sort fallback (UPDATE)**. `pkg/rancher-desktop/components/SortableTable/sorting.js:66-86` picks the default sort column in this order: explicit `defaultSort: true`, then a header named `name`, then the first non-state header, then `id`. Why: tables with metadata columns that don't match the underlying record key (e.g., `title` vs `id`) are vulnerable; the bug is invisible until someone sorts the data and notices the mismatch.

### I1. **Duplicated `extensionLink` logic forks Docker Hub fallback into two files** — `pkg/rancher-desktop/pages/extensions/installed.vue:120-132`, `pkg/rancher-desktop/components/MarketplaceCard.vue:120-135` (important, regression)

```ts
extensionLink(ext: ExtensionMetadataSource): string {
  const preferredURL = this.extensionLabel(ext, LABEL_MORE_INFO);

  if (preferredURL) {
    return preferredURL;
  }

  if (!/^[^./]+\//.test(ext.id)) {
    return `https://${ ext.id }`;
  }

  return `https://hub.docker.com/extensions/${ ext.id }`;
},
```

`MarketplaceCard.vue:120-135` contains the same three-branch fallback (preferred label, bare-host fallback, Docker Hub fallback) keyed on `slug` instead of `id`. Per `store/extensions.ts:30`, "`id` … also known as 'slug'", so these are conceptually the same value. @mook-as raised this in review ("adapt `MarketplaceCard.vue` to make it usable here"); the response moved helpers from `metadata.ts` into `installed.vue` but kept the duplicate. Any future change to the URL policy (new label name, hub.docker.com → marketplace.docker.com, scheme validation) must now land in two files and risks divergence.

Fix: extract a shared helper, e.g. `pkg/rancher-desktop/utils/extensionMetadata.ts`, and call it from both components.

```ts
// utils/extensionMetadata.ts
export function extensionMoreInfoURL(slug: string, labels?: Record<string, string>): string {
  const preferred = labels?.['io.rancherdesktop.extension.more-info']?.trim();
  if (preferred) return preferred;
  if (!/^[^./]+\//.test(slug)) return `https://${ slug }`;
  return `https://hub.docker.com/extensions/${ slug }`;
}
```

  target="_blank"
  rel="noopener noreferrer"
>
```

`row.moreInfo` is derived from the extension image's `io.rancherdesktop.extension.more-info` label, which is fully controlled by whoever publishes the image. Electron's `setWindowOpenHandler` (`pkg/rancher-desktop/window/index.ts:86-94`) routes `target="_blank"` clicks through `shell.openExternal`, which deflects `javascript:` URIs but happily opens `file://`, `vscode://`, `mailto:`, and similar schemes via the OS handler. Pre-existing in `MarketplaceCard.vue:22`; the PR replicates the pattern.

Fix: validate the parsed URL's protocol before returning it from `extensionLink`:

```ts
try {
  const url = new URL(preferredURL);
  if (url.protocol === 'http:' || url.protocol === 'https:') {
    return preferredURL;
  }
} catch { /* fall through */ }
```

### S4. **Test exercises method internals via `(InstalledExtensions as any).methods`, not via Vue reactivity** — `pkg/rancher-desktop/pages/extensions/__tests__/installed.spec.ts:20-21` (suggestion, gap)

```ts

The spec calls helpers directly off the options object, so `this` is the methods bag rather than a Vue instance. A `shallowMount` of the component would let you assert against `installedExtensionRows` and the rendered template, which are the actual user-visible behavior — neither is covered.

Fix: prefer mounting the component (using `@vue/test-utils`) and reading `vm.installedExtensionRows`; assert that the rendered table contains the expected cell text.

### S6. **Empty `vendor`/`description` cells show a literal `-` glyph instead of an em-dash or accessible placeholder** — `pkg/rancher-desktop/pages/extensions/installed.vue:207-220` (suggestion, enhancement)

# Design Observations

## Concerns

**D1. Shared metadata helpers belong in a non-component module (in-scope)** — Both `MarketplaceCard.vue` (catalog) and `installed.vue` (installed) need the same three pieces of information from an OCI label bag: a title with `id`/`slug` fallback, a "more info" URL with the Docker Hub fallback, and label-trim semantics. Today both files reimplement the URL logic, MarketplaceCard reads `extension.labels[...]` directly without the safe-trim helper, and only `installed.vue` extracts vendor and description. A small `utils/extensionMetadata.ts` exporting `extensionTitle(meta)`, `extensionVendor(meta)`, `extensionDescription(meta)`, `extensionMoreInfoURL(meta)` would: (a) collapse I1 to a one-line import on each side; (b) be unit-testable without the `methods.X` hack flagged in S4; (c) leave a single place to enforce S3's protocol allow-list. This matches the @mook-as suggestion to make `MarketplaceCard.vue` reusable, while keeping the components' rendering responsibilities separate from the label-parsing logic.

## Strengths

- Sort key arrays (`['vendor', 'title', 'id']`, etc.) fall through to the next key when the primary is empty — gives stable ordering even when many extensions lack metadata.
- Using `?.trim() || fallback` correctly treats whitespace-only labels as missing, matching the spirit of the existing `MarketplaceCard.vue` logic.
- Computed `installedExtensionRows` recomputes derived fields when `installedExtensions` changes, keeping the table reactive to upgrade/uninstall.

# Testing Assessment

Untested scenarios, ranked by risk:

1. **No template-level assertion that the `More information` link is actually rendered for every row.** The test could regress to an empty cell without failing.
2. **`installedExtensionRows` computed property is not tested.** A regression that drops `description` from the row object would not fail any current assertion.
3. **Whitespace-only labels** (e.g. `' '`) are not covered; the `?.trim() || fallback` branch is exercised only with the all-empty case.
4. **Mixed label sets**: an extension that has `vendor` but no `description` is not covered; only "all labels" and "no labels" cases exist.
5. **Sort key fall-through** (e.g., two extensions with empty `vendor` ordering by `title`) is not asserted; this is the kind of behavior that subtly breaks when refactoring the sort spec.
6. **Method dispatch via Vue (`this` = component instance)** is bypassed entirely (see S4). The helpers happen to be `this`-light today; a future refactor that pulls a value off `this.$store` will break in production and not in the test.

### [Gemini] — Pass 1

### Executive Summary
The PR adds metadata columns (vendor, description, more info) to the installed extensions table. The implementation functionally addresses the requirement but introduces an XSS vulnerability by binding untrusted OCI image labels directly to URL `href` attributes, and relies on brittle unit testing by directly calling Vue component internal methods.

### Findings

#### Critical Issues
None.

#### Important Issues
I1. **XSS vulnerability via untrusted OCI labels** — `pkg/rancher-desktop/pages/extensions/installed.vue:117-121`

```typescript
    extensionLink(ext: ExtensionMetadataSource): string {
      const preferredURL = this.extensionLabel(ext, LABEL_MORE_INFO);

      if (preferredURL) {
        return preferredURL;
      }
```

The `more-info` URL is extracted from the extension's OCI image labels and bound directly to the `href` attribute without scheme validation. Since any user can install an arbitrary Docker extension, a malicious extension can supply a `javascript:` or `data:` URL in the `io.rancherdesktop.extension.more-info` label. Clicking the "More information" link would execute arbitrary JavaScript in the Rancher Desktop UI renderer, which operates in a higher privilege zone than the extension's isolated iframe. 

Fix: Validate that the URL scheme is HTTP or HTTPS before returning it.

```diff
     extensionLink(ext: ExtensionMetadataSource): string {
       const preferredURL = this.extensionLabel(ext, LABEL_MORE_INFO);
 
-      if (preferredURL) {
+      if (preferredURL && /^https?:\/\//i.test(preferredURL)) {
         return preferredURL;
       }
```
(important, regression)

I2. **Brittle component method testing** — `pkg/rancher-desktop/pages/extensions/__tests__/installed.spec.ts:20-21`

```typescript
const { default: InstalledExtensions } = await import('@pkg/pages/extensions/installed.vue');
const methods = (InstalledExtensions as any).methods;
```

The test extracts and calls Vue component `.methods` directly. This tests internal implementation details rather than the component's public interface and relies on the methods remaining pure functions. If a method is ever updated to access component state (e.g., `this.t()` for translations or `this.installedExtensions`), the test will crash because `this` evaluates to the local `methods` object, not a Vue instance.

Fix: Rather than testing component internals, extract the metadata parsing logic into a pure utility module (e.g., `pkg/rancher-desktop/utils/extensionMetadata.ts`) and test that module directly. This also allows sharing the extraction logic with `MarketplaceCard.vue` rather than duplicating it.
(important, regression)

#### Suggestions
S1. **Incorrect Docker Hub registry inference for custom ports** — `pkg/rancher-desktop/pages/extensions/installed.vue:123-125`

```typescript
      if (!/^[^./]+\//.test(ext.id)) {
        return `https://${ ext.id }`;
      }
```

The regex checks whether the first segment of the extension ID lacks a dot (`.`), inferring that `publisher/extension` belongs to Docker Hub and `ghcr.io/publisher/extension` does not. However, it fails to account for colons (`:`). Under Docker reference parsing rules, a first segment containing a colon (e.g., `localhost:5000/extension`) is also a custom registry. The current regex matches `localhost:5000/`, causing local registries to incorrectly generate Docker Hub URLs (`https://hub.docker.com/extensions/localhost:5000/...`). This same logic gap exists in `pkg/rancher-desktop/components/MarketplaceCard.vue:130`.

Fix: Add `:` to the negated character class in both files.

```diff
-      if (!/^[^./]+\//.test(ext.id)) {
+      if (!/^[^./:]+\//.test(ext.id)) {
```
(suggestion, gap)
