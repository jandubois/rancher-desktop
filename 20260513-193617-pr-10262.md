## Executive Summary

The Round 1 squash addresses every prior-round finding cleanly: the protocol fix lands with table-driven tests, the iptables probes distinguish absent from transient errors via `isIptablesRuleAbsent`, the regex compiles once at package level, and the `portState{delegated bool}` design eliminates the "safe-by-accident" cleanup the original Gemini headline flagged. Three new gaps remain — all flowed from agents tracing the post-squash code paths rather than re-litigating Round 1.

Structure: the diff is one squashed commit on three files; the new code carries unusually thorough block comments and the test file pins the regex behavior (including the intentional range-form miss) at 23 subtests. **I1** is the architecturally significant finding: `engineManagesLoopbackBinding` correctly returns the `managed` boolean and the caller correctly suppresses the iptables Append, but the caller still calls `tracker.Add` with procnet's synthetic ID. If procnet wins the race against the engine's events handler (engine has written `CNI-HOSTPORT-DNAT` or `DOCKER` but has not yet called `tracker.Add` for the container ID), procnet ends up owning the gvisor-tap-vsock proxy under a synthetic key whose lifetime is tied to the transient OCI-hook listener — when that listener disappears, procnet's cleanup tears the proxy down for a port the container still needs. The fix is one branch: when `managed=true`, mark delegated and skip `tracker.Add` entirely. **I2** is the symmetric retry gap: the Delete path retains `added[port]` and retries on transient iptables failure (with an explanatory comment); the Append path logs the error and unconditionally writes the success state, so an xtables-lock collision at Append time leaves `--network=host` loopback permanently unreachable until the listener cycles. **I3** is the partial-failure leak: when `apiForwarder.Expose` succeeds but `wsl-proxy.Send` fails, `portStorage[synthetic]` is already populated; the next-tick retry sees `Expose` return "proxy already running" and procnet mis-classifies it as engine delegation, then never calls `tracker.Remove` on cleanup — `portStorage` and the gvisor-tap-vsock proxy leak until RD restart.

The suggestions cover the safety-net hole around substring-matching gvisor-tap-vsock's error text (no test pins the contract), a misleading docstring claim about a non-existent `isIptablesRuleAbsent` unit test, a behavior regression where persistent `tracker.Add` failures now log every 3 s, and a small `iptables --wait` hardening that would reduce the failure surface I1/I2/I3 all sit on.

## Important Issues

I1. **Engine-managed ports still claimed by procnet under a synthetic tracker ID** — `src/go/guestagent/pkg/procnet/scanner_linux.go:174-205` [Codex GPT 5.5] (important, gap)

```go
managed, err := p.engineManagesLoopbackBinding(port, bindings)
if err != nil {
    log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
    continue
}
if err := p.tracker.Add(utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port())), nat.PortMap{
    port: bindings,
}); err != nil {
    if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
        added[port] = portState{bindings: bindings, delegated: true}
        continue
    }
    log.Errorf("/proc/net scanner failed to add port: %s", err)
    continue
}
log.Infof("/proc/net scanner added port: %s -> %+v", port, bindings)
if !managed {
    if err := p.execLoopbackIPtablesRule(bindings, port, Append); err != nil {
        log.Errorf("/proc/net scanner creating loopback iptable rules for portbinding: %v failed: %s", bindings, err)
    }
}
added[port] = portState{bindings: bindings, delegated: false}
```

The probe at line 174 establishes that an engine chain (`CNI-HOSTPORT-DNAT` or `DOCKER`) already manages the port, but the caller still issues `tracker.Add` at line 179 with procnet's synthetic ID `proto/port`. The `managed` flag only suppresses the PREROUTING append at line 200; on success the port lands in `added` as `delegated: false` at line 205. If procnet wins the race against the engine's events handler (engine has written the chain rule but has not yet called `tracker.Add` for the container ID), `apiForwarder.Expose` succeeds and `portStorage[synthetic]` owns the proxy. Cleanup then follows `/proc/net` observation rather than the container lifecycle — when the transient OCI-hook listener disappears, procnet runs `tracker.Remove(synthetic)` and tears down a proxy the container still needs.

Tracing the engine writers confirms the asymmetry. `tracker/apitracker.go:81-141` stores the binding under whichever caller-supplied ID succeeded first, and `containerd/events_linux.go` and `docker/events.go` use the real container ID. The substring path at line 182 catches the inverse race (engine adds first) but not this one (procnet adds first). The current Resolution note about "mismatched-key Remove is safe" applies only when the engine wrote `portStorage[containerID]`; here `portStorage[synthetic]` is the only entry, and the Remove tears it down.

Fix: make the `managed` result a hard delegation decision before touching tracker state. When the engine chain already references the port, mark the port delegated and skip `tracker.Add` entirely.

```diff
 managed, err := p.engineManagesLoopbackBinding(port, bindings)
 if err != nil {
     log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
     continue
 }
+if managed {
+    added[port] = portState{bindings: bindings, delegated: true}
+    continue
+}
 if err := p.tracker.Add(utils.GenerateID(...), nat.PortMap{port: bindings}); err != nil {
```

After the fix the substring path becomes second-line defense for the rare case where the engine's `tracker.Add` lands before its chain rule lands.

I2. **Append-side retry is asymmetric with Delete-side: transient iptables errors silently lose the rule** — `src/go/guestagent/pkg/procnet/scanner_linux.go:200-205` [Gemini 3.1 Pro] (important, regression)

```go
if !managed {
    if err := p.execLoopbackIPtablesRule(bindings, port, Append); err != nil {
        log.Errorf("/proc/net scanner creating loopback iptable rules for portbinding: %v failed: %s", bindings, err)
    }
}
added[port] = portState{bindings: bindings, delegated: false}
```

The Append path logs the error and writes `added[port]` unconditionally. On the next tick, `alreadyAdded` short-circuits the port, so the rule never gets retried — a transient `iptables` failure (xtables lock contention, fork/exec failure) leaves the `--network=host` loopback listener permanently unreachable from the Windows host until the listener cycles. The Delete path at line 226-234 takes the opposite stance with an explanatory comment: "Retain the `added` entry so we retry the iptables Delete on the next tick. Without this, a transient probe error leaks a real PREROUTING rule." The Append path needs the same shape.

Fix: add a `pendingAppend` (or `appendFailed`) flag to `portState` and retry the append on the next tick when set. Combine with S1 below — `iptables --wait` shrinks the failure window at the source, the retry covers what slips through.

```diff
 type portState struct {
     bindings      []nat.PortBinding
     delegated     bool
+    appendFailed  bool
 }
```

```diff
 if !managed {
-    if err := p.execLoopbackIPtablesRule(bindings, port, Append); err != nil {
+    appendErr := p.execLoopbackIPtablesRule(bindings, port, Append)
+    if appendErr != nil {
         log.Errorf("/proc/net scanner creating loopback iptable rules for portbinding: %v failed: %s", bindings, err)
     }
 }
-added[port] = portState{bindings: bindings, delegated: false}
+added[port] = portState{bindings: bindings, delegated: false, appendFailed: appendErr != nil}
```

The retry path then reads `state.appendFailed` at the top of the loop and reissues `execLoopbackIPtablesRule(..., Append)` without re-invoking `tracker.Add`. Add a fixture for this case once the loop is unit-testable.

I3. **Partial `tracker.Add` failure triggers false delegation and a `portStorage`/proxy leak** — `src/go/guestagent/pkg/procnet/scanner_linux.go:179-198`, `src/go/guestagent/pkg/tracker/apitracker.go:118-134` [Gemini 3.1 Pro] (important, gap)

```go
if err := p.tracker.Add(utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port())), nat.PortMap{
    port: bindings,
}); err != nil {
    if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
        log.Debugf("/proc/net scanner port %s already exposed elsewhere, delegating", port)
        added[port] = portState{bindings: bindings, delegated: true}
        continue
    }
    log.Errorf("/proc/net scanner failed to add port: %s", err)
    continue
}
```

`APITracker.Add` is not atomic.

```go
if len(successfullyForwarded) != 0 {
    a.portStorage.add(containerID, successfullyForwarded)
    portMapping := guestagentTypes.PortMapping{
        Remove: false,
        Ports:  successfullyForwarded,
    }
    log.Debugf("forwarding to wsl-proxy to add port mapping: %+v", portMapping)
    err := a.wslProxyForwarder.Send(portMapping)
    if err != nil {
        return fmt.Errorf("sending port mappings to wsl proxy error: %w", err)
    }
}
```

In `tracker/apitracker.go:118-134`, `apiForwarder.Expose` and `portStorage.add` succeed before `wslProxyForwarder.Send` runs. If the wsl-proxy step fails, `Add` returns an error that does *not* match `portAlreadyExposedSubstring`, so procnet logs and continues without recording the port in `added`. On the next tick, retry — `apiForwarder.Expose` sees the same `Local:port` already in gvisor-tap-vsock and returns "proxy already running" wrapped through `verifyResponseBody → "%w: %s"`. The substring matches; procnet marks the port `delegated: true`. When the listener disappears, the cleanup loop skips `tracker.Remove` (because `delegated`), so `portStorage[synthetic]` and the gvisor-tap-vsock proxy persist until RD restart. wsl-proxy never learned about the port at all.

Narrow precondition (wsl-proxy partial failure), but the failure is silent and leaks resources past container exit. Fix: distinguish "I have already added this port myself" from "someone else added it." The simplest probe is the `Tracker.Get` method already in the interface:

```diff
 if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
+    syntheticID := utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port()))
+    if len(p.tracker.Get(syntheticID)) > 0 {
+        log.Errorf("/proc/net scanner port %s already added by procnet; partial-failure leak: %s", port, err)
+        added[port] = portState{bindings: bindings, delegated: false}
+        continue
+    }
     log.Debugf("/proc/net scanner port %s already exposed elsewhere, delegating", port)
     added[port] = portState{bindings: bindings, delegated: true}
     continue
 }
```

With Codex's I1 fix applied, this finding shrinks further: the substring path then exists only for the engine-Add-first race, and procnet never owns `portStorage[synthetic]` in any other branch. Both fixes compose cleanly.

## Suggestions

S1. **Pass `--wait` to every `iptables` invocation** — `src/go/guestagent/pkg/procnet/scanner_linux.go:313`, `351`, `416` [Gemini 3.1 Pro]

```go
iptablesCmd := exec.CommandContext(p.context,
    "iptables",
    "--table", "nat",
    fmt.Sprintf("--%s", action), "PREROUTING",
    ...
```

None of the three iptables shell-outs (`execLoopbackIPtablesRule`, `engineChainManagesPort`, `preroutingHasLoopbackRule`) uses the `--wait` flag. iptables exits status 4 the moment it cannot acquire the `xtables` lock; with `--wait` the binary blocks until the lock is free. Adding the flag shrinks the failure window I2 and I3 sit on at the source.

```diff
 iptablesCmd := exec.CommandContext(p.context,
     "iptables",
+    "--wait",
     "--table", "nat",
```

Apply to all three call sites. Default `--wait` blocks indefinitely; if the loop already tolerates a one-tick deferral, pair with `--wait 2` (seconds) so a hung lock does not stall the scanner.

S2. **Substring-matching `proxy already running` has no test pinning the contract** — `src/go/guestagent/pkg/procnet/scanner_linux.go:59`, `182` [Claude Opus 4.7]

```go
const portAlreadyExposedSubstring = "proxy already running"
...
if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
```

The delegation marker is a `strings.Contains` against an error message that originates in `github.com/containers/gvisor-tap-vsock/pkg/services/forwarder/ports.go`. The wrap chain `verifyResponseBody → "%w: %s"` in `forwarder/serviceapi.go` keeps the substring intact today, but no compile-time guarantee enforces this. If upstream rewords the error (e.g., to `port already exposed`) the check silently fails and procnet writes the rogue rule again — the exact regression this PR exists to fix. No unit test triggers the delegation path against an actual upstream response body.

Two-step hardening:

1. Add a test that constructs an `APIForwarder` against an `httptest.Server` returning the upstream's actual response shape, and asserts the substring path triggers.
2. Promote the substring to a typed sentinel inside `forwarder.verifyResponseBody`. Have it detect the response body and return `forwarder.ErrPortAlreadyExposed`; then `procnet` switches from `strings.Contains` to `errors.Is`. The coupling stays one layer deep instead of two.

S3. **Docstring claims a unit test that does not exist** — `src/go/guestagent/pkg/procnet/scanner_linux.go:345-347` [Claude Opus 4.7, Codex GPT 5.5]

```go
// Shells out to iptables directly; end-to-end behavior is covered by
// published-ports.bats. Distinguishing absent from transient errors
// is unit-tested through isIptablesRuleAbsent.
```

The `engineChainManagesPort` docstring asserts that `isIptablesRuleAbsent` is unit-tested. `scanner_linux_test.go` contains only `TestDnatChainContainsPort`. A reader who trusts the comment skips the test and misses a regression in stderr-string matching.

Fix: add the asserted test, or remove the claim. The test is one table-driven function:

```go
func TestIsIptablesRuleAbsent(t *testing.T) {
    tests := []struct {
        name string
        err  error
        want bool
    }{
        {"nil", nil, false},
        {"non-exit error", errors.New("fork failure"), false},
        {"missing chain", &exec.ExitError{Stderr: []byte("iptables: No chain/target/match by that name.\n")}, true},
        {"missing rule", &exec.ExitError{Stderr: []byte("iptables: Bad rule (does a matching rule exist in that chain?).\n")}, true},
        {"xtables lock", &exec.ExitError{Stderr: []byte("Another app is currently holding the xtables lock.\n")}, false},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            require.Equal(t, tt.want, isIptablesRuleAbsent(tt.err))
        })
    }
}
```

Constructing `exec.ExitError` directly works in tests because the struct is exported and the only field this helper reads is `Stderr`.

S4. **Persistent `tracker.Add` failures now log every 3 s** — `src/go/guestagent/pkg/procnet/scanner_linux.go:196-197` [Claude Opus 4.7] (regression)

```go
log.Errorf("/proc/net scanner failed to add port: %s", err)
continue
```

The pre-PR code recorded the port unconditionally in `previousPortMap = newPortMap`, so a persistent `tracker.Add` failure (host-switch down, wsl-proxy crashed, malformed response) logged once and was masked on subsequent scans. The new code records the port in `added` only on success, so the failure re-fires every `scanInterval` for as long as the listener persists. For a single port: ~28,000 lines per day at the default 3 s interval. The block comment at lines 104-139 documents the latency tradeoff but not this retry tradeoff.

Fix options:

1. Record a `failed` state in `portState` and log at `Debug` after the first attempt.
2. Back off after N consecutive failures.
3. Extend the comment block so the next reader recognizes the retry is intentional.

Option 1 pairs cleanly with the I2 fix (both want richer state on `portState`).

S6. **`dnatRuleRe` anchors `tcp`/`udp` anywhere on the line rather than the protocol column** — `src/go/guestagent/pkg/procnet/scanner_linux.go:81-83` [Claude Opus 4.7]

```go
var dnatRuleRe = regexp.MustCompile(
    `\b(tcp|udp)\b[^\n]*?(?:dpt:([0-9]+)|dports ([0-9]+(?:,[0-9]+)*))\b`,
)
```

The first capture group matches any standalone `tcp` or `udp` token on the line and then non-greedily consumes characters up to `dpt:`/`dports`. For a hypothetical rule whose comment contains `tcp` before a real `udp dpt:` token (impossible in iptables's own output, but achievable through annotations or chain-name collisions), the parser misclassifies the protocol. A stricter regex anchors the protocol directly against the match-module token:

```diff
-`\b(tcp|udp)\b[^\n]*?(?:dpt:([0-9]+)|dports ([0-9]+(?:,[0-9]+)*))\b`
+`(?:\s|^)(tcp|udp)\s+(?:dpt:([0-9]+)|(?:multiport\s+)?dports ([0-9]+(?:,[0-9]+)*))\b`
```

Theoretical for shipped iptables output; flag as future hardening.

## Design Observations

### Concerns

- *(in-scope)* The Append/Delete asymmetry surfaced by I2 reflects a broader pattern: the loop now juggles three maps (`seenLastScan`, `added`, `newPortMap`) and four transition rules (gate, delegation, append-retry, delete-retry), but only the regex helper is unit-testable. Round 1's S4 already recommended extracting a `portStateTracker` for unit tests; I2's appendFailed flag and I3's `tracker.Get` check both make that extraction more attractive. [Claude Opus 4.7]
- *(in-scope)* The substring contract with gvisor-tap-vsock and the stderr-fragment matching in `isIptablesRuleAbsent` together establish two parallel "string contracts" with code outside this package. Both are documented; neither is asserted by a test. A small `Doer` interface for the iptables shell-outs plus a sentinel error from `APIForwarder` would let the caller depend on typed errors instead of substrings, and would unlock unit tests for cleanup paths that today live only behind `published-ports.bats`. [Claude Opus 4.7]
- *(future)* The two-scan gate's correctness depends on the OCI hook's reservation never spanning more than one tick. `scanInterval` is configurable through `NewProcNetScanner`; a future caller that plumbs in a shorter interval (e.g., 500 ms) could let a still-reserved port through. Worth a comment that `scanInterval ≥ ~2 s` is part of the contract. [Claude Opus 4.7]

### Strengths

- *(in-scope)* The `portState{delegated bool}` design makes the "engine wins the race" path explicit at every call site, rather than leaving readers to reverse-engineer it from "tracker.Remove on a synthetic ID happens to be a no-op." [Claude Opus 4.7]
- *(in-scope)* Splitting the regex parsing out as a pure function with a 23-case table-driven test (including IP-octet-as-port, cross-protocol, and the intentional range-form miss) is the right shape. The author thought about parser failure modes and locked them in. [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]
- *(in-scope)* The 35-line "Alternative considered" block records the rejected pid/cgroup/inode classification path and why. A future "we should classify by pid" suggestion can land on the prior reasoning rather than restart it. [Claude Opus 4.7, Codex GPT 5.5]
- *(in-scope)* The delete-side retry design (keep the port in `added` until iptables Delete succeeds, with an explanatory comment) is the right tradeoff. I2 calls for symmetry on the Append side. [Claude Opus 4.7]

## Testing Assessment

Coverage of the new shell-out helpers and the state machine is zero from unit tests; all behavior verification flows through `published-ports.bats`. Specific untested scenarios, ranked by risk:

The PR's manual test plan correctly identifies `bats/tests/containers/published-ports.bats` and the two manual smoke tests (bridge-network published port from Windows; `--network=host` loopback). Those were not run from this review.

### Skill improvements

- **When a function returns a "managed" or "owned" boolean that gates one side of a downstream call, trace every other side-effect the same caller produces.** A flag that correctly suppresses one mutation (here, the iptables Append) may leave a parallel mutation unchanged (here, the tracker.Add), opening a race window the prose of the helper's docstring would not predict. ## Appendix: [Gemini]

### Executive Summary
This PR resolves a bug where the `/proc/net` scanner creates a loopback `PREROUTING` DNAT rule that shadows the authoritative CNI or Docker engine rule, causing external connections to published ports to hang. The solution introduces a stability gate to defer short-lived listeners and checks container engine iptables chains before appending the procnet rule. While the approach effectively addresses the shadowing, there are edge cases in the error handling paths for transient `iptables` failures and partial API exposure failures that can lead to permanent port leaks or un-forwarded ports.

### Findings

#### Critical Issues
None.

#### Important Issues

I1. **Transient `Append` failures cause permanent skip of loopback routing** — `src/go/guestagent/pkg/procnet/scanner_linux.go:198-204`

```go
				if !managed {
					if err := p.execLoopbackIPtablesRule(bindings, port, Append); err != nil {
						log.Errorf("/proc/net scanner creating loopback iptable rules for portbinding: %v failed: %s", bindings, err)
					}
				}
				added[port] = portState{bindings: bindings, delegated: false}
```

If `execLoopbackIPtablesRule` fails (e.g., due to xtables lock contention, which the PR explicitly acknowledges can happen), the error is logged and the port is unconditionally tracked in the `added` map. Because the `added` map prevents retries on subsequent scans, the `Append` action is never retried, leaving the container's loopback port permanently inaccessible from the host.

Fix: Either pass `--wait` to the `iptables` command to prevent lock contention (see S1), or track `appendPending` in `portState` so the loop can retry only the iptables write on the next tick. Do not simply `continue` on error, as that would trigger the false-delegation bug described in I2 on the next scan.

I2. **Partial `tracker.Add` failure triggers false delegation and resource leak** — `src/go/guestagent/pkg/procnet/scanner_linux.go:181-190`

```go
				if err := p.tracker.Add(syntheticID, nat.PortMap{port: bindings}); err != nil {
					if strings.Contains(err.Error(), portAlreadyExposedSubstring) {
						// ...
						added[port] = portState{bindings: bindings, delegated: true}
						continue
					}
```

The delegation logic assumes that if `tracker.Add` returns an error containing `"proxy already running"`, the container engine's event handler already exposed the port, so it marks it as `delegated: true`. However, this error also occurs if *procnet itself* attempted to expose the port on the previous tick and partially failed (e.g., the API forwarder succeeded but the WSL proxy step failed, returning an error). 

On the next tick's retry, `tracker.Add` will return `"proxy already running"` because of the previous tick's partial success. The scanner misinterprets this as engine delegation, marks the port as `delegated: true`, skips `Append`, and—crucially—skips `tracker.Remove` when the port closes. This permanently leaks the port forwarding.

#### Suggestions

S1. **Add `--wait` to iptables commands to prevent lock contention** — `src/go/guestagent/pkg/procnet/scanner_linux.go:310`

```diff
 		iptablesCmd := exec.CommandContext(p.context,
 			"iptables",
+			"--wait",
 			"--table", "nat",
```

The `iptables` commands for `Append`/`Delete`, `--list`, and `--check` do not use the `--wait` (or `-w`) flag. This leads to spurious `exit status 4` (xtables lock contention) failures in concurrent environments. Adding `--wait` mitigates transient failures at the source and reduces the need for the scanner to handle and defer on these errors. This should be applied to `engineChainManagesPort` and `preroutingHasLoopbackRule` as well.

### Design Observations

**Strengths**
- The `dnatChainContainsPort` logic and its corresponding tests are robust and cover numerous edge cases (such as IP octets looking like ports and cross-protocol shadowing). 
- The stability gate is a clean, non-invasive way to handle the transient OCI hook race without needing fragile cgroup or process inspection.

### Testing Assessment
1. **End-to-end integration of partial `tracker.Add` failures:** There are no tests covering the state machine when `tracker.Add` fails partially and retries on the next tick, which currently leads to the false delegation bug (I2).
2. **Recovery from transient `iptables` failures:** There are no unit or integration tests asserting that the `Append` and `Delete` paths correctly retry (or fail to retry, as in I1) when `execLoopbackIPtablesRule` fails due to lock contention.

### Documentation Assessment
No gaps identified. The inline comments clearly explain the timing logic, edge cases, and rationale for deferrals.

### Acknowledged Limitations

- **Port-range syntax limitations:** The `dnatChainContainsPort` regex intentionally does not match `dpts:LO:HI` or ports within an embedded multiport range. A comment notes that the shipped engines emit per-port rules, so the gaps are theoretical for shipped engines but real for hand-rolled setups (`src/go/guestagent/pkg/procnet/scanner_linux.go:420-424`).
- **Hardcoded engine chains:** `engineChainManagesPort` only checks `CNI-HOSTPORT-DNAT` and `DOCKER`. A comment acknowledges that adding a third engine will require updating this list or the original shadow-DNAT bug will reappear (`src/go/guestagent/pkg/procnet/scanner_linux.go:323-326`).
- **Stability gate latency:** The stability gate adds a 3-second delay before genuine `--network=host` listeners reach the host. The comment explicitly weighs this against the complexity of cgroup inspection and defers a fix unless it becomes a user complaint (`src/go/guestagent/pkg/procnet/scanner_linux.go:121-127`).
