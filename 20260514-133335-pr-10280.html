Executive Summary ¶

This is Round 4. The commit under review (410039856) folds in the three Round 3 Suggestion fixes, all of which verify against the current code: APITracker.Remove now reports forwarder.ErrUnexposeAPI and tracker.ErrWSLProxy under distinct sentinels joined with errors.Join (S1), an iptablesCommand helper pins LC_ALL=C on every iptables invocation (S2), and a separate probeErrorLogged marker keeps the engine-probe failure from muting the first tracker.Add error (S3). TestRemoveWSLProxyFailureIsDistinguishable and the reworked append-retry tests cover the new behavior.

One new Important finding landed this round. retryAppend's engine-takeover branch marks a port delegated:true even when the applyLoopbackRules(..., Delete) call that is supposed to remove the committed procnet rule fails — and a delegated port's cleanup path never touches iptables again, so a committed PREROUTING DNAT can survive to RD restart and shadow the engine's chain. That is the exact bug this PR exists to fix, reachable in a narrow multi-failure error path the tests do not exercise.

The three Suggestions are minor: a self-contradicting doc comment, an untested first-error short-circuit in applyLoopbackRules that is benign under today's binding semantics, and a parity gap where the sibling APITracker.RemoveAll still drops wslProxyErrs the way Remove used to. Codex's initial-append race and Gemini's regex-hijack finding were both reclassified on verification — see the Reconciliation paragraph in the retro.

S2. applyLoopbackRules returns on the first per-binding error — benign today, but the semantics are untested Gemini Claude¶

// applyLoopbackRules dispatches an Append or Delete to the iptables
// runner for every 127.0.0.1 binding of the port. Returns the first
// per-binding error; remaining bindings are skipped so the caller can
// defer to a later tick.
func (pst *portStateTracker) applyLoopbackRules(bindings []nat.PortBinding, port nat.Port, act action) error {
	for _, binding := range bindings {
		if binding.HostIP != loopbackIP {
			continue
		}
		if err := pst.iptables.ApplyLoopbackRule(port.Proto(), binding.HostPort, act); err != nil {
			return err
		}
	}
	return nil
}

func syntheticIDFor(port nat.Port) string {
	return utils.GenerateID(fmt.Sprintf("%s/%s", port.Proto(), port.Port()))
}

applyLoopbackRules returns on the first failing binding, skipping the rest. For Append that is correct — the caller records appendFailed and retryAppend re-runs the whole binding list. For Delete the caller (retireDisappeared, retryAppend) drops the local marker regardless, so any binding after the failing one would be skipped permanently.

In practice this is unreachable: applyLoopbackRules acts only on bindings whose HostIP is exactly 127.0.0.1, and a nat.Port can have at most one such binding — two LISTEN sockets cannot share 127.0.0.1:<port>. Gemini rated this Important on the assumption of multiple loopback bindings per port; that assumption does not hold for the procfs-derived port map, so it is downgraded to a Suggestion. The residual value is that the "first-error skips the remainder" semantics are untested and would silently become a real leak if binding collection ever changed.

Fix (optional): accumulate per-binding errors with errors.Join so the loop always processes every binding, and add a fixture with two loopback bindings to pin the behavior.

Design Observations ¶

Concerns ¶

  • (future) exposeNew and handleAlreadyExposed do not re-probe the engine chain between the initial probe and the Append — port_state_tracker_linux.go:176-232 Codex. exposeNew probes the engine chain once, then runs tracker.Add, then applyLoopbackRules(..., Append) with no re-probe; handleAlreadyExposed's resume path appends with no fresh probe at all. If the engine chain lands during the Add-to-Append window, procnet installs a shadowing rule. This restates the Round 3 design observation "exposeNew does not re-probe the engine chain after a healthy tracker.Add" from the Add-window angle, and the code in this path is unchanged since Round 3. It is the same netns-blind / stability-gate-bypass limitation the portState doc comment already documents, and — unlike I1 — it self-heals: the port stays delegated:false, so when the transient listener disappears retireDisappeared deletes the rule. Codex rated it Important; reclassified as a design observation, consistent with Round 3. The fix (a re-probe immediately before each first Append) is the same shape retryAppend already uses and would be worth doing if the Add window is ever measured to be non-trivial.
  • (future) dnatChainContainsPort scans human-readable iptables output with a non-greedy line regex — scanner_linux.go:70-72, scanner_linux.go:264-281 Gemini. dnatRuleRe matches \b(tcp|udp)\b[^\n]*?(?:dpt:N|dports N…); the [^\n]*? scans the whole line and the first dpt:/dports token wins, so a dpt:/dports token appearing earlier on the line would mask the real port (a false negative → procnet installs a shadowing rule). Gemini raised this as a Critical regression on the theory that a container named dpt:80 injects that token into the iptables comment. That vector is not reachable: the CNI-HOSTPORT-DNAT comment carries the CNI network name and the CNI container ID (/* dnat name: "bridge" id: "default-abc" */), not the user's --name; and network/container names are validated to [a-zA-Z0-9][a-zA-Z0-9_.-]* — no colon, no space — so none of them can produce dpt:N or dports N. The finding was dropped as a Critical (see Reconciliation), but the underlying fragility of parsing presentation output is real and is the same point Round 3 recorded — iptables -S <chain> emits the machine-parseable rule-spec form and remains the fallback if a future iptables bump breaks the regex.
  • (future) procnet now couples to APITracker's error wrapping at three points — port_state_tracker_linux.go:206, :330 Claude. The state machine depends on the portAlreadyExposedSubstring substring match and on errors.Is against ErrWSLProxy/ErrUnexposeAPI. The sentinel coupling is sound — typed, in-repo, and pinned by TestRemoveWSLProxyFailureIsDistinguishable. The substring match is the fragile one: "proxy already running" originates in the third-party gvisor-tap-vsock forwarder and is not an exported symbol. TestPortAlreadyExposedSubstringIsReachable pins the wrap chain but, by its own comment, cannot guard an upstream rename. Worth a comment on the gvisor-tap-vsock dependency in go.mod; no in-scope action.

Review Process Notes ¶

Skill improvements ¶

  • When an error from a cleanup, release, or rollback step is only logged — not propagated and not retried — check what state the surrounding code moves the entity into next. A logged-and-ignored cleanup failure is recoverable only if some later code path re-attempts it; if the code immediately transitions the entity to a terminal or delegated state whose handlers short-circuit the retry path, the failure is a permanent leak, not a transient one. Flag a logged-but-unhandled error from a cleanup step when the very next statement transitions the entity into a state that forecloses retry — the log line makes it look handled, but the resource is stranded until process restart.
