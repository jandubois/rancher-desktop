High — `curl | bash` equivalents and self-fetches from a moving ref ¶

scripts/windows-setup.ps1:49
Invoke-WebRequest -UseBasicParsing -Uri 'https://get.scoop.sh' | Invoke-Expression. PowerShell equivalent of curl … | sh; whoever controls get.scoop.sh gets arbitrary code execution on the developer machine. Pin the URL to a 40-hex commit on the scoop installer repo, download to disk, verify a stored sha256, then execute. (See wiki §3.5; PowerShell threat model identical to shell.)
scripts/windows-setup.ps1:74
fetches install-wsl.ps1, restart-helpers.ps1, sudo-install-wsl.ps1, uninstall-wsl.ps1 from https://raw.githubusercontent.com/rancher-sandbox/rancher-desktop/main/scripts/windows/$file. The main branch is mutable; anyone with push access controls these bytes. Replace with a relative path (these files live next to the caller) or pin to a 40-hex SHA.

Retro: dependency types not explicitly covered ¶

R1. PowerShell IWR | iex with multi-line continuation goes undetected. The script's curl_bash_equivalents[] is empty even though scripts/windows-setup.ps1:49-50 clearly contains the pattern:

Invoke-WebRequest -UseBasicParsing -Uri 'https://get.scoop.sh' `
    | Invoke-Expression

The trailing backtick splits the pipeline across two lines; the script's regex appears to require both halves on the same line. Proposed change: before pattern matching, concatenate lines whose previous line ends in (PowerShell line-continuation), then run the existing detector. Alternatively, add a second pass that finds Invoke-WebRequest/iwr references and looks ahead up to ~3 lines for a | Invoke-Expression/| iex` consumer.

R2. Yarn-berry conditions: filtering treated as missing-integrity. The script reports 87 descriptors as missing_integrity in yarn.lock, but every one of them is a platform-specific optional dep gated by conditions: os=X & cpu=Y and never installed on the host platform — yarn-berry intentionally omits the checksum: line in that case. Proposed change: when walking yarn-berry blocks, skip entries whose body contains conditions:. Optionally surface them under a separate platform_conditioned: <count> key so the report can mention them without flagging.

R3. Custom-manifest no-op delegates aren't traced automatically. The script reports empty_checksum_blocks: 3 and lists the entry names, but stops there. In this scan I had to grep manually (scripts/dependencies/tools.ts) to find that the three entries delegate to go run, scripts/spelling.sh, and resources/setup-spin. Proposed change: when an entry name has an empty checksum block, look for a same-name class in the repo's dependency-management directory (scripts/dependencies/*.ts here, but configurable) and surface the download() body — when it is Promise.resolve(), find string literals that look like script paths (scripts/*.sh, resources/*) and emit them as delegate: <path> so Phase 2 can inline-download-rule them straight away.
