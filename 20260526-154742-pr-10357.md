## Important Issues

I1. **EADDRINUSE rollback unexposes the port from host-switch and contradicts the docstring** — `src/go/guestagent/pkg/procnet/scanner_linux.go:186-217`, `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:64-74` [Claude Opus 4.7, Codex GPT 5.5] (important, regression)

```go
if err := p.forwarder.Add(p.ctx, port.Proto(), uint16(portNum)); err != nil {
    p.logAddFailure(port, fmt.Sprintf("loopback forwarder %s/%s: %s", port.Proto(), b.HostPort, err))
    if removeErr := p.tracker.Remove(id); removeErr != nil {
        p.logAddFailure(port, fmt.Sprintf("rollback after forwarder.Add failure: %s", removeErr))
    }
    return err
}
```

On a `--network=host` container holding both `127.0.0.1:N` and `0.0.0.0:N`, `scanned` carries one port key with both bindings. `tracker.Add` (line 188) succeeds and host-switch routes Windows traffic to `tapIP:N`; the loopback iteration then calls `forwarder.Add`, which fails because the engine-namespace `0.0.0.0` listener already covers `tapIP:N`. `publish` rolls back via `tracker.Remove` (line 212), removing the host-switch exposure the wildcard listener relied on. Every tick repeats the cycle, flapping host-switch state every ~3 s.

The docstring at `loopback_forwarder_linux.go:73` claims "such ports stay reachable from Windows through the wildcard listener via eth0" — true only while the tracker entry stays installed. Round 2's S2 declined the noise concern; the throttle fixes noise, not reachability, and the new docstring claim in `b76cd5c7a` is unmet.

Fix: skip `forwarder.Add` for the loopback entry when `bindings` includes `0.0.0.0` for the same port; update the docstring.

```diff
 for _, b := range bindings {
-    if b.HostIP != loopbackIP {
+    if b.HostIP != loopbackIP || hasWildcardBinding(bindings) {
         continue
     }
```

I2. **IPv6 docstring overstates dual-stack coverage** — `src/go/guestagent/pkg/procnet/scanner_linux.go:25-30` [Claude Opus 4.7] (important, regression)

```go
IPv6 limitation: the scanner reads /proc/net/{tcp,udp} only, not the
tcp6/udp6 variants. A --network=host container that listens
exclusively on [::1]:port is invisible to the scanner and is not
reachable from Windows. Containers that listen on both 127.0.0.1
and [::1] (the common dual-stack pattern) remain reachable through
the IPv4 listener.
```

The "common dual-stack pattern" on Linux is `bind([::]:port)` with `IPV6_V6ONLY=0` (Go's `net.Listen("tcp", ":port")`, Python's `socket.AF_INET6` defaults, most container apps). That socket appears in `/proc/net/tcp6` only, never `/proc/net/tcp`; the `case procnettcp.TCP:` switch at `scanner_linux.go:303-310` filters it out. A `--network=host` Go HTTP server using `:8080` is invisible to the scanner — exactly the failure mode the comment claims is covered.

`git blame` confirms the misleading docstring landed in `b76cd5c7a` (the round-2 resolution commit); the underlying tcp6/udp6 gap is pre-existing.

Fix: rewrite the second sentence so the docstring matches the code.

```diff
-Containers that listen on both 127.0.0.1
-and [::1] (the common dual-stack pattern) remain reachable through
-the IPv4 listener.
+Containers that bind dual-stack (e.g. [::]:port with
+IPV6_V6ONLY=0, the Go and Python default) are also invisible
+because the socket appears only in /proc/net/tcp6. Only listeners
+that bind 127.0.0.1, 0.0.0.0, or both v4 and v6 separately are
+reachable from Windows.
```

Closing the underlying gap (extending `addValidProtoEntryToPortMap` to handle `procnettcp.TCP6`/`UDP6`) is out of scope for this PR.

## Suggestions

S1. **`pending` stores bindings the code never reads** — `src/go/guestagent/pkg/procnet/scanner_linux.go:147-164` [Claude Opus 4.7] (suggestion, enhancement)

```go
p.pending = make(nat.PortMap)
for port, bindings := range scanned {
    if _, ok := p.published[port]; ok {
        continue
    }
    p.pending[port] = bindings
}
```

The publish loop at line 147 reads `scanned[port]` for the bindings, not `p.pending[port]` — intentional, per the `b76cd5c7a` commit message. The rebuild above copies bindings from `scanned` into `pending`, and no later code reads them. The throttle sweep (lines 170-178) checks key membership only.

Fix: switch `pending` to `map[nat.Port]struct{}`. The rebuild becomes `p.pending[port] = struct{}{}`.

S2. **Shape change before publish bypasses the two-scan stability gate** — `src/go/guestagent/pkg/procnet/scanner_linux.go:147-156` [Gemini 3.1 Pro] (suggestion, gap)

```go
for port := range p.pending {
    bindings, ok := scanned[port]
    if !ok {
        continue
    }
    if err := p.publish(port, bindings); err != nil {
        continue
    }
```

`Tick` matches `pending` against `scanned` by port key alone. A port observed once with `[127.0.0.1]` and once with `[0.0.0.0]` clears the gate on its second sighting and publishes the second tick's shape. The gate's stated purpose — filter the OCI-hook reservation socket — is preserved (the OCI hook vanishes by the next tick). The remaining trigger is narrow: same port reused across consecutive ticks with different bind shapes. Worst case: a transient is published, then unpublished on the next tick. Self-healing.

Downgraded from Gemini's Important because the harm is bounded (briefly publish-then-unpublish, vs DNAT-era persistent shadowing) and the trigger is narrow.

Fix: compare `scanned[port]` against `pending[port]` before publishing. If they differ, leave the port in pending so the new shape gets its own two-tick gate.

```diff
 for port := range p.pending {
-    bindings, ok := scanned[port]
-    if !ok {
+    bindings, ok := scanned[port]
+    if !ok || !bindingsEqual(p.pending[port], bindings) {
         continue
     }
```

This conflicts with S1 (which proposes dropping bindings from `pending`); pick one approach.

S3. **`acceptTCP` Error-logs every transient Accept failure after backing off** — `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:170-196` [Claude Opus 4.7] (suggestion, enhancement)

```go
log.Errorf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
select {
case <-ctx.Done():
    return
case <-time.After(backoff):
}
backoff *= 2
```

Under sustained FD pressure (EMFILE) the loop saturates at the 5 s cap and emits one Error every 5 s indefinitely. The publish path uses `logAddFailure` (scanner_linux.go:229-240) to bound this exact pattern; apply the same template to `acceptTCP`.

Fix: log the first Error at full level, then degrade to Debug until Accept next succeeds.

```diff
-log.Errorf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
+if !loggedAcceptError {
+    log.Errorf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
+    loggedAcceptError = true
+} else {
+    log.Debugf("loopback forwarder accept tcp/%d: %s (retry in %s)", port, err, backoff)
+}
```

Reset `loggedAcceptError` to `false` after the next successful Accept.

S4. **PR description says "30-second idle timeout" but the UDP path uses 90 s** — PR body, `src/go/guestagent/pkg/procnet/loopback_forwarder_linux.go:100` [Claude Opus 4.7] (suggestion, gap)

The PR body's "What's added" section says UDP "multiplexes concurrent flows by source address with a per-flow outbound socket and a 30-second idle timeout." Round 1 swapped the hand-rolled demuxer for `forwarder.UDPProxy`, whose flow timeout is `UDPConnTrackTimeout = 90 * time.Second`. ## Design Observations

### Concerns

- **Scanner does not unwind tracker state on agent shutdown** *(future)* [Claude Opus 4.7]

  `ForwardPorts` exits on `ctx.Done()` and `defer p.forwarder.Close()` releases sockets, but `published` is not unwound through `tracker.RemoveAll()`. host-switch keeps its proxy table populated from the dead agent; on restart the new instance re-issues `Expose` for the same ports and meets "proxy already running" until something else clears the stale entry. Pre-existing — not introduced by this PR — but the new code structure is cleaner, so a `RemoveAll` shutdown hook fits naturally. One extra line and an API audit.

### Strengths

- **`Tick(scanned)` extraction makes every state transition unit-testable** without sockets. The tests exercise gate promotion, gate restart, shape change before and after publish, throttle hysteresis, rollback symmetry, and end-to-end forwarding through the scanner-to-forwarder seam.
- **Delegating UDP to `forwarder.UDPProxy`** (round 1's S1 fix) eliminates a class of hand-rolled conntrack bugs and matches Lima's own implementation choice.
- **`entriesToPortMap` bindIP self-filter** (`scanner_linux.go:289-300`) handles forwarder-owned sockets without coupling the scanner to the forwarder's internal state, and the trade-off is documented inline.
- **`addErrorLogged` throttle sweep** (`scanner_linux.go:170-178`) closes the unbounded-growth seam under listener churn — every port that leaves both pending and published also leaves the throttle map.
