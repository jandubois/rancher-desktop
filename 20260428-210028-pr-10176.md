## Suggestions

S1. **`readDependencyManifest` accepts any non-array object as the version field, deferring shape validation to download time** — `scripts/lib/dependencies.ts:131-158` [Claude Opus 4.7]

```ts
const valid = typeof version === 'string' ||
  (typeof version === 'object' && version !== null && !Array.isArray(version));

if (!valid) {
  throw new Error(`Entry ${ name } in ${ path } has invalid version ${ JSON.stringify(version) }; expected a string or { apiVersion, commit }`);
}
```

A hand-edit that produces e.g. `lima.version: { foo: 'bar' }` parses cleanly, then surfaces several layers later as a 404 with `[object Object]` in the URL. The compound shape is documented in the new `MANIFEST_HEADER`, but the parser does not enforce it.

Fix: dispatch on `name` and validate the expected per-key shape (e.g., for `mobyOpenAPISpec`, require `apiVersion` is a string and `commit` matches `/^[0-9a-f]{40}$/i`). The `commit` regex already lives at `scripts/dependencies/moby-openapi.ts:32`; pulling it up to the parser catches the bad entry where the user is editing rather than where the URL fetch fails.

S2. **`downloadAndHash` writes mismatch bytes to a fixed path in `os.tmpdir()`, colliding silently across distinct URLs with the same basename** — `scripts/lib/dependencies.ts:257-268` [Claude Opus 4.7]

```ts
const basename = path.basename(new URL(url).pathname) || 'artifact';
const keepPath = path.join(os.tmpdir(), `rddepman-mismatch-${ basename }`);

await fs.promises.copyFile(tempPath, keepPath);
throw new Error(`Upstream checksum mismatch for ${ url }: ...`);
```

A repeated run against the same URL overwrites the prior file (good), but two distinct URLs that happen to share `basename` (`checksums.txt` is a common one) overwrite each other. The error message names the keep-path, so a maintainer triaging two mismatches in sequence finds only the latter file.

Fix: include a short content hash in the filename, e.g. `rddepman-mismatch-${ basename }-${ actualChecksum.slice(0, 8) }`, or use `mkdtemp` to give each mismatch a unique directory.

S3. **`7z e -y archive.xz` silently overwrites any sibling file inside `internalDir` matching the archive's entry name** — `scripts/dependencies/wsl.ts:42` [Claude Opus 4.7]

```ts
await simpleSpawn('7z', ['e', '-y', archivePath], { cwd: context.internalDir });
```

The `-y` flag (added Round 7) auto-confirms overwrite prompts, which is necessary for the partial-failure-retry case the comment names. It also, however, silently overwrites unrelated files in `internalDir` if 7z's archive ever contains an entry whose name collides. xz archives carry a single entry today, so the failure mode is purely hypothetical — but if upstream packaging changes (e.g. switches to a tar-inside-xz layout), the silent overwrite happens without diagnostic.

Fix: extract to a unique tmp dir like the `downloadTarGZ`/`downloadZip` flows (`fs.mkdtempSync`), then move the single expected entry into place. Aligns with the rest of the file and removes the implicit single-entry assumption.

S4. **`download()` cache check hashes only `${destPath}.unsigned` for codesigned binaries; tampering with the signed `destPath` is undetected** — `scripts/lib/download.ts:69-94` [Claude Opus 4.7]

```ts
const hashTarget = options.codesign ? `${ destPath }.unsigned` : destPath;

if (!overwrite) {
  try {
    await fs.promises.access(destPath, access);
    if (expectedChecksum) {
      try {
        const actualChecksum = await hashFile(hashTarget, 'sha256');
        if (actualChecksum === expectedChecksum) {
          console.log(`${ destPath } already exists with expected checksum, not re-downloading.`);
          return;
        }
```

The codesign branch checks that `destPath` exists, then hashes the sibling `.unsigned`. If the signed binary at `destPath` is replaced with a corrupt or different file but `.unsigned` is left intact, the cache hit succeeds and the bad `destPath` survives. This is a developer-trusted-tree concern, not an attack vector — the practical likelihood is low. Pre-PR there was no checksum verification at all, so the new behaviour is a net improvement; flagging as Suggestion only because it inverts the strongest property the new code claims to provide.

Fix: delete `${destPath}.unsigned` after a successful codesign, so the cache shortcut is inapplicable for codesigned binaries. The next run re-downloads — the same cost the current logic already pays whenever `.unsigned` is missing.

S5. **`MANIFEST_HEADER` warns "the next bump will strip them" but the warning is only visible in the regenerated file itself** — `scripts/lib/dependencies.ts:167-176` [Claude Opus 4.7]

```ts
const MANIFEST_HEADER = `# Regenerated by rddepman on every dependency bump.  Document non-obvious
# version pins in scripts/dependencies/<name>.ts instead of inline comments
# here — the next bump will strip them.
...
`;
```

A developer who hand-edits a comment in `dependencies.yaml` and later sees it disappear after a bump is unlikely to have read the file's first lines. The natural place to look for the warning is `CONTRIBUTING.md` or a README near `dependencies.yaml`. Mirroring the rule there would close the loop for a reader approaching the file fresh.

Fix (optional): add a one-paragraph note in `CONTRIBUTING.md` (or whichever doc currently covers contributor workflows) pointing at `scripts/dependencies/<name>.ts` as the durable home for non-obvious pin justifications.

S6. **`MANIFEST_HEADER` source string and the live `dependencies.yaml` header are duplicated and will drift on the next edit** — `scripts/lib/dependencies.ts:167-176` and `pkg/rancher-desktop/assets/dependencies.yaml:1-9` [Claude Opus 4.7]

The header text is a literal in the source and an identical copy in the YAML. They match today; future edits to one without the other will desync until the next `rddepman` bump rewrites the YAML. Low risk because every bump regenerates the file, but readers between bumps see whichever copy was last written.

Fix (optional): accept the drift and rely on the next `rddepman` bump to refresh the YAML (current behaviour), or have `writeDependencyManifest` always rewrite the header even when the rest of the file is unchanged. The first option costs nothing.

## Design Observations

### Concerns

**D1. Two parallel structures (`DependencyVersions` and `DependencyManifest`) for the same data** *(in-scope)* [Claude Opus 4.7]

Carry-over from Round 6 D2 / mook-as inline review. `readDependencyManifest` produces `DependencyManifest` and `readDependencyVersions` flattens compound entries to a `name → version` map for `lint-go.ts` and `docker-cli-monitor.ts`. A single structure indexed by `name` with a `version` getter would let both callers share a model and remove the keep-them-in-sync risk. The two callers could read `manifest[name].version` directly without much ceremony. Worth pursuing as a follow-up if not blocking.

**D2. `MobyOpenAPISpec`'s composite version is a workaround for fetching from a mutable `master` URL** *(future)* [Claude Opus 4.7]

The deeper issue is that the source URL (`raw.githubusercontent.com/moby/moby/master/api/docs/...`) is mutable; the PR's pinned-commit pattern is correct but it is the only dependency that needs it. Vendoring the moby spec into the repo (one yaml file) would eliminate the API call to resolve commits, the schema special case, and the validation surface — at the cost of someone remembering to refresh it. The current design is a fine compromise; listing here as a future option.

### Strengths

**D3. Round 6 backlog substantially cleared** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

Round 6's S1 (`7z e -y`), S2 (`Candidate` type for `MobyOpenAPISpecVersion`), S3 (5 callsites with "upstream does not publish a checksum file" comments), S6 (EACCES catch dropped), S12 (lowercase-normalisation JSDoc), and S13 (manifest-header migration note) all landed cleanly. The Round 7 diff is six small files / 39 net lines and reads as deliberate cleanup rather than design churn.

**D4. `Sha256Checksum` template-literal type plus `parseSha256Checksum` normalisation gives a single chokepoint** *(in-scope)* [Claude Opus 4.7]

The brand type narrows what counts as a valid checksum, the parser rejects malformed input and normalises to lowercase, and the new JSDoc on both points at the comparison contract in `download()`. Good fence-post hygiene; a future test or fixture that constructs `Sha256Checksum` outside `parseSha256Checksum` is the only remaining regression hazard.

**D5. Compute-checksums-before-branching keeps failed bumps from leaving orphan branches** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

`scripts/rddepman.ts:291-294` runs `dep.getChecksums(latestVersion)` before any `git checkout -b`, so an upstream-mismatch failure aborts the bump before any git state changes. The right ordering, and a clear improvement over the pre-PR shape.

**D6. `downloadAndHash` keeps mismatched bytes for triage** *(in-scope)* [Gemini 3.1 Pro]

When `downloadAndHash` rejects on upstream-checksum mismatch, the bytes survive in `os.tmpdir()/rddepman-mismatch-${basename}` rather than disappearing. Vastly accelerates root-cause analysis when a CDN returns HTML or a release contains the wrong artifact. (See S2 for the basename-collision wrinkle that limits the convenience.)

## Review Process Notes

### Repo context updates

- **[repo] Verify a flagged class actually inherits the method before flagging "missing override".** When a finding claims that a subclass should override a base method, walk the inheritance chain and confirm the method is on it. In this codebase, `VersionedDependency` is the abstract base; `GitHubDependency extends VersionedDependency` and adds `versionToTagName`; concrete classes pick one to extend. A class that extends `VersionedDependency` directly does not see methods that live only on `GitHubDependency`. Reason: a Round-7 Important finding claimed a class should override `versionToTagName` to handle its object-valued version, but the class extends the base and never inherits the method at all — the entire premise was structurally false. How to apply: before flagging "missing override," grep the class declaration line for `extends`, then confirm the named method is defined on (or above) that class in the hierarchy.

### Important Issues

**I1. `download()` cache check hashes only `.unsigned` for codesigned binaries; tampering with `destPath` is undetected** — `scripts/lib/download.ts:69-94`

```ts
const hashTarget = options.codesign ? `${ destPath }.unsigned` : destPath;

if (!overwrite) {
  try {
    await fs.promises.access(destPath, access);
    if (expectedChecksum) {
      try {
        const actualChecksum = await hashFile(hashTarget, 'sha256');
        if (actualChecksum === expectedChecksum) {
          console.log(`${ destPath } already exists with expected checksum, not re-downloading.`);
          return;
        }
```

For codesigned downloads the cache decides whether to skip based on the digest of `${destPath}.unsigned` (line 69). If the signed binary at `destPath` is replaced with a corrupt or malicious file but `.unsigned` is left intact, the re-download is skipped (`access(destPath)` succeeds at line 73, and the hash of `.unsigned` matches at line 76-79). The intent of the verified-cache path is "the bytes the user has match the digest" — for codesigned files, this is no longer guaranteed.

In practice the source tree is developer-trusted, so the practical likelihood is low; flagging because it inverts the strongest property the new code claims to provide. Pre-PR there was no checksum verification at all, so this is a net improvement, but a partial one.

Fix: hash both `destPath` (sans signature, e.g., `codesign --remove-signature` is too invasive — alternative below) and `.unsigned`, OR record the digest of the signed binary as a second cache key, OR drop `.unsigned` after a successful sign and always re-download on the next run when codesign is in play. The simplest move is to delete `.unsigned` once codesign succeeds and treat the cache-hit shortcut as inapplicable for codesigned binaries — they have to re-download every time anyway under the current logic if `.unsigned` is missing, so deleting `.unsigned` only changes the cost from "always-skip" to "always-re-download," which matches the implicit promise.

(important, regression)

**I2. `MobyOpenAPISpec.getAvailableVersions` returns objects with `commit: ''` placeholders that flow through `rcompareVersions` and `printable` if anyone calls them outside `latestVersion`** — `scripts/dependencies/moby-openapi.ts:106-127`

```ts
async getAvailableVersions(): Promise<MobyOpenAPISpecVersion[]> {
  ...
  return candidates.map(c => ({ ...c, commit: '' }));
}

override get latestVersion(): Promise<MobyOpenAPISpecVersion> {
  return (async() => {
    const winner = await super.latestVersion as MobyOpenAPISpecVersion;
    return { ...winner, commit: await this.latestCommitForApiVersion(winner.apiVersion) };
  })();
}
```

The contract for `getAvailableVersions` advertises full `MobyOpenAPISpecVersion` objects; in practice the `commit` is empty until `latestVersion` rewrites it (only on the winner). Anyone who treats the items in `getAvailableVersions` as full versions, or who indexes by `versionToTagName(v)`/`printable(v)`, will see `v[object Object]`/empty commits and produce broken URLs (e.g., `https://raw.githubusercontent.com/moby/moby//api/docs/v1.54.yaml` if the empty commit slipped into a URL). The current callers happen to be safe (rddepman's `getBody` early-returns for non-string versions; `unreleased-change-monitor.ts` does not list this dependency), so this is latent.

Fix: have `getAvailableVersions` resolve the real commit per candidate, OR define a typed `Candidate` shape that omits `commit` and only widen to `MobyOpenAPISpecVersion` on the winner, so any future caller that touches a candidate fails at the type level rather than at runtime. The author's comment explains the resolved-on-winner choice as a cost-of-API-calls tradeoff; tightening the type without changing the call pattern preserves that.

**I3. The base `versionToTagName(version: Version): string { return \`v${ version }\` }` produces `v[object Object]` for object-valued versions** — `scripts/lib/dependencies.ts:552-554`

```ts
versionToTagName(version: Version): string {
  return `v${ version }`;
}
```

The Version union now includes both `AlpineLimaISOVersion` and `MobyOpenAPISpecVersion`. The base implementation only works for string versions; subclasses that take object versions must override. AlpineLimaISO does (line 241). MobyOpenAPISpec does not. The two existing callers that touch object-valued versions (rddepman `getBody`, unreleased-change-monitor) happen to early-return or not include MobyOpenAPISpec, but the next caller someone adds will silently produce a bad URL.

```diff

-   return `v${ version }`;
- }
+ versionToTagName(version: Version): string {
+   return `v${ VersionedDependency.versionString(version) }`;
+ }
```

For MobyOpenAPISpec the resulting tag (`v1.54`) is not the actual moby release tag, but it matches what rddepman writes into branch names today, so this is consistent with current behavior — and it kills the `[object Object]` failure mode.

(important, gap)

**I4. `downloadTarGZ` / `downloadZip` always re-extract on every postinstall, even when the destination is correct** — `scripts/lib/download.ts:202-247, 258-292`

The previous behavior short-circuited extraction when `destPath` already existed and was accessible. The new behavior keeps a checksum-verified cache of the archive at `${destPath}.tar.gz`/`${destPath}.zip` but always reruns `tar` / `unzip` and `copyFile`/`chmod` on `destPath`. This costs an extra extraction per dependency per postinstall and, more meaningfully, will fail with `EBUSY`/`EPERM` (Windows) or `ETXTBSY` (Linux) when the consuming binary is held open by Rancher Desktop or another shell. The comment at lines 218-223 / 274-279 acknowledges the failure mode and instructs the user to "stop the consuming process and retry."

The checksum cache at `download()` already enforces "the archive bytes match." Extraction is deterministic given those bytes, so re-extraction adds correctness only if `destPath` could drift independently. If that drift case is the motivation, hashing `destPath` (or stamping it the way `RancherDashboard` does — `tools.ts:404-451`) achieves the same goal without paying the extraction cost on every run.

Fix: re-introduce a cheap cache check, e.g., a `.source-sha256` stamp file alongside `destPath` (mirroring the `RancherDashboard` pattern at `tools.ts:409-415,449`), so postinstall runs without held-open binaries are still no-ops on cache-hit. Alternatively, document the new contract loudly in the README/CONTRIBUTING; the comment-only acknowledgement does not reach a developer who has not yet hit the failure.

(important, regression)

## Design Observations

**Concerns**

- **Two parallel structures (`DependencyVersions` and `DependencyManifest`) for the same data** *(in-scope)*. `mook-as` raised this in review (https://github.com/rancher-sandbox/rancher-desktop/pull/10176#discussion_r3149342472) and there's no follow-up commit. The conversion path `readDependencyManifest` → `readDependencyVersions` flattens compound entries to versions for two callers (`lint-go.ts`, `docker-cli-monitor.ts`). A single structure indexed by `name` with a `version` getter would let both callers share a model and remove the keep-them-in-sync risk. Worth pursuing in a follow-up if it's not a blocker for this PR.

- **`MobyOpenAPISpec`'s composite version is a workaround for fetching from a `master` URL** *(future)*. The deeper issue is that the source URL (`raw.githubusercontent.com/moby/moby/master/api/docs/...`) is mutable; the PR's pinned-commit pattern is correct but it's the only dependency that needs it. If the project is willing, the moby spec could be vendored into the repo (one yaml file) and treated as part of the codebase rather than an external dependency. Eliminates the API call to resolve commits, the schema special case, and the validation surface. Cost: someone has to remember to refresh it. The current design is a fine compromise; listing here as a future option.

**Strengths**

- The `Sha256Checksum` template-literal type plus `parseSha256Checksum` normalization gives a single chokepoint for "what counts as a valid checksum," and the `lookupChecksum` helper localizes the prefix-stripping responsibility. Good fence-post hygiene.
- Caching the manifest read at the module level (`depManifestCache`) and invalidating after `updateManifest` keeps the rddepman flow predictable; the explicit "read from disk" comment in `updateManifest` is a clear signal of intent.
- Computing checksums *before* branching in rddepman (`scripts/rddepman.ts:291-294`) is exactly the right ordering — an upstream-mismatch failure now aborts before any git state changes, avoiding stranded branches.

## Testing Assessment

No new unit tests accompany the change. The high-impact untested scenarios, in order:

1. **`fetchUpstreamChecksums` parser** — both GNU and BSD lines, the basename-aliasing branch, the algorithm-mismatch (sha256 sidecar pointed at a sha512 URL) detection, and the empty-result throw. The parser is the security boundary for the bump-time cross-check; it deserves tests against synthetic input fixtures.

5. **`readDependencyManifest`** error paths — missing `version`, non-object/non-string version values.

A few of these (parser, parseSha256Checksum) are pure functions and trivial to test; tests would catch regressions on the next refactor cheaply.
