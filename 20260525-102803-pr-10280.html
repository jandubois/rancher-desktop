Executive Summary ¶

Round 8 rebases Round 7's commit on current main; the PR diff is byte-identical to Round 7. Round 7's Important (delegated state never re-probes after the engine releases ownership) and four Suggestions remain unaddressed. Codex surfaces a sibling concern: ports procnet owns are never re-evaluated when the engine chain later appears. Gemini raised one Critical that re-states an already-acknowledged limitation (wsl-proxy re-notify), downgraded here. Five new minor Suggestions cover error-wrap symmetry, predicate clarity, comment hygiene, performance under burst container start, and a shadowed loop variable.

Suggestions ¶

S1. engineDNATChains is a mutable package-level var Claude¶
src/go/guestagent/pkg/procnet/scanner_linux.go

// loopbackIP is the host-local address procnet redirects --network=host
// loopback listeners to.
const loopbackIP = "127.0.0.1"

// engineDNATChains is the closed list of container-engine PREROUTING
// chains procnet probes for an existing DNAT to a host port. Adding a
// third engine (e.g. a future containerd-native portmap implementation)
// requires extending this list, or engineChainManagesPort silently
// returns false for ports the new engine handles -- recreating the
// original shadow-DNAT bug. NewProcNetScanner logs the active list at
// startup so a support-bundle reader sees what procnet considered.
var engineDNATChains = []string{"CNI-HOSTPORT-DNAT", "DOCKER"}

// portAlreadyExposedSubstring is the substring tracker.Add returns when
// another component has already exposed the port (typically the
// containerd or docker events handler on /tasks/start). The string
// originates in the /services/forwarder/expose API response and survives

Go has no const slice, so a var is the only option for a multi-element list. The docstring treats this as a closed list, but any code in the same package can still mutate it at runtime. Fix: replace with a function returning a fresh slice each call (engineDNATChainList() []string), or declare as a fixed-length array ([2]string) so the slice cannot grow. Skip the change if "swap in a fake chain list" testability is anticipated.

// cases it covers: an engine owns the proxy (delegate), or procnet owns

The [^\n]*? clause encodes the invariant "each rule occupies one line." Real iptables --list <chain> --numeric output respects this. S8. engineChainManagesPort shells out twice per newly-observed port Gemini¶
src/go/guestagent/pkg/procnet/scanner_linux.go
// the port to a later scan; mirrors the precedent at
// pkg/iptables/iptables.go that treats exit status 4 as transient.
// Shells out to iptables directly; end-to-end behavior is covered by
// published-ports.bats. Distinguishing absent from transient errors
// is unit-tested through isIptablesRuleAbsent.
func engineChainManagesPort(ctx context.Context, protocol, port string) (bool, error) {
	for _, chain := range engineDNATChains {
		cmd := iptablesCommand(ctx,
			"--wait", "2",
			"--table", "nat",
			"--list", chain,
			"--numeric")
		out, err := cmd.Output()
		if err != nil {
			if isIptablesRuleAbsent(err) {
				continue
			}
			return false, fmt.Errorf("iptables --list %s: %w", chain, wrapExitError(err))
		}
		if dnatChainContainsPort(string(out), protocol, port) {
			return true, nil
		}
	}
	return false, nil
}

// dnatChainContainsPort reports whether `iptables --list <chain>
// --numeric` output references port as a DNAT destination for protocol.
// It matches both the single-port form (`tcp dpt:<port>`) and the
// multiport list form (`tcp multiport dports <port>[,…]`). Word

For each newly-observed port, the helper spawns iptables --list CNI-HOSTPORT-DNAT --numeric and iptables --list DOCKER --numeric — two subprocesses contending on the xtables lock with --wait 2. During a burst (docker-compose up with many published ports), the per-tick fork count scales with new-port count and stretches the tick. Fix: dump the NAT table once per tick with iptables-save -t nat, parse it once, and answer all engine-chain queries against the in-memory snapshot. Skip if real-world load profiles do not hit this; the cost is added parsing code.

Design Observations ¶

Strengths

  • The injectable portStateTracker + fake iptablesRunner test design continues to pay off: every state transition has a direct unit test, and adding a new transition or regression case touches one fixture rather than refactoring shared scaffolding. (in-scope) Claude Codex
  • Switching APITracker.Add/Remove/RemoveAll to errors.Join over separate sentinels fixes a pre-existing class of silently-dropped errors (Expose failures swallowed by a concurrent wsl-proxy failure, and vice versa). The new tests TestAddBothFailuresAreDistinguishable and TestRemoveAllBothFailuresAreDistinguishable lock the contract end-to-end. (in-scope) Claude
  • removeReportedProxyDestroyed centralises the wsl-proxy-only vs host-switch-still-bound classification used at three sites (exposeNew engine delegation, retryAppend engine takeover, retireDisappeared default branch). TestRemoveReportedProxyDestroyed pins the predicate so a future refactor cannot silently desync the three call sites. (in-scope) Claude
  • The 86-line portState block comment doubles as a state-machine specification covering every transition, the cleanup-rule decision matrix, and the partial-failure rationale. Future maintainers reading the file have the design they need without leaving the file. (in-scope) Claude Gemini

Concerns

  • Proxy ownership is inferred through side effects across procnet, docker, and containerd — (future) Codex: each handler reads iptables chain rules, tracker keys, and "proxy already running" strings to figure out who owns the proxy. A durable claim protocol in tracker (an explicit Claim/Release pair keyed on container ID and synthetic ID, with a queryable owner) would erase several race-specific branches in all three call sites and make handoff testable without duplicating engine-timing in procnet.
  • tracker.Add/Remove's new joined-error contract is documented only inside apitracker.go — (in-scope) Claude: procnet now depends on ErrExposeAPI/ErrUnexposeAPI/ErrWSLProxy as distinguishable sentinels, but the engine handlers in containerd/events_linux.go:123-128 and docker/events.go:100-103 still log and continue. Promoting the contract into the Tracker interface docstring (and exposing the sentinels from the tracker package) would force those callers to confront the partial-failure semantics on the next touch.

Testing Assessment ¶

Coverage of the documented state machine is comprehensive. Notable gaps for next round:

  1. No coverage for I1 (either lifecycle): a delegated port whose engine chain disappears while the listener persists, and a successful-Add port whose engine chain appears later then loses its Add race. Both transitions could be driven through Tick with a fake engineManaged toggle (true → false and false → true) that the existing fakeIptables fixture trivially supports. The author may want to add both tests regardless of whether the fix lands, since either outcome (re-probe or accept-as-trade-off) deserves a regression test.
  2. fakeIptables does not model preroutingHasLoopbackRule's idempotency probe — acknowledged in port_state_tracker_linux_test.go:951-959. Append-on-rule-present sequences (a procnet rule survives across guestagent restarts and the same listener reappears in-session) are not unit-testable until the fake mirrors production's short-circuit.
  3. End-to-end iptables-nft chain-terminating behavior is exercised by published-ports.bats but not asserted directly. The scanner_linux.go:158-169 docstring flags this as a foundation-of-the-design risk: a future WSL2 kernel that defaults to iptables-legacy would silently invalidate the design. Adding iptables-save -t nat | grep '^:PREROUTING ACCEPT' plus a kernel-version check to the bats setup would catch a regression on the next WSL2 kernel bump.
  4. Mixed-binding edge case: the scanner only emits non-loopback bindings for --network=host listeners, and the engine never manages those, so anyEngineBinding's loopback-only probe is correct in practice. A two-line invariant comment on addEntryToPortMap would prevent a future contributor from emitting an engine-managed non-loopback binding by accident.

Acknowledged Limitations ¶

  • handleAlreadyExposed RESUME branch cannot re-notify wsl-proxy after a partial-Add — port_state_tracker_linux.go:73-77, 350-364. apiForwarder.Expose succeeded on a prior tick (gvisor proxy bound, portStorage[synthetic] populated) and wsl-proxy.Send failed; the next tick's tracker.Add returns proxy already running, RESUME installs the iptables rule and resumes ownership, but the wsl-proxy notification is not retried because the Tracker interface exposes no idempotent re-notify hook. Windows-side traffic for that port stays disconnected until RD restart. *Gemini raised this as Critical+regression this round; verified as the same well-documented limitation referenced in the Round 7 review and pinned by the portState docstring.*
  • Engine-delegation tracker.Remove can tear down the only host-switch proxy under a Local-string collision — port_state_tracker_linux.go:240-269, pinned by TestEngineDelegationAcceptsLeakWhenRemoveFails. Trigger: stability-gate bypass under heavy load PLUS transient wsl-proxy.Send failure PLUS HostIP=127.0.0.1 publish (or non-admin install). Documented as intentional trade-off.
  • Stability gate bypass under heavy load — the two-scan gate filters the OCI hook reservation socket on normal load; a heavily-loaded system that takes >3s between procnet polls could see the reservation socket twice and act on it. Documented as the underlying limitation the retryAppend/engine-takeover paths ride along with.
  • In-session Delete leak — port_state_tracker_linux.go:282-285,305-307 document that the Delete path has no retry, so a transient iptables-Delete failure leaks a procnet rule until RD restart. The I1 reconciliation in exposeNew's if-managed branch self-heals this whenever the engine later republishes the same host port.
  • iptables-nft kernel default assumption — scanner_linux.go:158-169 documents that the entire design depends on iptables-nft's PREROUTING chain-terminating-on-DNAT behavior, with no live assertion of the assumption.
  • engineDNATChains closed list — scanner_linux.go:55-62 and engineChainManagesPort docstring at :227-231 document that adding a third engine without extending the list silently re-introduces the shadow-DNAT bug. NewProcNetScanner logs the active list at startup for diagnosability.
  • Port-range regex gap — scanner_linux.go:269-279 documents that dpts:LO:HI and multiport-embedded ranges are intentionally not matched. RD's shipped engines emit per-port rules, so this is theoretical for shipped engines but real for hand-rolled iptables setups against the same chains.
  • Engine chain visibility ≠ engine has a host-switch proxy — port_state_tracker_linux.go:93-101 documents that the CNI chain is installed before the engine's portTracker.Add, so a chain-present probe does not prove the engine owns a proxy. Acknowledged in the retryAppend engine-takeover comment.
  • Cross-session leftover Delete in handleAlreadyExposed's RESUME branch — port_state_tracker_linux.go:365-376 documents the choice not to reconcile because the rule's origin is ambiguous (cross-session leftover vs deliberately retained from a Remove-failure-on-Unexpose). The I1 reconciliation in exposeNew's if-managed branch covers the common cross-session case.
