## Executive Summary

This PR fixes the Vue 3 `StringList` regression that broke the no-proxy list UI, adds validation for noproxy entries (IPs, CIDRs, domains, wildcards), and extends the moproxy backend to support domain-based proxy bypass through `policy.rules`. The implementation cleanly splits IP entries (iptables) from domain entries (moproxy policy), with 165 new test lines covering validation. Two initd script issues need fixing: IPv6 noproxy defaults never activate because the backend always writes `MOPROXY_NOPROXY6=""` and the shell expansion uses `-` instead of `:-`, and the new `ip6tables` calls can abort the entire service if IPv6 kernel support is unavailable.

## Important Issues

1. **IPv6 noproxy defaults never activate** — `moproxy.initd:26` + `wsl.ts:802` (important, regression) [Claude Opus 4.6]

```bash
# moproxy.initd line 26
: ${noproxy6_rules:=${MOPROXY_NOPROXY6-"::1/128,fc00::/7,fe80::/10"}}
```

```typescript
// wsl.ts lines 800-803
await this.modifyConf('moproxy', {
  MOPROXY_NOPROXY:  ipv4Entries.join(','),
  MOPROXY_NOPROXY6: ipv6Entries.join(','),
});
```

The default noproxy list in `config/settings.ts` contains only IPv4 entries, so `ipv6Entries` is always empty, producing `MOPROXY_NOPROXY6=""` in `/etc/conf.d/moproxy`. The inner expansion on line 26 uses `-` (substitute only when *unset*), not `:-` (substitute when unset *or empty*). Since the variable is set to empty, the defaults `::1/128,fc00::/7,fe80::/10` never activate. All IPv6 TCP traffic on ports 80/443 — including loopback and link-local — gets redirected through moproxy.

The `:-` to `-` change on line 25 for `MOPROXY_NOPROXY` is intentionally correct: when all entries are domains, the IPv4 iptables list should be empty. But `MOPROXY_NOPROXY6` has a different invariant — no user-facing mechanism adds IPv6 entries, so falling back to sane defaults is always right.

Fix: use `:-` for `MOPROXY_NOPROXY6`:

```diff
-: ${noproxy6_rules:=${MOPROXY_NOPROXY6-"::1/128,fc00::/7,fe80::/10"}}
+: ${noproxy6_rules:=${MOPROXY_NOPROXY6:-"::1/128,fc00::/7,fe80::/10"}}
```

2. **Service startup fails if ip6tables is unavailable** — `moproxy.initd:66-74` (important, gap) [Gemini 3.1 Pro]

```bash
enable_redirection_to_moproxy_chain() {
    local chain="$1"
    local cmd
    for cmd in iptables ip6tables; do
        if ! redirect_to_moproxy_chain "$cmd" check "$chain" &> /dev/null; then
            redirect_to_moproxy_chain "$cmd" append "$chain"
        fi
    done
}
```

This PR adds `ip6tables` throughout the initd script (the old version used only `iptables`). If `ip6tables` fails — for example, when IPv6 is disabled system-wide in WSL2 and the kernel module is unavailable — the `append` command at line 71 returns non-zero. Because `enable_redirection_to_moproxy_chain PREROUTING` is the last call in `enable()` (line 109), and `enable()` is called from `start_post()` (line 120), a non-zero exit propagates to OpenRC, which marks the service as failed. This breaks the proxy entirely, even for IPv4.

The same risk applies to `create_moproxy_chain` (line 88), `add_noproxy_rules` (line 62), `add_redirect_rule ip6tables` (line 107), and `delete_moproxy_chain` (line 94).

Fix: add `|| true` to ip6tables calls or add `return 0` at the end of functions that loop over both commands:

```diff
 enable_redirection_to_moproxy_chain() {
     local chain="$1"
     local cmd
     for cmd in iptables ip6tables; do
         if ! redirect_to_moproxy_chain "$cmd" check "$chain" &> /dev/null; then
             redirect_to_moproxy_chain "$cmd" append "$chain"
         fi
     done
+    return 0
 }
```

2. **Missing `eend` in `reload()`** — `moproxy.initd:127-137` (suggestion, gap) [Gemini 3.1 Pro]

```bash
reload() {
    # SIGHUP reloads the proxy server list only, not the --policy file.
    # Domain noproxy rules require a full stop+start cycle.
    ebegin "Reloading ${name}"
    start-stop-daemon --signal HUP --pidfile "$pidfile"

    ip6tables --table nat --flush MOPROXY
    add_noproxy_rules
    add_redirect_rule iptables
    add_redirect_rule ip6tables
}
```

The `reload` function calls `ebegin` at line 130 but never calls `eend $?`, leaving OpenRC status output malformed. The backend no longer calls `reload` (it uses `stop+start`), but `reload` remains in `extra_started_commands` for manual use.

Fix:

```diff
     add_redirect_rule iptables
     add_redirect_rule ip6tables
+    eend $?
 }
```

3. **Semantic noproxy duplicates not detected** — `settingsValidator.ts:720` (suggestion, gap) [Claude Opus 4.6]

`*.example.com` and `.example.com` are semantically equivalent — both bypass `example.com` and its subdomains. `findDuplicates` uses string comparison, so both pass as non-duplicates. After `stripNoproxyPrefix` strips the prefixes, both produce `dst domain example.com direct` in the policy file. Duplicate moproxy rules are harmless, so this is cosmetic. Low priority.

4. **Unquoted `$3` in `redirect_to_moproxy_chain`** — `moproxy.initd:44` (suggestion, gap) [Gemini 3.1 Pro]

```bash
redirect_to_moproxy_chain() {
    run_iptables "$1" --$2 $3 --protocol tcp --match multiport --dports "${ports_redirected}" --jump MOPROXY
}
```

The `$3` parameter (chain name) is unquoted. It receives only hardcoded values (`OUTPUT`, `PREROUTING`), so this is safe in practice but inconsistent with the quoting of `$1` and `${ports_redirected}`.

Fix: `run_iptables "$1" "--$2" "$3" --protocol ...`

#### Important Issues

1. **IPv6 noproxy defaults are dead code** — `moproxy.initd:26` + `wsl.ts:802` (important, regression)

```bash
# moproxy.initd line 26 — defaults never activate
: ${noproxy6_rules:=${MOPROXY_NOPROXY6-"::1/128,fc00::/7,fe80::/10"}}
```

```typescript
// wsl.ts lines 800-803 — always writes MOPROXY_NOPROXY6, even when empty
await this.modifyConf('moproxy', {
  MOPROXY_NOPROXY:  ipv4Entries.join(','),
  MOPROXY_NOPROXY6: ipv6Entries.join(','),
});
```

The default noproxy list in `config/settings.ts` (line 159) contains only IPv4 entries. So `ipv6Entries` is always empty for a default configuration, producing `MOPROXY_NOPROXY6=""` in the conf file. `modifyConf` (line 1181) always writes this key to `/etc/conf.d/moproxy`.

In the initd script (line 26), the inner expansion `${MOPROXY_NOPROXY6-"defaults"}` uses `-` (not `:-`). As confirmed by testing, `-` only substitutes the default when the variable is *unset* — an empty string is still "set". Since the conf file always sets `MOPROXY_NOPROXY6=""`, the initd defaults `::1/128,fc00::/7,fe80::/10` never take effect.

Consequence: all IPv6 TCP traffic on ports 80/443 — including to `::1` (loopback) and `fe80::/10` (link-local) — gets redirected through moproxy. This is unlikely to cause immediate breakage in most WSL2 environments (IPv4 predominates), but it's wrong and could bite users with IPv6-enabled local services.

The `:-` to `-` change for `MOPROXY_NOPROXY` (line 25) is intentionally correct — when all entries are domains, the IPv4 list should truly be empty. But `MOPROXY_NOPROXY6` has a different invariant: the user has no default IPv6 entries, so falling back to sane IPv6 defaults is always the right behavior.

Fix (either approach):

```diff
 # Option A: use :- for MOPROXY_NOPROXY6 so empty falls back to defaults
-: ${noproxy6_rules:=${MOPROXY_NOPROXY6-"::1/128,fc00::/7,fe80::/10"}}
+: ${noproxy6_rules:=${MOPROXY_NOPROXY6:-"::1/128,fc00::/7,fe80::/10"}}
```

```diff
 # Option B: don't write MOPROXY_NOPROXY6 when empty (backend side)
+    const conf: Record<string, string> = { MOPROXY_NOPROXY: ipv4Entries.join(',') };
+    if (ipv6Entries.length > 0) {
+      conf.MOPROXY_NOPROXY6 = ipv6Entries.join(',');
+    }
-    await this.modifyConf('moproxy', {
-      MOPROXY_NOPROXY:  ipv4Entries.join(','),
-      MOPROXY_NOPROXY6: ipv6Entries.join(','),
-    });
+    await this.modifyConf('moproxy', conf);
```

Option A is simpler and self-contained. Option B requires also handling the case where MOPROXY_NOPROXY6 was set by a previous call and needs clearing — `modifyConf` merges but doesn't remove keys. I'd recommend Option A.

1. **Semantic noproxy duplicates not detected** — `settingsValidator.ts:720` (suggestion, gap)

```typescript
const duplicateValues = this.findDuplicates(desiredValue);
```

`*.example.com` and `.example.com` are semantically equivalent (both mean "bypass example.com and subdomains"), but `findDuplicates` does string comparison, so they pass as non-duplicates. ### Important Issues

1. **Service failure risk if `ip6tables` is disabled** — `pkg/rancher-desktop/assets/scripts/moproxy.initd:68-74` (important, gap)

```bash
enable_redirection_to_moproxy_chain() {
       local chain="$1"
       local cmd
       for cmd in iptables ip6tables; do
               if ! redirect_to_moproxy_chain "$cmd" check "$chain" &> /dev/null; then
                       redirect_to_moproxy_chain "$cmd" append "$chain"
               fi
       done
}
```

If `ip6tables` fails (e.g. if IPv6 is disabled system-wide in WSL2 and the kernel module is unavailable), the `append` command returns a non-zero exit code. Because this `for` loop is the last statement in `enable_redirection_to_moproxy_chain`, and `enable_redirection_to_moproxy_chain PREROUTING` is the last statement in `enable()`, the `enable` function will return non-zero. This causes the `start_post` OpenRC hook to fail, marking the moproxy service as failed and completely breaking the proxy, even for IPv4.

Fix: Add `return 0` at the end of the function to ensure a partial IPv6 failure doesn't abort the entire service startup.

```diff
 enable_redirection_to_moproxy_chain() {
        local chain="$1"
        local cmd
        for cmd in iptables ip6tables; do
                if ! redirect_to_moproxy_chain "$cmd" check "$chain" &> /dev/null; then
                        redirect_to_moproxy_chain "$cmd" append "$chain"
                fi
        done
+       return 0
 }
```

### Suggestions

1. **Unquoted parameter in `run_iptables` call** — `pkg/rancher-desktop/assets/scripts/moproxy.initd:43` (suggestion, gap)

```bash
redirect_to_moproxy_chain() {
       run_iptables "$1" --$2 $3 --protocol tcp --match multiport --dports "${ports_redirected}" --jump MOPROXY
}
```

The parameter `$3` (the chain name, e.g. `OUTPUT` or `PREROUTING`) is unquoted. While it currently only receives safe, hardcoded values, it is best practice to quote all variables.

Fix: Quote the `$3` argument.

```diff
-       run_iptables "$1" --$2 $3 --protocol tcp --match multiport --dports "${ports_redirected}" --jump MOPROXY
+       run_iptables "$1" --$2 "$3" --protocol tcp --match multiport --dports "${ports_redirected}" --jump MOPROXY
```

2. **Missing `eend` in `reload()`** — `pkg/rancher-desktop/assets/scripts/moproxy.initd:127-137` (suggestion, gap)

```bash
reload() {
       # SIGHUP reloads the proxy server list only, not the --policy file.
       # Domain noproxy rules require a full stop+start cycle.
       ebegin "Reloading ${name}"
       start-stop-daemon --signal HUP --pidfile "$pidfile"
       iptables --table nat --flush MOPROXY
       ip6tables --table nat --flush MOPROXY
       add_noproxy_rules
       add_redirect_rule iptables
       add_redirect_rule ip6tables
}
```

The `reload` function calls `ebegin "Reloading ${name}"` but never calls `eend $?`. This leaves the OpenRC status output malformed if `reload` is invoked manually. 

Fix: Add `eend $?` at the end of the `reload` function.

### Design Observations

**Strengths**
- Changing the default expansion in `moproxy.initd` from `:-` to `-` is a subtle but excellent move. It ensures that if the user explicitly clears the noproxy list in the UI, `MOPROXY_NOPROXY=""` is respected rather than falling back to the default list.
- Shifting from `reload` to `stopService` followed by `startService` in `wsl.ts` accurately accounts for moproxy's inability to reload the `--policy` file via `SIGHUP`.
