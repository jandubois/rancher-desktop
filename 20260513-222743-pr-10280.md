## Executive Summary

This PR extracts the `/proc/net` scanner's per-port heuristic stack out of an
inlined loop in `ForwardPorts` into a unit-tested `portStateTracker` state
machine, behind injectable `tracker.Tracker` and `iptablesRunner` seams. It
also wires a `test:unit:guestagent` target into `test:unit`, closing a real CI
gap where the `guestagent` module's tests — including those added in the
superseded #10262 — never ran.

The extraction is the right call: it turns "fix one race, expose another" into
a directly testable state machine, the `portState` doc comment is an unusually
clear spec, and the test suite walks every transition with fakes. All three
reviewers agree the structure is sound and found no Critical issues.

Two Important issues remain, and both sit in the cleanup/retry paths that are
net-new in this PR. **I1** (regression, confirmed against the merge-base):
`retireDisappeared` now deletes the loopback DNAT rule even when
`tracker.Remove` returned an error — the pre-PR loop did `continue` and kept
the rule. A failed host-switch unexpose therefore leaves a stale proxy with no
DNAT, and the reappearance path then misclassifies the port as engine-managed,
so a `--network=host` loopback listener stays unreachable until RD restart.
**I2** (gap): `retryAppend`'s engine-takeover branch releases tracker ownership
without issuing the idempotency-gated Delete, so if the prior failed Append had
actually committed its rule, that rule shadows the engine's chain — the exact
bug this PR exists to fix. Both fixes are a few lines and make the paths
symmetric with code that already exists elsewhere in the same file.

Structure of findings: 0 Critical, 2 Important, 3 Suggestions.

## Important Issues

I1. **Failed `tracker.Remove` still deletes the loopback DNAT, breaking host-network ports on reappearance** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:276-291` [Codex GPT 5.5]

```go
removeErr := pst.tracker.Remove(syntheticIDFor(port))
delete(pst.added, port)
if removeErr != nil {
	log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
}
if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
	log.Errorf("/proc/net scanner deleting loopback iptable rules for portbinding: %v failed: %s", state.bindings, err)
}
```

`tracker.Remove` is not atomic. `APITracker.Remove` clears `portStorage` via a
leading `defer a.portStorage.remove(containerID)`, but the host-switch unexpose
HTTP call can fail and still return `ErrUnexposeAPI` (`apitracker.go:150-198`).
In that state the proxy may still be bound, yet `retireDisappeared` deletes
procnet's PREROUTING DNAT anyway.

This is a regression. The merge-base `ForwardPorts` did `continue` after a
failed `Remove`, leaving the DNAT rule in place:

```go
err := p.tracker.Remove(...)
if err != nil {
	log.Errorf("/proc/net scanner failed to remove port: %s", err)
	continue
}
if err = p.execLoopbackIPtablesRule(previousBindings, port, Delete); err != nil { ... }
```

The bad sequence: failed unexpose → stale proxy + no DNAT. The listener
reappears; `tracker.Add` returns `proxy already running` from the stale proxy;
because `portStorage` was cleared, `handleAlreadyExposed` sees an empty
`tracker.Get` and marks the port `delegated` (`:215-230`). No DNAT is
installed, no engine chain owns it, and a `--network=host` loopback listener
stays unreachable from the Windows host until RD restarts.

The `delete(pst.added, port)` on line 284 must stay unconditional — keeping the
marker deadlocks the Add path on reappearance — but the Delete must not run
when `Remove` failed.

Fix: skip the Delete after a failed `Remove`, restoring the merge-base contract:

```diff
 removeErr := pst.tracker.Remove(syntheticIDFor(port))
 delete(pst.added, port)
 if removeErr != nil {
 	log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
+	continue
 }
 if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
```

A retained DNAT keeps the (still-bound) proxy reachable and self-cleans on RD
restart, which the code already treats as acceptable. If retaining the rule
after a wsl-proxy-only `Remove` failure is undesirable, the larger fix is to
have `tracker.Remove` return a typed result distinguishing "host-switch
unexpose succeeded" from "may still be running" and Delete only in the former
case.

I2. **`retryAppend` engine-takeover branch can leak a shadowing PREROUTING rule** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:245-251` [Claude Opus 4.7]

```go
if managed {
	log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
	if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
		log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
	}
	pst.added[port] = portState{bindings: state.bindings, delegated: true}
	return
}
```

This branch fires when a port is in `appendFailed=true` state and the engine
chain has appeared since the failed Append. It releases the tracker entry but
never issues an iptables Delete — it assumes `appendFailed=true` implies no
procnet rule is installed.

That assumption contradicts the code's own idempotency design.
`applyLoopbackIPtablesRule`'s docstring (`scanner_linux.go:140-143`) states the
Append idempotency probe exists "because a previous tick's iptables call
committed the rule but returned non-zero." When that happens, `appendFailed` is
`true` *and* the rule exists. `retryAppend` then marks the port `delegated`;
`retireDisappeared` later drops a delegated port without a Delete (`:267-274`),
so the rogue `-A PREROUTING ... -j DNAT --to-destination 127.0.0.1:<port>` rule
shadows `CNI-HOSTPORT-DNAT` until RD restarts — precisely the bug this PR
exists to fix.

`retireDisappeared` handles the symmetric case correctly: it always calls
`applyLoopbackRules(..., Delete)` for non-delegated ports, and the Delete is
idempotency-gated by `preroutingHasLoopbackRule`, so it is a harmless no-op
when no rule exists. `retryAppend`'s engine-takeover branch is the only place
that skips it.

Fix: issue the idempotency-gated Delete before releasing ownership, so the
branch is correct whether or not the prior Append committed.

```diff
 	if managed {
 		log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
 		if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
 			log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
 		}
+		// The failed Append may have committed the rule (iptables committed
+		// but returned non-zero). Issue the idempotency-gated Delete so a
+		// leaked procnet rule cannot shadow the engine's chain.
+		if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
+			log.Errorf("/proc/net scanner deleting loopback iptable rules after engine takeover for portbinding: %v failed: %s", state.bindings, err)
+		}
 		pst.added[port] = portState{bindings: state.bindings, delegated: true}
 		return
 	}
```

Reconciliation: the raising agent labeled this a regression. Reclassified as a
gap — the merge-base has no append-retry path at all (`port_state_tracker_linux.go`
is net-new), and the pre-PR code shadowed CNI far more broadly. This is an
incompletely-closed corner of new code, not a behavior the PR broke.
Probability is lower than I1 (it needs the rare commit-but-errored iptables
shape), but the impact is the headline bug and the fix is three lines.

## Suggestions

S1. **Engine-chain probe failure in `exposeNew` logs at Error every tick, with no log-once guard** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:159-162` [Claude Opus 4.7, Gemini 3.1 Pro]

```go
managed, err := pst.anyEngineBinding(port, bindings)
if err != nil {
	log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
	return
}
```

A port whose engine-chain probe keeps failing (persistent xtables lock
contention, fork/exec failure) never reaches `pst.added`, so `exposeNew`
re-runs and logs at `Error` every ~3 s indefinitely. The PR deliberately added
the `addErrorLogged` log-once mechanism for the parallel `tracker.Add`
retry-storm (`:176-181`), and `retryAppend`'s own engine-probe failure logs at
`Debug` (`:242`) — so this path is an inconsistent gap in that same effort.

Fix: route this through the `addErrorLogged` log-once bookkeeping (loud first
error, quiet thereafter, cleared when the port is added or disappears), or drop
it to `Debug` to match `retryAppend`.

S2. **`fakeIptables` does not model installed-rule state, so the commit-but-errored interaction is untested** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux_test.go:88-140` [Claude Opus 4.7]

`fakeIptables.ApplyLoopbackRule` records the call and returns the injected
error but never tracks whether a rule is "installed."
`TestAppendRetryReleasesOwnershipWhenEngineChainAppears` therefore exercises
only the lock-contention shape of `appendFailed` (no rule installed) — it
cannot reach the commit-but-errored shape that I2 describes.

Fix: when fixing I2, give the fake a rule-presence model (Append sets it,
Delete clears it, the idempotency probe reads it) and add a regression test
asserting the engine-takeover branch issues a Delete when a rule was committed.

S3. **No regression test for a failed `Remove` followed by a "proxy already running" reappearance** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux_test.go:315-337` [Codex GPT 5.5]

`TestCleanupReleasesAddedOnRemoveFailure` lets the next `tracker.Add` succeed,
so it misses the realistic shape I1 describes: a failed unexpose leaves a stale
proxy, and the reappearance `tracker.Add` returns `portAlreadyExposedSubstring`
with an empty `tracker.Get`.

Fix: when fixing I1, add a test where `Remove` returns an unexpose error, the
listener reappears, the next `Add` returns `portAlreadyExposedSubstring`, and
`tracker.Get(syntheticID)` is empty — asserting the loopback DNAT is not
deleted (or is reinstalled) so the port stays reachable.

## Acknowledged Limitations

- **wsl-proxy failure not retried on the resume-ownership path** — `port_state_tracker_linux.go:54-59`, `:208-213`. The resume path installs the iptables rule the partial-failure path skipped but does not re-send the port mapping to wsl-proxy, because `Tracker` exposes no idempotent re-notify hook. Documented in the `portState` doc comment.
- **Undeleted PREROUTING rule persists until RD restart** — `port_state_tracker_linux.go:282-283`. The cleanup path drops the local marker unconditionally (retaining it deadlocks Add on reappearance); a stranded rule self-cleans on restart. (I1 and I2 are distinct: they cover *deleting* the rule when it should be kept, and *keeping* it when it should be deleted.)
- **Port-range iptables syntax not matched** — `scanner_linux.go:233-243`. `dpts:LO:HI` and multiport-embedded ranges are intentionally not matched; shipped engines emit per-port DNAT rules, and the `tcpRangeChain` / `multiportEmbeddedRange` fixtures lock the behavior.
- **Engine chain list closed to `CNI-HOSTPORT-DNAT` and `DOCKER`** — `scanner_linux.go:190-194`. Documented, with an explicit "revisit when adding a third engine" note.
- **English-locale stderr parsing** — `scanner_linux.go:305-316`. `isIptablesRuleAbsent` matches exact English iptables phrases; safe within RD's controlled Lima/WSL utility VMs.

## Review Process Notes

### Skill improvements

- When a code path creates or deletes one resource conditioned on the success of an operation on a *different* resource, check the partial-failure matrix: can the conditioning operation fail in a way that leaves its own state half-applied, so its return value no longer predicts whether the dependent resource should exist? Flag paths where an error from operation A is treated as "A did nothing" when A actually mutates shared state before returning that error, or where success of A is treated as "the dependent resource is now consistent" when A only partially completed. This is the shape behind a whole class of cleanup/retry bugs and is not obvious from reading either side in isolation.

### Repo context updates

- [repo] `tracker.APITracker` mutates shared state before returning errors, so its error returns do not cleanly map to "nothing happened." `Remove` clears `portStorage` via a leading `defer` even when the host-switch unexpose API fails — a non-nil error means "storage cleared, proxy may still be bound." `Add` runs `portStorage.add` and `apiForwarder.Expose` before `wslProxyForwarder.Send`, so a non-nil error can mean "storage populated and proxy exposed, wsl-proxy not notified." When reviewing `guestagent` port-forwarding code (`procnet`, `iptables`, callers of `tracker.Tracker`), check that any caller installing or removing a downstream artifact (an iptables PREROUTING rule, a local marker) keyed off `Add`/`Remove` success accounts for these partial-failure states.

### Findings

#### Critical Issues

None.

#### Important Issues

**I1. `retryAppend` engine-takeover branch can leak a shadowing PREROUTING rule — `port_state_tracker_linux.go:245-251`** *(important, regression)*

```go
if managed {
    log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
    if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
        log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
    }
    pst.added[port] = portState{bindings: state.bindings, delegated: true}
    return
}
```

This branch fires when a port is in `appendFailed=true` state and the engine chain has appeared since the failed Append. It releases the tracker entry but never issues an iptables Delete — it assumes `appendFailed=true` implies no procnet rule is installed.

That assumption contradicts the code's own idempotency design. `applyLoopbackIPtablesRule`'s docstring (`scanner_linux.go:140-143`) explicitly states the Append probe exists "because a previous tick's iptables call committed the rule but returned non-zero, or an external party wrote the same rule." When that happens, `appendFailed=true` *and* the rule exists. `retryAppend` then marks the port `delegated`, drops it on the next `retireDisappeared` pass without a Delete (`port_state_tracker_linux.go:267-273`), and the rogue `-A PREROUTING ... -j DNAT --to-destination 127.0.0.1:<port>` rule shadows `CNI-HOSTPORT-DNAT` until RD restarts — precisely the bug this PR exists to fix.

The symmetric cleanup path handles this correctly: `retireDisappeared` (`port_state_tracker_linux.go:288`) always calls `applyLoopbackRules(..., Delete)` for non-delegated ports, and the Delete is idempotency-gated by `preroutingHasLoopbackRule`, so it is a harmless no-op when no rule exists. `retryAppend`'s engine-takeover branch is the only place that skips it.

Fix: issue the idempotency-gated Delete before releasing ownership, so the branch is correct whether or not the prior Append committed.

```diff
 	if managed {
 		log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
 		if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
 			log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
 		}
+		// The failed Append may still have committed the rule (iptables
+		// committed but returned non-zero). Issue the idempotency-gated
+		// Delete so a leaked procnet rule cannot shadow the engine's chain.
+		if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
+			log.Errorf("/proc/net scanner deleting loopback iptable rules after engine takeover for portbinding: %v failed: %s", state.bindings, err)
+		}
 		pst.added[port] = portState{bindings: state.bindings, delegated: true}
 		return
 	}
```

#### Suggestions

**S1. Engine-probe failure in `exposeNew` logs at Error every tick, with no log-once guard — `port_state_tracker_linux.go:159-162`** *(suggestion, gap)*

```go
managed, err := pst.anyEngineBinding(port, bindings)
if err != nil {
    log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
    return
}
```

A port whose engine-chain probe keeps failing (persistent xtables lock contention, fork/exec failure) never reaches `pst.added`, so `exposeNew` re-runs every tick and logs at `Error` every ~3 s indefinitely. The PR deliberately added the `addErrorLogged` log-once mechanism for the parallel `tracker.Add` retry-storm (`port_state_tracker_linux.go:176-181`), and `retryAppend`'s own engine-probe failure logs at `Debug` (`port_state_tracker_linux.go:242`) — so this path is an inconsistent gap in that same effort. ### Executive Summary

This PR moves `/proc/net` forwarding into a tested state machine and adds CI coverage for the guestagent module. The main correctness issue is in the cleanup failure path: the new state machine can delete the loopback DNAT even when the host-switch proxy was not actually unexposed, then later misclassify the stale proxy as engine-owned.

### Findings

#### Critical Issues

#### Important Issues

I1. **Failed `Remove` can leave a stale proxy without the loopback DNAT** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:276-288` (important, regression)

```go
removeErr := pst.tracker.Remove(syntheticIDFor(port))
delete(pst.added, port)
if removeErr != nil {
	log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
}
if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
	log.Errorf("/proc/net scanner deleting loopback iptable rules for portbinding: %v failed: %s", state.bindings, err)
}
```

`tracker.Remove` at line 276 is not a single external operation. `APITracker.Remove` clears storage with `defer a.portStorage.remove(containerID)` at `src/go/guestagent/pkg/tracker/apitracker.go:152`, but the host-switch unexpose call can fail at `src/go/guestagent/pkg/tracker/apitracker.go:167-174` and still return `ErrUnexposeAPI` at `src/go/guestagent/pkg/tracker/apitracker.go:193-194`. In that state, `retireDisappeared` deletes procnet’s local marker at line 284 and still deletes the PREROUTING DNAT at line 288, even though the host-switch proxy may still be bound.

That creates a bad reappearance path for host-network loopback listeners. The next `tracker.Add` can return `proxy already running`; because storage was cleared, `handleAlreadyExposed` sees no synthetic entry at `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:215` and marks the port delegated at `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:230`. The stale proxy remains, but the loopback DNAT is gone, so Windows traffic no longer reaches the `127.0.0.1` listener. The current regression test for remove failure at `src/go/guestagent/pkg/procnet/port_state_tracker_linux_test.go:325-336` lets the next add succeed, so it misses the realistic `proxy already running` shape left by a failed unexpose.

Fix: do not delete the loopback rule after a failed `tracker.Remove` unless the tracker can prove the host-switch proxy was actually removed. A minimal safer change is:

```diff
removeErr := pst.tracker.Remove(syntheticIDFor(port))
delete(pst.added, port)
if removeErr != nil {
	log.Errorf("/proc/net scanner failed to remove port: %s", removeErr)
+	continue
}
if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
	log.Errorf("/proc/net scanner deleting loopback iptable rules for portbinding: %v failed: %s", state.bindings, err)
}
```

If the intent is to avoid retaining stale DNAT rules after wsl-proxy-only failures, extend `tracker.Remove` to return a typed result that distinguishes “host-switch unexpose succeeded” from “host-switch may still be running,” then delete the DNAT only in the former case. Add a unit test where `Remove` returns an unexpose error and the subsequent `Add` returns `portAlreadyExposedSubstring`.

S1. **Missing log suppression for transient engine-chain probe errors** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:153-156` (suggestion, gap)

```go
	managed, err := pst.anyEngineBinding(port, bindings)
	if err != nil {
		log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
		return
	}
```

The state machine explicitly uses `addErrorLogged` to prevent log spam when `tracker.Add` fails transiently across multiple ticks. However, if `anyEngineBinding` fails (e.g., due to prolonged xtables lock contention), the error is logged loudly as an `Error` on every 3-second tick because the port is never added to `pst.added` and the suppression marker is not set. 

Fix: Apply the same log suppression mechanism to engine-chain probe failures so that only the first error is loud, and subsequent failures remain quiet until the condition clears.
```diff
 	managed, err := pst.anyEngineBinding(port, bindings)
 	if err != nil {
-		log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
+		if pst.addErrorLogged[port] {
+			log.Debugf("/proc/net scanner engine-chain probe still failing for %s: %s", port, err)
+		} else {
+			log.Errorf("/proc/net scanner engine-chain probe failed for %s; deferring: %s", port, err)
+			pst.addErrorLogged[port] = true
+		}
 		return
 	}
```
