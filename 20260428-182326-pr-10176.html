S10. lookupChecksum returns string, throwing away Sha256Checksum¶

/**
 * Returns the stored sha256 for the given artifact as raw hex, with the
 * `sha256:` prefix stripped so it slots straight into `download()`.  Throws
 * if the artifact has no recorded checksum.
 */
export function lookupChecksum(
  context: DownloadContext,
  name: keyof DependencyVersions,
  artifactName: string,
): string {
  const entry = context.dependencies[name];

  if (!entry) {
    throw new Error(`Dependency "${ name }" is not present in ${ DEP_VERSIONS_PATH }.`);
  }
  const prefixed = entry.checksums?.[artifactName];

  if (!prefixed) {
    const available = Object.keys(entry.checksums ?? {}).sort().join(', ') || '(none)';

    throw new Error(
      `No checksum recorded for ${ name } artifact "${ artifactName }" in ${ DEP_VERSIONS_PATH }. ` +
      `Available: ${ available }`,
    );
  }

  return prefixed.slice('sha256:'.length);
}

S12. parseSha256Checksum lowercases hex without documenting the normalization Claude¶
scripts/lib/dependencies.ts

/**
 * Parses a raw string as a {@link Sha256Checksum}.  Throws unless the value
 * has the form `sha256:<64 hex chars>`.
 */
export function parseSha256Checksum(value: unknown): Sha256Checksum {
  if (typeof value !== 'string' || !/^sha256:[0-9a-f]{64}$/i.test(value)) {
    throw new Error(`Invalid sha256 checksum ${ JSON.stringify(value) }; expected "sha256:<64 hex chars>"`);
  }

  return value.toLowerCase() as Sha256Checksum;
}

/**
 * The compound entry recorded for each dependency in `dependencies.yaml`:
 * the version (typed per dependency) plus a map from artifact filename to
 * its stored {@link Sha256Checksum}.  Filenames match what upstream

actualChecksum from crypto.createHash().digest('hex') is always lowercase; expectedChecksum reaches download() only via lookupChecksum, which normalises through parseSha256Checksum. So download.ts:78, 121 (actualChecksum !== expectedChecksum) is robust today. A test or fixture that constructs Sha256Checksum outside parseSha256Checksum (the brand allows sha256:${ string } literally) could regress this silently.

Fix: document the lowercase normalization in the Sha256Checksum JSDoc and the parseSha256Checksum body.

readDependencyManifest requires {version, checksums} per entry. Design Observations ¶

Concerns ¶

D1. Unconditional re-extraction trade-off in downloadTarGZ/downloadZip (in-scope) Claude

The author's choice ("destPath always reflects the current archive") pays a developer-install cost on every run, even when nothing changed. The RancherDashboard.download .source-sha256 stamp pattern is a natural template — short-circuit on stamp match — that would recover the pre-PR skip-on-cache-hit while still detecting partial extractions. The current divergence between RancherDashboard (stamps) and downloadTarGZ/downloadZip (always re-extract) is itself a design smell. The author has accepted this trade-off in this round; documenting it via the new comment is the minimum, but converging the three archive-extraction call sites onto the stamp pattern is the cleaner long-term shape.

D2. Two parallel manifest types (DependencyVersions vs DependencyManifest) (future) Claude

mook-as raised this in the PR review and the structure has not changed. The split is ergonomic for the two call sites that only need versions (lint-go.ts, docker-cli-monitor.ts), but those callers could trivially read manifest[name].version themselves. Folding DependencyVersions into the per-key version field of DependencyManifest would yield a single source of truth.

D3. Bump-time getChecksums materialises every artifact for every platform (future) [carry-over from round 5 D3]

Each getChecksums() downloads each artifact, hashes, then deletes. A --all rebuild downloads gigabytes. If this becomes a hot path, switch downloadAndHash to a streaming hash that pipes the response body straight into crypto.createHash('sha256').

Strengths ¶

D4. Round 5 backlog cleared cleanly (in-scope) Claude Codex Gemini

Round 5's S1, S2, S3, S5, S6, S7, S8, and S11 all landed: 40-char hex commit guard, algorithm-aware sidecar parsing, .xz cache retention, codesign error detail, trust-on-bump comment, EACCES comment rewrite, throw-on-zero-entries, codesign filtered out of inner archive download. The compute-checksums-before-branching pattern in rddepman.ts:291-297 keeps a failed bump from leaving an orphan branch.

D5. The pinned-commit design for MobyOpenAPISpec (in-scope) Claude Codex Gemini

apiVersion for comparison and commit for URL immutability is the right separation for a master-branched upstream. Resolving the commit only on the winner (latestVersion getter) avoids one listCommits per historical version. The 40-char hex guard added this round closes the immutability hole.

D6. .unsigned codesign cache is the cleanest option for in-place mutation (in-scope) Claude Codex

Keeping the verified bytes at ${destPath}.unsigned and hashing them on cache checks integrates digest verification with codesign's in-place rewrite without re-downloading or trusting the post-signing bytes.

D7. Schema evolution preserves install-time simplicity (in-scope) Codex Gemini

Storing every digest as sha256: even when upstream publishes sha512 confines multi-algorithm parsing to rddepman at bump time. Install-time download() only handles sha256. The compound entry shape mirrors the on-disk YAML, so the in-memory model and the file stay in sync without translation.

Testing Assessment ¶

No unit tests cover the new helpers. The build-script directory has minimal test infrastructure (scripts/lib/__tests__/build-utils.spec.ts is the only pre-existing spec). Specific scenarios that would catch the issues raised here:

  1. fetchUpstreamChecksums parser: GNU vs BSD lines, with/without * binary marker, path-prefixed entries, mixed sha256/sha512 widths, BSD prefix mismatch (covers S12).
  2. parseSha256Checksum: rejects empty, wrong-length, non-hex, no-prefix; accepts uppercase and normalises to lowercase (covers S12).
  3. readDependencyManifest: rejects missing version, rejects array version, accepts string and object forms, handles missing checksums field.
  4. lookupChecksum artifact-name parity: each dependency's getChecksums() must produce filenames that its download() later passes to lookupChecksum (covers S8).
  5. downloadTarGZ/downloadZip re-extract semantics; Moproxy.download 7z e non-interactive behaviour (covers S1).
  6. MobyOpenAPISpec.download() rejects non-SHA commit; getAvailableVersions candidate type round-trip (covers S2).
  7. rddepman dry-run: getChecksums() failure aborts before any git mutation; updateManifest preserves the compound YAML shape.

End-to-end, the PR's primary functional test is "rddepman runs and produces a working bump PR" — demonstrated by 9d35ac4cc and 53aa0796a in the history (lima 1.2.1.rd2 → 1.2.1.rd3 was generated by this branch).

Documentation Assessment ¶

The header comment in dependencies.yaml, the codesign comment block at scripts/lib/download.ts:66-69, the new MobyOpenAPISpec.getChecksums trust-on-bump comment, and the Moproxy.download cache-retention comment are exemplary — they explain why, not what. The new downloadTarGZ/downloadZip "stop the consuming process and retry" comments convert the I1 trade-off into a documented limitation, which is the right shape.

The remaining gap is the manual-edit workflow for the compound schema: a maintainer who hand-edits a version (e.g., bumping the mobyOpenAPISpec.commit between automated bumps) must also recompute the affected sha256 entries. No external docs or YAML header callout describes this. A short note at pkg/rancher-desktop/assets/dependencies.yaml's manifest header, or a yarn rddepman --refresh-checksums=<name> helper, would close the loop.

Acknowledged Limitations ¶

  • Re-extraction overwrites locked binaries on Windows / Linux (scripts/lib/download.ts:215-225, 268-279, round 5 I1 declined): "copyFile overwrites destPath unconditionally, so a postinstall run while Rancher Desktop or another shell holds the binary fails with EBUSY/EPERM on Windows or ETXTBSY on Linux; stop the consuming process and retry." The author accepted the trade-off this round by adding the comment instead of adopting the RancherDashboard stamp pattern. macOS is unaffected (in-place overwrite of mapped binaries is allowed).
  • Stamp absence on crash starts over (scripts/dependencies/tools.ts:405-408): "A crash between mkdir and tar leaves the stamp absent, so the next run starts over."
  • Forcing sha256 over upstream algorithms (scripts/dependencies/lima.ts:81 review thread): @mook-as questioned not using upstream sha512; @jandubois explained that uniform sha256 storage simplifies install-time verification by confining multi-algorithm parsing to rddepman at bump time. @mook-as accepted ("It's fine to use whatever is easier for you").
  • WiX has no upstream checksum file (scripts/dependencies/wix.ts:33-34): "WiX doesn't publish upstream checksum files; we rely on the digest recorded in dependencies.yaml at bump time."
  • Spin shim has no upstream checksum file (scripts/dependencies/tools.ts:605-606).
  • Docker buildx omits darwin from upstream verification (scripts/dependencies/tools.ts:208-210): upstream checksums.txt omits darwin entries (https://github.com/docker/buildx/issues/945).
  • Upstream linux/arm64 not yet published for QEMU (scripts/dependencies/lima.ts:96-99, 119): the download() skip and the getChecksums() matrix omission mirror each other.
  • MobyOpenAPISpec commit is invisible to upstream security fixes (scripts/dependencies/moby-openapi.ts:131-136): rcompareVersions ignores commit, so a security fix to v1.54.yaml on master does not auto-bump. Adopting a newer commit on the same apiVersion requires a manual dependencies.yaml edit.
