Executive Summary ¶

The Round 3 backlog landed cleanly: the codesign + cache-hit interaction that re-downloaded Docker CLI on every macOS postinstall is fixed by keeping the verified bytes at ${destPath}.unsigned and hashing that on re-runs (Round 3 I1); MobyOpenAPISpec overrides latestVersion so only the winner's commit is resolved (Round 3 S2); parseSha256Checksum now accepts case-insensitive hex and normalises to lowercase (Round 3 S3); downloadAndHash preserves the served bytes to os.tmpdir() on mismatch with the path included in the error (Round 3 S6); download() ignores EACCES on the cache-hit access check (Round 3 S7); readDependencyManifest rejects array/scalar version values (Round 3 S9); the gnuLine regex bounds the hex prefix (Round 3 S10); and dependencies.yaml now carries an in-file "do not edit comments" header (Round 3 S12).

Two new Important findings surface from the Round 3 → Round 4 changes:

  1. The Round 3 S1 follow-up was applied to downloadTarGZ and downloadZip — they now persist the archive next to destPath and re-extract on every postinstall — but the same skip-on-cache-hit pattern still lives in RancherDashboard.download() at scripts/dependencies/tools.ts:406, where an existing resources/rancher-dashboard directory short-circuits before download() is ever called (I1, Codex). The PR's stated invariant ("every install path verifies the recorded digest") therefore does not hold for that one dependency.
  2. The "always re-extract" policy downloadTarGZ/downloadZip now use means a developer running yarn postinstall on Windows while Rancher Desktop is running will fail when fs.copyFile tries to overwrite a locked helm.exe / docker.exe (I2, Gemini). Pre-PR the helpers skipped extraction entirely on cache hit, so the lock was never touched.

The remaining surface is suggestion-grade: bump-time double-hashing in downloadAndHash (every artifact is sha256'd twice when the upstream sidecar is also sha256), three Round 3 Suggestions still outstanding (windows/arm64 in getChecksums matrices, MACOS_RUNNER hard-coding, Sha256Checksum return type round-trip), per-version archive caches that accumulate after a bump (wix314rtm.zip, lima.*.v1.2.1.rd2.tgz), a hand-edit hazard where { apiVersion, commit: '' } produces a malformed Moby URL, and a few defensive nits.

Verdict: merge with the two Important fixes folded in. Everything else is non-blocking polish.

/**
 * A GlobalDependency is a dependency where the version is managed in the file
 * {@link DEP_VERSIONS_PATH}.
 */
export function GlobalDependency<T extends abstract new(...args: any[]) => VersionedDependency>(Base: T) {
  abstract class GlobalDependency extends Base {
    /** The name of this dependency; it must be a key in DEP_VERSIONS_PATH. */
    abstract name: keyof DependencyVersions;

    get currentVersion(): Promise<Version> {
      return getCachedManifest().then(m => m[this.name].version);
    }

    async updateManifest(newVersion: Version, newChecksums: Record<string, Sha256Checksum>): Promise<Set<string>> {
      // Clone so writes here do not mutate the value other readers see.
      const manifest = structuredClone(await getCachedManifest());

      // The cast trusts the subclass to pass a Version that matches its
      // own DependencyVersions field; rddepman builds the call from
      // `dep.getChecksums(latestVersion)` whose return type already
      // depends on the dependency's own version shape.
      (manifest as any)[this.name] = { version: newVersion, checksums: newChecksums };
      await writeDependencyManifest(DEP_VERSIONS_PATH, manifest);

      return new Set([DEP_VERSIONS_PATH]);
    }
  }

  return GlobalDependency;
}

Design Observations ¶

Concerns ¶

D1. MobyOpenAPISpec commit pinning has a soft failure mode for upstream security fixes (future) Claude

The design correctly notes that pinning commit makes the install URL immutable, and rcompareVersions ignoring commit means a new commit on the same apiVersion does not auto-bump. Implication: a security fix to v1.54.yaml on master is invisible until someone manually edits dependencies.yaml. Worth documenting in the maintenance steps so the manual-edit case is not lost.

Strengths ¶

D2. Round 3 backlog cleared and the codesign repair is well-targeted (in-scope) Claude Codex Gemini

Round 3's I1 (codesign + cache-hit) is fixed by keeping the verified bytes at ${ destPath }.unsigned and hashing that on cache-hit checks (scripts/lib/download.ts:67-94, 130-143). Round 3 S2/S3/S6/S7/S9/S10/S12 all landed. The codesign repair preserves the verified-before-sign ordering D3 (Round 3) noted, and adds a deterministic re-entry point.

D3. Computing checksums before git checkout -b aborts the bump cleanly (in-scope) Claude

rddepman.ts:291-297 resolves the new checksums before any branch is created, so an upstream verification failure leaves no orphan branch on disk. Right ordering for a long-running automation script.

D4. Mismatch error preserves the served bytes for inspection (in-scope) Claude

downloadAndHash copies the unmatched bytes to os.tmpdir()/rddepman-mismatch-… and includes the path in the error message. Most CDN swap-in failures look like an HTML error page; the saved file makes the cause obvious without re-running the bump.

D5. DependencyManifest mirrors the YAML shape; DependencyVersions is a derived view (in-scope) Claude

The on-disk format and the in-memory type now match shape; the flat DependencyVersions projection (used by lint-go.ts and docker-cli-monitor.ts via readDependencyVersions) keeps legacy callers working without translation. mook-as's review thread on splitting DependencyVersions is addressed by this projection.

D6. Sha256Checksum template literal type plus parseSha256Checksum form a clean validation boundary (in-scope) Claude

Sha256Checksum = \sha256:${ string }\ plus the regex parser constrains the on-disk shape at the YAML edge. Type-system enforcement of the prefix keeps the encoding asymmetry (store with prefix, consume without) local to lookupChecksum.

Review Process Notes ¶

Skill improvements

  • Add to the review dimensions a sub-bullet under "Backwards compatibility / regressions": *When a prior round's fix changes a cached path's contract from "skip on existence" to "always re-do," re-examine every host-OS-specific failure mode the old skip protected against. The classic shape is exclusive-lock filesystems (Windows): the skip silently dodged EBUSY on a held-open binary, the always-re-do path now hits it on every run.*

Why this generalises: I2 caught Round 4's Windows regression because Gemini reasoned about the OS-level lock semantics that Round 3 S1's "always verify" suggestion silently changed. Equivalent shapes: a chmod made conditional now races a concurrent process; a temp-file rename made unconditional now collides with a reader holding the old inode; an extract-over-symlink hits noexec mounts. The underlying principle is "removing a guard relaxes a precondition the platform may still enforce."

  • Add to the review dimensions, under "Cross-component carry-through": When a finding is fixed for one consumer of a shared helper, enumerate every other consumer of the same pattern and check that the fix applies there too. The originating helper is one of N callers; an architectural fix at a single call site leaves the pattern open in the others.

Why this generalises: I1 caught the RancherDashboard gap because Codex carried the Round 3 S1 fix forward across all consumers of the "extract + cache" pattern. Single-call-site fixes invite this exact shape of regression.
