## Executive Summary

The branch adds a default Info tab driven by `docker inspect`. Three reviewers converged on lifecycle gaps: the child fetches before the store's client and namespace land and then never retries; the unsubscribe path leaves stale `inspectData`; an empty `inspect` result returns silently; templates render `Mounts.length` and `Args.length` without null-safety; and cached previous data masks the next fetch's error. The feature is close to landing once these five issues are addressed.

All three reviewers ran a single pass. Five Important findings cluster around watcher dependencies and template robustness; ten Suggestions cover style, i18n, and CLI hardening. The branch has an open PR (#9989); reconciliation of its prior review comments appears under Acknowledged Limitations and Unresolved Feedback below.

## Important Issues

I1. **Inspect dispatch races store client/namespace; never retries** — `pkg/rancher-desktop/components/ContainerInspect.vue:291` (important, gap) [Claude Opus 4.7, Codex GPT 5.5]

```ts
watch(() => props.containerId, fetchInspect, { immediate: true });
```

`ContainerInfo.vue` activates the Info tab by default (`activeTab` initial = `'tab-info'`) and mounts the child immediately when `route.params.id` is present. The child's `immediate: true` watcher dispatches `fetchContainerInspect` synchronously, but the parent's `subscribe()` only sets `state.client` after the asynchronous `settings-read` IPC resolves. On a direct URL navigation, page reload, or any path bypassing `Containers.vue`, the dispatch throws "Container client is not available" (`store/container-engine.ts:408-409`). `props.containerId` never changes afterward, so the watcher does not refire when the client later becomes ready and the error persists. The namespace case is symmetric: when `props.namespace` lands after `settings-read` resolves, the watch source ignores it.

Fix: include client readiness and namespace in the watch source.

```diff
+const clientReady = computed(() => !!store.state['container-engine'].client);
-watch(() => props.containerId, fetchInspect, { immediate: true });
+watch(
+  [() => props.containerId, () => props.namespace, clientReady],
+  () => clientReady.value && fetchInspect(),
+  { immediate: true },
+);
```

I2. **Unsubscribe path leaves stale `inspectData`** — `pkg/rancher-desktop/store/container-engine.ts:315-321` (important, gap) [Codex GPT 5.5]

```ts
unsubscribe({ commit, state }) {
  state.subscriber?.destroy();
  state.subscriber = null;
  state.containers = null;
  state.volumes = null;
},
```

`inspectData` is initialized at line 242 and cleared in `SET_PARAMS` when the namespace or backend changes, but the unsubscribe path — called from `ContainerInfo.vue:286` on unmount and on backend-unready transitions — clears only `containers` and `volumes`. After unsubscribe, a later inspect failure still shows the previous container's data because `ContainerInspect.vue:266` reads from `inspectData[containerId]` and the template prefers cached data over the error branch (see I5).

Fix: clear `inspectData` alongside the other lifecycle fields.

```diff
     state.containers = null;
     state.volumes = null;
+    state.inspectData = {};
```

I3. **Empty `parsed` result silently leaves the user with no data** — `pkg/rancher-desktop/store/container-engine.ts:415-420` (important, gap) [Claude Opus 4.7]

```ts
const parsed = result.parseJsonObject() as ContainerInspectData[];

if (parsed[0]) {
  commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] });
}
```

`docker inspect <id>` returns `[]` when the container was removed between the list query and the click, when the lookup runs against the wrong namespace under containerd, or when `fetchContainers` already pruned the entry. In every such case the action returns normally with no commit and no throw: the caller's `finally` flips `loading=false`, the computed `data` is `undefined`, and the user sees the generic "An unknown error has occurred" string with no retry path and no specific message.

Fix: throw on empty result so the catch path surfaces a meaningful message.

```diff
-if (parsed[0]) {
-  commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] });
-}
+if (!parsed?.[0]) {
+  throw new Error(`Container ${ containerId } not found.`);
+}
+commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] });
```

I4. **Template renders nullable array fields without null-safety** — `pkg/rancher-desktop/components/ContainerInspect.vue:58, 61, 145, 147` (important, gap) [Gemini 3.1 Pro]

```vue
Mounts <span class="count">({{ data.Mounts.length }})</span>
...
<tr v-if="data.Args.length">
  <th>Args</th>
  <td>{{ data.Args.join(' ') }}</td>
</tr>
```

`ContainerInspectData` (`store/container-engine.ts:13-48`) declares `Mounts: ContainerMount[]` and `Args: string[]` as required arrays, but nerdctl's Docker-compatible output serializes uninitialized Go slices as `null`. The existing `Ports` typing in the same file already acknowledges this pattern (`Record<…, … | null>`). When either field arrives as `null`, the template throws `TypeError: Cannot read properties of null (reading 'length')` and the whole component fails to render with no recovery path.

Fix: add optional chaining and a fallback at each access site, and widen the types in `ContainerInspectData` to match the wire format.

```diff
-Mounts <span class="count">({{ data.Mounts.length }})</span>
+Mounts <span class="count">({{ data.Mounts?.length ?? 0 }})</span>
...
-<tr v-if="data.Args.length">
+<tr v-if="data.Args?.length">
   <th>Args</th>
-  <td>{{ data.Args.join(' ') }}</td>
+  <td>{{ data.Args?.join(' ') ?? '' }}</td>
 </tr>
```

I5. **Cached data masks the next fetch's error** — `pkg/rancher-desktop/components/ContainerInspect.vue:6-15, 244-251` (important, gap) [Gemini 3.1 Pro, Claude Opus 4.7]

```vue
<div v-if="loading" ... >Loading container details…</div>
<template v-else-if="data">
  <!-- summary + sections -->
</template>
<div v-else class="error-state" data-testid="info-error">
  {{ error || 'An unknown error has occurred' }}
</div>
```

The template gates on `loading → data → error`. When the user re-enters the Info tab with `inspectData[containerId]` already cached, `v-else-if="data"` wins as soon as `loading` flips false — even when the new fetch failed and `error.value` is set. The user sees stale data with no signal that the refresh failed. The spinner also flashes briefly on remount despite the cache, which defeats the cache's purpose.

Fix: gate the spinner on `loading && !data`, and place the error block above the data block so failures take precedence over the previous cache.

```diff
-<div v-if="loading" ... >
+<div v-if="loading && !data" ... >
   Loading container details…
 </div>
+<div v-else-if="error" class="error-state" data-testid="info-error">
+  <i class="icon icon-warning" />
+  {{ error || 'An unknown error has occurred' }}
+</div>
 <template v-else-if="data">
```

## Suggestions

S1. **CLI stderr swallowed by generic error message** — `pkg/rancher-desktop/components/ContainerInspect.vue:284-285` (suggestion, gap) [Codex GPT 5.5]

```ts
} catch (err: any) {
  error.value = err?.message ?? 'Failed to load container details.';
}
```

The extension `exec` wrapper throws the raw `ExecResult` on non-zero exit (`preload/extensions.ts:286, 421`); that object carries `stderr`/`stdout` fields but no `message`. The catch therefore converts useful failures such as "No such container" or daemon errors into the generic fallback string.

Fix: surface `stderr`/`stdout` before falling back.

```diff
-error.value = err?.message ?? 'Failed to load container details.';
+error.value = err?.message ?? err?.stderr ?? err?.stdout ?? 'Failed to load container details.';
```

S2. **`parsed[0]` should match the requested container ID** — `pkg/rancher-desktop/store/container-engine.ts:416-419` (suggestion, gap) [Claude Opus 4.7]

```ts
const parsed = result.parseJsonObject() as ContainerInspectData[];
if (parsed[0]) {
  commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] });
}
```

The type cast skips runtime validation, and `parsed[0]` assumes the first element matches `containerId`. If a future caller passes a partial ID or name that matches multiple containers, the keyed cache associates the wrong record with the requested key. PR review comment `r3002719697` raised the same suggestion.

Fix: select by full ID match.

```diff
-if (parsed[0]) {
-  commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] });
-}
+const entry = Array.isArray(parsed) ? parsed.find(c => c?.Id === containerId) : undefined;
+if (entry) {
+  commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: entry });
+}
```

S3. **Container ID passed without `--` separator** — `pkg/rancher-desktop/store/container-engine.ts:411-414` (suggestion, gap) [Claude Opus 4.7]

```ts
const args = [
  ...(namespace ? [`--namespace=${ namespace }`] : []),
  containerId,
];
```

`containerId` originates from `route.params.id` with no validation. A route param starting with `-` (e.g. `--help`) is parsed by docker/nerdctl as a flag and produces confusing errors instead of the expected "no such container" response. No security implication (no shell interpolation), but the guard is cheap.

Fix: terminate flag parsing with `--`.

```diff
 const args = [
   ...(namespace ? [`--namespace=${ namespace }`] : []),
+  '--',
   containerId,
 ];
```

S4. **Env entries with empty key collide on `:key`** — `pkg/rancher-desktop/components/ContainerInspect.vue:109, 297-301` (suggestion, gap) [Claude Opus 4.7]

```ts
const envEntries = computed(() => envVars.value.map((e) => {
  const idx = e.indexOf('=');
  return idx === -1 ? [e, ''] : [e.slice(0, idx), e.slice(idx + 1)];
}));
```

Template line 109 uses `:key="key"`. A docker `Env` entry with no `=` or one beginning with `=` produces an empty `key`; two such entries collide, Vue logs a duplicate-key warning, and the row renders with an empty `<th>`.

Fix: key by index.

```diff
-<tr v-for="([key, value]) in envEntries" :key="key">
+<tr v-for="([key, value], i) in envEntries" :key="i">
```

S5. **New user-visible strings bypass i18n** — `pkg/rancher-desktop/components/ContainerInspect.vue` (suggestion, gap) [Claude Opus 4.7]

The component hard-codes every user-visible string ("Name", "ID", "IP Address", "Mounts", "Environment", "Loading container details…", "None", "An unknown error has occurred", etc.). The repo ships an i18n surface at `pkg/rancher-desktop/assets/translations/{en-us,de,zh-hans}.yaml` and a global `t()` helper.

Caveat: `ContainerLogs.vue` and `ContainerShell.vue` already hard-code their strings, so this PR follows the established (non-i18n) pattern in this folder. The folder is the gap, not this PR — but the gap doubles in surface area here. Worth flagging to the maintainer who owns the i18n migration.

S6. **No refetch when container state changes** — `pkg/rancher-desktop/components/ContainerInspect.vue:291` (suggestion, gap) [Claude Opus 4.7]

The watcher refetches only when `containerId` changes. If the container is stopped, started, renamed, or otherwise updated while the Info tab is open (events the parent's subscriber already receives), the inspect view keeps showing the prior `Status`, `StartedAt`, `IPAddress`, etc., until the user navigates away and back.

Fix: re-dispatch when `currentContainer.state` changes, or hook the inspect refresh into the existing event stream.

S7. **Missing `defineOptions({ name: ... })`** — `pkg/rancher-desktop/components/ContainerInspect.vue:254` (suggestion, gap) [Claude Opus 4.7]

```ts
// ContainerLogs.vue:46
defineOptions({ name: 'ContainerLogs' });
// ContainerShell.vue:52
defineOptions({ name: 'ContainerShell' });
```

`ContainerInspect.vue` does not declare its name. Vue infers a name from the filename in most tooling, but the explicit declaration is the established pattern in this directory and aids devtools and `<keep-alive>` matching.

Fix: add `defineOptions({ name: 'ContainerInspect' });` at the top of the `<script setup>` block.

S8. **Namespace prop type style differs from siblings** — `pkg/rancher-desktop/components/ContainerInspect.vue:260-263` (suggestion, gap) [Claude Opus 4.7]

```ts
// ContainerInspect.vue
const props = defineProps<{
  containerId: string;
  namespace:   string | undefined;
}>();
```

`ContainerShell.vue:54-58` uses `namespace?: string` and `ContainerLogs.vue:48-52` uses `namespace?: string | null`. The `?:` shorthand and explicit `string | undefined` are not equivalent in Vue's dev-mode prop validation: the explicit form marks the prop as required (parent must always pass it), while `?:` marks it as optional.

Fix: switch to `namespace?: string` to match the surrounding convention.

## Design Observations

**Concerns**

- `fetchContainerInspect` diverges from the surrounding actions in two ways: it does not commit failures to `SET_ERROR` and it does not classify them against an `ErrorSource`. The reviewer flagged this in PR comment `r3011823415` and suggested reusing `'containers'`; the action still surfaces errors only through the local component `error` ref. `(in-scope)` [Claude Opus 4.7]
- `commit('SET_INSPECT_DATA', { ...state.inspectData, [containerId]: parsed[0] })` replaces the whole map on every fetch, which defeats fine-grained reactivity for components observing other keys. A `MERGE_INSPECT_DATA(state, { id, data })` mutation would centralise the immutability contract and shorten the call site. `(future)` [Claude Opus 4.7]

**Strengths**

- Stale-data cleanup in `fetchContainers` (lines 389-396) hooks the new cache into the existing container lifecycle without adding a separate eviction path — the reviewer specifically asked for this reuse and the implementation matches. `(in-scope)` [Claude Opus 4.7, Codex GPT 5.5]
- Gating the inspect dispatch on `activeTab === 'tab-info'` means Logs and Shell keep their existing lifecycle behaviour while Info is fetched only when its tab is visible. `(in-scope)` [Codex GPT 5.5]
- The `<!-- rendered in .tab-content so we can mix v-if and v-show -->` comment at `ContainerInfo.vue:11-15` documents the intent behind the tabbed lifecycle mix — the kind of "why" comment that survives refactors. `(in-scope)` [Claude Opus 4.7]

### Repo context updates

Reviewers should diff new components against `ContainerLogs.vue` and `ContainerShell.vue` for the canonical pattern and flag drift.

Treat a non-nullable collection type in this file as a likely bug.
- **[repo] i18n migration is partial in `pkg/rancher-desktop/components/`.** The repo ships an i18n surface (`pkg/rancher-desktop/assets/translations/{locale}.yaml` plus a global `t()` helper), but components in this folder (`ContainerLogs.vue`, `ContainerShell.vue`, and now `ContainerInspect.vue`) still hard-code English strings. New components in this folder follow the established non-i18n pattern; flag missing i18n at Suggestion severity, not Important — the folder is the gap, not the individual component.

## Important Issues

I1. **Inspect dispatch races the store's `client` initialization** — `pkg/rancher-desktop/components/ContainerInspect.vue:291` and `pkg/rancher-desktop/store/container-engine.ts:405-410` (important, gap)

```ts
// ContainerInspect.vue
watch(() => props.containerId, fetchInspect, { immediate: true });

// store/container-engine.ts
async fetchContainerInspect({ commit, state }, { containerId, namespace }) {
  const { client } = state;
  if (!client) {
    throw new Error('Container client is not available');
  }
```

# Design Observations

**Concerns**

- The store action `fetchContainerInspect` diverges from the surrounding actions in two ways: it does not wrap the `try` in a `try/catch` that commits to `SET_ERROR`, and it does not classify the error against an `ErrorSource`. The reviewer flagged this (`#discussion_r3011823415`) and suggested reusing `'containers'`; the suggestion was not addressed.
