## Important Issues

I1. **Ports in `pst.added` never re-evaluate engine-chain status across ticks** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:186-200` [Codex GPT 5.5]

```go
func (pst *portStateTracker) addObserved(newPortMap nat.PortMap) {
    for port, bindings := range newPortMap {
        if state, alreadyAdded := pst.added[port]; alreadyAdded {
            if state.appendFailed {
                pst.retryAppend(port, state)
            }
            continue
        }
        if _, sawLastScan := pst.seenLastScan[port]; !sawLastScan {
            ...
```

`addObserved` re-probes the engine chain only for `appendFailed=true`; `delegated=true` (lines 303, 378) and owned-success short-circuit forever. Two lifecycles go wrong:

- **Delegated → not managed** (round 7 I1 unaddressed): `--network=host` listener co-tenants with a bridge container's CNI rule. Bridge exits, chain torn down, listener persists — procnet never installs the loopback DNAT.

- **Owned-success → engine takeover**: a long-lived nerdctl reservation survives the gate; procnet wins and Appends. CNI installs its rule later. The engine's events handler (`containerd/events_linux.go:123-128`, `docker/events.go:100-103`) calls `tracker.Add`, gets "proxy already running", logs, and gives up. When the reservation closes, `retireDisappeared` removes procnet's proxy — no engine proxy survives.

Fix: re-probe `anyEngineBinding` every tick for ports in `pst.added`, symmetric with `retryAppend`. Add flap-protection. Accepting as a trade-off (matching R6-I1) needs documentation covering both lifecycles.

## Suggestions

S1. **`engineDNATChains` is a mutable package-level var** — `src/go/guestagent/pkg/procnet/scanner_linux.go:55-62` [Claude Opus 4.7] (round 7 S3 unaddressed)

```go
var engineDNATChains = []string{"CNI-HOSTPORT-DNAT", "DOCKER"}
```

Go has no const slice, so a `var` is the only option for a multi-element list. The docstring treats this as a closed list, but any code in the same package can still mutate it at runtime. Fix: replace with a function returning a fresh slice each call (`engineDNATChainList() []string`), or declare as a fixed-length array (`[2]string`) so the slice cannot grow. Skip the change if "swap in a fake chain list" testability is anticipated.

S2. **`node -e` cross-platform notice is fragile and tied to `go test` success** — `package.json:38` [Claude Opus 4.7] (round 7 S2 unaddressed)

```
"test:unit:guestagent": "cd ./src/go/guestagent/ && go test ./... && node -e \"process.platform !== 'linux' && console.warn(...)\""
```

The `&& node -e ...` runs only when `go test ./...` succeeds, so a developer whose tests fail on macOS sees no "skipped on non-Linux" notice. The wording also implies tests were skipped at runtime, but `_linux.go` files are not compiled at all off Linux. Fix: drop the notice and rely on the file-name suffix, or print unconditionally with the wording "Linux-only files were not compiled on `<platform>`."

S3. **`realIptablesRunner` stores `context.Context` as a struct field** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:135-146` (round 7 S1 unaddressed)

```go
type realIptablesRunner struct {
    ctx context.Context
}
```

`go vet`'s `containedctx` rule (and Go style guidance) prefers passing `context.Context` per call instead of storing on a struct: the lifetime of the runner is tied to one outer call (`ForwardPorts`). The risk is small today but invites future code to reuse the runner with a stale context. Fix: extend `iptablesRunner` methods to accept `ctx context.Context` as the first parameter; drop the field. The fake updates mechanically. A `//nolint:containedctx` with a rationale comment closes the audit trail if the change is too noisy this late.

S4. **`dnatRuleRe` assumes one iptables rule per line** — `src/go/guestagent/pkg/procnet/scanner_linux.go:74-81` (round 7 S4 unaddressed)

```go
var dnatRuleRe = regexp.MustCompile(
    `\b(tcp|udp)\b[^\n]*?(?:dpt:([0-9]+)|dports ([0-9]+(?:,[0-9]+)*))\b`,
)
```

The `[^\n]*?` clause encodes the invariant "each rule occupies one line." Real `iptables --list <chain> --numeric` output respects this. The risk is theoretical: a future iptables version, an unexpected piped paginator, or a malformed exec wrapper that collapses lines could cause one `tcp` match to consume a rule boundary. Fix: add a table-driven test case with two `tcp dpt:` clauses on one line and assert the regex matches only the first — or anchor on the `target` column ordering if multi-rule lines should still parse correctly.

S5. **`removeReportedProxyDestroyed` does not collapse the `err == nil` case** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:547-554` [Claude Opus 4.7]

```go
func removeReportedProxyDestroyed(err error) bool {
    return errors.Is(err, tracker.ErrWSLProxy) && !errors.Is(err, forwarder.ErrUnexposeAPI)
}
```

Three call sites (`exposeNew` engine-managed branch, `retryAppend` engine-takeover branch, `retireDisappeared` default branch) ask "did the host-switch unexpose succeed?" The predicate returns false for `err == nil`, so each call site repeats `case removeErr == nil: ... case removeReportedProxyDestroyed(removeErr): ...` with both arms falling through. Fix: rename to `proxyDestroyed(err error) bool` and return true for nil — three switches collapse to a single boolean expression each.

S6. **`RemoveAll`'s wsl-proxy error wrap is asymmetric with `Remove`'s** — `src/go/guestagent/pkg/tracker/apitracker.go:244-262` [Claude Opus 4.7]

```go
// RemoveAll inner wrap
wslProxyErrs = append(wslProxyErrs,
    fmt.Errorf("sending port mappings to wsl proxy error: %w", wslProxyError))
```

In `Remove` (line 202) the wsl-proxy failure wraps as `fmt.Errorf("%w: %w", ErrWSLProxy, err)` — both `ErrWSLProxy` and the underlying `err` unwrap. In `RemoveAll`, the inner wrap uses the legacy prefix and the outer wrap uses `%+v` over the slice (line 262), so individual underlying errors stringify rather than unwrap. A future caller writing `errors.Is(retErr, originalSendErr)` works in `Remove` but not `RemoveAll`. Fix: drop the inner prose prefix; mirror `Remove`'s `%w: %w` shape; keep the outer `%+v` wrap.

S7. **Review-round identifiers (`R5-I3`, `I1/I2/I3 regression test`) appear in source as stable labels** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:255-258, 370` and `port_state_tracker_linux_test.go:101,137,169-173,1640,1718-1725,1858-1860` [Claude Opus 4.7]

```go

// the synthetic ID; engine's later Add forever failing with
// "proxy already running"), so this regression rides along
```

A maintainer six months from now reading `// is the I2 regression test` gets no signal — the review thread is the only key. Fix: replace each identifier with the concrete description (`engine-takeover-during-append-retry regression test`, `same-port partial-Add gvisor-proxy-stranded leak`) and let the test names continue to carry the rest of the meaning.

S8. **`engineChainManagesPort` shells out twice per newly-observed port** — `src/go/guestagent/pkg/procnet/scanner_linux.go:241-260` [Gemini 3.1 Pro] (downgraded from important)

```go
func engineChainManagesPort(ctx context.Context, protocol, port string) (bool, error) {
    for _, chain := range engineDNATChains {
        cmd := iptablesCommand(ctx, "--wait", "2", "--table", "nat", "--list", chain, "--numeric")
```

For each newly-observed port, the helper spawns `iptables --list CNI-HOSTPORT-DNAT --numeric` and `iptables --list DOCKER --numeric` — two subprocesses contending on the xtables lock with `--wait 2`. During a burst (`docker-compose up` with many published ports), the per-tick fork count scales with new-port count and stretches the tick. Fix: dump the NAT table once per tick with `iptables-save -t nat`, parse it once, and answer all engine-chain queries against the in-memory snapshot. Skip if real-world load profiles do not hit this; the cost is added parsing code.

S9. **Shadowed loop variable in `APITracker.RemoveAll`** — `src/go/guestagent/pkg/tracker/apitracker.go:213-241` [Gemini 3.1 Pro]

```go
for _, portMapping := range a.portStorage.getAll() {
    // ...
    portMapping := guestagentTypes.PortMapping{
        Remove: true,
        Ports:  portMapping,
    }
```

The outer `portMapping` is a `nat.PortMap`; the inner is `guestagentTypes.PortMapping`. The names coincide but the types do not. The compiler is happy because the right-hand side evaluates before the new binding takes scope, but a reader has to look twice. Fix: rename the outer loop variable to `portMap` (matching `Remove`'s vocabulary) or the inner struct to `wslPortMapping`.

## Design Observations

**Strengths**

- The injectable `portStateTracker` + fake `iptablesRunner` test design continues to pay off: every state transition has a direct unit test, and adding a new transition or regression case touches one fixture rather than refactoring shared scaffolding. *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5]

- Switching `APITracker.Add`/`Remove`/`RemoveAll` to `errors.Join` over separate sentinels fixes a pre-existing class of silently-dropped errors (Expose failures swallowed by a concurrent wsl-proxy failure, and vice versa). The new tests `TestAddBothFailuresAreDistinguishable` and `TestRemoveAllBothFailuresAreDistinguishable` lock the contract end-to-end. *(in-scope)* [Claude Opus 4.7]

- `removeReportedProxyDestroyed` centralises the wsl-proxy-only vs host-switch-still-bound classification used at three sites (`exposeNew` engine delegation, `retryAppend` engine takeover, `retireDisappeared` default branch). `TestRemoveReportedProxyDestroyed` pins the predicate so a future refactor cannot silently desync the three call sites. *(in-scope)* [Claude Opus 4.7]

- The 86-line `portState` block comment doubles as a state-machine specification covering every transition, the cleanup-rule decision matrix, and the partial-failure rationale. Future maintainers reading the file have the design they need without leaving the file. *(in-scope)* [Claude Opus 4.7, Gemini 3.1 Pro]

**Concerns**

- **Proxy ownership is inferred through side effects across procnet, docker, and containerd** — *(future)* [Codex GPT 5.5]: each handler reads `iptables` chain rules, `tracker` keys, and `"proxy already running"` strings to figure out who owns the proxy. A durable claim protocol in `tracker` (an explicit `Claim/Release` pair keyed on container ID and synthetic ID, with a queryable owner) would erase several race-specific branches in all three call sites and make handoff testable without duplicating engine-timing in procnet.

- **`tracker.Add`/`Remove`'s new joined-error contract is documented only inside `apitracker.go`** — *(in-scope)* [Claude Opus 4.7]: procnet now depends on `ErrExposeAPI`/`ErrUnexposeAPI`/`ErrWSLProxy` as distinguishable sentinels, but the engine handlers in `containerd/events_linux.go:123-128` and `docker/events.go:100-103` still log and continue. Promoting the contract into the `Tracker` interface docstring (and exposing the sentinels from the `tracker` package) would force those callers to confront the partial-failure semantics on the next touch.

## Acknowledged Limitations

- **`handleAlreadyExposed` RESUME branch cannot re-notify wsl-proxy after a partial-Add** — `port_state_tracker_linux.go:73-77, 350-364`. `apiForwarder.Expose` succeeded on a prior tick (gvisor proxy bound, `portStorage[synthetic]` populated) and `wsl-proxy.Send` failed; the next tick's `tracker.Add` returns `proxy already running`, RESUME installs the iptables rule and resumes ownership, but the wsl-proxy notification is not retried because the `Tracker` interface exposes no idempotent re-notify hook. Windows-side traffic for that port stays disconnected until RD restart. *Gemini raised this as Critical+regression this round; verified as the same well-documented limitation referenced in the Round 7 review and pinned by the `portState` docstring.*

- **Engine-delegation `tracker.Remove` can tear down the only host-switch proxy under a `Local`-string collision** — `port_state_tracker_linux.go:240-269`, pinned by `TestEngineDelegationAcceptsLeakWhenRemoveFails`. Trigger: stability-gate bypass under heavy load PLUS transient `wsl-proxy.Send` failure PLUS HostIP=127.0.0.1 publish (or non-admin install). Documented as intentional trade-off.

- **Stability gate bypass under heavy load** — the two-scan gate filters the OCI hook reservation socket on normal load; a heavily-loaded system that takes >3s between procnet polls could see the reservation socket twice and act on it. Documented as the underlying limitation the `retryAppend`/engine-takeover paths ride along with.

- **In-session Delete leak** — `port_state_tracker_linux.go:282-285,305-307` document that the Delete path has no retry, so a transient iptables-Delete failure leaks a procnet rule until RD restart. The I1 reconciliation in `exposeNew`'s if-managed branch self-heals this whenever the engine later republishes the same host port.

- **iptables-nft kernel default assumption** — `scanner_linux.go:158-169` documents that the entire design depends on iptables-nft's PREROUTING chain-terminating-on-DNAT behavior, with no live assertion of the assumption.

- **`engineDNATChains` closed list** — `scanner_linux.go:55-62` and `engineChainManagesPort` docstring at `:227-231` document that adding a third engine without extending the list silently re-introduces the shadow-DNAT bug. `NewProcNetScanner` logs the active list at startup for diagnosability.

- **Port-range regex gap** — `scanner_linux.go:269-279` documents that `dpts:LO:HI` and multiport-embedded ranges are intentionally not matched. RD's shipped engines emit per-port rules, so this is theoretical for shipped engines but real for hand-rolled iptables setups against the same chains.

- **Engine chain visibility ≠ engine has a host-switch proxy** — `port_state_tracker_linux.go:93-101` documents that the CNI chain is installed before the engine's `portTracker.Add`, so a chain-present probe does not prove the engine owns a proxy. Acknowledged in the `retryAppend` engine-takeover comment.

- **Cross-session leftover Delete in `handleAlreadyExposed`'s RESUME branch** — `port_state_tracker_linux.go:365-376` documents the choice not to reconcile because the rule's origin is ambiguous (cross-session leftover vs deliberately retained from a Remove-failure-on-Unexpose). The I1 reconciliation in `exposeNew`'s if-managed branch covers the common cross-session case.

### Executive Summary

The PR substantially improves procnet ownership tracking and error classification, with broad unit coverage around the new state machine. One important gap remains: if a reservation socket survives the stability gate and procnet successfully installs ownership before the engine chain appears, the later engine takeover path is never detected.

### Findings

#### Important Issues

I1. **Successful procnet ownership never re-checks for engine takeover** — `port_state_tracker_linux.go:188-192` (important, gap)

After a long-lived nerdctl reservation survives the two-scan gate, procnet can successfully call `tracker.Add` at line 307 and append its PREROUTING DNAT at line 322 before CNI appears. Once the port is in `added`, lines 188-192 only re-probe engine chains when `appendFailed` is true, so a later `CNI-HOSTPORT-DNAT`/`DOCKER` rule is ignored. The engine event handler can lose its one `tracker.Add` attempt (`containerd/events_linux.go:123-127`, `docker/events.go:100-103`). When the listener disappears, procnet removes the synthetic proxy and deletes its rule, leaving no engine-owned host-switch proxy until restart.

Fix: add an owned-success engine-takeover transition and regression test.

### Design Observations

- *Concern*: Proxy ownership inferred through side effects across procnet, docker, containerd — a durable claim protocol in `tracker` would remove several race-specific branches. *(future)*
- *Strength*: Extracting `portStateTracker` behind fake tracker/iptables runners makes the formerly inline scanner behavior much easier to reason about.

### [Gemini] — Pass 1

### Executive Summary

The change extracts the `procnet` scanner logic into a testable state machine (`portStateTracker`) to reliably prevent procnet's `--network=host` loopback rules from shadowing container-engine DNAT rules. While the architecture and state management are highly robust, a critical flaw exists in the retry mechanism: partial failures when communicating with the WSL proxy are never retried, leaving the port permanently unreachable from the host.

### Findings

#### Critical Issues

C1. **WSL Proxy notification is never retried after a partial failure** — `port_state_tracker_linux.go:340` (critical, regression)

The PR description claims that if `wsl-proxy.Send` fails, the retry will come on the next tick, but `handleAlreadyExposed`'s RESUME branch installs the iptables rule and resumes ownership without re-notifying wsl-proxy (the docstring at lines 73-77 acknowledges this). The port stays disconnected from the Windows host until RD restart.

*Reconciled during consolidation: this gap is the same well-documented limitation referenced in prior rounds' Acknowledged Limitations and pinned by the file's own state-machine docstring. Not a regression (wsl-proxy notification has always been best-effort). Moved to Acknowledged Limitations rather than re-raised as Critical.*

#### Important Issues

I1. **N+1 subprocess execution for engine chain probes** — `scanner_linux.go:210` (important, performance)

Each `exposeNew` call spawns two `iptables --list` subprocesses (one per chain). Burst container start spikes CPU and stretches the tick.

*Reconciled during consolidation: downgraded to S8 (the cost only materialises under burst container start; `--wait 2` bounds the worst case).*

#### Suggestions

S1. **Shadowed loop variable** in `APITracker.RemoveAll` at `apitracker.go:237` (kept as S9).
