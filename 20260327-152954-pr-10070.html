Executive Summary ¶

This PR fixes the broken no-proxy list UI (missing Array.isArray guard on the StringList @change handler), adds input validation for noproxy entries, and routes domain-based entries through moproxy's --policy flag while keeping IP/CIDR entries on the iptables path. The implementation is well-structured, and the validator has thorough test coverage. The main concerns are that the validator accepts IPv6 addresses but the init script feeds them to IPv4-only iptables, and that the policy.rules fallback content uses ; as a comment character, which may not be valid in moproxy's policy parser.

Critical Issues ¶

None.

Important Issues ¶

1. IPv6 entries pass validation but cause iptables failures Codex Gemini ¶

The validator accepts IPv6 addresses and CIDRs (with dedicated test cases at settingsValidator.spec.ts:1020-1029), but writeProxySettings in wsl.ts routes all IP entries to MOPROXY_NOPROXY. The init script at moproxy.initd:45 feeds each entry to iptables --destination, which supports only IPv4. An IPv6 entry like ::1 causes iptables to emit host/network '::1' not found. Since the init script does not use ip6tables to redirect IPv6 traffic either, IPv6 is already not proxied — making the bypass a no-op that only produces errors.

settingsValidator.ts
  protected static isIPAddressOrCIDR(entry: string): boolean {
    const slashIndex = entry.indexOf('/');

    if (slashIndex === -1) {
      return net.isIP(entry) !== 0;
    }
    const address = entry.substring(0, slashIndex);
    const prefixStr = entry.substring(slashIndex + 1);
    const ipVersion = net.isIP(address);

    if (ipVersion === 0) {
      return false;
    }
    if (!/^\d+$/.test(prefixStr)) {
      return false;
    }
    const prefix = parseInt(prefixStr, 10);
    const maxPrefix = ipVersion === 4 ? 32 : 128;

    return prefix >= 0 && prefix <= maxPrefix;
  }

2. Policy.rules fallback uses ; comment character, which may crash moproxy Claude Gemini ¶

When no domain rules exist (the default state), this writes ; no domain noproxy rules to the policy file. The ; character is a comment delimiter in INI files (like proxy.ini), but moproxy's policy parser uses # for comments. If the parser rejects ;, moproxy fails to start for every user who does not configure domain-based noproxy entries — that is, the common case.

wsl.ts
    const ipEntries: string[] = [];
    const domainRules: string[] = [];

    for (const entry of proxy.noproxy) {
      if (net.isIP(entry) !== 0 || entry.includes('/')) {
        ipEntries.push(entry);
      } else {
        const domain = entry.startsWith('*.') ? entry.substring(2) : entry;

        domainRules.push(`dst domain ${ domain } direct`);

3. Domain noproxy rules may not take effect on live settings update Claude ¶

When the user changes proxy settings at runtime, handleSettingsUpdate writes the new policy.rules file and then sends SIGHUP via reload. SIGHUP causes moproxy to re-read its proxy server list (--list), but there is no documented behavior for reloading --policy. The subsequent startService calls start --ifnotstarted, which is a no-op because moproxy is already running. IP/CIDR changes take effect immediately through the iptables rebuild in the init script's reload(), but domain changes may require a full application restart.

wsl.ts
  async handleSettingsUpdate(newConfig: BackendSettings): Promise<void> {
    const proxy = newConfig.experimental.virtualMachine.proxy;

    await this.writeProxySettings(proxy);
    if (this.currentAction === Action.NONE && this.process) {
      if (proxy.enabled && proxy.address && proxy.port) {
        await this.execService('moproxy', 'reload', '--ifstarted');
        await this.startService('moproxy');
      } else {
        await this.stopService('moproxy');
      }
    }
  }

4. Single-label hostnames rejected despite backend support Codex Gemini ¶

The domain validator requires at least two labels, rejecting single-label hostnames like localhost, intranet, or wiki. Corporate environments commonly bypass the proxy for internal short-names. Moproxy's dst domain policy mechanism handles single-label domains without issue. The test at settingsValidator.spec.ts:1062-1068 explicitly codifies this rejection.

settingsValidator.ts
  protected static isDomainName(entry: string): boolean {
    const domain = entry.startsWith('*.') ? entry.substring(2) : entry;

    if (domain.length === 0 || domain.length > 253) {
      return false;
    }
    const labels = domain.split('.');

    if (labels.length < 2) {
      return false;
    }
    const labelRE = /^[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?$/;

    return labels.every(label => labelRE.test(label));
  }

Fix: Remove the two-label requirement:

-    if (labels.length < 2) {
-      return false;
-    }
     const labelRE = /^[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?$/;

Suggestions ¶

1. Support .example.com wildcard convention Gemini ¶

The validation accepts *.example.com but rejects .example.com, because splitting on . produces an empty first label that fails the regex. Since .example.com is a common convention in NO_PROXY environment variables (curl, wget, and others accept it), users who paste existing proxy bypass lists will hit validation errors.

settingsValidator.ts

      return false;
    }

Fix: Also strip a leading .:

Both *.example.com and example.com produce the identical rule dst domain example.com direct. Whether moproxy's dst domain matches subdomains determines whether the wildcard form is meaningful. A one-line comment would prevent future readers from mistaking this for a bug.

Design Observations ¶

Concerns ¶

1. Validation and backend classification may drift — The validator (settingsValidator.ts) and the backend classifier (wsl.ts:780-793) independently decide what constitutes an IP vs. a domain entry. They already diverge on IPv6: the validator accepts it, but the backend routes it to iptables. A shared classifier returning { ipv4Entries, ipv6Entries, domainEntries } would eliminate this class of mismatch. Codex

Strengths ¶

1. Clean separation of concerns — The validator checks format; the backend classifies entries by routing mechanism. Each layer does one job. The split between iptables (IP/CIDR) and policy rules (domains) matches the underlying capabilities. Claude

2. Pattern reuse — The Array.isArray guard matches the established fix in ContainerEngineAllowedImages.vue, and checkNoproxyList follows the validator's existing conventions. Claude

3. Thorough validator tests — Eighteen test cases cover valid inputs, invalid inputs, edge cases, and platform gating. Claude

Testing Assessment ¶

The validator is well-tested. The following scenarios lack coverage, ranked by risk:

  1. Backend entry classification (writeProxySettings at wsl.ts:780-795) — no tests cover the IP-vs-domain split, *. stripping, or policy.rules file format generation. This gap allowed the IPv6/iptables mismatch to pass undetected.
  2. Live settings update with domain entries — no integration test verifies that changing domain noproxy entries at runtime takes effect in moproxy.
  3. Moproxy startup with comment-only policy.rules — no test verifies that the fallback content is valid moproxy syntax.

Documentation Assessment ¶

The command-api.yaml (lines 757-760) and the UI copy in en-us.yaml (lines 432-434) describe noproxy as a hostname list. The PR now supports IPs, CIDRs, domains, and wildcards — those descriptions should be updated. Codex
