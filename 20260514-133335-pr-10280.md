## Executive Summary

This is Round 4. The commit under review (`410039856`) folds in the three
Round 3 Suggestion fixes, all of which verify against the current code:
`APITracker.Remove` now reports `forwarder.ErrUnexposeAPI` and
`tracker.ErrWSLProxy` under distinct sentinels joined with `errors.Join`
(S1), an `iptablesCommand` helper pins `LC_ALL=C` on every iptables
invocation (S2), and a separate `probeErrorLogged` marker keeps the
engine-probe failure from muting the first `tracker.Add` error (S3).
`TestRemoveWSLProxyFailureIsDistinguishable` and the reworked append-retry
tests cover the new behavior.

One new Important finding landed this round. `retryAppend`'s
engine-takeover branch marks a port `delegated:true` even when the
`applyLoopbackRules(..., Delete)` call that is supposed to remove the
committed procnet rule fails — and a delegated port's cleanup path never
touches iptables again, so a committed `PREROUTING` DNAT can survive to RD
restart and shadow the engine's chain. That is the exact bug this PR
exists to fix, reachable in a narrow multi-failure error path the tests do
not exercise.

The three Suggestions are minor: a self-contradicting doc comment, an
untested first-error short-circuit in `applyLoopbackRules` that is benign
under today's binding semantics, and a parity gap where the sibling
`APITracker.RemoveAll` still drops `wslProxyErrs` the way `Remove` used to.
Codex's initial-append race and Gemini's regex-hijack finding were both
reclassified on verification — see the Reconciliation paragraph in the
retro.

Structure of findings: 0 Critical, 1 Important, 3 Suggestions.

## Important Issues

I1. **`retryAppend` engine-takeover branch strands the shadowing rule when the cleanup Delete fails** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:279-294` [Claude Opus 4.7]

```go
if managed {
	log.Infof("/proc/net scanner engine chain appeared for %s during append retry; releasing ownership", port)
	if removeErr := pst.tracker.Remove(syntheticIDFor(port)); removeErr != nil {
		log.Errorf("/proc/net scanner failed to release ownership of %s after engine took over: %s", port, removeErr)
	}
	if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
		log.Errorf("/proc/net scanner deleting loopback iptable rules after engine takeover for portbinding: %v failed: %s", state.bindings, err)
	}
	pst.added[port] = portState{bindings: state.bindings, delegated: true}
	return
}
```

`retryAppend` runs when a prior tick set `appendFailed=true`. The
`applyLoopbackIPtablesRule` and `retryAppend` comments both state that a
failed Append "may have committed the rule (iptables committed but
returned non-zero)" — so the procnet `PREROUTING` DNAT may already be
installed. When the engine chain has appeared since, this branch deletes
that rule and marks the port `delegated:true`. But if
`applyLoopbackRules(..., Delete)` fails transiently — the same xtables-lock
contention that produced the original `appendFailed` — the error is only
logged, and the port still transitions to `delegated:true`.

From that state the rule is unreachable for cleanup:

`addObserved` (`port_state_tracker_linux.go:162-167`) sees `alreadyAdded`
with `appendFailed == false`, so `retryAppend` never runs again for this
port.

`retireDisappeared` (`port_state_tracker_linux.go:310-316`) sees
`delegated:true` and drops the local marker "without touching either" the
tracker entry or the iptables rule.

So a committed procnet `PREROUTING` DNAT survives, shadowing the engine's
`CNI-HOSTPORT-DNAT`/`DOCKER` chain, until an RD restart — the exact failure
this PR exists to eliminate. The `portState` doc comment
(`port_state_tracker_linux.go:64-75`) acknowledges a *proxy*-orphan window
in this branch, but says nothing about a Delete failure leaving the
shadowing *rule* in place; that case is genuinely unhandled. Unlike the
acknowledged proxy orphan, it does not self-heal — `retireDisappeared`
will never delete a delegated port's rule.

Reachability is low — it needs the stability gate bypassed under load, a
commit-but-errored Append, the engine chain appearing, *and* a transient
Delete failure (or a real host-port conflict plus two transient iptables
failures) — which is why this is Important rather than Critical.
`TestAppendRetryEngineTakeoverDeletesCommittedRule` covers the *success*
of this Delete; the fake `iptablesRunner` already supports per-action
error injection (`applyErrors[applyKey("tcp","8080",Delete)]`), so the
failure branch is one test stage away.

Fix: keep the port in `appendFailed` state when the Delete fails, so the
next tick re-enters `retryAppend`, re-probes the still-managed engine
chain, and re-attempts the Delete. `tracker.Remove` is a documented silent
no-op after the first call, so the re-entry is safe.

```diff
 		if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
 			log.Errorf("/proc/net scanner deleting loopback iptable rules after engine takeover for portbinding: %v failed: %s", state.bindings, err)
+			// Leave appendFailed=true so the next tick re-probes the
+			// engine chain and re-attempts the Delete. Marking the port
+			// delegated here would strand the shadowing rule until RD
+			// restart -- retireDisappeared never deletes a delegated
+			// port's rule.
+			return
 		}
 		pst.added[port] = portState{bindings: state.bindings, delegated: true}
 		return
```

## Suggestions

S1. **`portState.bindings` doc comment says "first sighting" but the set is captured at the second** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:33` [Claude Opus 4.7]

```go
//	bindings: the binding set captured at first sighting.
```

The state-machine description immediately below says the first sighting
only defers (`log.Debugf("...deferring port %s on first sighting")`,
line 169) and records nothing. `exposeNew` — the only writer of
`portState{bindings: bindings, ...}` — runs on the *second* sighting, so
the binding set reflects the second scan. The doc comment contradicts the
state machine it heads.

Fix: `// bindings: the binding set captured when the port was first acted on (its second sighting).`

S2. **`applyLoopbackRules` returns on the first per-binding error — benign today, but the semantics are untested** — `src/go/guestagent/pkg/procnet/port_state_tracker_linux.go:393-403` [Gemini 3.1 Pro, Claude Opus 4.7]

```go
func (pst *portStateTracker) applyLoopbackRules(bindings []nat.PortBinding, port nat.Port, act action) error {
	for _, binding := range bindings {
		if binding.HostIP != loopbackIP {
			continue
		}
		if err := pst.iptables.ApplyLoopbackRule(port.Proto(), binding.HostPort, act); err != nil {
			return err
		}
	}
	return nil
}
```

`applyLoopbackRules` returns on the first failing binding, skipping the
rest. For `Append` that is correct — the caller records `appendFailed` and
`retryAppend` re-runs the whole binding list. For `Delete` the caller
(`retireDisappeared`, `retryAppend`) drops the local marker regardless, so
any binding after the failing one would be skipped permanently.

In practice this is unreachable: `applyLoopbackRules` acts only on bindings
whose `HostIP` is exactly `127.0.0.1`, and a `nat.Port` can have at most
one such binding — two `LISTEN` sockets cannot share `127.0.0.1:<port>`.
Gemini rated this Important on the assumption of multiple loopback bindings
per port; that assumption does not hold for the procfs-derived port map, so
it is downgraded to a Suggestion. The residual value is that the
"first-error skips the remainder" semantics are untested and would silently
become a real leak if binding collection ever changed.

Fix (optional): accumulate per-binding errors with `errors.Join` so the
loop always processes every binding, and add a fixture with two loopback
bindings to pin the behavior.

S3. **`APITracker.RemoveAll` still drops `wslProxyErrs` when `apiErrs` is non-empty** — `src/go/guestagent/pkg/tracker/apitracker.go:250-258` [Gemini 3.1 Pro]

```go
if len(apiErrs) != 0 {
	return fmt.Errorf("%w: %+v", forwarder.ErrUnexposeAPI, apiErrs)
}

if len(wslProxyErrs) != 0 {
	return fmt.Errorf("%w: %+v", ErrWSLProxy, wslProxyErrs)
}
```

This PR fixed exactly this shape in `APITracker.Remove` — it now joins
`forwarder.ErrUnexposeAPI` and `tracker.ErrWSLProxy` with `errors.Join` so
neither is swallowed. The sibling `RemoveAll` keeps the old early-return:
when `apiErrs` is non-empty it returns immediately and `wslProxyErrs` is
lost, hiding the wsl-proxy failure from the caller.

`RemoveAll` is pre-existing code untouched by this PR, and its callers
(shutdown paths) are not on the procnet shadow-DNAT path, so this is a
consistency Suggestion rather than an in-scope defect — Gemini rated it
Important; downgraded. Still worth folding in: the PR already proves the
`errors.Join` pattern in the same file, and leaving the two siblings
divergent invites the next reader to trust the wrong one.

Fix: apply the same `errors.Join` accumulation `Remove` now uses.

## Design Observations

### Concerns

- *(future)* **`exposeNew` and `handleAlreadyExposed` do not re-probe the engine chain between the initial probe and the Append** — `port_state_tracker_linux.go:176-232` [Codex GPT 5.5]. `exposeNew` probes the engine chain once, then runs `tracker.Add`, then `applyLoopbackRules(..., Append)` with no re-probe; `handleAlreadyExposed`'s resume path appends with no fresh probe at all. If the engine chain lands during the Add-to-Append window, procnet installs a shadowing rule. This restates the Round 3 design observation "`exposeNew` does not re-probe the engine chain after a healthy `tracker.Add`" from the Add-window angle, and the code in this path is unchanged since Round 3. It is the same netns-blind / stability-gate-bypass limitation the `portState` doc comment already documents, and — unlike I1 — it self-heals: the port stays `delegated:false`, so when the transient listener disappears `retireDisappeared` deletes the rule. Codex rated it Important; reclassified as a design observation, consistent with Round 3. The fix (a re-probe immediately before each first Append) is the same shape `retryAppend` already uses and would be worth doing if the Add window is ever measured to be non-trivial.
- *(future)* **`dnatChainContainsPort` scans human-readable iptables output with a non-greedy line regex** — `scanner_linux.go:70-72`, `scanner_linux.go:264-281` [Gemini 3.1 Pro]. `dnatRuleRe` matches `\b(tcp|udp)\b[^\n]*?(?:dpt:N|dports N…)`; the `[^\n]*?` scans the whole line and the first `dpt:`/`dports` token wins, so a `dpt:`/`dports` token appearing earlier on the line would mask the real port (a false negative → procnet installs a shadowing rule). Gemini raised this as a Critical regression on the theory that a container named `dpt:80` injects that token into the iptables comment. That vector is not reachable: the `CNI-HOSTPORT-DNAT` comment carries the CNI *network* name and the CNI *container ID* (`/* dnat name: "bridge" id: "default-abc" */`), not the user's `--name`; and network/container names are validated to `[a-zA-Z0-9][a-zA-Z0-9_.-]*` — no colon, no space — so none of them can produce `dpt:N` or `dports N`. The finding was dropped as a Critical (see Reconciliation), but the underlying fragility of parsing presentation output is real and is the same point Round 3 recorded — `iptables -S <chain>` emits the machine-parseable rule-spec form and remains the fallback if a future iptables bump breaks the regex.
- *(future)* **procnet now couples to `APITracker`'s error wrapping at three points** — `port_state_tracker_linux.go:206`, `:330` [Claude Opus 4.7]. The state machine depends on the `portAlreadyExposedSubstring` substring match and on `errors.Is` against `ErrWSLProxy`/`ErrUnexposeAPI`. The sentinel coupling is sound — typed, in-repo, and pinned by `TestRemoveWSLProxyFailureIsDistinguishable`. The substring match is the fragile one: `"proxy already running"` originates in the third-party gvisor-tap-vsock forwarder and is not an exported symbol. `TestPortAlreadyExposedSubstringIsReachable` pins the wrap chain but, by its own comment, cannot guard an upstream rename. Worth a comment on the gvisor-tap-vsock dependency in `go.mod`; no in-scope action.

## Review Process Notes

### Skill improvements

- When an error from a cleanup, release, or rollback step is only logged —
  not propagated and not retried — check what state the surrounding code
  moves the entity into next. A logged-and-ignored cleanup failure is
  recoverable only if some later code path re-attempts it; if the code
  immediately transitions the entity to a terminal or delegated state whose
  handlers short-circuit the retry path, the failure is a permanent leak,
  not a transient one. Flag a logged-but-unhandled error from a cleanup step
  when the very next statement transitions the entity into a state that
  forecloses retry — the log line makes it look handled, but the resource
  is stranded until process restart.

### Repo context updates

- [repo] The `CNI-HOSTPORT-DNAT` chain's jump-rule comment renders in
  `iptables --list` output as `/* dnat name: "<network>" id: "<container-id>" */`
  — the CNI *network* name and the CNI *container ID*, never the container's
  user-chosen `--name`. Docker/nerdctl container and network names are
  validated to `[a-zA-Z0-9][a-zA-Z0-9_.-]*` (no colon, no whitespace). A
  finding that a user-chosen name can inject tokens such as `dpt:N` or
  `dports N` into iptables `--list` output, hijacking a parser regex, is not
  reachable through the engines Rancher Desktop ships — verify the claimed
  payload can actually pass name validation before rating such a finding.

# Code Review: PR #10280 — procnet: don't shadow CNI portmap with PREROUTING DNAT

### Executive Summary

This PR extracts the `/proc/net` scanner's per-port logic from an inlined heuristic in `ForwardPorts` into a unit-tested `portStateTracker` state machine, behind injectable `tracker.Tracker` and `iptablesRunner` interfaces. It adds a two-scan stability gate, engine-chain delegation (probing `CNI-HOSTPORT-DNAT`/`DOCKER`), partial-failure recovery, and append-retry — all to stop procnet from installing a `PREROUTING` DNAT that shadows the engine's portmap chain on iptables-nft. The change is well-structured and thoroughly tested; the one substantive finding is a rare error path in `retryAppend` that can leave the exact shadowing rule the PR sets out to eliminate.

### Findings

#### Critical Issues

None.

#### Important Issues

**I1. `retryAppend` engine-takeover branch strands the shadowing rule on a Delete failure — `pkg/procnet/port_state_tracker_linux.go:290-294`**

```go
		if err := pst.applyLoopbackRules(state.bindings, port, Delete); err != nil {
			log.Errorf("/proc/net scanner deleting loopback iptable rules after engine takeover for portbinding: %v failed: %s", state.bindings, err)
		}
		pst.added[port] = portState{bindings: state.bindings, delegated: true}
		return
```

When the engine chain appears during an append-retry, `retryAppend` deletes any committed procnet rule and marks the port `delegated:true`. But if `applyLoopbackRules(..., Delete)` fails transiently (the same xtables-lock contention that caused the original `appendFailed`), the error is only logged — the port still transitions to `delegated:true`. - When the listener disappears, `retireDisappeared` (line 310-316) sees `delegated:true` and drops the marker *without touching iptables*.

So a committed procnet `PREROUTING` DNAT survives — shadowing the engine's `CNI-HOSTPORT-DNAT`/`DOCKER` chain — until an RD restart. That is exactly the bug this PR exists to fix. Practical likelihood is low (it needs the stability gate bypassed under load, a commit-but-errored Append, the engine chain appearing, *and* a transient Delete failure — or a real host-port conflict plus two transient iptables failures), so this is Important rather than Critical. But unlike the acknowledged proxy-orphan, it does not self-heal on the next tick — it persists to restart, and the fix is trivial.


C1. **CNI `id` annotation can hijack `dnatRuleRe` causing false negatives** — `src/go/guestagent/pkg/procnet/scanner_linux.go:68` (critical, regression)


The regular expression `dnatRuleRe` (line 68) uses the non-greedy `[^\n]*?` to scan a line for `dpt:` or `dports `. If a user creates a container with a name containing `dpt:80` (e.g., `nerdctl run --name dpt:80 -p 8080:80 nginx`), the regex will match `dpt:80` inside the comment and terminate early.
