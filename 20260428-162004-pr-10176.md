## Executive Summary

Most of the Round 4 backlog landed in `e707246d7`: `RancherDashboard.download()` now uses a `.source-sha256` stamp (Round 4 I1); `downloadAndHash` reuses the sha256 it already computes when the upstream verify algorithm is sha256 (Round 4 S1); `GlobalDependency.updateManifest` re-reads from disk and invalidates the module cache after writing (Round 4 S5); `codesign` exit status is checked and the unsigned cache is dropped on failure (Round 4 S6); the mismatch artifact uses a constant filename so subsequent failures overwrite (Round 4 S7); the gnuLine regex now anchors to sha256/sha512 widths exactly (Round 4 S10); and `MobyOpenAPISpec.download()` rejects empty `apiVersion` or `commit` (Round 4 S4).

One Important regression survives: `downloadTarGZ`/`downloadZip` re-extract on every postinstall (Round 4 I2). On Windows, `fs.copyFile` then fails with `EBUSY`/`EPERM` overwriting a locked `helm.exe`/`docker.exe` while Rancher Desktop or another shell holds the binary; on Linux a running binary trips `ETXTBSY`. The pattern that `RancherDashboard.download()` adopted in this round (`.source-sha256` stamp + short-circuit) extends naturally to the archive helpers and is the recommended fix.

The remaining surface is suggestion-grade. The Round 4 S4 fix accepts any non-empty `commit` string, so `master` or a typo silently builds a mutable upstream URL and trips an "Upstream checksum mismatch" only at install — a 40-char hex regex would catch the misedit at the source. `fetchUpstreamChecksums` parses BSD `SHA256 (foo) = …` lines without checking the algorithm name, so a `.sha512sum` URL pointed at a sha256 sidecar parses successfully and produces a confusing diagnostic at verify time. `Moproxy.download()` deletes its archive immediately after extraction, so the `download()` cache check never fires and `xz` is re-fetched every postinstall. `AlpineLimaISO`'s old `updateManifest` runtime guard against string versions is gone now that the common implementation lives in `GlobalDependency`. Seven Round 4 suggestions remain unaddressed and carry forward.

Verdict: merge with the one Important fix folded in. Everything else is non-blocking polish that has accumulated across rounds.

## Important Issues

I1. **`downloadTarGZ`/`downloadZip` always re-extract, overwriting locked destinations** — `scripts/lib/download.ts:215-242, 263-281` [Claude Opus 4.7, Gemini 3.1 Pro] (round 4 I2 unaddressed)

```ts
// Re-extract on every postinstall so destPath always reflects the
// current archive, even after a version bump that re-downloaded
// tgzPath above.
const workDir = fs.mkdtempSync(path.join(os.tmpdir(), `rd-${ path.basename(destPath, '.exe') }-`));
const args = ['tar', '-zxf', tgzPath, '--directory', workDir, fileToExtract];

try {
  …
  await simpleSpawn(args[0], args.slice(1));
  await fs.promises.copyFile(path.join(workDir, fileToExtract), destPath);
  await fs.promises.chmod(destPath, mode);
}
```

The pre-PR helpers short-circuited when `destPath` already existed; Round 4 removed that guard so the verified archive's contents always reach `destPath`, but the unconditional `copyFile` clobbers any existing `destPath` even when nothing changed. A developer running `yarn postinstall` on Windows while Rancher Desktop is running fails with `EBUSY`/`EPERM` when `copyFile` tries to overwrite a locked `resources/win32/bin/helm.exe`, `docker.exe`, `kubectl.exe`, or any other Round-4-affected binary; on Linux the same scenario trips `ETXTBSY` for a running executable. macOS allows in-place overwrite of mapped binaries, so it is benign there.

The `download()` helper at `scripts/lib/download.ts:74-89` and the new `RancherDashboard.download()` (`scripts/dependencies/tools.ts:407-416, 449`) both implement the same checksum-stamp short-circuit; the archive helpers can adopt it directly:

```diff
   await download(url, tgzPath, { ...options, access: fs.constants.W_OK });
+
+  const stampPath = `${ destPath }.source-sha256`;
+  const stamp = await fs.promises.readFile(stampPath, 'utf-8').catch(() => '');
+  if (options.expectedChecksum && stamp === options.expectedChecksum) {
+    try {
+      await fs.promises.access(destPath, access);
+      return destPath;
+    } catch { /* fall through to extract */ }
+  }
   …
   await fs.promises.copyFile(path.join(workDir, fileToExtract), destPath);
   await fs.promises.chmod(destPath, mode);
+  if (options.expectedChecksum) {
+    await fs.promises.writeFile(stampPath, options.expectedChecksum, 'utf-8');
+  }
```

The new `electron-builder.yml` exclusions extend naturally to `*.source-sha256`. (important, regression)

## Suggestions

S1. **`MobyOpenAPISpec.download()` accepts any non-empty `commit`, defeating immutability** — `scripts/dependencies/moby-openapi.ts:30-37` [Claude Opus 4.7]

```ts
const { apiVersion, commit } = context.dependencies.mobyOpenAPISpec.version;

if (!apiVersion || !commit) {
  throw new Error(`mobyOpenAPISpec entry must specify both apiVersion and commit; got ${ JSON.stringify({ apiVersion, commit }) }`);
}
const baseUrl = `https://raw.githubusercontent.com/${ this.githubOwner }/${ this.githubRepo }/${ commit }/api/docs`;
```

The Round 4 S4 fix rejects empty `commit`, but the truthy check still admits `master`, `main`, a branch name, or a typo. `raw.githubusercontent.com/<owner>/<repo>/<ref>/...` accepts a branch ref and serves whatever that branch points at today. The recorded sha256 then almost certainly fails the install-time verification (so the security property holds), but the maintainer chases an "Upstream checksum mismatch" at install time instead of seeing a manifest-shape error at the source. The PR's stated immutability property — "the pinned commit makes the install URL immutable" (`scripts/dependencies/moby-openapi.ts:130-133`) — is not enforced by code.

Fix: tighten the guard to require a 40-char hex SHA:

```diff
-    if (!apiVersion || !commit) {
-      throw new Error(`mobyOpenAPISpec entry must specify both apiVersion and commit; got ${ JSON.stringify({ apiVersion, commit }) }`);
-    }
+    if (!apiVersion || !/^[0-9a-f]{40}$/i.test(commit ?? '')) {
+      throw new Error(`mobyOpenAPISpec entry must specify apiVersion and a 40-char commit SHA; got ${ JSON.stringify({ apiVersion, commit }) }`);
+    }
```

S2. **`fetchUpstreamChecksums` BSD branch ignores the declared algorithm name** — `scripts/lib/dependencies.ts:286-308` [Claude Opus 4.7]

```ts
const gnuLine = /^([0-9a-fA-F]{64}|[0-9a-fA-F]{128})\s+\*?(.+?)\s*$/;
const bsdLine = /^[A-Z0-9]+\s*\((.+?)\)\s*=\s*([0-9a-fA-F]+)\s*$/;
```

The BSD form embeds the algorithm name (`SHA256 (foo) = …` vs `SHA512 (foo) = …`) in `[A-Z0-9]+`, but the parser drops it. A caller that passes `verify: { algorithm: 'sha512', expected: <hex> }` against a misconfigured `.sha256sum` URL would parse successfully, then `downloadAndHash` hashes with sha512 and compares a 128-char digest against a 64-char string — the diagnostic reads `expected sha512:<sha256-hex>, got sha512:<real-sha512-hex>`. The mismatch is caught (no security loss), but the error misleads.

Fix: pass the requested algorithm into `fetchUpstreamChecksums(url, algorithm)` and reject lines whose width or BSD-name does not match.

S3. **`Moproxy.download()` deletes its archive, defeating the cache check** — `scripts/dependencies/wsl.ts:25-33` [Gemini 3.1 Pro]

```ts
await download(
  moproxyURL,
  archivePath,
  {
    expectedChecksum: lookupChecksum(context, this.name, archiveName),
    access:           fs.constants.W_OK,
  });

// moproxy uses xz with no tar wrapper; just decompress it manually.
await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
await fs.promises.rename(path.join(context.internalDir, binName), moproxyPath);
await fs.promises.rm(archivePath);
```

The `fs.promises.rm(archivePath)` at the end ensures `download()`'s cache-hit check never sees a hit on the next run — `archivePath` is gone, so `download()` re-fetches the xz archive from GitHub on every `yarn postinstall`. Pre-PR `moproxyURL` had the same shape but without the verified checksum, so the redundant download is not new; Round 4's I1 fix made every other dependency cache its archive, leaving `Moproxy` as the lone outlier.

Fix: keep `archivePath` after extraction so the next run hits `download()`'s cache, or check `moproxyPath` existence and short-circuit before calling `download()` (mirroring `RancherDashboard.download()`'s stamp pattern). Removing `fs.promises.rm(archivePath)` is the smallest change.

S4. **`AlpineLimaISO`'s string-rejecting `updateManifest` is gone with no runtime substitute** — `scripts/dependencies/lima.ts` (deleted lines per diff vs `27b856a46:scripts/dependencies/lima.ts:168-172`) [Claude Opus 4.7]

The pre-PR override threw `TypeError` if `updateManifest` was called with a string for `alpineLimaISO`:

```ts
if (typeof newVersion === 'string') {
  throw new TypeError(`AlpineLimaISO.updateManifest does not support string version ${ newVersion }`);
}
```

The new common implementation in `GlobalDependency` (`scripts/lib/dependencies.ts:492-507`) accepts whatever `Version` the caller passes and writes it through `(manifest as any)`. TypeScript still narrows the call chain — `AlpineLimaISO.latestVersion` returns an `AlpineLimaISOVersion` object — so no current caller can hit this path. The hole is defensive: a future refactor or a direct caller bypassing the type narrowing would silently write a malformed entry, after which every install fails on `version.alpineVersion` access.

Fix (optional): re-assert the per-name shape in `GlobalDependency.updateManifest` or in `writeDependencyManifest`. The check is one switch-or-table for a known set of entries.

S5. **`codesign` failure swallows `result.error` in the diagnostic** — `scripts/lib/download.ts:130-148` [Claude Opus 4.7]

```ts
const result = spawnSync(
  'codesign',
  ['--force', '--sign', '-', destPath],
  { stdio: 'inherit' },

if (result.status !== 0) {
  await fs.promises.rm(unsignedPath, { force: true });

When `spawnSync` itself fails (binary missing, permission denied, signal-killed), `result.status` is `null` and `result.error` carries the reason. The thrown message reads `codesign failed for X (exit null)` and discards the actionable detail. Cosmetic; the cache-cleanup branch still fires correctly.

Fix:

```diff

+    const detail = result.error ? `: ${ result.error.message }` : '';
+    throw new Error(`codesign failed for ${ destPath } (exit ${ result.status })${ detail }`);
```

S6. **`MobyOpenAPISpec.getChecksums` skips upstream cross-check without explanation** — `scripts/dependencies/moby-openapi.ts:92-97` [Claude Opus 4.7]

```ts

A future reader may "fix" the perceived gap by adding a verify call against a sidecar that does not exist for `raw.githubusercontent.com` paths.

S7. **`fetchUpstreamChecksums` warns and continues when the sidecar parses to zero entries** — `scripts/lib/dependencies.ts:330-332` [Claude Opus 4.7]

S8. **`download()`'s EACCES comment overstates the cause** — `scripts/lib/download.ts:96-101` [Claude Opus 4.7]

```ts
} catch (ex: any) {
  // EACCES means destPath exists but lacks the requested mode (e.g.,
  // a previous run wrote 0644 where this run wants X_OK).  Treat as
  // stale state and re-download.
  if (ex.code !== 'ENOENT' && ex.code !== 'EACCES') {
    throw ex;
  }
}
```

`EACCES` from `fs.promises.access` can also originate from the parent directory lacking search permission, not just the file mode. The fall-through behaviour is correct (the next `mkdir`/`rename` surfaces the real error), but the comment misleads anyone tracing a permission failure.

Fix: rewrite the comment to cover both cases — "EACCES from the parent directory or the destination itself; either way fall through and let `mkdir`/`rename` surface a clearer error."

S9. **Several `getChecksums` matrices omit windows/arm64** — `scripts/dependencies/tools.ts:58-62, 126-130, 169-174, 212-216, 555-559, 696-700, 371-375` [Gemini 3.1 Pro] (round 4 S2 unaddressed)

`KuberlrAndKubectl`, `Helm`, `DockerCLI`, `DockerBuildx`, `ECRCredHelper`, `Steve`, and `SpinCLI` enumerate a five-entry matrix (linux × {amd64,arm64}, darwin × {amd64,arm64}, windows × amd64); peers `DockerCompose` and `DockerProvidedCredHelpers` enumerate a six-entry matrix that includes windows/arm64. Each `download()` requests the windows arm64 artifact when `M1=1`, so a Windows ARM64 cross-build trips "No checksum recorded". The package workflow only sets `M1=1` on darwin/arm64, so no shipping CI build hits this path; the asymmetry remains a foot-gun for any developer cross-building on Windows ARM64.

Fix: drop `isM1` from the windows-only `download()` paths (so windows-arm64 fails up-front) or extend each `getChecksums` matrix with the missing entry.

S10. **Per-version archive caches accumulate after a bump** — `scripts/dependencies/wix.ts:45`, `scripts/dependencies/lima.ts:49,107` [Claude Opus 4.7] (round 4 S3 unaddressed)

`Wix`, `Lima`, and `Qemu` embed the version in the cache filename (`wix314rtm.zip`, `lima.macos-15.amd64.v1.2.1.rd3.tgz`, `qemu.v9.2.1.tgz`), so each bump leaves the prior archive on disk forever. The `.unsigned` sidecars introduced for codesigned binaries compound this. `electron-builder.yml:20-22` excludes them from the package, so this is a developer-disk-only concern.

Fix: drop the version from the cache filename, or sweep stale archives at postinstall start (keeping only the one whose basename matches the current `dependencies.yaml` version).

S11. **`downloadTarGZ`/`downloadZip` propagate `codesign` to the inner `download()` if a future caller passes it** — `scripts/lib/download.ts:214, 264` [Claude Opus 4.7] (round 4 S9 unaddressed)

## Design Observations

### Concerns

**D1. `MobyOpenAPISpec` commit pinning is invisible to upstream security fixes** *(future)* [Claude Opus 4.7]

`rcompareVersions` ignores `commit`, so a security fix to `v1.54.yaml` on `master` does not auto-bump. The recorded digest stays valid only against the pinned commit; adopting a newer commit on the same `apiVersion` requires a manual `dependencies.yaml` edit. Worth documenting in the maintenance notes so the manual-edit case is not lost.

**D2. `globalDependencies` and the `postinstall.ts` per-platform lists drift independently** *(future)* [Claude Opus 4.7]

`scripts/dependencies/global.ts:11-37` enumerates every class whose version+checksums live in `dependencies.yaml`; `scripts/postinstall.ts:43-107` partitions the same classes by platform/runtime. New deps must touch both files. A shared per-name map with a `.platforms` field would collapse the two and surface mismatches at compile time.

**D3. Bump-time `getChecksums` materialises every artifact for every platform** *(future)* [Claude Opus 4.7]

Each `getChecksums()` downloads each artifact, hashes, then deletes. A `--all` rebuild (e.g. recording every checksum after a process change) downloads gigabytes. If this becomes a hot path, switch `downloadAndHash` to a streaming hash that pipes the response body straight into `crypto.createHash('sha256')` — same correctness, no temp file.

**D4. Runtime Spin setup remains outside the digest model** *(future)* [Codex GPT 5.5]

`resources/setup-spin` still installs templates from GitHub tarball URLs and the kube plugin from a manifest under `refs/heads/main` (`resources/setup-spin:38, 41, 53`). That path runs at user-runtime, not build-time `postinstall`, so it is outside this PR's scope but is the remaining place where dependency-sourced bytes are not digest-pinned. If the immutability goal expands beyond build-time artifacts, this script needs its own pinned manifest.

### Strengths

**D5. Round 4 backlog cleared cleanly** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5, Gemini 3.1 Pro]

Round 4 I1, S1, S5, S6, S7, S10, and S4 all landed. The `RancherDashboard` stamp pattern is the right shape and is the natural template for the remaining I1; computing checksums before `git checkout -b` (`scripts/rddepman.ts:291-297`) keeps a failed bump from leaving an orphan branch.

**D6. The `MobyOpenAPISpec` commit-pinning design** *(in-scope)* [Claude Opus 4.7, Gemini 3.1 Pro]

Resolving the commit only on the winner (`latestVersion` getter) avoids one `listCommits` per historical version. `apiVersion` for comparison and `commit` for URL immutability is the right separation for a `master`-branched upstream.

**D7. `.unsigned` codesign cache is the cleanest option for in-place mutation** *(in-scope)* [Claude Opus 4.7, Codex GPT 5.5]

Keeping the verified bytes at `${destPath}.unsigned` and hashing them on cache checks integrates digest verification with `codesign`'s in-place rewrite without re-downloading or trusting the post-signing bytes. The cache-cleanup branch on codesign failure (Round 4 S6) keeps the failure recoverable.

**D8. Schema evolution preserves install-time simplicity** *(in-scope)* [Codex GPT 5.5, Gemini 3.1 Pro]

Storing every digest as `sha256:` even when upstream publishes sha512 confines multi-algorithm parsing to `rddepman` at bump time. Install-time `download()` only handles sha256. The compound entry shape mirrors the on-disk YAML, so the in-memory model and the file stay in sync without translation.

## Testing Assessment

No unit tests cover the new helpers. The build-script directory has minimal test infrastructure (`scripts/lib/__tests__/build-utils.spec.ts` is the only pre-existing spec). Specific scenarios that would catch the issues raised here:

1. `fetchUpstreamChecksums` parser: GNU vs BSD lines, with/without `*` binary marker, path-prefixed entries, mixed sha256/sha512 widths, lines with stray hex prefixes (covers S2).
2. `parseSha256Checksum`: rejects empty, wrong-length, non-hex, no-prefix; accepts uppercase and normalises.
3. `readDependencyManifest`: rejects missing `version`, rejects array `version`, accepts string and object forms, handles missing `checksums` field.
4. `lookupChecksum` artifact-name parity: each dependency's `getChecksums()` must produce filenames that its `download()` later passes to `lookupChecksum` (covers S9).
5. `download()` cache-hit with codesign=true and missing `.unsigned` triggers re-download; `downloadTarGZ`/`downloadZip` skip extraction when the stamp matches (covers I1).
6. `MobyOpenAPISpec.download()` rejects non-SHA `commit` (covers S1).
7. `rddepman` dry-run: `getChecksums()` failure aborts before any git mutation; `updateManifest` preserves the compound YAML shape.

End-to-end, the PR's primary functional test is "rddepman runs and produces a working bump PR" — demonstrated by `9d35ac4cc` and `53aa0796a` in the history (lima 1.2.1.rd2 → 1.2.1.rd3 was generated by this branch).

## Acknowledged Limitations

- **Forcing sha256 over upstream algorithms** (`scripts/dependencies/lima.ts:81` review thread): @mook-as questioned not using upstream sha512; @jandubois explained that uniform sha256 storage simplifies install-time verification by confining multi-algorithm parsing to `rddepman` at bump time. @mook-as accepted ("It's fine to use whatever is easier for you"). The implementation matches.
- **WiX has no upstream checksum file** (`scripts/dependencies/wix.ts:33-34`): "WiX doesn't publish upstream checksum files; we rely on the digest recorded in dependencies.yaml at bump time."
- **Spin shim has no upstream checksum file** (`scripts/dependencies/tools.ts:605-606`): "Upstream does not publish a checksum file, so we record the sha256 we observe at bump time."
- **Docker buildx omits darwin from upstream verification** (`scripts/dependencies/tools.ts:208-210`): "Upstream checksums.txt omits darwin entries (https://github.com/docker/buildx/issues/945), so we hash darwin without upstream verification."
- **Upstream linux/arm64 not yet published for QEMU** (`scripts/dependencies/lima.ts:96-99, 119`): the `download()` skip and the `getChecksums()` matrix omission mirror each other; both reference the same TODO.
- **Stamp absence on crash starts over** (`scripts/dependencies/tools.ts:405-408`): "A crash between mkdir and tar leaves the stamp absent, so the next run starts over." Same property would apply to the I1 fix.

**I2. `fetchUpstreamChecksums` ignores the algorithm declared by BSD-format sidecars** — `scripts/lib/dependencies.ts:286-308`

```ts

The BSD form embeds the algorithm name (`SHA256 (foo) = ...` vs `SHA512 (foo) = ...`) in `[A-Z0-9]+`, but the parser drops it. A caller that asks for sha512 verification will silently consume a SHA256 line if the sidecar happens to be sha256 — `verify: { algorithm: 'sha512', expected: <sha256-hex> }` then hashes the file with sha512, gets a 128-char digest, compares it to a 64-char string, and dies with a confusing `expected sha512:<sha256>, got sha512:<real-sha512>` message. The GNU branch has the same exposure: width disambiguates between sha256 and sha512, but if the wrong-width sidecar URL is configured, parse succeeds without diagnostic. (important, gap)

Fix: capture the algorithm name and reject lines whose width or BSD-name does not match the caller's request. Pass the requested algorithm into `fetchUpstreamChecksums(url, algorithm)` and use it both to filter lines and to make error messages accurate.

## Design Observations

### Concerns

**Bump-time artifact downloads are O(deps × platforms)** *(future)* — Every `getChecksums()` call downloads every artifact for every platform, hashes, then throws the bytes away. For a single-dep bump this is acceptable; a `--all` rebuild (e.g., re-recording every checksum after a process change) downloads gigabytes. If this becomes a hot path, switch `downloadAndHash` to a streaming hash that doesn't materialize the file (`response.body.pipeTo(stream.Writable.toWeb(crypto.createHash('sha256')))`) — same correctness, no temp file, no `workDir`.

**`postinstall.ts`/`global.ts` duplication invites drift** *(future)* — `globalDependencies` enumerates every class whose version+checksums live in `dependencies.yaml`; `postinstall.ts` enumerates per-platform partitions of the same classes. New deps require touching both. A shared per-name map, with a `.platforms` field on each class describing where it runs, would collapse the two.

### Strengths

- The `MobyOpenAPISpec` design — `apiVersion` for comparison, `commit` for download URL immutability — is the right way to handle a `master`-branched upstream. Pinning the commit only on the winner (`latestVersion` getter) avoids one `listCommits` call per historical version.
- `download.ts` codesign refactor — keep `.unsigned` next to `destPath`, hash that for cache checks — is the cleanest way to integrate digest verification with in-place mutation. Good comment placement at `scripts/lib/download.ts:66-69`.
- `downloadAndHash`'s persistent `keepPath` for mismatches is exactly the right diagnostic; HTML error pages from CDNs are far easier to triage with the bytes than from a digest mismatch alone.
- The `readDependencyManifest` runtime validator is robust (rejects missing `version`, rejects array values, validates each checksum) and `parseSha256Checksum` is strict. Together they catch most hand-edit corruption at parse time, not at install time.

## Testing Assessment

No unit tests cover the new helpers, and the build-script directory has minimal test infrastructure (`scripts/lib/__tests__/build-utils.spec.ts` is the only pre-existing spec). Specific scenarios that would benefit from tests:

**Strengths** — aspects of the design that are particularly well done.

### [Gemini]

```markdown
This PR successfully upgrades Rancher Desktop's dependency management to verify downloads against pinned checksums and immutable references instead of fetching mutable upstream checksum files at install time. This significantly hardens the supply chain against man-in-the-middle attacks or upstream repository compromises. However, the caching logic modifications introduce a significant performance regression for `yarn postinstall`, and there is an unhandled edge case for Qemu on Linux ARM64.

### Findings

#### Critical Issues
None.

#### Important Issues

I1. **Unconditional archive extraction degrades `postinstall` performance** — `scripts/lib/download.ts:206-224`

```typescript
  // Persist the verified archive next to destPath; download() handles
  // the cache-hit checksum check against the manifest.
  await download(url, tgzPath, { ...options, access: fs.constants.W_OK });

  // Re-extract on every postinstall so destPath always reflects the
  // current archive, even after a version bump that re-downloaded
  // tgzPath above.
  const workDir = fs.mkdtempSync(path.join(os.tmpdir(), `rd-${ path.basename(destPath, '.exe') }-`));
  const args = ['tar', '-zxf', tgzPath, '--directory', workDir, fileToExtract];
```

By removing the `fs.promises.access(destPath)` short-circuit that previously existed, `downloadTarGZ` and `downloadZip` now unconditionally execute `tar` and `unzip` extractions every time `yarn postinstall` runs. While relying on `download()` to avoid redownloading the archive fixes the caching bug during version bumps, running disk-heavy extractions for all dependencies (Helm, Steve, Trivy, WSLDistro, etc.) on every build significantly slows down local development. 

**Fix**: Implement a `.stamp` file caching mechanism for the extracted output, similar to the one introduced in this PR for `RancherDashboard.download`.
```typescript
  const expectedChecksum = lookupChecksum(context, /* ... */);
  const stampPath = `${ destPath }.source-sha256`;
  const stamp = await fs.promises.readFile(stampPath, 'utf-8').catch(() => '');

  if (stamp === expectedChecksum) {
    console.log(`${ destPath } already extracted with expected checksum.`);
    return destPath;
  }

  // ... extract ...

  await fs.promises.writeFile(stampPath, expectedChecksum, { encoding: 'utf-8' });
```

I2. **Qemu download crashes on Linux ARM64 due to missing checksum** — `scripts/dependencies/lima.ts:114`

```typescript
  async getChecksums(version: string): Promise<Record<string, Sha256Checksum>> {
    const baseUrl = `https://github.com/${ this.githubOwner }/${ this.githubRepo }/releases/download/v${ version }`;
    // Upstream does not yet publish linux/arm64; mirror the skip in download().
    const platforms: [string, string][] = [
      ['darwin', 'x86_64'], ['darwin', 'aarch64'],
      ['linux', 'x86_64'],
    ];
```

The comment claims to "mirror the skip in download()", but `Qemu.download` does NOT skip Linux ARM64. If run on a Linux ARM64 host, `Qemu.download` computes the `archiveName` as `qemu-...-linux-aarch64.tar.gz`, passes it to `lookupChecksum()`, and immediately throws an error because `getChecksums` correctly omitted it from `dependencies.yaml`.

**Fix**: Add the missing platform guard to `Qemu.download`:
```diff
   async download(context: DownloadContext): Promise<void> {
     if (context.platform === 'win32' || context.dependencyPlatform === 'wsl') {
       return;
     }
+    if (context.platform === 'linux' && context.isM1) {
+      return; // Upstream does not yet publish linux/arm64
+    }
```

#### Suggestions

S1. **Moproxy download prevents caching by deleting its archive** — `scripts/dependencies/wsl.ts:25-33`

```typescript
    await download(
      moproxyURL,
      archivePath,
      {
        expectedChecksum: lookupChecksum(context, this.name, archiveName),
        access:           fs.constants.W_OK,
      });

    // moproxy uses xz with no tar wrapper; just decompress it manually.
    await simpleSpawn('7z', ['e', archivePath], { cwd: context.internalDir });
    await fs.promises.rename(path.join(context.internalDir, binName), moproxyPath);
    await fs.promises.rm(archivePath);
```

By deleting `archivePath` at the end of the method, `Moproxy.download` guarantees that `download(moproxyURL, archivePath, ...)` will never hit the disk cache. As a result, the `xz` binary is redundantly downloaded from the network every time `postinstall` runs.

**Fix**: Do not delete `archivePath` after extracting it, so that future cache hits succeed. Alternatively, adopt the `.stamp` check approach to avoid hitting the download path entirely.
