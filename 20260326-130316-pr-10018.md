## Executive Summary

This PR replaces Steve's hardcoded port 9443 with dynamically allocated ephemeral ports, resolving a long-standing conflict with other software (notably Logitech GHub). The implementation is clean: `getAvailablePorts()` binds all ports simultaneously to guarantee distinct values, the dashboard proxy is recreated per Steve start via wrapper functions that survive route re-registration, and a new `/api/steve-port` endpoint lets the dashboard discover the port at runtime. The health check was iteratively improved from raw TCP to an HTTPS probe against `/v1/namespaces`. All three agents agreed the code is well-structured; the only important finding (dashboard button enabled despite Steve failure) is already filed as #10060.

1. **Steve startup failure still enables the dashboard UI** — `background.ts:1278-1297` [Codex GPT 5.4, Gemini 3.1 Pro] (important, regression)

```typescript
if (enabledK8s) {
  try {
    const [stevePort] = await getAvailablePorts(1);
    setSteveCertPort(stevePort);
    await Steve.getInstance().start(stevePort);

```

If `Steve.start()` throws, the catch block at line 1288 logs the error but execution continues to `mainEvents.emit('k8s-check-state', mgr)` at line 1296. The renderer enables the dashboard button solely based on `k8sState === K8s.State.STARTED`, so the dashboard appears accessible even though Steve is down. `git blame` attributes the swallowing `catch` to `ba16a032a`. Already filed as #10060 and acknowledged by reviewer — not blocking merge.

Fix: either gate `k8s-check-state` emission on Steve success, or add a separate Steve-ready signal. The deferred approach (let Steve self-select a port and report it) would also solve this.

## Suggestions

1. **Race condition on fast K8s restart with port mismatch** — `steve.ts:72-78` [Gemini 3.1 Pro] (suggestion, regression)

```typescript
public async start(httpsPort: number) {
  const { pid } = this.process || { };
  if (this.isRunning && pid) {
    console.debug(`Steve is already running with pid: ${ pid }`);
    return;
  }
```

If K8s restarts quickly before the old Steve process emits its `close` event, `isRunning` is still `true` when `start()` is called with a new port. The method silently returns, and `background.ts` proceeds to update the proxy to the new (unused) port. In practice, K8s restart cycles take many seconds while Steve exits promptly on SIGINT, making this window negligible. Rated as suggestion because the practical likelihood is near zero.

Fix: either make `Steve.stop()` return a promise that resolves on process exit, or guard against mismatched ports:
```diff
 if (this.isRunning && pid) {
+  if (this.httpsPort !== httpsPort) {
+    throw new Error(`Steve is shutting down on port ${ this.httpsPort }`);
+  }
   console.debug(`Steve is already running with pid: ${ pid }`);
   return;
 }
```

2. **Test asserts behavior outside `getAvailablePorts` contract** — `networks.spec.ts:20-33` [Codex GPT 5.4] (suggestion, regression)

-      // ... rebind attempt
-    }

4. **Dashboard proxy not cleared when Steve stops** — `background.ts:1299-1301` [Gemini 3.1 Pro] (suggestion, gap)

```typescript
if (state === K8s.State.STOPPING) {
  Steve.getInstance().stop();
}
```

When Steve stops, `DashboardServer` still holds proxy middleware targeting the old ephemeral port. If another process binds that port, requests would be proxied to the wrong service. In practice, when K8s is STOPPING the dashboard button is disabled, so no user-initiated requests reach the proxy. Defense-in-depth, not a real exposure.

Fix: `DashboardServer.getInstance().setStevePort(0)` after `Steve.stop()`.

5. **Process error listener remains attached after spawn** — `steve.ts:110-112` [Gemini 3.1 Pro] (suggestion, gap)

```typescript
this.process.once('error', (err) => {
  reject(new Error(`Failed to spawn Steve: ${ err.message }`, { cause: err }));
});
```

The `.once('error')` listener is meant to catch spawn failures. If the process spawns successfully, this listener remains attached and would silently swallow a later runtime error by calling `reject()` on an already-resolved promise. Harmless in practice (Node.js ignores duplicate promise resolutions).

Fix: remove the listener upon successful spawn.

6. **Inaccurate timeout message in `waitForReady`** — `steve.ts:140-142` [Gemini 3.1 Pro] (suggestion, regression)

```typescript
throw new Error(`Steve did not become ready after ${ maxAttempts * delayMs / 1000 } seconds`);
```

The error message calculates `60 * 500 / 1000 = 30` seconds, but each loop iteration also awaits `isPortReady()` which has its own 1000ms timeout. If every attempt times out, actual elapsed time is up to 90 seconds, not 30.

Fix: use attempt count instead of calculated time:
```diff
-throw new Error(`Steve did not become ready after ${ maxAttempts * delayMs / 1000 } seconds`);
+throw new Error(`Steve did not become ready after ${ maxAttempts } attempts`);
```

7. **Redundant `setCertificateVerifyProc` on each health check** — `steve.ts:158-173` [Claude Opus 4.6] (suggestion, enhancement)

```typescript
const session = Electron.session.fromPartition('steve-healthcheck', { cache: false });
session.setCertificateVerifyProc((request, callback) => { ... });
```

`Electron.session.fromPartition()` returns the same session for a given partition name. The `setCertificateVerifyProc` call overwrites the previous callback on every iteration (up to 60 times in `waitForReady`). The function is identical each time, so this is harmless but redundant.

Fix: set the callback once in the constructor or on first call.

### Design Observations

**Concerns**

1. `(future)` The app still pre-allocates Steve’s port in `getAvailablePorts()` and then races Steve to bind it. ### Gemini 3.1 Pro Review

### Executive Summary

This PR modifies the Steve API to listen on a dynamically assigned ephemeral port instead of a hardcoded port to prevent conflicts with other applications. Overall, the dynamic port binding and proxy middleware updates are implemented well; however, the change introduces critical race conditions with Steve's lifecycle that can misalign the dashboard proxy and break the UI, as well as a security risk where the proxy continues routing to released ephemeral ports after Steve stops.

### Critical Issues

1. **Dashboard proxy and Steve state break during fast K8s restarts** — `background.ts:1280-1287` `(critical, regression)`

```typescript
// background.ts
            const [stevePort] = await getAvailablePorts(1);
            setSteveCertPort(stevePort);
            await Steve.getInstance().start(stevePort);
            DashboardServer.getInstance().setStevePort(stevePort); // recreate proxy middleware

// pkg/rancher-desktop/backend/steve.ts
  public async start(httpsPort: number) {
    const { pid } = this.process || { };

    if (this.isRunning && pid) {
      console.debug(`Steve is already running with pid: ${ pid }`);
      return;
    }
```

When K8s stops, `Steve.getInstance().stop()` sends an asynchronous `SIGINT` to the process. `this.isRunning` remains `true` until the child process emits its `close` event. If K8s restarts quickly and emits `STARTED` before the old Steve process has fully exited, `background.ts` unconditionally allocates a *new* dynamic port and calls `Steve.start(newPort)`. 

Because `this.isRunning` is still `true`, `Steve.start()` silently ignores the request and returns immediately. `background.ts` then proceeds to update the dashboard proxy and certificate handlers to point to the newly allocated (and unused) port. Moments later, the old Steve process finally exits. The result: Steve is dead, and the proxy points to a dead port. Because ports were previously hardcoded, Steve would eventually just restart on the same port; with dynamic ports, the proxy completely loses track of Steve's location.

**Fix:** 
Either make `Steve.stop()` asynchronous so callers wait for it to exit, or make `Steve.start()` handle restarts. To minimally prevent proxy corruption, `Steve.start()` should throw an error if called with a mismatched port while running:
```diff
     if (this.isRunning && pid) {
+      if (this.httpsPort !== httpsPort) {
+        throw new Error(`Steve is shutting down or running on a different port (${ this.httpsPort })`);
+      }
       console.debug(`Steve is already running with pid: ${ pid }`);
 
       return;
     }
```

### Important Issues

2. **Dashboard proxy routes to released ephemeral ports after Steve stops** — `background.ts:1299-1301` `(important, gap)`

```typescript
      if (state === K8s.State.STOPPING) {
        Steve.getInstance().stop();
      }
```

When K8s stops, Steve is killed and its dynamic port is returned to the OS. However, `DashboardServer` is never notified, so its active proxy middleware continues attempting to route `/v1`, `/k8s`, and other API requests to `127.0.0.1:<released-port>`. If another local application binds to that ephemeral port (which is highly likely, as ephemeral ports are frequently reassigned), any background requests or interactions in an open Rancher Dashboard window will be proxied directly to the unrelated application.

**Fix:** explicitly clear the proxy configuration when Steve is stopped so that connections fall through or fail safely.

```diff
       if (state === K8s.State.STOPPING) {
         Steve.getInstance().stop();
+        DashboardServer.getInstance().setStevePort(0);
       }
```

### Suggestions

3. **Process error listener remains attached after successful spawn** — `pkg/rancher-desktop/backend/steve.ts:110-112` `(suggestion, gap)`

```typescript
      this.process.once('error', (err) => {
        reject(new Error(`Failed to spawn Steve: ${ err.message }`, { cause: err }));
      });
```

The `.once('error')` listener is intended to reject the startup promise if Steve fails to spawn. If the process spawns successfully, the promise resolves, but this listener remains attached. If the process emits an `error` event later during its runtime (e.g., a failure to kill the process), this dangling listener will catch it and call `reject()` on the already-resolved promise, silently swallowing the runtime error instead of logging it. 

**Fix:** remove the listener upon successful spawn, or use `.on('error')` to log post-startup errors.
```typescript
    await new Promise<void>((resolve, reject) => {
      const onSpawn = () => {
        this.isRunning = true;
        this.process.removeListener('error', onError);
        resolve();
      };
      const onError = (err: Error) => {
        reject(new Error(`Failed to spawn Steve: ${ err.message }`, { cause: err }));
      };
      this.process.once('spawn', onSpawn);
      this.process.once('error', onError);
    });
```

4. **Inaccurate timeout calculation in `waitForReady` loop** — `pkg/rancher-desktop/backend/steve.ts:140-142` `(suggestion, regression)`

```typescript
      await setTimeout(delayMs);
    }

    throw new Error(`Steve did not become ready after ${ maxAttempts * delayMs / 1000 } seconds`);
```

The error message hardcodes the timeout logic as `60 * 500 / 1000 = 30` seconds. However, the loop awaits `this.isPortReady()`, which has its own internal timeout of 1,000ms if the HTTP request hangs. If `isPortReady()` times out on every attempt, each loop iteration takes `1000ms + 500ms = 1.5s`, meaning the loop could actually take up to 90 seconds before throwing. 

**Fix:** Record the actual start time before the loop or change the error message to reflect the number of attempts.
```diff
-    throw new Error(`Steve did not become ready after ${ maxAttempts * delayMs / 1000 } seconds`);
+    throw new Error(`Steve did not become ready after ${ maxAttempts } attempts`);
```

### Design Observations

**Strengths:**
- **Strict WebSocket upgrade matching:** The rewriting of the WebSocket routing in `DashboardServer`'s `upgrade` event correctly tightens the logic from `.startsWith('/v1')` to exact or path-prefix matches (`req.url === key || req.url?.startsWith(key + '/')`). This prevents accidental upgrades for unrelated routes.
- **Robust port allocation:** `getAvailablePorts` handles dynamic binding cleanly, and mapping `Promise.all` alongside `.close()` accurately guarantees port release without risking unhandled promise rejections, as `net.Server.close` correctly passes an error into the resolve callback for non-listening servers instead of throwing synchronously.
- **Efficient session management:** Using `Electron.session.fromPartition` with a static partition name (`'steve-healthcheck'`) efficiently reuses the same session object across all health checks without leaking memory.

### Acknowledged Limitations

- **Dashboard button enablement** (`background.ts:1295`): "Steve: Do not enable button if Steve fails to start correctly". Because `k8s-check-state` emits `STARTED` regardless of exceptions in `Steve.start()`, the UI dashboard button is enabled even if Steve fails. This was explicitly acknowledged by the author and filed as `#10060`.
- **Orphaned timers in health check** (`pkg/rancher-desktop/backend/steve.ts:195`): The 1000ms timer in `isPortReady` is left running even if the health check request succeeds early. The reviewer noted that calling `req.abort()` on a completed request is a documented safe no-op and resolving the promise twice is correctly ignored by Node.js.
