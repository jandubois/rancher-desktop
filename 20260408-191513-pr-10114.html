Executive Summary¶

This PR rewrites the Latch utility from a thin new Promise() wrapper into a resettable thenable with generic type parameters, adding a reset() method for reconnection scenarios. The implementation uses a dual-promise state machine (Promise.allSettled over value and reset signals) that handles concurrent waiters, resets, and rejection cleanup correctly. One test targets the wrong method and should be fixed; the remaining findings are minor style and coverage improvements.

Important Issues¶

I1. Test exercises resolve() instead of reset() for unhandled rejection prevention Gemini 3.1 Pro¶

The comment claims the test verifies that internal promise rejection does not cause an UnhandledPromiseRejectionWarning. But resolve() never rejects a promise — it resolves both state.promise and state.reset. The method that internally rejects the old promise is reset(), where oldPromise.promise.catch(() => {}) guards against the warning. This test passes vacuously and does not verify the guard.

pkg/rancher-desktop/utils/__tests__/latch.spec.ts
  it('can be resolved without any handlers', async() => {
    // If the latch resolves and it internally rejects a promise, it should not
    // cause an UnhandledPromiseRejectionWarning even if we never await the latch.
    Latch().resolve();

    // Wait a tick to allow any unhandled microtask rejections to surface.
    await new Promise(resolve => setTimeout(resolve, 10));
  }, 100);

The resolve() method (lines 124-127) resolves both underlying promises; it never triggers rejection:

pkg/rancher-desktop/utils/latch.ts
    resolve(value: T) {
      state.promise.resolve(value);
      state.reset.resolve(ResetState.SETTLED);
    },

The reset() method (lines 138-141) is where the rejection guard lives:

pkg/rancher-desktop/utils/latch.ts
      oldReset.resolve(ResetState.RESET);
      // Prevent UnhandledPromiseRejectionWarning, because we don't know what
      // dummy value to resolve with.
      oldPromise.promise.catch(() => {});

I2. Missing test: multiple concurrent awaiters on the same Latch Claude Opus 4.6¶

Every test exercises a single caller awaiting the latch. In production, multiple callers await the same Latch concurrently — protocolsRegistered is awaited at two sites in background.ts, and hasCredentials is awaited by every concurrent fetchAPI call in credentials.ts. The new implementation starts an independent waitForSettled() loop per .then() call. While correct, this fan-out behavior through the custom thenable protocol should be pinned by a test.

Fix: add a test:

it('supports multiple concurrent awaiters', async() => {
  const latch = Latch<number>();
  const p1 = expect(latch).resolves.toBe(42);
  const p2 = expect(latch).resolves.toBe(42);
  setTimeout(() => latch.resolve(42), 0);
  await Promise.all([p1, p2]);
}, 100);

Design Observations¶

Strengths¶

  • Dual-promise state machine. The waitForSettled loop using Promise.allSettled over value and reset signals is an elegant mechanism for resettable promises. Existing awaiters transparently loop to the new generation after a reset, with no caller coordination required. Claude Opus 4.6 Codex GPT 5.4 Gemini 3.1 Pro
  • Correct rejection cleanup. The reset() method at line 140 attaches .catch(() => {}) before rejecting the old promise, preventing UnhandledPromiseRejectionWarning when no consumer has attached a handler. Claude Opus 4.6 Gemini 3.1 Pro
  • Backward-compatible API. The rewrite preserves the existing await latch pattern while adding reset() and generic typing. Existing call sites (credentials.ts, k3sHelper.ts, iterator.ts, protocols.ts) require no changes. Codex GPT 5.4

Testing Assessment¶

The test suite covers the core Latch lifecycle: resolve, reject, reset, chained then/catch/finally, pre-resolution, invalid callbacks, and reset-while-pending.

Untested scenarios ranked by risk:

  1. Multiple concurrent awaiters (I2) — exercises the fan-out behavior of the custom then under the production usage pattern.
  2. Unhandled rejection on reset (I1) — the existing test targets the wrong method and proves nothing about the guard.
  3. finally on rejected latch (S4) — exercises the rejection propagation path in finally.
  4. Idempotent resolve/reject — calling resolve() twice or resolve() then reject() should be a no-op for the second call.
  5. .then(onFulfilled, onRejected) where latch rejects — the onrejected parameter of then is never tested directly.

Documentation Assessment¶

The JSDoc on the Latch interface and factory function is adequate. Inline comments in waitForSettled and reset explain the non-obvious mechanisms. The misleading comment at line 68 is flagged in S1.

Acknowledged Limitations¶

  • Line 73: // This should never reject. — documents that state.reset is never expected to reject; the throw r.reason is defensive. The change does not alter the likelihood of this path being reached.
