### Custom dependency manifest

`pkg/rancher-desktop/assets/dependencies.yaml` — 25 entries, 22 with digest-verified downloads. Every download path goes through `download()` in `scripts/lib/download.ts:61`, which streams the bytes to a temp file, computes `sha256`, and refuses to rename to the destination unless the digest matches `expectedChecksum`. The expected value is fetched via `lookupChecksum()` (`scripts/lib/dependencies.ts:202`), which throws when the entry has no recorded checksum — so a regression that drops a checksum block fails fast at install time. `rddepman` (the auto-bumper) cross-checks every artifact against the upstream `*.sha256sum`/`*.sha512sum` sidecar in `downloadAndHash` (`scripts/lib/dependencies.ts:232`) before writing the new digest.

## Open issues

### Top tier — `curl|bash` equivalents and self-fetches from a moving ref

- `scripts/windows-setup.ps1:49` — `Invoke-WebRequest -UseBasicParsing -Uri 'https://get.scoop.sh' | Invoke-Expression` is the PowerShell equivalent of `curl … | sh`: the Scoop installer's bytes are fetched and executed with no checksum pin and no signature verification. A maintainer's machine doing fresh dev-environment setup is fully exposed if `get.scoop.sh` is ever served corrupted bytes. Pin the installer to a SHA (download to disk, verify a stored sha256, then dot-source).
- `scripts/windows-setup.ps1:74` — `https://raw.githubusercontent.com/rancher-sandbox/rancher-desktop/main/scripts/windows/$file`. The dev-setup script downloads four PowerShell installers from its own repo's `main` branch via `raw.githubusercontent.com`. Anyone with push access to `main` (or a compromised CI bot) can rewrite `install-wsl.ps1` and the next `windows-setup.ps1` run elevates to admin and executes the new bytes. Pin to a 40-hex SHA, vendor the four scripts into `windows-setup.ps1`'s sibling directory, or load them via relative path from the cloned working tree.

### Mid tier — inline downloads, run-from-network, and tool-archive installs

- `bats/Makefile:39` — `wget --no-verbose $(JQ_URL)/jq-osx-amd64 -O $@` downloads `jq` 1.7.1 with no integrity check. The matching Linux fetch sits at line 44 (same shape, same gap). Pin the upstream `sha256sum` next to the URL and run `sha256sum --check` between `wget` and the `chmod +x`.
- `.github/actions/setup-environment/action.yaml:45` — `curl --location --output "$bindir/yq.exe" https://github.com/mikefarah/yq/releases/download/v4.43.1/yq_windows_amd64.exe` runs in a composite action used by every Windows CI job. mikefarah/yq publishes a `checksums` sidecar per release; download it, verify, then `chmod +x`.
- `.github/workflows/smoke-test.yaml:66` — `curl -L -o rancher-desktop.AppImage "${OBS_DOWNLOAD_URL}${artifact_name}"`. The follow-up comment (`# The AppImage does not have a checksum; make one up.`) makes the gap explicit: smoke-test then runs `sha512sum rancher-desktop.AppImage > rancher-desktop.AppImage.sha512sum` so the verification loop in `smoke-test.sh:50` finds *something* to check, but that "checksum" is computed locally from the freshly downloaded bytes and verifies nothing. Either publish a real `.sha512sum` from OBS and consume it here, or note that this AppImage path is unauthenticated and gate it behind a separate "untrusted smoke" mode.
- `.github/workflows/smoke-test/install-from-repo.sh:21` — `curl -s https://download.opensuse.org/repositories/isv:/Rancher:/dev/deb/Release.key | gpg --dearmor > "${keyLocation}/keyrings/isv-rancher-dev-archive-keyring.gpg"`. The downloaded key bytes anchor every subsequent `apt-get install rancher-desktop`, but the script trusts whatever OBS serves on first contact (TOFU) — there is no `gpg --show-keys` against a hard-coded fingerprint constant, the way HashiCorp/Docker/Kubernetes pin their APT keys. Add a `gpg --show-keys --with-fingerprint` check against the upstream Rancher signing key fingerprint before importing.
- `.github/workflows/linux-release.yaml:36` — `curl -L -o "dist/${release_zip_name}" "https://github.com/${repository}/releases/download/${version_with_v}/${release_zip_name}"`. Triggered by `release: published`, downloads the just-published Linux zip and re-uploads to S3 + triggers OBS. No digest check — if a release author swaps the asset between publish and this workflow's run, OBS gets the new bytes. Either consume the `.sha512sum` artifact (already published alongside) or have OBS pull from S3 with a known-good digest.
- `.github/workflows/test.yaml:20` — `go install mvdan.cc/sh/v3/cmd/shfmt@latest` is the only `@latest` ref in the run-from-network set; every other `go install` in the repo pins a concrete version. Pin to the `shfmt` version that matches the `bats/Makefile` lint expectations.
- `resources/setup-spin:39` — `"$spin" templates install --update --tar "$url"` where `$url` is a `spinframework/${repo}/archive/refs/tags/${SPIN_TEMPLATES_TAG}.tar.gz` URL, with a fallback to `archive/refs/heads/main.tar.gz` (line 42) when the tag fetch fails. The `spinKubePlugin` entry in `dependencies.yaml` carries an empty `checksums:` block and explicitly delegates to this script (`scripts/dependencies/tools.ts:730`), so the `dependencies.yaml` digest discipline does not extend here. The fallback to `main` is the bigger problem — a transient 404 on the tagged tarball silently downgrades to whatever `main` ships today. Fetch the tarball with `curl`, verify against a stored sha256, then point `spin templates install --tar` at the verified file; remove the `main`-branch fallback or stop letting it overwrite the destination silently.
- `scripts/spelling.sh:66` — `git -c advice.detachedHead=false clone --branch "$version" --depth 1 "$repo" "$checkout"` against `https://github.com/check-spelling/check-spelling`. `$version` comes from `dependencies.yaml`'s `check-spelling.version` (currently a tag), and the entry's `checksums:` block is empty (`scripts/dependencies/tools.ts:286`). Tags are mutable; clone via `git fetch` followed by `git checkout <40-hex>` recorded under a new `commit:` field in the manifest (mirrors what `mobyOpenAPISpec` already does in `scripts/lib/dependencies.ts:40`).

## Retro: dependency types not explicitly covered

### R1. Verifier outside the ±30-line window paired with a downstream script

**Where**: `.github/workflows/smoke-test.yaml:48` (`gh release download "$RELEASE_TAG" --pattern '*.dmg' --pattern '*.dmg.sha512sum' …`) — the verification loop lives in `.github/workflows/smoke-test/smoke-test.sh:50` (`for checksum in *.sha512sum; do … sha512sum --check --quiet --strict "$checksum"; done`), invoked many steps later from a different file.

**Why missed**: Phase 2's inline-download check scans ±30 lines in the same file. Cross-file verification is invisible to it.

**Proposed change**: Extend the inline-downloads check to track `*.sha*sum` artifact patterns alongside the primary asset; when both are downloaded together (same `gh release download` or `curl` invocation), record a "deferred-verifier" hint in `inventory.json` so Phase 2 can confirm the sister script actually consumes the sums. Alternatively, allow `# scan:audited verified-by=<path>:<line>` annotations next to the download, then look up the cited line and enforce it carries the verification primitive. The latter is simpler and works for any cross-file pattern.

### R2. PowerShell `Start-Job -ScriptBlock { … IWR … | iex }` block scope

**Where**: `scripts/windows-setup.ps1:46-50` — the `IWR | iex` lives inside a `Start-Job -ScriptBlock { … }` block, which the script correctly flagged. But the *self-fetch* finding (`raw.githubusercontent.com/.../main/...`) at line 74 also lives in a top-level `if (! (Get-Command wsl …)) { … }` block. The script flags both (good), but the bullet under `language_source_fetches[]` for line 76 is a near-duplicate of the `self_fetch_unpinned[]` entry for line 74 (same URL, two-line offset).

**Why missed**: Two detectors firing on overlapping byte ranges (URL extraction + PowerShell `Invoke-WebRequest` primitive) produce duplicate-looking entries in different sections.

**Proposed change**: When `self_fetch_unpinned[]` already covers a URL, suppress matching `language_source_fetches[]` entries within ±5 lines whose primitive is the language-level wrapper for the same fetch. The two findings are the same threat described twice.

### R3. `gpg --dearmor` of an unpinned upstream key

**Where**: `.github/workflows/smoke-test/install-from-repo.sh:21-23` — `curl -s …Release.key | gpg --dearmor > …keyring.gpg`. The script's APT-key recipe already exists in the wiki (`gpg --show-keys` against a hardcoded fingerprint), but `gpg --dearmor` without a sibling `gpg --show-keys` looks like a verifier from a regex's perspective ("gpg" is the right primitive) when it actually only reformats the key bytes.

**Why missed**: The verifier-detection regex matches any `gpg` invocation in the ±30-line window; `gpg --dearmor` is non-verifying.

**Proposed change**: Tighten the GPG verifier regex to require `gpg --verify`, `gpg --show-keys` paired with a hardcoded `[0-9A-F]{40}` fingerprint constant in the same window, or `rpm --import` paired with a fingerprint check. Plain `gpg --dearmor` should not satisfy the verifier check.

### R4. Custom dependency manifest with documented no-op delegates

**Where**: `pkg/rancher-desktop/assets/dependencies.yaml` — three entries (`golangci-lint`, `check-spelling`, `spinKubePlugin`) carry empty `checksums: {}` blocks because their `download()` methods return `Promise.resolve()` and delegate to a separate script. Phase 2 already calls this out as "track the delegate down," but the mechanical inventory cannot tell whether the delegate verifies its bytes.

**Why missed**: `inventory.py` records `empty_checksum_blocks` but does not link to the delegate's source file. Phase 2 had to read `scripts/dependencies/tools.ts` and grep for `Promise.resolve` to find the delegate path each time.

**Proposed change**: When an entry's `download()` body matches `return Promise.resolve()` or `return Promise.resolve(undefined)`, scan the comment immediately preceding it for a reference to a sibling script (`scripts/spelling.sh`, `resources/setup-spin`, etc.) and surface the path under the manifest entry in `inventory.json`. Phase 2 then knows exactly which file to apply the inline-download rules to. Alternatively, normalize the convention: every empty-`checksums` entry must have a `scan:delegate=<path>` comment in the manifest itself, parsed at scan time.
